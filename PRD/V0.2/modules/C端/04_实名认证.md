# 实名认证

> **来源**: P1-5
> **核心**: 必须客户本人,不可门店代替

> **V0.2 Peer Review 修订(2026-05-27)v1.2**:
> - 补充实名认证未完成情况下的恢复逻辑(关闭小程序 / 杀进程 / 跨端切换 / 实名超过 24 小时未完成)。
> - 明确实名成功后 `return_context` 的有效期与失效后的兜底跳转。

> **V0.2 Peer Review 修订(2026-05-27)v1.1**:
> - 实名认证与下单动作解耦。
> - 客户实名成功后回到进入实名前的商品详情、订单确认页或订单待办页,不自动创建订单、不自动提交订单。
> - 客户需再次点击“立即下单 / 确认订单 / 继续办理”等明确按钮后,系统才进入下一步。

## 1. 业务背景

客户首次办单或资料过期时,需做实名认证:
- 身份证 OCR 识别
- 人脸识别活体检测

## 2. 流程

```
1. 客户上传身份证正反面照片
   - 实拍 + 防翻拍检测
   - OCR 自动识别姓名 / 身份证号 / 有效期 / 地址
   - 校验身份证号格式 + 有效期
2. 人脸识别(活体检测)
   - 调用支付宝/微信/三方刷脸 SDK
   - 客户做指定动作(眨眼、张嘴、摇头等)
   - 服务方比对 OCR 出的身份证照与本人是否一致
3. 通过 → 客户实名状态变 verified
   通过失败 → 重试(最多 3 次)
   3 次失败 → 转人工(客服处理)
```

### 2.1 与下单链路的关系

实名认证是客户身份前置校验,不是“下单提交”动作。

| 进入实名的来源 | 实名成功后返回 | 后续动作 |
|---|---|---|
| 商品详情页点击立即下单时发现未实名 | 原商品详情页 | 客户再次点击立即下单 |
| 订单确认页提交前发现未实名 | 原订单确认页,保留已选套餐和资料草稿 | 客户点击确认订单 |
| 支付/签约前实名过期 | 原订单待办页 | 客户点击继续支付或继续签约 |
| 个人中心主动实名 | 个人中心实名状态页 | 无自动下单动作 |

系统必须携带 `return_context` 记录来源页面、商品、SKU、套餐、渠道来源和办单员工,用于实名后恢复页面状态。`return_context` 只恢复展示和草稿,不得触发订单创建、订单提交、合同发起或支付。

### 2.2 状态刷新

实名成功返回原页面后,前端应自动刷新一次实名状态,并允许用户下拉刷新。

| 状态 | C 端提示 |
|---|---|
| verified | “实名认证已完成,请继续办理” |
| pending | “认证结果处理中,请稍后下拉刷新” |
| failed | “认证未通过,请重新认证或联系客服” |

### 2.3 实名未完成的恢复逻辑 ⭐ v1.2 新增

实名认证过程中,客户可能因为各种原因中断(关闭小程序 / 杀进程 / 跨端切换 / 网络中断 / 接电话等)。系统必须处理这些场景,避免客户回来时找不到入口。

#### 2.3.1 `return_context` 的有效期

| 场景 | `return_context` 状态 | 客户下次进入时 |
|---|---|---|
| ≤ 30 分钟内回来 | 有效 | 直接恢复到原页面继续 |
| > 30 分钟 ≤ 24 小时 | 半有效 | 弹窗提示"您之前在浏览 XXX 商品,是否继续?"[继续][重新选择] |
| > 24 小时 | 失效 | 回到小程序首页,但保留实名进度;实名状态完成的话直接显示已实名 |

#### 2.3.2 实名中断的兜底跳转

| 中断场景 | 客户下次进入 | 系统行为 |
|---|---|---|
| 上传身份证后关闭 | 小程序首页 | 跳转回身份证上传页,已上传图片保留 |
| 等待 OCR 处理时关闭 | 小程序首页 | 跳转回实名进度页;若 OCR 已完成 → 进入下一步 |
| 等待人脸识别时关闭 | 小程序首页 | 跳转回人脸识别入口 |
| 人脸识别失败后关闭 | 小程序首页 | 跳转到失败页,显示剩余重试次数 |
| 实名已成功但未点击"继续办理"就关闭 | 小程序首页 | 弹窗"您已完成实名认证,继续之前的操作吗?"[继续 XXX 下单][稍后再说] |

#### 2.3.3 跨端切换(支付宝 ↔ 微信)的实名复用

| 场景 | 处理 |
|---|---|
| 客户在支付宝完成实名,改用微信进入 | 通过手机号 + 实名信息匹配,自动识别为已实名 |
| 同一手机号在两端有不同实名(身份证不一致) | 异常,转客服处理 |
| 客户更换手机号 | 实名状态保留,但需要新手机号完成短信验证才能下单 |

#### 2.3.4 数据模型扩展

```sql
customer_kyc_session
- session_id            bigint PK
- customer_id           bigint FK
- channel               enum -- alipay / wechat / h5 / app
- return_context        json -- 商品ID / SKU / 套餐 / 渠道码 / 办单员工等
- step                  enum -- id_card_uploaded / ocr_processing / face_pending / face_failed / verified
- attempt_count         int
- started_at            datetime
- last_active_at        datetime
- expires_at            datetime -- 默认 24 小时
- completed_at          datetime
- status                enum -- active / expired / completed / abandoned
```

#### 2.3.5 异常告警

| 异常 | 处理 |
|---|---|
| 同一客户 24 小时内创建超过 3 个 kyc_session | 提示"您今日尝试次数过多,请明天再试或联系客服" |
| 同一手机号关联多个不同身份证 | 风控告警,转客服核实 |
| 实名进度页加载失败 | 客户可点 [联系客服] |

## 3. 必须客户本人

- 身份证拍照可门店帮举手机但客户拿身份证
- 人脸识别**必须客户本人**,不可代替

## 4. 中控台对接

- 身份证 OCR / 人脸识别都通过中控台对接(P0-4)
- 中控台异步计费
- 本地缓存 + 降级模式

## 5. 失败处理

| 失败类型 | 处理 |
|---|---|
| 身份证 OCR 失败 | 可重试 3 次,或手工填(客服核对) |
| 人脸识别不匹配 | 可重试 3 次,3 次后进"待客服介入" |
| 身份证过期 / 无效 | 提示客户更新身份证 |
| 黑名单(已被加黑) | "您当前无法办理此业务,请联系平台客服" |

## 6. 实名状态

- 客户实名一次,后续办单复用(不需每次重做)
- 实名信息有效期:身份证有效期内
- 身份证更换 / 过期:需重新实名

## 7. 隐私

- 实名信息默认脱敏展示
- 完整查看需权限 + 日志

## 8. 数据模型

- `customer.kyc_status` (verified / unverified / expired)
- `customer.kyc_verified_at`
- `customer.id_card_number`(加密存)
- `customer_kyc_log`(实名历史记录)
- `customer_kyc_session`(实名过程会话,详见 §2.3.4) ⭐ v1.2 新增
