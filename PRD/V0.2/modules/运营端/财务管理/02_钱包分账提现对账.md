## 2. 核心口径(v1.3 命名最终统一)

1. **三种合作模式**(`cooperation_mode`)都进入财务总账:
   - **商家订单**(`self_operate`):门店全额出资 + 平台综合服务
   - **联营订单**(`joint_venture`):双方按比例共同出资,按出资比例分润
   - **平台订单**(`receivables_assignment`):平台支付应收账款转让对价,平台拿全部应收

2. **平台综合服务费**统一按客户租金的固定比例收(默认 2%,后台可配),三种模式都适用。**留在平台对公账户**(平台收入,不穿透备付金)。

3. **设备使用费部分(客户租金 - 服务费)**按合作模式分账:
   - 商家订单:100% 给门店
   - 联营订单:按出资比例分给门店和平台台账
   - 平台订单:100% 给平台台账

4. **门店端只有一个"我的钱包"**(门店结算账户),所有归属门店的钱(订单结算款 / 月度分账 / 平台奖励 / 平台返利 / 财务调整)统一进入。

5. **资金穿透原则**:平台收到任何归属门店的钱,实时(< 1 分钟)按 1:1 充值到通联备付金,**平台对公账户不沉淀**。

6. 所有钱包变动都必须有来源单据:订单、账单、支付流水、退款、冲正、提现、奖励规则、返利规则、调账审批。

7. 财务动作不得只改展示字段,必须生成流水和日志。

8. 通联备付金账户技术实现:**单备付金主账户 + 系统内门店子台账 + 平台子台账**(详见 07 文档 §3.1)。

9. **内部资金来源台账**:平台后台维护 `funder` 主表(平台自营 / 蓝海银行 / 其他资方),记录每个资方的授信额度、应收余额、已用额度。内部资金来源台账(funder_receivable_ledger)记录每单订单的资方资金流转(详见 `资方管理/01` + 07 §7)。

10. **撤单退款资金链路**:已分账订单撤单强制走线下退款工单(详见 07 §6 + 10 §3)。

## 4. 账户模型(v1.3 新增内部资金来源台账)

| 账户类型 | 账户主体 | 入账来源 | 出账来源 | 可提现 |
|---|---|---|---|---|
| **门店结算账户(我的钱包)** ⭐ | 商家(商家级,一个商家一个账户) | 订单结算款(联营/平台订单生效)/ 月度分账(三模式)/ 平台奖励 / 返利 / 财务调整 | 提现 / 平台扣款 / 撤单退款扣减 / 财务调整 | 是,自助提现(单笔 500-20000 元;超 20000 或异常提现需人工审核) |
| 渠道账户 | 渠道推广主体 | 联营/平台订单佣金 | 提现 / 冲正 | 是 |
| 平台收入账户 | 平台 | 综合服务费 / 应收账款受让溢价 / 利润分成 / 接口计费 | 退款 / 成本结算 | 否 |
| 平台成本账户 | 平台 | 内部记账 | 通联通道费 / 信用审核成本 / 接口成本 | 否 |
| **内部资金来源台账** ⭐ v1.3 新增 | 资方(平台自营 / 蓝海银行 / 其他) | 应收账款转让对价回收 / 联营出资款回收(从客户租金穿透) | 撤单退款时还本 / 资方实际提取(平台自营无提取,蓝海银行有提取动作) | 后台动作,不直接面向商家 |
| 通联备付金账户(平台名义,持牌通道) | 平台(在通联) | 平台对公账户穿透充值 | 通联代付(到门店银行卡 / 资方银行卡 / 客户) | - (技术性账户) |

### 4.3 内部资金来源台账的余额模型(v1.3 新增)

参考 `资方管理/01` + 07 §7:

```sql
funder
- funder_id
- funder_name              -- "平台自营资金" / "蓝海银行"
- funder_type              -- platform_self / bank / trust / private
- credit_limit_total       -- 总授信额度
- credit_limit_used        -- 已用额度(在外应收余额合计)
- credit_limit_left        -- 剩余可用额度
- status                   -- active / paused / closed

funder_receivable_ledger
- ledger_id
- funder_id
- order_id
- transferred_amount       -- 资方支付的转让对价 / 联营出资款
- collected_amount         -- 已从客户租金中回收
- refunded_amount          -- 撤单退回的金额
- outstanding_balance      -- = transferred - collected - refunded
- status                   -- active / collected / refunded
```

闭环验证:
- 每个资方的 `credit_limit_used = SUM(outstanding_balance)`
- 每个资方的总投入 = 总回收 + 总应收余额

## 6. 分账规则(v1.3 命名同步)

### 6.1 客户每期租金的通用拆分

```
客户支付 ¥500
   ↓
平台对公账户(过路)
   ↓
拆分(合同口径):
   ├─ 平台综合服务费 = ¥500 × 服务费率(默认 2%)= ¥10  → 留对公户(平台收入)
   └─ 设备使用费 = ¥490                                  → 穿透通联备付金
```

平台综合服务费**留在平台对公账户**(合规)。

### 6.2 商家订单分账(`self_operate`)

| 步骤 | 规则 |
|---|---|
| 1 | 客户支付每期租金 ¥500 |
| 2 | 扣平台综合服务费 ¥10(留对公户) |
| 3 | 剩余 ¥490 全额穿透到门店结算账户(`entry_type=monthly_split`) |
| 4 | 写流水 + 通联充值凭证 |

**商家订单不触发 `order_settlement` 入账**,只有 `monthly_split`。

### 6.3 联营订单分账(`joint_venture`,以 50:50 为例)

#### 阶段 1:订单生效(**内部资金来源放款条件全部满足**,详见资方 04)

| 步骤 | 规则 |
|---|---|
| 1 | 系统检查所有内部资金来源放款前置条件 |
| 2 | 全部满足 → 平台支付应收账款转让对价 = ¥5,000 × 50% = ¥2,500 |
| 3 | ¥2,500 穿透到门店结算账户(`entry_type=order_settlement`) |
| 4 | 同步更新内部资金来源台账(平台自营):transferred_amount += ¥2,500 |
| 5 | 同步更新内部资金来源额度:credit_limit_used += ¥2,500 |

#### 阶段 2:客户每期付款

| 步骤 | 规则 |
|---|---|
| 1 | 客户支付每期租金 ¥500 |
| 2 | 扣平台综合服务费 ¥10(留对公户) |
| 3 | 设备使用费 ¥490 穿透到通联备付金 |
| 4 | 按出资比例分配:¥245 → 门店结算账户 + ¥245 → 平台子台账 |
| 5 | 同步更新内部资金来源台账:collected_amount += ¥245(从客户租金回收的资方部分) |
| 6 | 写流水(门店:`monthly_split` ¥245;平台:内部记账) |

### 6.4 平台订单分账(`receivables_assignment`)

#### 阶段 1:订单生效(**内部资金来源放款条件全部满足**)

| 步骤 | 规则 |
|---|---|
| 1 | 内部资金来源放款条件全部满足 |
| 2 | 按 funding_source 分支:平台自营资金 OR 蓝海银行放款 |
| 3 | ¥5,000(100%)穿透到门店结算账户 |
| 4 | 同步更新内部资金来源台账(平台自营 OR 蓝海银行):transferred_amount += ¥5,000 |
| 5 | 同步更新内部资金来源额度:credit_limit_used += ¥5,000 |

#### 阶段 2:客户每期付款

| 步骤 | 规则 |
|---|---|
| 1 | 客户支付每期租金 ¥500 |
| 2 | 扣平台综合服务费 ¥10(留对公户) |
| 3 | 设备使用费 ¥490 穿透到通联备付金 |
| 4 | 全额进平台子台账(应收账款受让回款) |
| 5 | 同步更新内部资金来源台账:collected_amount += ¥490 |
| 6 | 门店不分账(应收账款已转让) |

### 6.5 渠道佣金

仅在联营订单 / 平台订单触发,商家订单不触发渠道佣金。

| 模式 | 渠道佣金来源 |
|---|---|
| 联营订单 | 从平台分润中扣除 |
| 平台订单 | 从受让溢价中扣除 |

## 8. 提现审核(V0.2.1 冻结口径)

### 8.1 提现来源

| 来源 | 提交端 | 审核端 | 审核规则 |
|---|---|---|---|
| **门店结算账户提现** ⭐ | 商家 PC 端 | 运营端财务 | **单笔 500-20000 元;超 20000 或异常提现进入人工审核** |
| 渠道提现 | 渠道 H5 | 运营端财务 | 全部需审核 |
| 财务调账 | 运营端财务 | 运营端财务主管 | 全部需审核 |

员工账号默认不能发起提现,也不能查看钱包。商家老板可单独授权"财务人员"查看我的钱包和发起提现,但财务人员不能修改结算账户。

### 8.2 V0.2.1 提现规则

| 项 | 规则 |
|---|---|
| 单笔最低 | 500 元 |
| 单笔最高 | 20000 元 |
| 每日次数 | 最多 3 次 |
| 超限处理 | 不直接提交通道,进入人工审核 |

| 发起权限 | 商家老板 / 被授权管理员 / 被授权财务人员 |
| 修改结算账户 | 仅商家老板提交,平台审核 |

## 9. 对账中心(v1.3 新增资方对账)

### 9.1 对账范围

| 对账对象 | 对账内容 |
|---|---|
| 支付通道 | 支付成功金额、退款金额、手续费、回调状态 |
| 代扣通道 | 代扣签约、扣款成功、扣款失败、解约 |
| 平台对公账户 ↔ 通联备付金账户(穿透对账) | 客户支付 → 平台对公账户入账;平台扣服务费后,设备使用费部分实时充值通联备付金。两边金额必须严格对账 |
| 通联备付金账户内部子台账(对账) | 通联备付金主账户余额 = 所有子台账之和(门店子台账 + 平台子台账) |
| **内部资金来源台账对账** ⭐ v1.3 | 每个资方:总投入 = 总回收 + 总应收余额;授信使用额度 = 应收余额合计 |
| **资方银行账户对账(蓝海银行)** ⭐ v1.3 | 平台与蓝海银行的资金对账(放款 + 账单支付)|
| 分账账单 | 分账金额、平台综合服务费、入账账户、失败重试 |
| 钱包余额 | 期初、入账、出账、冻结、提现、期末 |
| 渠道佣金 | 佣金规则、订单归属、结算状态 |
| 门店结算账户余额(系统记录) ↔ 通联子台账余额 | 严格一致,差额需立即查 |

### 9.2 五边对账架构(v1.3 升级)

```
五边对账(从 v1.2 四边对账升级):
   1. 平台对公账户(银行)
   2. 通联备付金账户(技术中介)
   3. 门店结算账户(系统记录)
   4. 业务订单系统(应收应付)
   5. 内部资金来源台账 ⭐ v1.3 新增(内部资金来源应收余额)
   
   五边任意两边不平 → 立即告警 + 财务复核
```

### 9.3 异常类型(v1.3 扩充)

| 异常 | 处理 |
|---|---|
| 支付成功但订单未更新 | 重试回调或人工关联 |
| 订单已更新但支付通道无流水 | 标记高风险,禁止分账 |
| 平台对公账户已收钱但通联充值失败 | 系统重试 + 财务异常队列 + 人工补穿 |
| 通联充值成功但门店子台账未更新 | 系统对账兜底 + 财务核实 |
| 门店结算账户余额(系统记录)与通联子台账余额不一致 | 禁止提现 + 财务复核 |
| **内部资金来源台账不平(内部资金来源应收余额 ≠ 授信已用额度)** ⭐ v1.3 | 财务复核所有订单 |
| **资方维度总投入 ≠ 总回收 + 总应收余额** ⭐ v1.3 | 财务紧急复核 |
| **蓝海银行账户对账差异(银行流水 vs 系统记录)** ⭐ v1.3 | 财务联系银行核对 |
| 同一订单收到重复监管锁回调 | 系统幂等校验 |
| **内部资金来源放款条件检查任务漏跑** ⭐ v1.3 | 5 分钟兜底任务自动复查 |
| 提现打款失败 | 退回余额,记录失败原因 |
| 渠道佣金重复 | 冻结重复佣金并等待复核 |
| 客户租金支付金额与应收金额不符 | 进入对账异常 |
| **撤单退款工单超时(门店未转账)** ⭐ v1.3 | 自动催办,7 天升级主管 |
