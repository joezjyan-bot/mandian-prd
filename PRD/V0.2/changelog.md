### V0.2.1 冻结口径

- V1 冻结安心用长租、体验租短租和 IM 客服基础能力。
- 体验租/短租作为独立业务线进入 V1,但不得混入长租办单助手、长租订单状态和长租设备档案。
- IM 客服进入 V1,采用统一客服中心 + 会话上下文 + 工单转化 + 业务跳转模式;IM 内不直接执行高危动作。
- C 端不暴露资方、合作模式、资金来源、风控字眼。
- 默认合同主线为三方租赁合同:甲方商家、乙方客户、丙方平台;资方仅为运营端和财务端内部配置。
- 普通提现单笔 500-20000 元,每日最多 3 次;超 20000 或异常提现进入人工审核。
- 长租只做设备识别码档案绑定;体验租短租使用独立车辆库存状态机。

## 2026-05-25(上午):P0 / P1 / P2 / P3 全部核心决策定稿

**背景**:Hudson 与 Claude 通过长会话讨论,完成 V0.2 体系的全部业务底座 + 工程基线决策。

**核心决策摘要**:

### P0 业务底座(4 项)

| 编号 | 主题 | 核心结论 |
|---|---|---|
| P0-1 | 资金链路口径 | 三类订单统一平台代收 → 钱包结算 → 提现;通联 + 直付通双通道 |
| P0-2 | 商家与门店关系 | 商家 1:N 门店(受控字典);四级角色;订单冻结门店归属快照 |
| P0-3 | 分账细节 | "分账有出退款必入";责任方钱包充值到位才能扣减;平台不兜底客户 |
| P0-4 | 中控台边界 | C 混合模式:直连第三方 + 中控台异步计费 + 本地缓存降级 |

### P1 关键决策(5 项)

| 编号 | 主题 | 核心结论 |
|---|---|---|
| P1-1 | 设备状态机 | 9 状态(新增预约保留 + 争议中);已下架不显示在库存列表 |
| P1-2 | 资方准入与分配 | 运营线下创建 + 简化偏好配置 + 系统推荐人工确认 |
| P1-3 | 价格调整 & 修改还款日 | 不做原订单改价,走取消重下;修改还款日仅运营客服可操作 |
| P1-4 | 合同链路 | 三方合同制 + 资方专属模板分层管理 + 平台必下载副本存档 |
| P1-5 | C 端核心流程 | 支付宝/微信/H5 三端 + 后期 App;断点续传 + 后台发起客户响应 |

### P2 工程基线(4 项)

| 编号 | 主题 | 核心结论 |
|---|---|---|
| P2-1 | 多租户/部署架构 | 独立部署 + 统一代码维护 + 灰度发布 |
