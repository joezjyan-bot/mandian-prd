# V0.2.1 核心数据模型总表

> **V0.2.2 开发冻结同步(2026-05-28)**: 本文档是 V1 开发的数据模型阅读入口。汇总安心用长租、体验租短租和 IM 客服基础能力必需模型，不改数据库字段名、API 路径和英文枚举。若页面级 PRD 与本文冲突，开发先按 `00_V0.2.2_开发冻结版总PRD.md`、`00_核心决策表.md` 和本文执行，并在验收报告中记录差异。

> **V0.2.2 全量整改补丁(2026-05-28)**: V0.2.2 要求四账分离、结算三条件、退款责任留痕、短租 V1、IM 客服、法诉证据包和幂等补偿均进入数据模型。本文只描述数据模型建议,不创建 migration、不修改业务代码。

### 0.5 体验租短租 V1

短租模型在本文 §5 已列出, V0.2.2 补充以下要求:

- `short_rent_orders` 必须保存经营模式、资产归属、履约门店、结算主体、保险/保单快照。
- `short_rent_exception_tickets` 必须覆盖事故、违章、损坏、丢车、超时、控车失败。
- `short_rent_deposit_records` 必须支持冻结、扣款审批、退还、争议、冲正。
- 事故、违章、损坏、丢车、保证金扣款细则已确认按推荐方案执行:平台不默认兜底,优先走保险/协议/证据责任划分;保证金扣款必须有规则、证据、审批、客户通知和申诉记录,字段和证据留存结构必须先建设。

## 1. V1 模型边界

| 范围 | 冻结口径 |
|---|---|
| V1 主链路 | 安心用长租 + 体验租短租 + IM 客服基础能力 |
| 长租订单 | 商家订单、联营订单、平台订单三类长租主营订单 |
| 短租订单 | 体验租独立订单、独立车辆库存、独立取还车核销、独立短租保证金和扣款链路 |
| IM 客服 | 统一会话、订单上下文、客服分配、转工单、会话归档 |
| 长租设备 | 发货/交付节点绑定 `IMEI/SN/VIN` 等设备识别码档案，不占用短租库存 |
| 短租设备 | 进入 V1,但仅服务体验租业务线,不得写入长租订单主状态机 |
| C 端展示 | 不展示资方、资金来源、合作模式、风控字眼；内部字段可保留兼容 |

## 2. 主体与权限模型

| 模型 | 用途 | 关键字段/关系 | V1 要求 |
|---|---|---|---|
| `tenants` | 多租户部署主体 | `id`、`name`、`status`、`config_version` | 每套部署独立数据域 |

## 6. 账单、资金与结算模型

| 模型 | 用途 | 关键字段/关系 | V1 要求 |
|---|---|---|---|
| `order_bills` | 账单计划 | `order_id`、`period_no`、`due_date`、`amount_due`、`status` | C 端展示账单支付/每期应付，不写账单支付 |
| `payment_records` | 支付流水 | `order_id`、`bill_id`、`channel`、`amount`、`status` | 平台代收，支持首期、账单、保证金、留购/提前结清 |
| `guarantee_deposit_records` | 保证金记录 | `order_id`、`amount`、`status`、`deducted_amount` | 全文使用保证金，不用保证金 |
| `settlement_entries` | 订单结算款入账 | `order_id`、`responsible_party`、`amount`、`entry_type` | 结算归属到商家/平台/内部台账 |
| `merchant_wallets` | 我的钱包 | `merchant_id`、`available_balance`、`frozen_balance` | 展示为我的钱包 |
| `wallet_ledgers` | 钱包流水 | `wallet_id`、`amount`、`direction`、`biz_type`、`ref_id` | 所有入账、冻结、冲正、提现可追溯 |
| `withdrawal_requests` | 提现申请 | `wallet_id`、`amount`、`status`、`review_status` | 单笔 500-20000，每日最多 3 次；超 20000 人工审核 |
| `merchant_settlement_accounts` | 商家结算账户 | `merchant_id`、`account_type`、`masked_account_no`、`status` | 修改账户必须高权限 + 二次确认 + 日志 |
| `refund_requests` | 退款申请 | `order_id`、`amount`、`responsibility_type`、`status` | 出退款必须入分账/钱包冲正闭环 |
| `refund_workflow_locks` | 退款流程锁 | `order_id`、`locked_steps`、`unlock_condition` | 退款处理中锁定相关业务动作 |
| `pending_fee_queues` | 待补扣费用队列 | `merchant_id`、`amount`、`reason`、`status` | 钱包余额不足时挂账，不强行负数处理 |
| `funder_assignments` | 内部资金来源分配 | `order_id`、`funder_id`、`funding_source`、`assigned_at` | 内部后台使用，C 端、合同、流水备注不展示 |
| `funder_receivable_ledgers` | 内部资金方/应收台账 | `order_id`、`amount`、`status`、`settled_at` | 只做内部核算和对账，不改变客户侧三方合同链路 |

## 10. 开发验收规则

1. 新增 V1 migration 时，不得把短租库存状态字段强行写入长租订单主链路。
2. C 端接口返回字段不得包含可直接展示的资方、资金来源、合作模式、风控文案。
3. 涉及英文枚举和历史字段兼容时，可以保留底层字段，但前端展示必须走 `modules/全局/10_全局措辞规范.md`。
4. 资金类表必须支持幂等、冲正、对账和操作日志。
5. 合同、签收、发货、账单、分账之间必须通过订单事件和状态流转闭环，不能只靠页面按钮跳转。
6. 体验租短租订单、车辆库存、取还车核销和保证金扣款必须有独立状态机和日志。
7. IM 客服会话必须能关联客户、订单、业务线和工单,高危动作只允许跳转到原业务模块执行。
