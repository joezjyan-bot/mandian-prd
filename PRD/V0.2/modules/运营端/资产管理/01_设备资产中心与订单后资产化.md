# 设备资产中心与订单后资产化

> **所属端**:运营端 > 资产管理(商家端/门店端只读或受限可见自有资产)
> **版本**:v1.0 / 2026-05-29
> **性质**:合规证据链核心模块。用于证明「每个安心用订单对应一台真实设备」,支撑真实设备租赁定性。
> **来源**:合规审查 P0 — 安心用长租不强制提前入库,但订单生成后必须沉淀设备资产记录。

> **关键定位(必读)**:
> 1. **安心用长租不要求提前入库**;改为「订单生成后自动资产化」:订单一旦生成,系统自动为该订单创建/绑定一条设备资产记录,形成设备资产、权属、交付、归还、购买、违约处置的完整记录与状态流转。
> 2. **轻松试短租走提前入库**:短租车辆使用既有「短租车辆库存 9 状态」(状态字典 §6.6),可提前入库、库存可租、归还后再次出租。本文档的资产化机制**仅针对安心用长租**,不改短租库存逻辑。
> 3. 设备资产状态机是**独立于订单主状态机的平行状态机**,不污染订单主状态(参照短租车辆库存与长租设备档案分离的既有原则,状态字典 §6.6)。

---

## 0. 为什么要这个模块

合规上要证明的是「设备真实、租赁真实、可归还、可续用、申请购买是独立自愿」。其中「设备真实」这一环,靠的就是:每个订单背后有一台可追溯的真实设备,有编号、有权属、有交付凭证、有全生命周期记录,而不是一笔纯金额账单。

安心用长租的实际业务不一定先入库(很多是订单成立后门店再配/再交付),所以**不能用「没有提前入库」判定为缺失**。正确做法是:**订单生成 → 自动资产化 → 交付时补设备识别码 → 全程记录**。

## 1. 安心用:订单后资产化机制

### 1.1 触发与时点

| 时点 | 动作 | 设备资产状态 |
|---|---|---|
| 订单生成(草稿/待审核) | 系统自动创建一条 `device_asset` 记录,绑定订单、商品、规格、设备价值;此时**尚无设备识别码** | `PENDING_ASSETIZED`(待实物绑定) |
| 门店交付填写 IMEI/SN/VIN(长租生命周期 08 阶段3) | 回填设备识别码到该资产记录,完成「资产↔实物」绑定 | `DELIVERED`(已交付) |
| 客户签收确认 | 资产进入出租状态 | `IN_LEASE`(出租中) |
| 客户归还 | 资产进入归还/验收 | `RETURNED`(已归还) |
| 客户申请购买完成 | 资产权属转移记录 | `PURCHASED`(已购买) |
| 违约/灭失/转卖等处置 | 处置记录 | `DISPOSED`(已处置) |

> **重要**:`PENDING_ASSETIZED` 阶段允许设备识别码为空——这是设计要求,不是缺陷。价值与商品信息在订单生成时已确定,具体哪一台(识别码)交付时才有。

### 1.2 设备识别码回填与一致性校验(与设备价值确认单联动)

1. 交付填写 IMEI/SN/VIN 时,系统把识别码回填到 `device_asset`,并把该资产与「客户在订单确认页确认的设备价值」(见 02_设备价值确认单)关联。
2. **一致性校验**:交付实物的型号/成色若与客户当初确认的型号/成色不一致(例:客户确认 95 新,实际发 9 成新),系统必须**触发重新确认**(回到设备价值确认流程),不得用旧确认记录直接放行。
3. 校验通过后,资产、设备价值确认、订单、交付证据形成闭环证据链。

### 1.3 资产记录字段

```text
device_asset
- asset_id           bigint PK
- order_id           bigint FK      -- 绑定订单(安心用长租)
- merchant_id / store_id            -- 归属商家/门店
- product_id / sku_id               -- 商品与规格
- brand_model        varchar        -- 品牌型号
- condition_label    varchar        -- 成色(全新/靓机/95新/9成新等)
- device_value_cent  bigint         -- 设备价值(= 该订单设备价格;办单助手填 或 客服后台填)
- source_type        enum           -- 来源:merchant_self/platform_supply/other(可扩展)
- device_identifier  varchar        -- IMEI/SN/VIN;PENDING_ASSETIZED 阶段可空,交付回填
- identifier_type    enum           -- IMEI/SN/VIN/OTHER
- ownership_status   enum           -- 权属:platform/merchant/customer/funder_internal(内部,C端不展示)
- delivery_evidence_ids  json       -- 交付证据附件(照片/凭证)关联
- asset_status       enum           -- 见 §1.1(独立资产状态机)
- created_at / updated_at

device_asset_status_log
- log_id / asset_id
- before_status / after_status
- trigger_source     enum           -- order_event/manual/callback
- operator_id / reason
- created_at
```

### 1.4 与订单状态的关系

- 资产状态机**独立**,由订单事件驱动(订单生成→资产化、交付→已交付、签收→出租中…),但不并入订单主状态。
- 一个安心用订单对应一条资产记录(单设备租赁);未来如需一单多设备,`device_asset` 与订单为多对一,预留。
- C 端不展示资产内部字段(来源类型、权属内部值、内部资金来源相关),客户只在订单里看到自己租的设备型号/规格。

## 2. 轻松试:提前入库与库存周转(沿用既有)

短租车辆库存沿用状态字典 §6.6 的 9 状态(待入库/在库可租/预约保留/已锁定/出租中/归还待验/维修中/争议中/已下架),支持日租/周租/月租、押金、检测、维修、再次上架。本文档不改短租库存逻辑,仅在此声明两套资产体系分离:

| 维度 | 安心用长租 | 轻松试短租 |
|---|---|---|
| 入库时点 | 订单后资产化(不强制提前入库) | 提前入库 |
| 资产表 | `device_asset`(订单绑定) | 短租车辆库存表(可复租) |
| 状态机 | 本文档 §1.1 | 状态字典 §6.6 |
| 复租 | 否(留购或归还后终态) | 是(归还检测后再上架) |

## 3. 权限与日志

| 动作 | 谁可操作 | 日志 |
|---|---|---|
| 订单后自动资产化 | 系统 | 资产创建日志 |
| 回填设备识别码 | 门店/商家/运营(交付环节) | 资产状态日志 + 交付证据 |
| 修改资产成色/价值 | 运营(受限,改值纳入审计) | 改前后值 + 操作人 |
| 资产处置(灭失/转卖/报废) | 运营主管(高危,2FA) | 处置原因 + 凭证 + 审计 |
| 查看自有资产 | 商家/门店(仅本商家) | — |

资产关键操作(识别码绑定、价值修改、处置)进不可删操作日志。

## 4. 接口建议(供研发落地)

| 接口 | 说明 |
|---|---|
| `POST /device-assets`(或订单事件内部触发) | 订单生成后创建资产记录(PENDING_ASSETIZED) |
| `POST /device-assets/{asset_id}/bind-identifier` | 交付时回填 IMEI/SN/VIN + 一致性校验 |
| `POST /device-assets/{asset_id}/status` | 资产状态流转(出租/归还/购买/处置) |
| `GET /device-assets/{asset_id}` | 资产详情(内部;C端不返回内部字段) |

## 5. 验收要点

1. 安心用订单生成后自动产生一条 `device_asset`(此时识别码可空),不依赖提前入库。
2. 交付填 IMEI/SN/VIN 后回填成功,资产与订单/价值确认绑定。
3. 交付实物成色/型号与客户确认不一致时触发重新确认,不放行。
4. 资产全状态(资产化/交付/出租/归还/购买/处置)可追溯,有状态日志。
5. 短租库存与长租资产分离,互不污染。
6. C 端不出现资产来源、权属内部值、内部资金来源等字段。
