## 1. 核心定性与术语对齐

### 1.1 合同口径下的核心术语

| 旧术语(必须清理)| 新术语(对齐合同)| 法律性质 |
|---|---|---|
| 采购款 / 采购账户 | **订单结算款 / 订单结算账户** | 应收账款转让对价 / 联营出资 |
| 采购账户提现 | **订单结算账户提现** | 门店从结算账户提现到银行卡 |
| 资方出资 / 资方代付 | **应收账款受让方付款 / 联营出资** | 应收账款转让对价 / 联营投资款 |
| 客户分期款 | **客户租金**(=设备使用费+平台服务费)| 租金 |
| 平台抽佣(2%)| **平台综合服务费** | 服务费收入(交 6% 增值税) |
| 软钱包 / 我的钱包(后台) | **结算台账 / 门店结算账户** | 通联备付金账户的子台账视图 |

### 1.2 三种合作模式

订单创建时,门店选择本订单的合作模式,系统按模式适用对应的资金 / 收益规则。

| 合作模式 (`cooperation_mode`) | 商家端 UI 用语 | 业务含义 | 法律性质 | 出资责任 |
|---|---|---|---|---|
| `self_operate` | **商家订单** | 门店全额出资,平台提供综合服务 | 门店自主经营 + 平台服务合同 | 门店 100% |
| `joint_venture` | **联营订单** | 门店和平台按比例共同出资 | 联营投资合作 | 门店 X% / 平台 (100-X)% |
| `receivables_assignment` | **平台订单** | 平台代付应收账款转让对价,门店履约 | 应收账款转让 | 平台 100% |

**关键设计**:
- 不管哪种合作模式,**客户合同永远是「门店出租 + 客户租赁 + 平台提供综合服务」三方租赁关系**(平台订单可能用赊销合同,见 1.4)
- 客户**完全感知不到**三种合作模式的内部区别(通过统一中性话术覆盖)
- 合作模式仅影响**门店和平台之间的内部资金流转与收益分配**

### 1.3 客户租金的法律拆分

按《租赁服务协议》定义条款:

> 【租金】乙方因接受甲方租赁服务应向甲方支付的每期应付款项的统称,**包含设备使用费及平台服务费两部分**

```
客户每期付租金 ¥500
   ↓
合同拆分:
   ├─ 设备使用费 ¥490(归门店,法律性质=应收租金)
   └─ 平台服务费 ¥10(归平台,法律性质=平台综合服务费收入)
```

**法律含义**:
- **设备使用费部分**:门店对客户享有的应收账款,需穿透通联备付金(因为这部分钱"还不是平台的")
- **平台服务费部分**:已是平台的服务收入,留在平台对公账户(不需穿透,合规)

### 1.4 资金来源(funding_source,后台用)⭐ v2.2 新增

订单经审批后,**后台根据风控规则分配资金来源**,商家不感知具体资金来源。

| `funding_source` | 说明 | 适用合作模式 |
|---|---|---|
| `platform_self` | 平台自营资金 | 联营订单 / 平台订单 |
| `funder_blue_ocean` | 蓝海银行(赊销融资)| 平台订单(部分,符合银行授信标准) |
| `funder_xxx`(预留)| 未来其他资方 | 视资方约定 |

**资金来源决定**:
- 客户合同模板(平台自营 → 租赁合同;蓝海银行 → 赊销合同)
- C 端关键按钮文案(留购 vs 提前结清,详见话术规范文档)
- 平台对账的资方账户

## 2. 资金路径总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                       │
│  应收账款受让方/联营资金     客户                                       │
│   │ 应收账款转让对价        │ 每月租金                                  │
│   │ 联营出资款              │ (设备使用费+平台服务费)                  │
│   ↓                        ↓                                          │
│  ┌──────────────────────────────────────┐                            │
│  │ 平台实体银行账户(对公户)              │                            │
│  │ - 收入金(订单生效时/客户支付)        │                            │
│  │ - 内部计算扣分(过路 < 1 分钟)         │                            │
│  └──────────────────────────────────────┘                            │
│                    │                                                   │
│         ┌──────────┼──────────┐                                       │
│         ↓ 实时穿透  ↓ 留对公户                                          │
│  ┌──────────────────────┐  ┌──────────────┐                          │
│  │ 通联备付金账户         │  │ 平台综合服务  │                          │
│  │ (平台名义,持牌通道)   │  │ 费收入(合规) │                          │
│  │ - 接收应收账款回款     │  │              │                          │
│  │ - 接收联营出资款       │  │              │                          │
│  │ - 系统子台账分账       │  │              │                          │
│  └──────────────────────┘  └──────────────┘                          │
│           │                                                            │
│           ↓ 通联代付                                                    │
│      门店银行卡 / 联营出资方银行卡                                       │
│                                                                        │
│  门店端 UI:                                                            │
│    "我的钱包"(一个账户,对应通联子台账视图)                            │
│     - 余额:¥X                                                          │
│     - 明细:订单结算款/月度分账/联营分润/平台奖励/返利等                  │
│     - 提现:一键提现 → 通联代付出款                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.1 穿透核心原则

> **平台收到任何归属门店的钱(应收账款转让对价 / 联营出资款 / 客户租金中的设备使用费部分 / 平台奖励 / 返利 等),实时(< 1 分钟)按 1:1 等额充值到通联备付金账户。平台实体账户只过路,不沉淀。**

**例外**:平台综合服务费(已是平台的服务收入)**留在平台对公账户**,这是平台的合法营业收入,不算客户资金沉淀,合规无问题。

### 2.2 备付金账户结构

**单备付金主账户 + 系统内子台账**(单账户模式,Q3 确认):

- 通联备付金账户(平台主体名义,1 个)
- 内部子台账(系统自维护):
  - 每个商家 1 个门店子台账(对应商家"我的钱包")
  - 平台子台账(联营订单 / 平台订单中归属平台的部分)
- 主账户余额 = 所有子台账之和

## 3. 数据模型

### 3.1 `merchant_settlement_account` 门店结算账户主表

```sql
merchant_settlement_account
- account_id              bigint PK
- merchant_id             bigint FK     -- 商家级,一个商家一个账户(对应"我的钱包")
- account_no              varchar(32)   -- 账户编号(系统生成,展示用,如 MSA20260527001)
- balance_total           decimal(18,2) -- 余额合计
- balance_pending         decimal(18,2) -- 在途余额(平台对公账户已收钱,通联充值确认中)
- balance_available       decimal(18,2) -- 可提现余额
- balance_withdrawing     decimal(18,2) -- 提现中余额
- total_received          decimal(18,2) -- 累计入账
- total_withdrawn         decimal(18,2) -- 累计提现
- total_fee               decimal(18,2) -- 累计手续费
- status                  enum          -- normal / frozen / disabled
- bank_account_id         bigint FK     -- 绑定的收款银行账户
- tonglian_subledger_id   varchar(32)   -- 通联备付金账户中的子台账 ID
- created_at / updated_at

-- 索引:(merchant_id) 唯一约束
```

### 3.2 `settlement_entry` 结算账户流水(统一承接所有入账/出账)

```sql
settlement_entry
- entry_id                bigint PK
- account_id              bigint FK
- direction               enum          -- in(入账)/ out(出账)
- entry_type              enum          -- 详见 §3.3
- amount                  decimal(18,2)
- order_id                bigint        -- 关联订单(若适用)
- order_no                varchar(32)
- cooperation_mode        enum          -- 订单合作模式快照(self_operate/joint_venture/receivables_assignment)
- funding_source          enum          -- 资金来源快照(platform_self/funder_blue_ocean/...)
- funder_id               bigint        -- 关联资方(若 funding_source != platform_self)
- source_funder_type      enum          -- 资金来源类型:assignment_payer / customer / joint_venturer / platform / merchant_offline_compensation
- platform_account_ref    varchar(64)   -- 平台对公账户过路凭证(银行流水号)
- tonglian_topup_id       varchar(64)   -- 通联充值流水号(入账时穿透凭证)
- tonglian_payout_id      varchar(64)   -- 通联代付流水号(出账时凭证)
- balance_before          decimal(18,2)
- balance_after           decimal(18,2)
- description             varchar(255)  -- 摘要(门店端 UI 展示)
- status                  enum          -- pending / completed / failed / reversed
- created_at / updated_at

-- 索引:(account_id, created_at) / (order_id) / (entry_type, status)
```

### 3.3 `entry_type` 枚举(9 类,v2.2 扩展)

| 枚举值 | 中文名 | 触发场景 | 资金方向 |
|---|---|---|---|
| `order_settlement` | 订单结算款 | 联营订单 / 平台订单监管锁回调 → 转让对价 / 联营出资款入账 | in |
| `monthly_split` | 月度分账 | 客户每期租金支付后,门店应得部分入账 | in |
| `joint_venture_profit` | 联营分润 | (并入 monthly_split,实际无独立流水)| - |
| `platform_reward` | 平台奖励 | 平台主动给门店的奖励(扶持金 / 业绩奖 等)| in |
| `platform_rebate` | 平台返利 | 渠道返利 / 推广提成 等 | in |
| `platform_deduction` | 平台扣款 | 违约金扣减 / 罚款 / 服务费补扣 等 | out |
| `refund_deduction` ⭐ | 撤单退款扣减 | 撤单退款时,从门店结算账户扣回已分账金额(罕用,仅余额够用时)| out |
| `withdrawal` | 提现 | 门店主动提现 | out |
| `financial_adjustment` | 财务调整 | 财务人工调账(经审批)| in/out |

## 4. 开户流程

### 4.1 开户时机

| 节点 | 动作 |
|---|---|
| 商家入驻审核通过 | 系统不自动开户,等待商家申请 |
| 商家在 PC 端 > 财务钱包 看到 [申请开通我的钱包] | 入驻完成后即可申请 |
| 商家点击申请 | 进入开户审核流程 |
| 财务审核通过 | 创建 `merchant_settlement_account` 记录 + 在通联备付金主账户创建子台账 |

**说明**:不分订单类型,**任何入驻商家都可申请开通**(因为商家订单也可能有平台奖励 / 返利等需要走结算账户)。

### 4.2 开户字段要求

| 字段 | 说明 |
|---|---|
| 商家主体 | 必须与营业执照一致 |
| 收款银行账户 | 户名必须与商家主体一致 |
| 法人身份证 | 必须与营业执照法人一致 |
| 实名验证状态 | 必须已实名 |
| 收款账户授权 | 收款账户非法人账户时需授权文件 |

### 4.3 开户流程

```
商家入驻审核通过
   ↓
商家在 PC 端 > 财务钱包 > 我的钱包 > [申请开通]
   ↓
确认收款银行账户(从已提交资料中选择,或新增)
   ↓
提交申请 → 运营端财务审核
   ↓
运营端财务审核:
  ✓ 收款账户户名与商家主体一致
  ✓ 法人身份证有效
  ✓ 收款账户授权完整
   ↓
  ├─ 通过 → 创建 merchant_settlement_account + 通联子台账
  │         → 商家端通知:"我的钱包已开通"
  └─ 驳回 → 通知商家补资料
```

### 4.4 开户接口

| 接口 | 用途 |
|---|---|
| `POST /api/v1/merchant/settlement-account/apply` | 商家发起开户申请 |
| `GET /api/v1/admin/settlement-account/applications` | 财务查看待审申请 |
| `POST /api/v1/admin/settlement-account/:id/approve` | 财务审核通过(同步创建通联子台账)|
| `POST /api/v1/admin/settlement-account/:id/reject` | 财务驳回 |
| `GET /api/v1/merchant/settlement-account` | 商家查看自己的账户 |

## 5. 入账触发(按合作模式分别处理)

### 5.1 总览:三种合作模式的入账触发

| 合作模式 | 入账类型 | 触发时点 | 资金来源 | 入账金额 |
|---|---|---|---|---|
| **联营订单**(joint_venture)| `order_settlement` 订单结算款 | 监管锁回调成功 | 平台联营出资款(对应平台出资比例)| 设备价 × 平台出资比例 |
| **平台订单**(receivables_assignment)| `order_settlement` 订单结算款 | 监管锁回调成功 | 平台支付的应收账款转让对价(可能来自平台自营或资方)| 设备价 × 100% |
| 三种模式 | `monthly_split` 月度分账 | 客户每期租金支付成功 | 客户租金 - 平台服务费,按合作模式分账 | 按分账规则计算(详见 §6)|

### 5.2 触发链路(以联营 / 平台订单的初始结算款为例)

```
客户 [确认收货] 完成
   ↓ 订单 PENDING_RETURN
商家给设备上锁
   ↓ 监管锁系统 Webhook 回调
平台监管锁服务收到回调
   ↓
1. 更新 lock_status = LOCKED
2. 根据订单 cooperation_mode 判断:
   ├─ self_operate(商家订单)→ 不触发入账
   ├─ joint_venture(联营订单)→ 触发"应收账款转让对价(联营出资比例部分)"穿透
   └─ receivables_assignment(平台订单)→ 触发"应收账款转让对价(全额)"穿透
3. 根据订单 funding_source 决定资金调度:
   ├─ platform_self → 平台对公账户出资
   └─ funder_blue_ocean → 蓝海银行打款到平台对公账户(详见资方管理 01 文档)
   ↓
进入资金穿透流程(详见 §5.3)
```

### 5.3 资金穿透流程(实时,< 1 分钟)

```
步骤 1:平台对公账户出资金到位
   ├─ platform_self → 平台自有资金 +¥X
   └─ funder_blue_ocean → 蓝海银行打款 +¥X(银行接口或线下)
   ↓
步骤 2:平台对公账户实时充值通联备付金
   通联备付金账户 +¥X
   备付金子台账:门店 +¥X
   ↓
步骤 3:门店结算账户余额更新
   merchant_settlement_account.balance_available +¥X
   ↓
步骤 4:写流水
   settlement_entry:entry_type=order_settlement, amount=¥X, funding_source=...
   platform_passthrough_log:direction=out, amount=¥X
   funder_receivable_ledger(若 funding_source != platform_self):transferred_amount=¥X
   ↓
步骤 5:商家端 IM 推送
   "订单结算款 ¥X 已到账"
```

**关键校验**:
- 必须先有"平台对公账户实际收到 X 元"的银行流水
- 才能触发"通联备付金充值 X 元"
- 严格 1:1 等额,**绝不允许平台对公账户长期沉淀**

### 5.4 入账金额计算(按合作模式)

#### 联营订单

```python
amount = device_price × platform_funding_ratio
```

例:设备价 ¥5,000,门店 50% / 平台 50% → 平台支付 ¥2,500 到门店结算账户

#### 平台订单

```python
amount = device_price × 1.0
```

例:设备价 ¥5,000,平台 100% → 平台支付 ¥5,000

#### 商家订单

不触发 `order_settlement` 入账(门店自己出钱采购设备,平台只在客户每月支付时做月度分账)。

### 5.5 不触发的场景

| 场景 | 处理 |
|---|---|
| 商家订单 | 不触发 order_settlement(只触发 monthly_split)|
| 订单已撤单 | 反向冲正(详见 §6 撤单退款资金链路)|
| 门店结算账户未开通 | 进入异常队列,提醒门店先开通 |
| 门店结算账户被冻结 | 暂挂入账,门店解冻后补打 |

### 5.6 异常兜底

| 异常 | 处理 |
|---|---|
| 监管锁系统超时未回调 | 客服催商家上锁(详见 08 §7.5)|
| 上锁回调到达但订单状态不对 | 异常队列,主管核实 |
| 平台对公账户已收钱但通联充值失败 | 重试 + 告警;3 次失败后人工介入 |
| 通联充值成功但子台账更新失败 | 系统自动重试 + 对账兜底 |
| 重复回调 | 系统幂等校验,只处理首次 |

## 6. 撤单退款资金链路 ⭐ v2.2 新增

### 6.1 核心规则

> **只要订单已有任何"对外分账"动作发生(包括 order_settlement 入账或 monthly_split 入账),撤单退款一律走线下退款模式 — 门店通过银行对公转账把对应金额还给平台,平台再处理对客户和对资方的资金还原。**

### 6.2 判断分支

```
撤单触发(客户/客服/主管发起)
   ↓
计算客户应退金额(按合同 §2.9 规则,扣除违约金)
   ↓
判断订单是否已有"对外分账":
   ├─ 没有(收货前撤单,无 order_settlement、无 monthly_split)
   │    ↓
   │    平台对公账户直接退给客户(钱还在过路阶段,未穿透)
   │    冲销 platform_passthrough_log
   │
   └─ 已有分账(收货后撤单,已有 order_settlement 或 monthly_split)
        ↓
        ⭐ 强制走【线下退款流程】(详见 §6.3)
```

### 6.3 线下退款流程(Q1=B:等门店转账到位再退给客户)

```
T0  撤单触发,系统生成"退款资金链路工单"
    计算各方应退/应补金额:
      - 客户应退:¥A
      - 门店应退给平台:¥B(已分账的订单结算款 + 月度分账)
      - 平台应退给资方:¥C(若 funding_source != platform_self)
    工单状态:待门店转账

T1  平台财务通知门店:
    "订单 OD2026001 已撤单,需您线下转账 ¥B 到平台对公账户。
     收到后我们将处理客户退款及资方还原。"

T2  门店执行线下银行对公转账 → 平台对公账户
    平台财务在后台手动确认到账,标记工单"门店转账已到位"

T3  平台同步执行:
    ① 平台对公账户出钱退给客户 ¥A
       客户收到退款,完成
    
    ② 反向冲销门店结算账户:
       写 settlement_entry:entry_type=refund_deduction, direction=out, amount=¥B
       (此为账面调整,实际资金已通过线下转账回到平台对公,通联备付金子台账同步扣减)
    
    ③ 若 funding_source != platform_self(资方资金):
       Q2=A:门店转账到位后立即处理 → 平台对公账户付款给资方 ¥C
       同步更新资方台账:funder_receivable_ledger.refunded_amount += ¥C
       资方授信额度还原:funder.credit_limit_used -= ¥C

T4  整个工单闭环,财务核对账面:
    平台对公账户净流入/流出 = 0(过路完成)
    通联备付金账户余额对齐
    资方应收余额还原
```

### 6.4 退款资金链路工单数据模型

```sql

- order_id                bigint FK
- refund_type             enum          -- before_settlement(收货前)/ after_settlement(已分账,需线下)
```

### 6.5 案例:联营订单 50/50 撤单(已分账 1 期)

```
原始状态:
  设备价 ¥5,000,联营 50/50,客户已付 1 期 ¥568.75
  - 资方台账(平台自营):transferred = ¥2,500, collected = ¥279.38, outstanding = ¥2,220.62
  - 门店结算账户:已入账 ¥2,500(order_settlement) + ¥245(monthly_split) = ¥2,745
  - 平台子台账:¥245(联营出资回收)
  - 平台对公账户:¥10(平台服务费收入)
  - 客户:已付 ¥568.75

撤单(假设客户应退 ¥0,违约金封顶等于已付金额):
  customer_refund_amount = ¥0
  merchant_payback_amount = ¥2,500(订单结算款) + ¥245(月度分账)= ¥2,745
  funder_refund_amount = ¥2,500(还资方本金,不含已回收的 ¥279.38,因为那部分已经回收)
                          但实际平台子台账有 ¥245 也要还给资方
                          ⇒ ¥2,500 - ¥245 = ¥2,255 (从对公账户出)
                          + ¥245(从平台子台账穿透到对公)
                          
  实际操作:
    ① 门店线下转账 ¥2,745 → 平台对公账户
    ② 平台对公账户付 ¥2,500 → 资方银行卡(还本)
       (¥2,255 来自门店转账,¥245 来自平台子台账穿透回对公)
    ③ 平台保留 ¥245 + ¥10 = ¥255 (¥245 是分润收益 + ¥10 是服务费收入)
       注:违约金封顶规则下,这部分作为平台的合法收益保留

撤单后账面:
  资方台账:transferred = ¥2,500, refunded = ¥2,500, outstanding = ¥0 ✓
  门店结算账户:扣减 ¥2,745(refund_deduction)✓
  平台对公账户净留存:¥255(联营分润 + 服务费)✓
  客户:已付 ¥568.75,退 ¥0(违约金封顶)
  
账面闭环 ✓
```

### 6.6 资方额度自动增减

```
订单生效时(监管锁回调,联营/平台订单):
  funder.credit_limit_used += transferred_amount
  funder.credit_limit_left -= transferred_amount

订单完成或撤单退款时:
  funder.credit_limit_used -= refunded_amount
  funder.credit_limit_left += refunded_amount
  funder_receivable_ledger.refunded_amount += refunded_amount
```

## 8. 客户每期租金的分账处理 ⭐ 核心算法 C

### 8.1 分账总流程

```
客户支付 ¥500(月供)
   ↓
[步骤 1] 进平台对公账户(过路)
   ↓
[步骤 2] 按合同口径拆分:
   ├─ 平台综合服务费 ¥10(留对公户,平台收入)
   └─ 设备使用费部分 ¥490(应收账款回款)
       ↓
[步骤 3] 设备使用费部分穿透通联备付金,按合作模式分账:
   ↓
   ├─ 商家订单:¥490 全部进门店结算账户
   ├─ 联营订单(门店 50% / 平台 50%):
   │    ├─ 门店 50% × ¥490 = ¥245 → 门店结算账户
   │    └─ 平台 50% × ¥490 = ¥245 → 通联备付金子台账(平台台账)
   │           + 同步更新资方台账 collected_amount += ¥245
   └─ 平台订单:¥490 全部进通联备付金子台账(平台台账)
        + 同步更新资方台账 collected_amount += ¥490
   ↓
[步骤 4] 写流水 + 推送
```

### 8.2 三种合作模式的分账规则

#### 商家订单(self_operate)

| 客户付 | 平台服务费 | 应收账款回款部分 | 给门店 | 给平台 |
|---|---|---|---|---|
| ¥500 | ¥10(留对公户)| ¥490 | ¥490(穿透到结算账户)| ¥10(服务费收入)|

#### 联营订单(joint_venture,以 50/50 为例)

| 客户付 | 平台服务费 | 应收账款回款部分 | 给门店 | 给平台 |
|---|---|---|---|---|
| ¥500 | ¥10(留对公户)| ¥490 | ¥245 → 结算账户 | ¥10 服务费 + ¥245 平台台账(资方应收回款 + 分润) |

#### 平台订单(receivables_assignment)

| 客户付 | 平台服务费 | 应收账款回款部分 | 给门店 | 给平台 |
|---|---|---|---|---|
| ¥500 | ¥10(留对公户)| ¥490 | ¥0(应收账款已转让) | ¥10 服务费 + ¥490 平台台账(应收账款受让回款)|

### 8.3 关键字段

订单创建时锁定:

```sql
order
- cooperation_mode              -- self_operate / joint_venture / receivables_assignment
- funding_source                -- 资金来源
- funder_id                     -- 关联资方
- merchant_funding_ratio        -- 门店出资比例
- platform_funding_ratio        -- 平台出资比例 = 100% - merchant_funding_ratio
- platform_service_fee_rate     -- 平台综合服务费率(下单时锁定快照)
- device_price                  -- 设备价
- total_rent                    -- 客户租金总额
- contract_template_id          -- 客户合同模板(继承资方)
- terminology_template_id       -- C 端话术模板(继承资方)
```

### 8.4 平台综合服务费率的来源

| 合作模式 | 默认费率 | 后台可配 |
|---|---|---|
| 商家订单 | 客户租金的 2% | 是 |
| 联营订单 | 客户租金的 2% | 是 |
| 平台订单 | 客户租金的 2% | 是 |

- 联营订单 / 平台订单的 `order_settlement` 入账,必须由监管锁回调触发(不允许人工跳过)
- 不冷冻、不自动黑名单冻结、不强制留存

### 11.2 异常情况下的人工冻结

| 场景 | 操作 |
|---|---|
| 商家欠款大额 | 运营主管手动冻结 |
| 商家被立案/起诉 | 运营主管手动冻结 |
| 重大风控告警 | 运营主管手动冻结 |
| 商家主动关店 | 运营主管手动冻结 |

冻结操作必须写日志(`merchant_settlement_account_log` action=freeze)。

## 13. 异常队列

| 异常 | 优先级 | 处理 |
|---|---|---|
| 监管锁回调成功但平台对公账户未到账 | 高 | 财务核对受让方/联营方付款 |
| 平台对公账户已到账但通联穿透失败 | 高 | 系统重试 + 主管手动补穿 |
| 通联充值成功但门店子台账未更新 | 高 | 系统对账兜底 |
| 客户租金支付成功但分账穿透失败 | 高 | 同上 |
| 同一订单收到重复监管锁回调 | 中 | 系统幂等,记录日志 |
| 通联代付出款失败 | 高 | 退回 balance_available + 通知商家 |
| 门店结算账户余额不平 | 高 | 禁止提现,财务复核 |
| 资方台账不平(资方维度) | 高 | 财务复核,核对所有订单 |
| 撤单退款工单超时未到位(门店未转账)| 中 | 财务催办,超 7 天升级主管 |
