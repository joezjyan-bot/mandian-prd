# 无界租登录与订单列表参考

> 来源：2026-05-24 用户提供的无界租后台截图与说明。  
> 用途：作为满点租赁系统重构时的登录页、订单列表页 UI 与信息架构参考。  
> 注意：原截图包含生产订单、姓名、手机号、地址、IP、订单号等敏感信息，本文只记录页面结构和字段类型，不记录真实业务数据。

---

## 1. 登录页参考

### 1.1 用户评价

用户明确表示：这个登录界面比较清爽，比较喜欢。

### 1.2 页面结构

```
┌──────────────────────────────────────────────────────────────┐
│ 浅蓝背景 / 简洁装饰                                           │
│                                                              │
│              ┌──────────────────────────────────┐            │
│              │ 管理后台登录                      │            │
│              │ 支持支付宝/微信扫码登录            │            │
│              │                                  │            │
│              │ [二维码]       验证码登录 | 密码登录 │            │
│              │ 支付宝/微信扫一扫                  │            │
│              │                用户名 / 手机号      │            │
│              │                密码 / 验证码        │            │
│              │                [登录]              │            │
│              └──────────────────────────────────┘            │
│                                                              │
│                 Copyright / 备案信息                         │
└──────────────────────────────────────────────────────────────┘
```

### 1.3 重构要求

| 项 | 要求 |
|---|---|
| 适用端 | 运营端、商家 PC 端、中控台登录页可复用同类结构 |
| 登录方式 | 支持扫码登录、验证码登录、密码登录 |
| 视觉 | 卡片居中、浅蓝背景、蓝色主按钮、表单区域干净 |
| 信息密度 | 登录页保持轻量，不堆说明文案 |
| 安全 | 登录失败、账号禁用、无权限、扫码过期、验证码错误要有明确提示 |

### 1.4 不能照搬的点

- 不复制原站真实二维码。
- 不把备案号、版权主体写死为无界租。
- 重构系统需要根据不同端显示不同系统名，例如“满点租赁运营平台”“商家中心”“中控台”。

## 2. 订单列表参考

### 2.1 用户评价

用户希望重构系统也做成这种类型的订单列表界面：列表内直接展示大量关键业务信息，包括接口回调、订单备注、已交租金、未交租金等，尽量减少点进详情页才能判断订单的情况。

### 2.2 页面结构

```
┌──────────────────────────────────────────────────────────────┐
│ 顶部栏：品牌 / 通知 / 用户                                    │
├────────────┬─────────────────────────────────────────────────┤
│ 左侧导航    │ 订单状态标签：全部 待付款 待审核 待发货 待归还... │
│ 首页        ├─────────────────────────────────────────────────┤
│ 店铺        │ 筛选：订单号 商品名称 用户姓名 手机号 [查询][展开] │
│ 商品        ├─────────────────────────────────────────────────┤
│ 用户        │ 批量操作：[分配人员] [订单分配]                   │
│ 订单        ├─────────────────────────────────────────────────┤
│ 投诉        │ 订单卡片 1：                                      │
│ 租后        │ - 头部：时间 / 订单号 / 商户 / 支付方式 / 合同      │
│ 营销        │ - 商品：图片 / 商品名 / 套餐 / 租金 / 增值服务      │
│ 财务        │ - 用户：认证、手机号、地址、风控、定位              │
│ 权限        │ - 价格：已付/总租金、押金、购买价、租期             │
│ 数据        │ - 状态：待审核、历史订单、转单记录、风控建议         │
│ 应用        │ - 操作：审核、分配、取消、加黑、交易快照、签署       │
│ 推广        │ - 备注/回调：商户备注、用户退款原因、系统回调        │
│ 设置        │                                                   │
└────────────┴─────────────────────────────────────────────────┘
```

### 2.3 列表必须直接展示的信息

| 信息组 | 字段 |
|---|---|
| 订单头部 | 下单时间、订单编号、订单类型、商户/门店、负责人、支付方式、代扣类型、下单渠道、合同服务商、公证状态 |
| 商品信息 | 商品图、商品名称、规格/套餐、数量、每期租金、首期租金、增值服务、运费 |
| 用户信息 | 昵称、实名信息脱敏、手机号脱敏、收货地址脱敏、认证状态、年龄/性别、审核资料、银行卡数、定位信息 |
| 租金与押金 | 已付租金、总租金、已付期数、未付期数、商品押金、补充押金、实际冻结 |
| 买断/留购 | 是否可购买、购买总价、到期购买价、修改购买价入口 |
| 租期 | 起租日期、到期日期、租期天数、总期数、下一期支付时间、修改租期入口 |
| 审核与风控 | 当前状态、历史订单数、转单记录、风控建议、风控信息入口、审核资料入口 |
| 操作按钮 | 订单详情、分配人员、订单分配、取消订单、审核通过、审核不通过、加黑名单、交易快照、发起授权签署、发起报告、发起补充合同 |
| 备注/回调 | 商户备注、平台备注、用户退款原因、支付回调、代扣签约回调、退款失败回调、系统操作记录 |
| 操作日志 | 操作来源、操作人、操作时间、记录条数 |

### 2.4 重构要求

1. 订单列表不是普通稀疏表格，而是“高密度订单卡片列表”。
2. 列表页要支持运营人员完成初筛、初审、分配、跟进，不强迫进入详情页。
3. 详情页只承载更深层信息：合同文件、审核资料原图、完整账单、完整日志、外部接口原始报文等。
4. 敏感字段默认脱敏展示，查看完整信息需要权限和日志。
5. 危险操作必须二次确认并写操作日志，包括审核通过、审核不通过、取消订单、加黑名单、修改价格、修改租期。
6. 接口回调不能隐藏在详情页底部，关键回调要在列表卡片中作为状态/备注直接露出。

### 2.5 与满点三类订单的映射

| 满点订单类型 | 列表展示重点 |
|---|---|
| 门店订单 | 门店自审状态、门店商品配置、门店账单、平台抽佣、门店发货状态 |
| 分红订单 | 门店出资比例、资方出资比例、资方分配状态、平台审核、分账明细 |
| 平台订单 | 平台资方分配、平台审核、平台/资方财务主控、商家发起进度查看 |

## 3. 设计结论

满点重构 UI 的基准不只是一套视觉皮肤：

- `finance-zhongtai` 作为整体企业中台视觉基准。
- 无界租登录页作为清爽登录结构参考。
- 无界租订单列表作为高密度订单运营列表参考。
- 后续订单管理页面必须优先把“列表即工作台”做出来，再补详情页。
