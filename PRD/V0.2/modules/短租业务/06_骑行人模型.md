# 体验租 · 06 骑行人模型

> 解决 P0 问题 Q3:骑行人字段散落在 02 流程文档中,但**没有数据建模**,没有定义"一个客户可以维护几个骑行人""骑行人是否必须实名""骑行人未成年或无证驾驶的责任"等关键规则。
> 本文统一定义骑行人数据模型、与客户/订单的关系、合规告知边界。

---

## 1. 概念定义

| 概念 | 含义 |
|---|---|
| **客户**(customer) | 在 C 端登录的账户主体,完成下单、支付、查询订单的人 |
| **骑行人**(rider) | 实际驾驶/使用租赁车辆的人,可能等于客户本人,也可能是客户为其他人下单 |
| **紧急联系人**(emergency_contact) | 骑行人遇紧急情况时联系的人,不参与下单 |

**核心原则**:
- 客户可以是 18 岁以上的成年人(下单主体)
- 骑行人**必须满足品类对应的法律要求**(电动车骑行人需 ≥ 16 岁,摩托车骑行人需持对应驾照)
- 客户和骑行人可以是同一人,也可以不同人

---

## 2. 客户与骑行人的关系模型

```
customer (1) ─── (N) rider
              │
              └── 一个客户可维护多个骑行人
              └── 同一骑行人可在多个客户下挂(例如父母为子女下单)
```

### 2.1 关系规则

| 规则 | 说明 |
|---|---|
| 一个客户可维护多个骑行人 | V1 上限 5 个;一般用于家庭成员、同事 |
| 骑行人需独立实名认证 | 不复用客户的实名结果(防止冒名) |
| 客户可指定默认骑行人 | 下单时自动带入 |
| 骑行人可被多个客户挂载 | 但同一时刻只能在一个订单的"在租中"状态 |
| 删除骑行人 | 仅可删除无在租订单的骑行人;有历史订单的骑行人记录保留(脱敏存档) |

---

## 3. 数据表

### 3.1 `rider` 骑行人主表

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| rider_id | bigint | 是 | 主键 |
| customer_id | bigint | 是 | 创建该骑行人的客户(关联) |
| rider_name | varchar(50) | 是 | 真实姓名 |
| id_card_type | enum | 是 | id_card(身份证)/ passport / hk_macau_id / other |
| id_card_no | varchar(50) | 是 | 证件号(加密存储) |
| id_card_no_hash | char(64) | 是 | 证件号 SHA256,用于查重和黑名单匹配 |
| phone | varchar(20) | 是 | 手机号 |
| birthday | date | 否 | 出生日期(从身份证号解析,用于年龄校验) |
| gender | enum | 否 | male / female / unknown |
| is_kyc_verified | boolean | 是 | 是否通过实名认证 |
| kyc_verified_at | datetime | 否 | 实名通过时间 |
| is_face_verified | boolean | 是 | 是否通过人脸验证 |
| face_verified_at | datetime | 否 | 人脸通过时间 |
| driver_license_no | varchar(50) | 否 | 驾照号(摩托车租赁场景必填,加密存储) |
| driver_license_type | varchar(20) | 否 | 驾照类型(C1/D/E/F) |
| driver_license_expiry | date | 否 | 驾照到期日 |
| is_default | boolean | 是 | 是否客户的默认骑行人 |
| status | enum | 是 | active / suspended(黑名单)/ deleted |
| created_at / updated_at | datetime | 是 | - |

**唯一约束**:`(customer_id, id_card_no_hash)` — 同一客户下不能有重复证件的骑行人。

### 3.2 `rider_emergency_contact` 紧急联系人

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| contact_id | bigint | 是 | 主键 |
| rider_id | bigint | 是 | 关联骑行人 |
| contact_name | varchar(50) | 是 | 联系人姓名 |
| contact_phone | varchar(20) | 是 | 联系人电话 |
| relationship | enum | 是 | spouse / parent / child / sibling / friend / colleague / other |
| created_at | datetime | 是 | - |

一个骑行人**至多 2 个紧急联系人**(V1)。

### 3.3 `short_rent_order` 关联字段

`short_rent_order.rider_id` 引用 `rider.rider_id`,详见 `05_体验租数据模型与状态机.md` §4.4。

## 4. 实名与人脸认证策略

### 4.1 认证强度

| 场景 | 客户实名 | 骑行人实名 | 骑行人人脸 |
|---|---|---|---|
| 客户=骑行人(默认场景) | ✅ | 复用客户实名 | 下单时一次 |
| 客户≠骑行人 | ✅ | ✅ 必须独立实名 | 取车时由骑行人本人完成 |
| 续租(原骑行人) | - | - | 跳过 |
| 续租(更换骑行人) | - | ✅ 新骑行人实名 | ✅ 新骑行人人脸 |

### 4.2 人脸的触发时机

| 时机 | 说明 |
|---|---|
| 首次下单 | 强制 |
| 距上次人脸 ≥ 30 天 | 强制 |
| 客户/骑行人换设备登录 | 建议(V2) |
| 取车环节(到店核销前) | 必须由骑行人本人完成 |
| 续租 | 复用上次结果(若 ≤ 30 天) |

### 4.3 失败处理

| 失败原因 | 处理 |
|---|---|
| 证件 OCR 识别失败 | 提示重拍 / 手动输入 |
| 实名三要素核验不通过 | 提示信息不一致,不允许重试,需联系客服 |
| 人脸活体检测失败 | 允许重试 3 次,超过 → 锁定 30 分钟 |
| 人脸与身份证比对不通过 | 标记 `kyc_failed_at`,引导联系客服;不可强制通过 |

## 5. 年龄与资质校验

### 5.1 品类对应的骑行人最低年龄

| 品类 | 最低年龄 | 法律依据 |
|---|---|---|
| 电动自行车(非机动车) | 16 岁 | 《道路交通安全法》 |
| 电动摩托车 | 18 岁 + C1 以上驾照 + 摩托车驾照(D/E/F) | 《机动车驾驶证申领和使用规定》 |
| 摩托车 | 18 岁 + 摩托车驾照 | 同上 |
| 手机/相机/数码(短租) | 18 岁 | 平台合规要求 |

**校验时机**:
- 下单时:校验骑行人 `birthday` 推算年龄
- 选择摩托车类品类时:额外校验 `driver_license_type` 是否在允许范围
- 校验失败:阻止下单,提示"骑行人不满足该品类的法律要求"

### 5.2 驾照过期校验

- 驾照过期日 < 还车日 → 阻止下单,提示"骑行人驾照将于 XX 过期,请更新后再下单"

## 6. 黑名单匹配

### 6.1 匹配维度

骑行人创建/更新/下单时,自动匹配以下黑名单:

| 黑名单类型 | 匹配字段 |
|---|---|
| 平台内置黑名单 | id_card_no_hash / phone |
| 商家自定义黑名单 | id_card_no_hash / phone |
| 第三方接入黑名单(V2) | id_card_no_hash |
| 司法失信被执行人(V2) | id_card_no_hash |

### 6.2 命中处理

| 等级 | 处理 |
|---|---|
| 严重 | 阻止下单,提示"该骑行人暂不支持下单,如有疑问联系客服" |
| 警告 | 允许下单,但订单进入运营人工审核 |
| 提醒 | 允许下单,记录风险标记 |

## 8. 合规告知

### 8.1 客户≠骑行人时的告知

下单页强制展示:

> **关于代他人下单**
>
> 您正在为他人(王**)下单租赁。请注意:
> - 您作为下单人需承担订单费用支付责任
> - 实际驾驶人(王**)需对车辆使用、交通法规遵守、安全负责
> - 若发生车辆损坏或交通违章,由实际驾驶人承担相关责任
> - 您与驾驶人之间的权利义务关系由您们自行约定,平台不介入

### 8.2 骑行人未成年(年满 16 岁不满 18 岁)告知

电动自行车租赁场景下,16-18 岁骑行人需额外勾选:

```
☐ 我(下单人)确认本订单驾驶人未满 18 岁,
   已征得其监护人同意,并由我对驾驶人安全负责。
   监护人姓名:____  关系:____  联系电话:____
```

## 9. 权限与隐私

| 数据 | 客户可见 | 商家可见 | 平台可见 |
|---|---|---|---|
| 骑行人姓名 | 自己创建的明文 | 脱敏(闫**) | 明文(权限查看) |
| 证件号 | 自己创建的明文(可隐藏) | 脱敏(37**********37) | 明文(权限查看) |
| 手机号 | 明文 | 脱敏 | 明文 |
| 驾照号 | 自己创建的 | 脱敏 | 明文(权限查看) |
| 人脸照片 | 不可下载 | 不可见 | 仅风控可见 |
| 紧急联系人 | 明文 | 仅订单异常时可见 | 明文(权限查看) |

**敏感字段加密**:`id_card_no` / `driver_license_no` 数据库列加密存储,应用层按权限脱敏。

## 10. API 设计要点

| API | 用途 |
|---|---|
| `POST /api/v1/rider` | 新增骑行人(需实名+人脸) |
| `GET /api/v1/riders` | 当前客户的骑行人列表 |
| `PUT /api/v1/rider/:id` | 修改骑行人(姓名/证件不可改,只能改手机/紧急联系人) |
| `DELETE /api/v1/rider/:id` | 删除骑行人(校验无在租订单) |
| `POST /api/v1/rider/:id/set-default` | 设默认 |
| `POST /api/v1/rider/:id/face-verify` | 人脸验证(取车前) |
| `POST /api/v1/rider/check-eligibility` | 校验骑行人是否满足品类要求 |
