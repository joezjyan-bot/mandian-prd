# 违约金账单与规则配置

> P0 业务文档(2026-05-26)。
> 长租订单逾期违约金的核心机制:**独立账单**(与租金分离)、**规则可配**(分商户自配 / 平台统配)、**手动减免**(客服协商场景)、**留购前必清**(最终结清节点)。
> 本文档定义违约金数据模型、规则配置后台、生成与计算机制、减免流程、客户 C 端展示规则。

---

## 1. 核心机制概览

| 机制 | 说明 |
|---|---|
| **支付独立** | 违约金 = 独立账单,不付违约金也可以支付租金(两条线并行)|
| **规则可配** | 后台配置违约金规则,支持两种算法(固定金额/天 OR 按逾期金额比例/天)|
| **配置粒度** | 门店订单 → 商家后台自配;分红 + 平台订单 → 平台后台统配(Q22)|
| **手动修改** | 运营可在后台手动调整客户的违约金 |
| **手动减免** | 违约金生成后,运营可减免(协商场景,留痕)|
| **客户 C 端展示** | 看到原始金额 + "已减免 ¥XX(运营调整)" + 实付金额(Q24)|
| **支付时机** | 平时累计;**留购或归还前必须清**(Q25)|

## 2. 违约金生命周期

```
租金某期逾期(到期日 + 1 天)
   ↓
系统定时扫描(每日凌晨 1 点)
   ↓
按订单规则计算违约金,生成违约金账单
   ↓
每日累计 +1 天,重算金额
   ↓
客户可选择:
   ├─ 支付违约金账单(完全自愿,平时不强制)
   └─ 暂不支付(继续累计;但留购或归还时必须清)
   ↓
运营可干预:
   ├─ 修改违约金(改原始金额)
   ├─ 手动减免(扣除一部分,实付金额降低)
   └─ 设置规则上限(单日封顶 / 总额封顶)
   ↓
客户支付租金后(消除逾期)
   ↓
违约金停止累计,转为"待支付"状态
   ↓
留购 / 归还触发"必须结清"
   ↓
所有违约金账单状态变为已支付(或主管全额减免清零)
```

## 3. 数据模型

### 3.1 违约金账单表

```sql
order_penalty
- penalty_id          bigint PK
- order_id            bigint FK
- source_period       int     -- 来源期数(第几期逾期)
- period_amount       decimal -- 当期金额(快照)
- start_date          date    -- 逾期起始日(到期日 + 1)
- end_date            date    -- 累计截止日(若已停止累计)
- rule_type           enum    -- fixed(固定金额/天)/ percent(按比例/天)
- rule_value          decimal -- rule_type=fixed:元/天;rule_type=percent:百分比/天
- rule_min_per_day    decimal -- 每日最低收(可选;比例计算时可能太少时兜底)
- rule_max_total      decimal -- 总额上限(可选)
- accumulated_days    int     -- 累计天数
- original_amount     decimal -- 系统算出的原始金额
- reduced_amount      decimal -- 累计减免金额
- final_amount        decimal -- 应付金额 = original - reduced
- paid_amount         decimal -- 已付金额
- status              enum    -- pending / partial_paid / paid / waived / closed
- rule_snapshot       json    -- 规则快照(留痕,后续改规则不影响)
- last_calc_at        datetime -- 上次计算时间
- created_at / updated_at

-- 索引建议:(order_id, status) / (status, last_calc_at)
```

### 3.2 违约金减免日志

```sql
order_penalty_reduction_log
- log_id              bigint PK
- penalty_id          bigint FK
- order_id            bigint FK
- reduction_amount    decimal  -- 本次减免金额
- reduction_reason    text     -- 减免原因(必填)
- reduced_by          bigint   -- 操作客服 ID
- reduced_by_role     varchar  -- 角色名(留痕)
- client_ip           varchar(45) -- 操作 IP
- created_at          datetime

-- 索引建议:(penalty_id, created_at) / (reduced_by, created_at)
```

### 3.3 违约金规则配置

```sql
penalty_rule_config
- rule_id             bigint PK
- owner_type          enum    -- platform(平台统配)/ merchant(商家自配)
- owner_id            bigint  -- platform 时为 NULL,merchant 时为商家 ID
- rule_name           varchar -- 规则名称
- rule_type           enum    -- fixed / percent
- rule_value          decimal -- 固定值 或 百分比
- rule_min_per_day    decimal -- 每日最低
- rule_max_total      decimal -- 总额上限
- applicable_orders   enum    -- store_orders(门店订单)/ dividend_orders(分红订单)/ platform_orders(平台订单)/ all
- status              enum    -- active / inactive
- effective_from      date    -- 生效日期
- created_by / created_at / updated_at

-- 索引建议:(owner_type, owner_id, applicable_orders, status)
```

**重要原则**:订单生成时,违约金规则**快照写入** `order_penalty.rule_snapshot`,**后续平台修改规则不影响已生效订单**。

## 4. 违约金计算

### 4.1 触发时机

- **定时任务**(每日凌晨 1 点)扫描所有"租用中"订单
- 检查每一期是否逾期
- 已逾期但未生成违约金账单 → 新建
- 已生成违约金账单且未结清 → 累计天数 +1,重算金额

### 4.2 计算公式

```
# 按规则类型分支
if rule_type == 'fixed':
    daily_amount = rule_value           # 例:¥10/天
else (rule_type == 'percent'):
    daily_amount = period_amount × rule_value%   # 例:¥568.75 × 0.05% = ¥0.28/天

# 单日封顶(可选)
if rule_min_per_day and daily_amount < rule_min_per_day:
    daily_amount = rule_min_per_day

# 累计计算
original_amount = daily_amount × accumulated_days

# 总额封顶(可选)
if rule_max_total and original_amount > rule_max_total:
    original_amount = rule_max_total

# 实付金额
final_amount = original_amount - reduced_amount
```

### 4.3 例子

**场景**:客户第 3 期 ¥568.75 逾期 5 天,规则为"按比例 0.05%/天"

| 日期 | 累计天数 | 日金额 | 原始金额 | 减免 | 实付 |
|---|---|---|---|---|---|
| Day 1 | 1 | ¥0.28 | ¥0.28 | ¥0 | ¥0.28 |
| Day 2 | 2 | ¥0.28 | ¥0.57 | ¥0 | ¥0.57 |
| Day 3 | 3 | ¥0.28 | ¥0.85 | ¥0 | ¥0.85 |
| Day 4 | 4 | ¥0.28 | ¥1.14 | ¥0 | ¥1.14 |
| Day 5 | 5 | ¥0.28 | ¥1.42 | ¥0 | ¥1.42 |

**场景**:同一情况,但规则为"固定 ¥10/天"

| 日期 | 累计天数 | 日金额 | 原始金额 | 减免 | 实付 |
|---|---|---|---|---|---|
| Day 1 | 1 | ¥10.00 | ¥10.00 | ¥0 | ¥10.00 |
| Day 5 | 5 | ¥10.00 | ¥50.00 | ¥20(运营手动减免) | ¥30.00 |

## 5. 违约金停止累计的触发

| 触发条件 | 处理 |
|---|---|
| 客户支付了对应租金期 | 违约金停止累计,`end_date = today`;状态保持"待支付"或"已支付" |
| 客户支付违约金(完全清)| 状态变为 `paid`,但若租金期仍逾期,继续累计新的违约金 |
| 主管全额减免(waived)| 状态变为 `waived`(等同已结清)|
| 订单撤单 / 关闭 | 状态变为 `closed`,违约金一并核销 |
| 留购前结清 | 客户必须支付所有违约金;状态变为 `paid` |
| 归还前结清 | 同上 |

**关键**:违约金累计与租金支付独立,但**租金已付** = 违约金停止累计的核心条件。

### 6.4 字段说明

| 字段 | 说明 |
|---|---|
| 计算方式 | `fixed`(固定金额/天) / `percent`(按比例/天) |
| 数值 | fixed 时为金额(元);percent 时为百分比 |
| 单日最低 | 可选,当 daily_amount 计算结果太低时兜底 |
| 总额上限 | 可选,累计金额超过此值不再增长(防止极端) |
| 适用范围 | 平台规则可勾选"分红订单/平台订单";商家规则只能"门店订单" |
| 生效日期 | 新规则的生效日期(已生成订单不影响) |

```
┌──────────────────────────────────────┐
│ 修改违约金 - 第 3 期                  │
```

修改后,系统重新计算 final_amount,客户 C 端立即同步。

### 7.2 手动减免

运营在订单详情 > 违约金账单点击 [减免]:

```
┌──────────────────────────────────────┐
│ 减免违约金 - 第 3 期                  │

│ 当前原始金额: ¥50.00                  │
│ 已减免:      ¥20.00                   │
│ 当前待付:    ¥30.00                   │
│                                       │
│ 本次减免金额: [____] 元                │
│ 减免原因:    [____________]           │
│                                       │
│  [取消]      [确认减免]                │
└──────────────────────────────────────┘
```

减免后:
- `reduced_amount += new_reduction`
- `final_amount = original_amount - reduced_amount`
- 客户 C 端立即同步,显示"已减免 ¥XX(运营调整)"

## 9. 留购 / 归还前的强制结清(Q25)

### 9.1 留购触发

客户在 C 端任意一期点 [申请留购] → 系统检查:

```
检查:
  ├─ 当前期是否已付清? → 未付 提示先付
  └─ 是否有未结清违约金?
       ├─ 否 → 弹出留购明细 → 继续
       └─ 是 → 弹出"请先结清违约金"提示
                ├─ 未结清违约金合计: ¥XX
                ├─ [一并结清后留购] → 留购金额 += 违约金合计 + 一笔支付
                └─ [先支付违约金] → 跳转违约金支付页
```

### 9.2 归还触发

客户在 C 端点 [申请归还] → 系统同样检查违约金是否结清。

```
归还检查:
  ├─ 是否有未结清违约金?
  │    ├─ 否 → 正常归还流程
  │    └─ 是 → 必须结清违约金才能完成归还
  │            ├─ [立即支付违约金 ¥XX] → 支付完成 → 归还
  │            └─ 拒绝 → 归还流程终止
  └─ ...
```

### 9.3 例外:运营主管全额减免

如果客户与平台协商,运营主管可全额减免该订单的所有违约金(批量操作)→ 状态变为 `waived` → 留购 / 归还不再检查。

## 10. 权限矩阵(Q23 - 统一一个角色)

按 Q23 决策,**统一一个减免角色**,不分级。

| 动作 | 客服 | 客服主管 | 运营主管 |
|---|---|---|---|
| 查看违约金账单 | ✅ | ✅ | ✅ |
| 修改违约金原始金额 | ✅(留痕)| ✅ | ✅ |
| 单期手动减免 | ✅(留痕)| ✅ | ✅ |
| 全额减免(单期清零) | ✅(留痕)| ✅ | ✅ |
| 批量减免(全订单清零) | ❌ | ❌ | ✅ |
| 修改平台规则 | ❌ | ❌ | ✅ |
| 修改商家规则 | 商家自己 | 商家自己 | ✅ |

**所有操作必须填写原因 + 写日志**(留痕足够,不分级)。

## 11. 异常场景

| 场景 | 处理 |
|---|---|
| 客户支付违约金但租金未付 | 违约金标记 `paid`,但租金期仍逾期 → 第二天继续生成新违约金 |
| 减免金额超过原始金额 | 系统阻止,`reduced_amount` 不可超过 `original_amount` |
| 规则修改与已生成账单冲突 | 已生成订单按 `rule_snapshot` 计算,不影响 |
| 累计天数过长(>180 天) | 进入运营主管预警,可能需要法律介入 |
| 多次减免后又恢复 | 不支持"撤销减免",只能通过 [修改违约金] 调整 |
| 主管批量减免误操作 | 进入异常队列,需财务复核 |
