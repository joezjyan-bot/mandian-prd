# 开发排期与接口优先级

> 所属：开发设计
> 目标：把 V0.2 从“完整需求”压成可启动开发的优先级顺序。优先保证主链路能跑通，再补运营便利性、报表和培训文档。

---

## 1. 优先级定义

| 优先级 | 定义 | 判断标准 |
|---|---|---|
| P0 | 系统不能缺的主链路 | 缺了就无法从商家入驻走到客户下单、审核、支付、发货、分账 |
| P1 | 日常运营高频能力 | 缺了能跑单，但人工成本高、容易出错 |
| P2 | 提效和扩展能力 | 不影响首版跑通，但影响长期规模化 |
| P3 | 后续预留能力 | 先留数据结构和接口边界，后续按业务节奏接入 |

---

## 2. P0 主链路

### 2.1 账号、权限、日志

| 能力 | 接口/任务 | 验收 |
|---|---|---|
| 多端登录 | 登录、刷新会话、退出登录 | 运营端、商家 PC、门店手机端可登录，登录保持规则生效 |
| 商家员工 | 创建员工、禁用员工、权限范围 | 店员只能办单和处理待办，看不到钱包和敏感财务 |
| 操作日志 | 所有写接口统一写日志 | 订单、财务、配置、审核动作都能追溯 |
| 敏感查看日志 | 查看身份证、收款账户、合同等敏感资料 | 有权限、二次验证、查看日志 |

### 2.2 商家入驻

| 能力 | 接口/任务 | 验收 |
|---|---|---|
| 入驻注册 | 创建入驻申请、短信验证码、邀请码归属 | 新商家从门店登录页进入入驻流程 |
| 资质资料 | 营业执照、法人身份证、门头照片、地址定位 | 资料进入附件中心，识别结果可编辑 |
| 收款账户 | 支付宝/银行卡收款账户 | 法人实名优先，不一致时走授权 |
| 签章授权 | e 签宝授权签章 | 运营端能同步看到授权状态 |
| 店铺审核 | 审核通过/拒绝 | 通过后开通商家 PC、门店手机端、钱包和默认配置 |

### 2.3 商品、办单助手、客户下单

| 能力 | 接口/任务 | 验收 |
|---|---|---|
| 商品新增 | 平台/商家商品、规格、增值服务 | 全新/二手是规格，可同时存在 |
| 商品审核 | 商家商品提交审核、运营审核 | 未审核商品不能同步办单助手 |
| 商品复制 | 平台商品复制给商家 | 商家可按自身情况启用或停用 |
| 办单助手三入口 | 门店订单、分红订单、平台订单 | 三入口都能生成报价和二维码 |
| 价格快照 | 押金、账单、服务费、公证费、留购价 | 客户扫码后后台金额和快照一致 |
| 客户扫码下单 | 读取报价、创建订单、提交资料 | 订单进入对应审核列表 |

### 2.4 审核、合同、支付

| 能力 | 接口/任务 | 验收 |
|---|---|---|
| 门店订单自审 | 商家/门店审核 | 不进入运营资方分配主流程 |
| 分红订单审核 | 运营审核、分配资方、比例分账规则 | 资方和门店占比写入订单快照 |
| 平台订单审核 | 运营审核、分配资方或平台主控 | 商家端只能看自己发起订单进度 |
| 补资料改价 | 审核中补资料、改套餐、改价 | 客户待办和补充合同能同步生成 |
| 合同公证 | 发起合同、公证、客户待办、回调 | 客户在订单详情完成，后台状态同步 |
| 支付账单 | 首期支付、账单计划、部分支付收款码 | 支付成功核销账单并触发分账 |

### 2.5 发货、设备、财务

| 能力 | 接口/任务 | 验收 |
|---|---|---|
| 默认门店发货 | 发货任务、线下交付、线上物流预留 | 发货责任按链路配置切换 |
| 设备码绑定 | 唯一设备、短租库存、出入库 | 订单和设备状态同步 |
| 交付照片 | 人机/人车合照、设备照片、配件清单 | 附件可在订单详情查看 |
| 钱包分账 | 平台抽佣、资方、门店、渠道、平台入账 | 金额平衡，不出现孤立流水 |
| 提现 | 商家/渠道提现、运营审核、打款回调 | 钱包冻结、打款、失败回退可追溯 |
| 退款冲正对账 | 退款、冲正、三边对账 | 原流水保留，反向流水可追踪 |

## 3. P1 高频运营能力

| 能力 | 原因 | 验收 |
|---|---|---|
| 订单列表高密度展示 | 审核人员不应频繁点详情 | 列表展示商品、用户、金额、押金、租期、备注、风控、回调、操作 |
| 订单搜索增强 | 运营每天靠搜索处理问题 | 支持订单号、客户、手机号、商家、门店、资方、渠道、日期、状态 |
| 链路配置中心 | 以后换通道不能重新开发 | 支付、合同、公证、风控、物流、分账链路可配置切换 |
| 店铺详情配置 | 店铺审核后需要配置能力 | 订单能力、授权状态、收款账户、渠道来源、接口计费可查看 |
| 财务对账中心 | 金额必须可核对 | 通道、业务、钱包三边对账，异常可处理 |
| 客服 IM 上下文 | 平台订单审核需要沟通 | 商家端联系客服能带订单信息 |
| 配置版本和灰度 | 防止配置误改影响全平台 | 配置发布、回滚、版本日志完整 |

## 4. P2 提效和扩展

| 能力 | 说明 |
|---|---|
| 渠道 H5 | 渠道查看推广入驻、分红订单、平台订单、佣金和提现 |
| 商家报表 | 商家查看经营、订单、财务、设备、员工办单数据 |
| 操作手册生成 | 按无界租式截图标注模板输出内部和商家培训文档 |
| 租后基础 | 内部逾期案件、回款记录、外部催收同步预留 |
| 客诉管理 | 支付宝投诉同步、订单上下文、回复处理 |
| 黑名单 | 脱敏匹配、疑似黑名单弹窗、外部黑名单库预留 |
| 监管锁 | 设备锁机、解锁、状态查询接口预留 |

## 5. P3 后续预留

| 能力 | 首版处理方式 |
|---|---|
| 资方只读端 | V1 不做完整资方端，先由运营端主控 |
| 中控台打通 | 租赁系统只预留接口边界，不把中控台并入系统 |
| 外部催收系统 | 租后模块先保留同步日志和案件字段 |
| 更多风控通道 | 链路配置中心预留 provider 和回调日志 |
| 多租户搭建 | 表字段保留 `tenant_id`，上线版本按单平台先跑通 |

## 6. 推荐开发节奏

| 周期 | 目标 | 交付 |
|---|---|---|
| 第 1 阶段 | 基础底座 | 登录、权限、附件、配置、日志、核心表 |
| 第 2 阶段 | 入驻和商品 | 商家入驻、店铺审核、商品、商品审核、商品复制 |
| 第 3 阶段 | 办单和下单 | 三入口办单、报价、二维码、客户扫码、订单创建 |
| 第 4 阶段 | 审核和合同支付 | 三类订单审核、资方分配、补资料、合同、公证、支付 |
| 第 5 阶段 | 发货和财务 | 发货、设备、签收、账单、分账、钱包、提现、退款、对账 |
| 第 6 阶段 | 运营扩展 | 渠道、租后、客诉、报表、操作手册 |

---

## 7. 开发验收红线

1. 不能只做列表展示，所有状态必须由真实业务事件推进。
2. 不能只在订单表存金额，账单、支付、分账、钱包、提现必须能互相追踪。
3. 不能只在前端判断权限，后端必须校验角色、数据范围和按钮动作。
4. 不能直接信任第三方回调，必须验签、幂等、入库、worker 消费。
5. 不能覆盖历史流水，退款和冲正必须生成反向记录。
6. 不能把门店订单、分红订单、平台订单混成一个审核逻辑。
7. 不能把普通门店端做成资金账户主控端，门店端保留收益、提现、流水和对账。
8. 不能把中控台混入租赁系统，二者只做接口打通。
