# 订单撤单与补充合同

> P0 业务文档(2026-05-27)。
> 定义两个关键的"订单修正"机制:
> 1. **订单撤单**— 客户主动取消订单的各阶段处理 + **撤单退款资金链路**
> 2. **补充合同**— 订单生效后需要变更某些信息时的合同修正机制

> **⚠️ V0.2 重大修订(2026-05-27)v2.0**:
> - **重写 §1 撤单退款资金链路**:分账已发生 → 强制走线下退款流程
> - **新增 §3 退款工单管理**(refund_workflow):退款资金链路工单全流程
> - 三种合作模式展示名统一为 **商家订单 / 联营订单 / 平台订单**
> - 资金链路引用 `财务管理/07 §6` + `财务管理/08`
> - C 端字眼按统一中性话术
> - 订单状态统一使用"履约中";费用文案按场景区分"撤单费用"与"逾期费用"
> - 详细话术见 `话术规范/01_平台统一中性话术规范.md`

> **⚠️ V0.2 修订(2026-05-26)v1.0**:撤单 7 阶段处理 + 补充合同流程(保留主体框架,资金链路 v2.0 重写)

## 1. 撤单机制

### 1.1 撤单可点的状态

| 订单状态 | C 端 [取消订单] 按钮 | 后台撤单权限 |
|---|---|---|
| 审核中(PENDING_REVIEW)| ✅ 客户直接可取消 | ✅ |
| 待付款(未付款)| ✅ 客户直接可取消 | ✅ |
| 待付款(已付首期)| ⚠️ 需要客服同意 | ✅(客服)|
| 待签约(已付首期未签)| ⚠️ 需要客服同意 | ✅(客服)|
| **待签约(已签合同)** | ⚠️ 需要客服同意,扣 1.5% 撤单费用 | ✅(客服)|
| 待发货 | ❌ 不可单方取消(走售后)| ✅(客服 + 主管批准)|
| 已发货 / 待收货 | ❌ 客户不可单方取消(走售后)| ✅(主管批准)|
| **履约中**(原"履约中")| ❌ 客户不可取消(走归还流程)| ❌(只能归还)|

### 1.2 撤单流程

#### 1.2.1 直接撤单(审核中 / 待付款未付款)

```
客户在 C 端点 [取消订单]
  ↓
弹窗确认:"确认取消订单?"
  ↓
客户填写取消原因(可选)
  ↓
[确认取消]
  ↓
订单 → 已完成(已取消)
退款:无需退款(未付款)
```

#### 1.2.2 需客服同意的撤单(已付款 / 已签约)

```
客户在 C 端点 [取消订单]
  ↓
提示:"该订单已付款,需联系客服处理"
  ↓
客户填写取消原因 → 提交申请
  ↓
进入运营端"撤单申请"队列
  ↓
客服审核:
  ├─ 同意 → 系统判断"是否已分账"(详见 §2)
  │         ├─ 未分账 → 平台对公直接退款(原路退回)
  │         └─ 已分账 → ⭐ 生成退款工单(详见 §3)
  └─ 拒绝 → 通知客户(给出理由)→ 订单继续
```

#### 1.2.3 后台主动撤单(已发货 / 异常)

```
运营主管在订单详情 → [关闭订单] 按钮
  ↓
弹窗:输入关闭原因 + 选择退款方案(系统按 §2 自动判断走原路退 vs 退款工单)
  ↓
[确认关闭]
  ↓
订单 → 已完成(已取消)
退款按 §2 规则执行
```

## 2. 撤单退款资金链路 ⭐ v2.0 核心重写

### 2.1 核心规则

> **判断分支**:订单是否已有"对外分账"动作?
> - **未分账** → 平台对公直接退给客户(钱还在过路阶段,原路退回)
> - **已分账** → ⭐ **强制走【线下退款工单】**(门店通过银行对公转账还给平台,平台再处理对客户和对资方的资金还原)

详细资金路径设计见 `财务管理/07_门店结算账户与资金穿透架构.md §6`。本文档承接业务流程层面。

### 2.2 "对外分账"的判断条件

"对外分账"包括:
- ✅ `order_settlement`(订单结算款)已入账门店(联营/平台订单监管锁回调后)
- ✅ `monthly_split`(月度分账)已入账门店(任何一期客户租金分账后)
- ✅ 资方应收账款台账已变动

判断逻辑(代码):

```python
def is_settlement_happened(order):
    return (
        SettlementEntry.exists(order_id=order.id, entry_type='order_settlement', status='completed')
        OR SettlementEntry.exists(order_id=order.id, entry_type='monthly_split', status='completed')
    )
```

### 2.3 未分账撤单(简单流程)

```
适用场景:
  - 商家订单:首期已付,但客户撤单(月度分账尚未触发)
  - 联营/平台订单:首期已付,但监管锁未回调(订单结算款未触发穿透)
  - 已签合同未发货

资金路径:
  ① 客户已付的钱在平台对公账户(过路状态,尚未穿透)
  ② 系统计算退款金额:
     - 客户已付 - 撤单费用(若已签合同扣 1.5%)
  ③ 平台对公账户 → 客户(原路退回支付通道)
  ④ 冲销 platform_passthrough_log

退款时效:支付宝/微信 1-3 工作日;银行卡 3-7 工作日
```

### 2.4 已分账撤单(强制线下流程)⭐

```
适用场景:
  - 监管锁已回调,订单结算款已入账门店
  - 客户已付若干期,月度分账已入账门店
  - 涉及资方应收账款台账已变动

流程概览(详见 §3 退款工单):

T0  客服审核同意撤单
    ↓
T1  系统生成"退款工单"(refund_workflow)
    计算各方应退/应补金额:
      - 客户应退:¥A(扣撤单费用后)
      - 门店应退还平台:¥B(已分账的订单结算款 + 月度分账)
      - 平台应内部清算:¥C(若 funding_source != platform_self)
    工单状态:pending(待门店转账)
    ↓
T2  平台财务通知门店线下转账 ¥B 到平台对公账户
    ↓
T3  门店执行线下银行对公转账
    ↓
T4  财务确认到账,标记工单"门店转账已到位"(merchant_paid)
    ↓
T5  ⭐ Q1=B 决策:等门店转账到位再退给客户
    平台对公账户出钱退给客户 ¥A
    工单状态:refunded_customer
    ↓
T6  ⭐ Q2=A 决策:门店转账到位后立即处理资方还原
    若 funding_source != platform_self:
      平台对公账户付款给资方 ¥C
      同步更新内部资金来源台账:refunded_amount += ¥C
      内部资金来源额度还原:credit_limit_used -= ¥C
    工单状态:refunded_funder
    ↓
T7  财务核对账面,所有台账归零或还原
    工单状态:completed
```

### 2.5 退款规则(资金链路对照表)

| 撤单阶段 | 退款金额计算 | 退款路径 |
|---|---|---|
| 审核中 | 无需退款 | - |
| 待付款(未付款)| 无需退款 | - |
| 待付款(已付首期未签)| **首期实付全额退** | 平台对公直接退回(未分账)|
| 待签约(已签合同)| **首期实付 - 订单总价 × 1.5%(撤单费用,平台收)** | 平台对公直接退回(未分账)|
| 已发货 / 待收货 | **首期实付 - 1.5% - 物流费**(此时监管锁未回调,未分账)| 平台对公直接退回 |
| **履约中(已分账)** ⭐ | **走退款工单(已分账,撤单费用 30% / 封顶规则,详见 §2.6)** | **线下退款工单** |

### 2.6 履约中撤单的撤单费用规则(对齐合同 §2.9)

按《租赁服务协议》:

> (一)乙方在订单生成后、确认收货前申请取消的,须向甲方支付**账单总租金 5%** 的撤单费用
> (二)乙方确认收货后申请退单的,须向甲方支付**账单总租金 30%** 的撤单费用

实操(履约中):

```
退款金额 = MAX(0, 客户已付 - 撤单费用 - 不可退费用)

例 1:客户已付 1 期 ¥568.75,撤单费用 = 账单总租金 × 30% = ¥6,000 × 30% = ¥1,800
     退款金额 = MAX(0, ¥568.75 - ¥1,800) = ¥0(撤单费用封顶 = 已付金额)
     此时客户不退款,但平台资金需还原

例 2:客户已付 6 期 ¥3,412.5,撤单费用 ¥1,800
     退款金额 = ¥3,412.5 - ¥1,800 = ¥1,612.5(部分退款)
     此时需要走退款工单流程
```

### 2.7 1.5% 撤单费用归属

| 项 | 说明 |
|---|---|
| 基数 | 订单总价(设备价 + 留购总价)|
| 比例 | 1.5%(待签约阶段)/ 30%(履约中阶段,按合同) |
| 归属 | **平台收**(非门店收)|
| 法律性质 | 撤单费用,平台综合服务费收入的一部分 |

### 2.8 撤单后的资产处理

| 项 | 处理 |
|---|---|
| 已发货设备 | 物流追回 / 客户原路退回 |
| **履约中设备** ⭐ | 客户必须归还设备给门店(物流或上门取)|
| 客户已下载的合同 | 合同失效,标记"已取消" |
| 客户的电子签约记录 | 永久保留(审计)|
| 客户已绑定的代扣 | 自动解绑 |
| **门店已收到的订单结算款** ⭐ | 通过退款工单线下追回 |
| **资方已支付的应收账款转让对价** ⭐ | 通过退款工单原路退回资方 |
| **内部资金来源额度** ⭐ | 自动还原 |
| 监管锁状态(若已上锁)| 远程解锁,设备状态恢复 DEVICE_AVAILABLE |

### 2.9 撤单 SLA

| 阶段 | SLA |
|---|---|
| 客户提交撤单申请 → 客服处理 | 24 小时 |
| 客服同意 → 退款触发 | 实时(未分账场景)|
| **退款工单生成 → 通知门店** ⭐ | 1 小时 |
| **门店线下转账 SLA** ⭐ | 3 个工作日(超时催办,7 天升主管)|
| **门店转账到位 → 退款客户** ⭐ | 实时 |
| **门店转账到位 → 退款资方** ⭐ | 实时 |
| 退款到账(支付宝/微信原路)| 1-3 工作日 |
| 退款到账(银行卡)| 3-7 工作日 |

### 3.2 工单状态流转


## 4. 补充合同机制 ⭐(保留 v1.0,字眼微调)

### 4.1 业务场景

订单生效后(已签主合同),发现某些信息错误或需要变更,但**不撤单**。通过"补充合同"修正。

### 4.2 适用场景

| 场景 | 是否适用补充合同 |
|---|---|
| 设备编码填错(IMEI/SN 错了)| ✅ 适用(主用例)|
| 客户地址变更(发货前)| ❌ 不需要合同,只改地址 |
| 客户地址变更(发货后,搬家)| ❌ 不需要合同,客服改地址 |
| 付款方式变更(代扣卡换了)| ❌ 不需要合同,客户在 C 端重绑 |
| 客户姓名变更(罕见,改名)| ✅ 适用(必须补充合同确认身份)|
| 商品规格变更(发货后发现型号不对)| ✅ 适用 |
| 月供 / 留购价调整 | ❌ **严禁**(已签合同不可改金额)|
| **资方变更(已签合同后)** | ✅ 适用(合同模板可能也要换)|
| 履约门店变更(主管改派)| ✅ 适用 |
| 商品到货发现质量问题 | ❌ 走售后流程,不走补充合同 |

### 4.3 触发权限

| 场景 | 谁可发起 |
|---|---|
| 设备编码改错 | 客服 / 客服主管 |
| 客户姓名变更 | 运营主管 |
| 商品规格变更 | 运营主管 |
| 资方变更 | 运营主管(注意:可能同时切换合同模板) |
| 履约门店变更 | 客服主管 / 运营主管 |
| 其他 | 运营主管 + 输入理由 |

### 4.4 补充合同状态

| 编码 | 名称 | 说明 |
|---|---|---|
| SUPP_NONE | 无补充合同 | - |
| SUPP_PENDING_REVIEW | 待审核 | 客服/主管发起,等待主管复核 |
| SUPP_PENDING_SIGN | 待签署 | 主管批准,等客户签 |
| SUPP_SIGNING | 签署中 | 客户在签 |
| SUPP_SIGNED | 已签署 | 补充合同生效 |
| SUPP_REJECTED | 已驳回 | 主管驳回或客户拒签 |

### 4.5 流程

```
1. 客服在订单详情 → [发起补充合同] 按钮
   ↓
2. 弹窗:
   - 选择变更类型(枚举:设备编码 / 客户姓名 / 商品规格 / 履约门店 / 资方 / 其他)
   - 旧值(只读,自动带入)
   - 新值(必填)
   - 变更原因(必填)
   - 上传证明材料(可选)
   ↓
3. [提交] → 补充合同进入 SUPP_PENDING_REVIEW
   ↓
4. 客服主管(或运营主管,按权限)审核
   ├─ 通过 → 系统生成补充合同 PDF(按订单当前合同模板)→ SUPP_PENDING_SIGN
   └─ 驳回 → SUPP_REJECTED → 通知发起客服
   ↓
5. C 端推送通知给客户:"您的订单需要签署补充合同"
   ↓
6. 客户在 C 端点击 → 调起 e 签宝 → 客户签
   ↓
7. e 签宝回调:
   ├─ 签署成功 → SUPP_SIGNED → 订单字段更新(如设备编码改为新值)
   └─ 客户拒签 → SUPP_REJECTED → 通知客服
```

### 4.6 客户感知

| 场景 | C 端展示 |
|---|---|
| 补充合同发起,等客户签 | 订单详情顶部出现紫色横幅:"请签署补充合同" + [立即签署] 按钮 |
| 客户签完 | 横幅消失;订单详情新增"补充合同 - 已签署"区块 |
| 客户拒签 | 系统通知客服 |

客户能看到补充合同**变更前后的字段对比**,以便明确知道改了什么。

```
补充合同确认                      
  变更项:  设备编码                 
  原值:    IMEI 1234567890123     
  新值:    IMEI 9876543210987     
                                    
  说明:发货时填写错误,现修正        
                                    
  [取消]  [立即签署]                  
```

### 4.7 补充合同的法律效力

- 补充合同与主合同**同等法律效力**
- 主合同条款若与补充合同冲突,**以补充合同为准**
- 一份订单可有**多份补充合同**(每次变更独立一份)
- 所有补充合同**永久保留**

### 4.9 限制

|---|---|

## 5. 撤单 vs 补充合同的选择

| 场景 | 选撤单 还是 补充合同 |
|---|---|
| 客户改主意了不想要 | 撤单 |
| 客户对商品不满意 | 撤单 / 售后(看阶段)|
| 客户对账单价格不满意 | 撤单(已签合同后不能改金额)|
| 设备编码填错 | 补充合同 |
| 客户姓名变更 | 补充合同(必要时撤单重做)|
| 客户地址变更 | 不需要合同,后台改 |
| 履约门店改派 | 补充合同 |
| 资方变更(已签后)| 补充合同(若涉及合同模板切换需谨慎)|

## 7. 异常场景

| 异常 | 处理 |
|---|---|
| 客户撤单申请超时未处理 | 进入主管复核队列 |
| 平台对公直接退款失败(未分账场景) | 财务异常队列,人工跟进 |
| **门店超 SLA 未转账(已分账场景)** ⭐ | 自动催办,7 天升主管 |
| **门店拒绝线下转账** ⭐ | 主管协调,可能进入法律程序 |
| **门店转账金额错误** ⭐ | 财务核对,要求门店补差或退多 |
| **资方退款失败** ⭐ | 财务异常队列,人工补打 |
| 补充合同客户拒签 | 通知客服,客服联系客户协商 / 撤单重做 |
| 补充合同生效后又要再改 | 允许,发起新一份补充合同(`supp_seq_no` 递增)|
| 设备编码改错后,补充合同已签,但 IMEI 仍对不上 | 启动技术排查;主管可发起紧急撤单 |
| **撤单时设备已留购完成** | 不允许撤单(已留购订单不可撤)|
