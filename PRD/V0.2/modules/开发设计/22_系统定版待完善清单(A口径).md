## 3. 二期挂起项(本期不开发,标记在案)

> 这些项**不阻塞本期开发与上线**。统一在二期"合规优化"时一并处理,届时多为"改配置/改公式结构/加展示",主框架不变。

### ⏸️ 3.1 购买价改折旧余值(残值保底)口径

- **现状(A 口径)**:申请购买价 = 剩余应付租金 + 保证金。
- **二期目标**:改为 `设备折旧余值 = max(设备价值×残值率, 设备价值 − (设备价值÷折旧月数)×已使用月数)`,使购买价锚定设备折旧而非剩余租金。
- **依赖**:① 残值率、折旧月数取值定稿(产品+法务);② 该口径能否被当地法院认可(律师);③ 是否被支付宝渠道接受(渠道对接人)。
- **改动面(届时)**:办单助手02 §1.2/§1.3/§1.4、C端12 §4.2、订单05 §8.2、全局13 新增 `depreciation_months`/`residual_rate` 两参数 + 折旧余值公式结构写死。
- **切换成本**:中。因购买价在系统中是"取快照某列值"的一层,改这一层的计算来源即可,不动下单/签约/履约/归还主框架。

### ⏸️ 3.2 设备无法返还赔偿计算

- **现状**:订单08 §0.5/§8 有逾期、坏账、回收完成、保证金抵扣;凭证结构已就位(订单05 §8.1.3 `return_fee_item`)。
- **二期目标**:把"设备无法返还赔偿额"做成自动计算:`赔偿 = 设备折旧余值 + 已到期未付应付款 + 逾期违约金 + (取回/检测/维修等 is_actual_incurred 费用累加) − 保证金抵扣`,保留计算快照。
- **依赖**:赔偿中的"设备折旧余值"部分依赖 §3.1 的折旧口径,故**随 §3.1 一起做**。实际费用部分(凭证累加)本期结构已可支撑。

### ⏸️ 3.3 平台服务费客户侧展示口径

- **现状**:办单助手账单按费率公式算综合月付;C端02 §2.4 价格方案区列了"服务费:¥XXX"。
- **二期/待确认**:按"平台服务费向门店收"的方向,客户侧是否移除/改名"服务费"展示,客户侧月付如何向支付宝与法院解释为"设备使用费"而非"分期"。
- **依赖**:律师 + 支付宝渠道对接人。
- **本期底线**:客户侧不新增"平台服务费"字样的强调;现有展示维持,待确认后再定。

## 5. 必须明确的边界

1. **A 口径"购买价 = 剩余应付租金 + 保证金"是已知的合规敏感口径**(在融资租赁认定上偏紧)。本期为开发进度先采用并上线,**已在 C端12 §4.2、订单05 §8.2 明确标注为"待二期合规复核、可切换折旧余值口径"**。这是工程上的有意决策,不是遗漏。
2. 折旧余值口径、残值率/折旧月数取值、平台服务费收门店的合规解释、账单列名是否被支付宝接受、最低服务期结清范围是否过重——**属合同/法务/渠道层,非系统层,待律师 + 支付宝对接人确认**。
3. 本清单为系统侧对齐与定版说明,**非法律意见**。合同本身仍为草案,系统按已拍板的 A 口径实现,二期按确认结果切换。
