# 平台订单 · 门店分配机制

> P0 业务诉求(2026-05-27):平台订单(扫平台通用商品码下单)创建时**没有门店信息**,需要平台客服在审核通过 + 内部资金来源分配后、合同发起前**手工分配门店**。
> 核心风险:门店数量多 + 门店名相似 → 打款打错门店,所以分配机制必须有**多维度精确匹配 + 二次确认 + 操作日志**。
> 本文档定义运营端门店分配的页面、字段、流程、状态机扩展、操作日志。

> **⚠️ V0.2 修订(2026-05-27)v1.2**:
> - 三种合作模式命名统一为 **商家订单 / 联营订单 / 平台订单**(对齐 04 / 08 文档)
> - **§2 状态机更新**:平台订单的标准流程是"风控审批 → 内部资金来源分配 → 门店分配 → 确认订单进入支付"
> - **§7.3 门店端"上锁"动作改为"内部资金来源放款条件之一"**(不再直接触发结算款穿透,详见资方 04 文档)
> - 新增"资金来源" / "合同模板"字段展示(后台用)
> - C 端"提货门店"展示规则保留(同 v1.1)

> **⚠️ V0.2 修订(2026-05-26)v1.1**:取消"申请改派"功能 / "门店等效出资" → "门店出资"(保留)

---

## 1. 业务场景

| 三类长租订单 | 扫码来源 | 谁操作 | 是否需要"分配门店" |
|---|---|---|---|
| **商家订单**(self_operate)| 扫**门店专属码** | 门店自接 + 自审 | ❌(扫码即门店)|
| **联营订单**(joint_venture)| 扫**门店专属码** | 门店发起 + 运营审 + 内部资金来源分配 | ❌(扫码即门店)|
| **平台订单**(receivables_assignment)| 扫**平台通用商品码** | **平台客服全程代办** | ✅ **本文档解决** |

平台订单的特点:
- 客户扫的不是某门店的码,而是平台二维码,订单创建时只有商品和客户
- 客户全程不需要门店配合,资料/实名/风控/合同/支付都跟客服互动
- 但订单**必须有一家门店**承担:发货、售后、收款(分账目标)
- 由于门店多 + 同主体多店、不同主体相似店名,**打款打错风险极高**

## 2. 订单状态机扩展(v1.2 更新)

在长租订单主状态机中新增一个状态:

| 状态编码 | 状态名称 | 说明 |
|---|---|---|
| **`PENDING_STORE_ASSIGN`** | **待分配门店** | 资料审核通过、风控通过、**资方已分配**,等待平台客服分配履约门店;**仅平台订单使用** |

### 2.1 状态流转(平台订单,v1.2 严格顺序)

```
PENDING_REVIEW(待审核)
   ↓ 客服资料审核通过
PENDING_FUNDER_ASSIGN(待分配内部资金来源)⭐ v1.2 明确独立阶段
   ↓ 客服按系统推荐确认资方(详见 资方管理/03)
   ↓ 锁定 funder_id / contract_template_id / terminology_template_id
PENDING_STORE_ASSIGN(待分配门店)⭐ v1.2 必须先有资方
   ↓ 客服分配门店成功 + 工号二次确认
   ↓ "确认订单进入支付"
PENDING_PAYMENT(待付款)
   ↓ 系统自动串行(支付 → 合同(按内部业务模板) → 代扣)
... 后续标准流程
```

### 2.2 v1.2 关键调整 ⭐

**先分配内部资金来源,再分配门店,顺序不可调换**:
- 内部资金来源分配决定了 `funding_source` / `funder_id` / `contract_template_id` / `terminology_template_id`
- 合同需要带商家主体 + 按资方对应模板渲染
- 只有资方确认后,才能合理估算分账,门店收益清晰

**例外**:商家订单 / 联营订单扫的是门店码,门店已确定,**只需分配内部资金来源**(联营订单 MVP 默认平台自营,不需要决策)。

### 2.3 反向流转

| 触发 | 流向 |
|---|---|
| 客服在分配门店时发现资料异常 | 回退到 `PENDING_SUPPLEMENT`(待补资料) |
| 客户主动取消 | `CLOSED` |
| 分配门店超时(默认 24 小时未操作) | 进入运营预警,**不自动关闭** |
| **资方审批不通过 / 授信不足** ⭐ v1.2 | 回退到 `PENDING_FUNDER_ASSIGN` 重新选资方 |

### 2.4 关键约束(v1.2 强化)

- 商家订单 / 联营订单**不进入** `PENDING_STORE_ASSIGN`
- 必须通过该状态才能进入 `PENDING_PAYMENT`(合同发起的前置硬约束)
- 分配后才可进入"确认订单进入支付",确保合同模板能正确带入经营主体名称
- **必须先有内部资金来源分配,才能分配门店**(v1.2 新增)

## 5. 改派规则(同 v1.1,保留)

(完整规则同 v1.1,无修改)

### 5.1 v1.1 核心变更

**取消门店申请改派功能**:
- 平台订单一旦分配,门店**必须履约**
- 门店端**没有"申请改派"按钮**
- 收到平台订单 = 履约义务(发货 + 售后)

**保留改派能力(仅供运营主管在极端异常下使用)**

### 5.2 改派的可用阶段(主管权限)

| 订单阶段 | 改派权限 |
|---|---|
| 待分配门店(`PENDING_STORE_ASSIGN`)| 直接重选,客服自助可改 |
| 已分配但未"确认订单进入支付"| ✅ 客服主管可改派 |
| 已确认进入支付但合同未发起 | ✅ 客服主管可改派 |
| 合同已发起 / 签署中 | ❌ 必须先撤销合同 → 才能由运营主管改派 |
| 合同已签署 | ❌ **不可改派**,只能撤单重下 |
| 已支付 / 已发货 / 在租中 | ❌ **绝对不可改派** |

## 6. 客户侧(C 端)展示(v1.2 字眼对齐)

### 6.1 客户在不同阶段看到什么

| 订单阶段 | C 端展示 | 字段口径 |
|---|---|---|
| 待审核(PENDING_REVIEW)| "审核中" | 不展示门店;不显示预估时间 |
| **待分配内部资金来源** ⭐ v1.2 | **"审核中"** | 客户感知与待审核一致,不暴露内部资金来源分配过程 |
| **待分配门店(PENDING_STORE_ASSIGN)** | **"审核中"** | 客户感知与待审核一致 |
| 待付款 | "请支付首期" | 合同还未发起,门店暂不展示 |
| 待签约 | "请阅读并签署合同" | 合同上会带商家主体名(法律必需,按资方对应合同模板渲染) |
| 已签约后 | **"提货门店:鼎租新街口店"** | **只展示门店名一个字段** |
| **履约中** ⭐ v1.2 | "提货门店:鼎租新街口店" | 同上 |

### 6.2 客户能看到的"提货门店"展示规则

**只在 C 端订单详情显示一行**:

```
提货门店:鼎租新街口店
```

**完全不展示**:
- ❌ 商家主体名称
- ❌ 收款账户信息
- ❌ 统一社会信用代码
- ❌ 合作模式(商家/联营/平台)
- ❌ **内部资金来源** ⭐ v1.2
- ❌ **合同模板 / 话术模板** ⭐ v1.2
- ❌ 分配客服的姓名 / 工号
- ❌ 改派记录(若发生)
- ❌ "门店出资" / "资方" / "分账" 等内部资金概念

**例外**:合同 PDF 内部展示商家主体全称 + 内部资金来源信息(法律必需,按资方对应合同模板),但客户看的是已签好的合同,不感知 C 端 UI 层面的"分配过程"。

## 10. 异常场景(v1.2 新增资方场景)

| 场景 | 处理 |
|---|---|
| 同主体多门店,客服误选另一家 | 客服主管在合同前可改派;合同后需走撤单重下 |
| 选了未开通安心用业务的门店 | 系统阻止 |
| 选了已关店 / 暂停的门店 | 系统阻止 |
| 选了跨城市门店 | 弹窗警告但不阻止,客服需输入跨城理由 |
| 门店收款账户最近被冻结 | 系统阻止 |
| 同一商家主体下所有门店均不开通 | 系统不展示该主体 |
| 门店突发停业 | 运营主管改派 |
| 门店超过 SLA 未履约 | 预警 + 客服催促 + 主管评估改派 |
| **内部资金来源额度突然不足(分配门店时)** ⭐ v1.2 | 系统提示,需先回到"待分配内部资金来源"重新选资方 |
| **资方在分配过程中被停用** ⭐ v1.2 | 同上 |
| **内部资金来源放款失败后触发降级**(蓝海→平台自营)⭐ v1.2 | 合同模板可能变化,需主管确认是否发起补充合同(详见资方 04 §6.4) |
