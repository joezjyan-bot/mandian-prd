# 平台订单 · 门店分配机制

> P0 业务诉求(2026-05-26):平台订单(扫平台通用商品码下单)创建时**没有门店信息**,需要平台客服在审核通过后、合同发起前**手工分配门店**。
> 核心风险:门店数量多 + 门店名相似 → 打款打错门店,所以分配机制必须有**多维度精确匹配 + 二次确认 + 操作日志**。
> 本文档定义运营端门店分配的页面、字段、流程、状态机扩展、改派规则、操作日志。

---

## 1. 业务场景

| 三类长租订单 | 扫码来源 | 谁操作 | 是否需要"分配门店" |
|---|---|---|---|
| 门店订单 | 扫**门店专属码** | 门店自接 + 自审 | ❌(扫码即门店)|
| 分红订单 | 扫**门店专属码** | 门店发起 + 运营审 + 分资方 | ❌(扫码即门店)|
| **平台订单** | 扫**平台通用商品码** | **平台客服全程代办** | ✅ **本文档解决** |

平台订单的特点:
- 客户扫的不是某门店的码,而是平台二维码,订单创建时只有商品和客户
- 客户全程不需要门店配合,资料/实名/风控/合同/支付都跟客服互动
- 但订单**必须有一家门店**承担:发货、售后、收款(分账目标)
- 由于门店多 + 同主体多店、不同主体相似店名,**打款打错风险极高**

## 2. 订单状态机扩展

在长租订单主状态机(全局 02 §3.1)中**新增**一个状态:

| 状态编码 | 状态名称 | 说明 |
|---|---|---|
| **`PENDING_STORE_ASSIGN`** | **待分配门店** | 资料审核通过、风控通过,等待平台客服分配履约门店;**仅平台订单使用** |

### 2.1 状态流转(仅平台订单)

```
PENDING_REVIEW(待审核)
   ↓ 客服审核通过
PENDING_STORE_ASSIGN(待分配门店) ⭐ 新增
   ↓ 客服分配门店成功 + 工号二次确认
PENDING_SIGN(待签约)
   ↓ 合同模板带入门店主体生成合同
... 后续标准流程
```

### 2.2 反向流转

| 触发 | 流向 |
|---|---|
| 客服在分配门店时发现资料异常 | 回退到 `PENDING_SUPPLEMENT`(待补资料) |
| 客户主动取消 | `CLOSED` |
| 分配门店超时(默认 24 小时未操作) | 进入运营预警,**不自动关闭**(防止客户体验受损)|

### 2.3 关键约束

- 门店订单 / 分红订单**不进入** `PENDING_STORE_ASSIGN`(它们扫码时已有门店)
- 必须通过该状态才能进入 `PENDING_SIGN`(合同发起的前置硬约束)
- 分配后才可进入 `PENDING_SIGN`,确保合同模板能正确带入经营主体名称

## 3. 数据模型

### 3.1 `orders` 表新增字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| **assigned_store_id** | bigint | 平台订单是 | 分配的履约门店 ID |
| **assigned_merchant_id** | bigint | 平台订单是 | 履约门店的商家主体 ID(冗余,便于查询)|
| **assigned_at** | datetime | 平台订单是 | 分配时间 |
| **assigned_by** | bigint | 平台订单是 | 执行分配的客服 ID |
| **assigned_by_employee_no** | varchar(16) | 平台订单是 | 客服工号(二次确认时输入,留痕)|
| **assigned_reassign_count** | int | 是 | 改派次数(初始 0)|

### 3.2 `order_store_assign_log` 分配日志表(新增)

每次分配 / 改派都写一条:

| 字段 | 类型 | 说明 |
|---|---|---|
| log_id | bigint | 主键 |
| order_id | bigint | 关联订单 |
| action | enum | `assign`(首次分配)/ `reassign`(改派)/ `revoke`(撤销分配)|
| previous_store_id | bigint | 上一个门店(改派/撤销时)|
| new_store_id | bigint | 新门店 |
| new_merchant_id | bigint | 新商家主体 |
| assigned_by | bigint | 客服 ID |
| assigned_by_employee_no | varchar(16) | 客服工号 |
| client_ip | varchar(45) | 客服 IP |
| reason | text | 操作原因(改派/撤销必填)|
| reassign_request_from | enum | 改派发起源:`platform_csr`(客服主动)/ `store_request`(门店申请)|
| created_at | datetime | 操作时间 |

**保留期**:永久保留(审计要求)。

### 4.3 搜索字段(模糊匹配)

| 搜索字段 | 匹配方式 | 说明 |
|---|---|---|
| 门店名称 | 模糊 | 部分匹配,如"鼎租"出全部含"鼎租"的门店 |
| 门店编码 | 精确 / 前缀 | 如 `DZ-BJ-001` 直接定位 |
| 商家主体名称 | 模糊 | 如"鼎租商贸"出该主体下全部门店 |
| 统一社会信用代码 | 精确 | 18 位精确匹配 |
| 收款账户末 4 位 | 精确 | 输 4 位定位账户 |
| 收款户名 | 模糊 | 与商家主体对照防错 |

### 4.4 搜索结果排序

默认按以下优先级:

1. 同省同市(以客户收货地址为准)
2. 近 30 天平台订单完成率高
3. 当前在租订单数少(避免单店过载)

但**不自动选定**,只是排序展示。

### 4.5 搜索结果展示字段

每家门店卡片**必须**展示:

| 字段 | 防错作用 |
|---|---|
| 门店名 + 门店编码 | 区分同名门店 |
| 商家主体全称 | 核对法律主体 |
| 统一社会信用代码(可选展示) | 终极唯一性 |
| **收款账户(脱敏)+ 持有人** | **核对打款目标** |
| 门店地址 | 核对地理位置 |
| 已开通业务线 | 核对是否能接长租 |
| 近 30 天完成率 + 订单数 | 辅助参考 |

## 5. 改派规则

### 5.1 可改派的阶段

| 订单阶段 | 改派 |
|---|---|
| 待分配门店(`PENDING_STORE_ASSIGN`)| 直接重选,不算改派 |
| 待签约前(`PENDING_SIGN` 但合同未发起)| ✅ 可改派 |
| 合同已发起 / 签署中(`CONTRACT_SIGNING`)| ❌ 不可改派 — 需先撤合同 → 再改派 |
| 合同已签署 | ❌ **不可改派**,只能撤单重下 |
| 已支付 / 已发货 / 在租中 | ❌ **绝对不可改派** |

### 5.2 改派触发场景

| 发起方 | 场景 |
|---|---|
| 客服主动 | 发现分配错误 / 客户投诉门店服务 |
| 门店申请 | 收到任务但无法履约(库存/人力/合规问题),需提交改派申请 |

### 5.3 门店申请改派的流程

```
门店 PC 端 / 手机端 "平台订单 - 待履约" → 点单 → 申请改派
    ↓
填写改派理由 + 上传证明(可选)
    ↓
进入运营端工单
    ↓
客服 24 小时内处理
    ├─ 同意改派 → 重新搜索分配 → 二次确认 → 写日志
    └─ 拒绝改派 → 通知门店必须履约 + 留痕
```

**门店不允许直接拒单**(没有拒单按钮),只能申请改派,以防恶意拒接高价订单。

### 5.4 改派的二次确认

改派 = 撤销旧分配 + 新分配,**两次操作都写日志**:

```
log 1: action=revoke, previous_store_id=旧门店, new_store_id=NULL, reason="..."
log 2: action=reassign, previous_store_id=旧门店, new_store_id=新门店, reason="..."
```

## 6. 客户侧(C 端)展示

### 6.1 客户在不同阶段看到什么

| 订单阶段 | C 端展示 | 字段口径 |
|---|---|---|
| 待审核(PENDING_REVIEW)| "平台审核中" | 不展示门店 |
| **待分配门店(PENDING_STORE_ASSIGN)** | **"平台审核中"** | **不展示分配状态,客户感知与待审核一致** |
| 待签约(PENDING_SIGN)| "请阅读并签署合同" | 合同上会带商家主体名(法律必需)|
| 已签约后 | "提货门店:鼎租新街口店" | **只展示门店名一个字段** |
| 在租中 | "提货门店:鼎租新街口店" | 同上 |

### 6.2 客户能看到的"提货门店"展示规则(P0 决策)

**只在 C 端订单详情显示一行**:

```
提货门店:鼎租新街口店
```

**不展示**:
- ❌ 商家主体名称
- ❌ 收款账户信息
- ❌ 统一社会信用代码
- ❌ "分红订单" / "平台订单" / "代客操作" 等内部分类
- ❌ 分配客服的姓名 / 工号
- ❌ 改派记录

**例外**:合同 PDF 内部必然展示商家主体全称(法律必需),但客户看的是已签好的合同,不会感知 C 端 UI 层面的"分配过程"。

## 8. SLA 与超时处置

### 8.1 客服分配 SLA

| 阶段 | 默认时长 | 超时处理 |
|---|---|---|
| 审核通过 → 待分配门店 | 立即(自动转入)| - |
| 待分配门店 → 客服首次操作 | 4 小时 | 进入预警 |
| 待分配门店 → 完成分配 | 24 小时 | 进入预警 |

预警显示在运营端工作台,不自动关闭订单。

### 8.2 门店接单 SLA(收到平台订单后)

| 阶段 | 默认时长 | 超时处理 |
|---|---|---|
| 分配成功 → 门店查看 | 6 小时 | 短信再次提醒 |
| 分配成功 → 门店处理(发货 / 申请改派)| 24 小时 | 进入运营预警 + 自动催促 |
| 分配成功 → 门店履约 | 48 小时 | 强制客服介入催促 |

## 10. 异常场景

| 场景 | 处理 |
|---|---|
| 同主体多门店,客服误选另一家 | 合同前可改派;合同后需走撤单重下 |
| 选了未开通安心用业务的门店 | 系统阻止,提示"该门店未开通安心用业务,无法承接长租订单" |
| 选了已关店 / 暂停的门店 | 系统阻止 |
| 选了跨城市门店(与客户地址不同省)| 弹窗警告但**不阻止**,客服需输入跨城理由 |
| 改派次数过多(> 3 次) | 进入主管复核 |
| 门店收款账户最近被冻结 | 系统阻止,提示更换收款账户后再尝试 |
| 同一商家主体下所有门店均不开通 | 系统不展示该主体 |

## 12. 数据接口

| 接口 | 用途 |
|---|---|
| `GET /api/v1/admin/order/:id/assignable-stores` | 列出可分配门店候选(带搜索、筛选、排序)|
| `POST /api/v1/admin/order/:id/assign-store` | 提交分配(带工号、目标门店、IP)|
