# 体验租业务总览

> **Stage 6 术语同步(2026-05-27)**: 本文档已按 Stage 6 统一为商家、联营、平台订单、订单结算款、我的钱包、履约中、逾期费用、留购、保证金等展示术语；数据库字段、API 路径、英文枚举保持不变。

> 体验租是短期租赁业务线，和安心用长租共用平台基础主体，但不复用长租订单状态、分期账单和办单助手字段。

> **V0.2.2 全量整改补丁(2026-05-28)**: 体验租短租 V1 与安心用长租、IM 客服基础能力一起进入一期。体验租必须独立入口、独立订单、独立车辆库存、独立保证金和取还车链路;短租事故、违章、损坏、丢车、保证金扣款细则已确认按推荐方案执行:平台不默认兜底,优先走保险/协议/证据责任划分,状态、证据、审批、客服介入和申诉结构必须先建。

## 0. V0.2.2 短租 V1 执行口径

### 0.1 C 端主路径

| 环节 | 页面/动作 | 关键内容 | 异常/提示 |
|---|---|---|---|
| 进入体验租 | C 端底部导航或首页分区 | 与安心用并列;展示城市、品类、附近门店、租期类型 | 不混入长租办单助手 |
| 选择门店 | 门店列表 / 地图 | 距离、营业时间、联系方式、导航、可租车型 | 无可租车辆时提示换门店/换周期 |
| 选择车型/车辆 | 车型列表 | 标准商品名、展示标题、图片、价格方案、可租数量 | 标准商品名启用后不可随意改 |
| 确认方案 | 订单确认页 | 小时租、日租、周租、月租;取还时间;总额;协议勾选 | 未勾选协议禁止提交 |
| 骑行人信息 | 骑行人页 | 证件、姓名、手机号、人脸、紧急联系人 | 二要素/实名异常进入人工处理 |
| 支付/保证金 | 支付页 | 租金支付、保证金实付/预授权/免押 | 失败保留支付单和重试入口 |
| 商家接单 | 订单详情 | 接单倒计时、联系商家 | 商家拒单/超时按推荐方案执行:支付成功未接单优先自动退款,退款异常或争议转客服 |
| 取车核销 | 到店核销 | 核销码、扫码绑车、唯一车辆码 | 车辆不一致禁止取车 |
| 用车中 | 订单详情 | 剩余时间、到期时间、续租、联系商家、联系客服、控车状态 | 控车异常进入 IM/工单 |
| 还车验收 | 门店端/订单详情 | 到店还车、验车、证据上传、费用确认 | 损坏/违章/丢车/争议进入异常 |

### 0.2 商家端和门店手机端路径

1. 商家端新增“体验租”模块:体验租订单、车辆管理、价格方案、保证金扣款/退还、收益流水、员工权限。
2. 门店手机端新增“体验租待办”:待接单、待取车、已取车、待还车、还车验收、保证金处理、异常上报。
3. 商家订单、联营订单、平台订单属于长租链路;体验租短租不使用三类长租办单助手。
4. 商家可接单/拒单、核销、绑车、验收、发起扣款申请;涉及扣款、退款、锁车、强制完成必须按权限和日志执行。

### 0.3 收益和分账【资金/法务高风险】

| 经营模式 | 收益归属 | 分账规则 |
|---|---|---|
| 商家自营短租 | 对应商家/门店 | 客户缴费后扣除平台服务费 2%(可配置),剩余进入商家钱包 |
| 平台自营短租 | 平台 | 不进入商家钱包;履约门店仅记录服务/交付动作 |
| 平台货源门店履约 | 平台 + 履约门店 | 履约门店服务费/履约费按商家协议和后台配置执行,不在 C 端展示 |

短租退款、保证金扣款、超时费、损坏费必须同时写订单业务账、支付流水、钱包流水和财务总账/分录账。

### 0.4 风险责任结构【已确认】【资金/法务高风险】

以下场景 V1 必须有状态、证据、审批和客服介入入口。责任承担按推荐方案执行:平台不默认兜底,先走保险/协议约定,再按证据划分客户、商家、平台或第三方责任;保证金扣款必须有规则、证据、审批、客户通知和申诉窗口。

| 场景 | 必备证据 | 系统动作 |
|---|---|---|
| 事故 | 现场照片、车辆照片、报警/保险材料、商家记录、IM 记录、保单/理赔材料 | 订单进入事故处理中;保证金可冻结;客服/运营介入;按保险和证据责任划分 |
| 违章 | 违章通知、车辆/骑行人绑定记录、用车时间证明 | 订单进入违章处理中;按证据向责任方追缴或扣款审批 |
| 损坏 | 取车前照片、还车照片、维修报价、门店验收记录 | 发起保证金扣款审批或争议单;客户可申诉 |
| 丢车 | 定位/控车记录、报警材料、客户确认、门店记录 | 订单进入丢车处理中;锁车/定位/法务材料归档;按协议和证据划责 |
| 超时 | 租期结束时间、还车时间、续租记录 | 按价格方案计算超时费;自动扣款需满足授权、通知和扣款规则 |

### 0.5 保险和保单

体验租 C 端应展示已购租车保障/保险信息和保单入口。具体保障字段、保单号、理赔入口和展示文案以实际对接保险公司要求为准;保单资料必须归入订单证据链。

### 0.6 IM 客服介入

体验租订单详情、用车中、支付失败、控车异常、保证金争议、事故/违章/损坏/丢车处理页均应提供客服入口。会话必须携带业务线、订单号、门店、车辆码、当前状态、异常原因和证据快照。

## 4. 经营模式

| 模式 | 使用场景 | 需要配置 |
|---|---|---|
| 门店自营 | 门店自己有车，自己接单和交付 | 门店短租开通、车辆库存、价格方案、保证金规则 |
| 平台自营 | 平台自己有车或集中采购，指定门店/仓交付 | 平台库存、履约门店、结算主体 |
| 平台货源门店履约 | 平台出车辆和方案，门店执行取还 | 资产归属、履约门店、门店服务费 |

后台不要把这三种模式做成三套系统。统一使用体验租订单，订单上记录经营模式、资产归属、履约门店和结算主体。

## 5. 商品与库存原则

1. 商品基础资料可以共用：品类、品牌、型号、图片、介绍、规格。
2. 长租配置和短租配置分开：
   - 安心用：设备价、首付、保证金、期数、每期租金、留购价、合同模板。
   - 体验租：时租/日租/周租/月租价格、保证金、预授权金额、免押规则、库存车辆。
3. 长租无需库存，发货或交付时录入设备识别码即可。
4. 短租必须有库存，车辆/设备以 VIN、SN、IMEI 或设备号唯一识别。
5. 同一设备唯一编码不能同时进入长租交付和短租可租库存。

### 5.1 体验租商品标准名锁定

体验租依赖“车型/商品 -> 门店价格方案 -> 车辆库存 -> 核销绑车”的链路。为了避免后期价格方案、库存、订单和控车记录无法对齐，适配体验租的商品必须有统一的标准商品名。

规则：

1. 商品一旦开启“支持体验租”，并且已绑定价格方案、车辆库存或产生体验租订单，`商品标准名` 不允许直接修改。
2. 商品标准名用于后台配置、库存绑定、价格方案、订单快照和数据统计。
3. C 端可以单独维护 `展示标题` 或 `营销标题`，用于活动页、商品卡片和卖点展示，但不得影响后台标准名。
4. 如果确实需要更换标准名，必须走“新建商品版本/新车型档案 + 迁移配置”的流程，不允许在原商品上直接改名。
5. 品牌、型号、核心类目、车型编号也按标准字段管理，已启用体验租后原则上不可随意修改。
6. 商家端新增体验租商品时，应优先从平台标准商品库选择；没有标准商品时提交新增申请，由运营审核后生成标准商品。

推荐字段：

| 字段 | 用途 | 是否允许修改 |
|---|---|---|
| 商品标准名 | 后台配置、库存、订单、统计 | 启用体验租后锁定 |
| 展示标题 | C 端商品卡片和详情页展示 | 可按权限修改 |
| 车型编号 | 价格方案和库存绑定 | 启用体验租后锁定 |
| 品牌/型号/类目 | 标准化检索和统计 | 产生配置或订单后锁定 |

## 6. 数据互通原则

体验租不能只是菜单，需要和以下模块互通：

| 数据 | 来源 | 消费方 |
|---|---|---|
| 门店经纬度 | 店铺管理 | C 端附近门店、运营端门店分配 |
| 车型/商品 | 商品管理 | 体验租选车、价格方案、车辆库存 |
| 车辆唯一编码 | 体验租车辆库存 | 取车核销、中控控车、还车验收 |
| 订单 | C 端/门店端/运营端 | 保证金、退款、控车、对账、操作日志 |
| 保证金/免押 | 支付和风控 | 扣款、退款、争议处理 |
| 控车记录 | 中控台 | 订单详情、异常处理、租期到期控制 |

## 7. 第一版建议范围

必做：

- 独立体验租入口
- 选城市/品类/门店/车型/方案
- 骑行人资料和人脸认证
- 用户协议勾选
- 租金支付和保证金/免押入口
- 门店确认/拒绝接单
- 取车核销并绑定车辆
- 租期内控车入口
- 到店还车
- 保证金扣款和退款
- 续租
- 操作日志和控车日志

暂缓：

- 异地还车
- 调度费
- 自动复杂逾期扣费
- 保险深度对接
- 多取还点自由切换
- 平台大规模车辆调度
