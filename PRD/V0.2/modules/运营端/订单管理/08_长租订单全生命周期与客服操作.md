## 0. V0.2.2 长租全生命周期执行口径

### 0.1 主流程闭环

| 阶段 | 触发端 | 必备动作 | 状态结果 | 异常处理 |
|---|---|---|---|---|
| 客户下单 | C 端/办单助手 | 客户确认商品、套餐、保证金、首期应付、订单周期 | 待客户提交资料 | 客户中途退出保留断点,超时按配置取消 |
| 资料提交 | C 端 | 实名、人脸、联系人、信用/信息授权、必要附件 | 待风控审核 | 缺资料进入待补充资料 |
| 风控/业务审核 | 商家端/运营端 | 商家订单可商家自主审核;联营/平台由运营主控 | 待签约 / 审核驳回 / 待补充资料 | 驳回和补资料必须有客户可理解原因 |
| 合同/公证 | C 端/运营端 | 选择默认或业务模板,发起签署/公证 | 待支付首付 / 签约失败 | 签约失败允许重发或人工处理 |
| 首期支付 | C 端/支付回调 | 首期、保证金、增值服务等按配置支付 | 待门店交付 | 支付失败保留支付单和补偿任务 |
| 门店交付 | 商家端/门店端/运营端 | 选择发货/自提方式,填写 IMEI/SN/VIN 等设备识别码,上传交付证据 | 待客户签收确认 | 设备型号不一致、交付失败进入异常 |
| 客户签收 | C 端 | 电子确认单或后台配置的 AI 问答确认 | 待监管锁校验 | 拒签、AI 异常、签收超时进入人工复核 |
| 监管锁校验 | 系统/运营 | 校验监管锁已上锁、激活锁已激活 | 待平台结算 | 未上锁或激活失败进入监管锁失败 |
| 平台结算 | 财务/系统 | 三条件满足后生成结算/钱包/总账记录 | 履约中 / 打款失败 | 打款失败冻结流程并补偿重试 |
| 履约中 | 系统 | 账单计划生效,客户按期账单支付 | 履约中 | 支付失败、逾期、部分支付进入对应子状态 |
| 提前留购/提前结清 | C 端/财务 | 按业务模板计算剩余金额并支付 | 正常结清 / 已留购 | 金额计算必须保留快照 |
| 退款 | 客服/财务/运营 | 记录责任方、金额、协商单、审批、凭证、冲正对象 | 退款中 / 已退款 | 通道失败进入补偿,不得只改订单状态 |
| 逾期/法诉/坏账 | 系统/外部 API/法务 | 逾期进入外部催收 API 或法诉证据包 | 催收中 / 法诉中 / 坏账核销 | 追回款项可回原订单做回收完成 |

### 0.2 结算三条件【资金/法务高风险】

平台结算、商家钱包入账、门店打款、联营/平台订单内部结算均必须校验:

1. `receipt_confirmation.status = passed`:客户签收确认已通过。
2. `lock_status = locked`:监管锁显示已上锁。
3. `activation_lock_status = active`:激活锁状态已激活。

校验失败时:

- 财务打款按钮禁用,并展示未满足原因。
- 可创建人工复核单,但人工放行必须具备权限、审批人、原因、证明附件和审计日志。
- 无监管锁品类的替代条件已确认按推荐方案执行:客户签收确认 + 设备识别码归档 + 交付照片 / 交付证据完整;高价值订单再叠加 AI 视频确认或人工复核。未满足替代验收条件不得直接套用标准结算。

### 0.3 客服改价与账单锁定

1. 客服可在账单生成前按权限修改订单价格、首期应付、保证金、设备价、增值服务等影响账单计划的字段。
2. 改价必须记录改前/改后快照、操作人、原因、客户确认状态。
3. 超过指导价无法下单,需提示联系客服核实;核实通过后仍需按权限记录。
4. 账单一旦生成,任何人不得直接修改账单金额或状态。
5. 账单生成后的调整只允许走退款、冲正、补收、减免、坏账核销、法诉回款或回收完成。

### 0.4 C 端敏感隔离

C 端和客户通知不得展示:资方、资金来源、合作模式、风控结论、内部审核备注、服务费拆分明细、商家结算价、平台利润、返点规则、内部标签、黑名单/灰名单结论。

联营订单和平台订单在 C 端只体现为正常订单流程;客户只看到商品、合同、保证金、每期应付、签收、账单支付、履约中、留购/提前结清、客服入口。

### 0.5 外部催收和坏账回收

催收能力 V1 不自建完整催收系统。逾期订单进入外部催收 API 对接队列,本系统保留案件同步状态、催收记录回写、承诺支付、部分支付和法诉转化入口。

法诉或坏账后追回款项时,允许在原订单上执行“回收完成”:

- 选择完成方式:客户补缴、法诉回款、线下回款、保证金抵扣、其他。
- 填写完成金额、凭证、操作原因。
- 系统新增支付/钱包/总账/回收流水,保留历史逾期、法诉、坏账核销记录。
- 完成后订单进入 `BAD_DEBT_RECOVERED` 或业务定义的已回收完成状态。

#### 3.1.1 客户重新确认时效(v1.7 新增)

商家修改订单后,客户必须重新确认当前有效快照。系统以 `order_change_log.expires_at` 记录确认截止时间,默认 72 小时。

| 时间区间 | 订单处理 | 客户动作 | 商家动作 |
|---|---|---|---|
| 0-24 小时 | `waiting_customer_reconfirm` 正常待确认 | 客户可直接点 [确认继续] | 可查看等待时长、重新发送提醒、撤回修改 |
| 24-72 小时 | 变更确认过期但可恢复 | 客户重新点确认后,需商家再次确认订单快照才可继续 | 商家需再次确认订单,确认后客户方可继续支付/签约 |
| 超过 72 小时 | 订单自动关闭为 CLOSED | 客户需重新下单 | 商家不可恢复,仅可查看关闭原因和日志 |

时效规则:
1. 客户在 24 小时内确认后,订单恢复原主流程,进入待付款或待签约。
2. 客户在 24-72 小时内返回,页面提示"订单变更确认已过期,需商家重新确认后继续办理"。
3. 商家再次确认后,系统重新生成当前快照并重置一次客户确认时效。
4. 超过 72 小时自动关闭时,如客户已支付款项,进入关闭退款流程;如未支付,直接 CLOSED。
5. 所有确认、过期、恢复、关闭动作都写入订单操作日志和 `order_change_log`。

#### 3.1.3 客户不在线时的触达

| 场景 | 处理 |
|---|---|
| 客户已退出小程序 | 通过订阅消息(微信)/ 小程序通知(支付宝)/ 短信(若授权)推送 |
| 客户没有授权订阅消息 | 短信通知,作为必选兜底 |
| 客户已删除小程序 | 短信通知 |
| 客户未读消息超过 48 小时 | 生成客服外呼待办(可配置开启) |

触达内容只描述"订单内容已更新,请回到订单详情确认",不得在短信或通知标题中展示资方、资金来源、风控、内部收益、分账等后台字段。

#### 3.1.4 商家撤回修改

客户长时间未确认时,商家可主动撤回未确认的修改:

```text
商家订单详情:
客户未确认变更,已等待 XX 小时
[重新发送客户确认] [撤回修改]
```

撤回规则:
1. 仅 `customer_confirm_status = pending / expired` 且客户未确认时可撤回。
2. 点击 [撤回修改] 后,订单恢复到修改前的有效快照。
3. 商家可选填撤回原因,系统写入 `withdrew_at / withdrew_reason`。
4. 撤回后客户侧"订单已变更"横幅消失,订单恢复到修改前的待付款或待签约状态。
5. 如撤回前已重新生成合同或支付单,必须作废对应未使用任务,避免客户误签或误付旧链路。

剩余期数重算规则:
1. 修改订单总额后,剩余未付期数按等额分摊生成默认方案。
2. 商家如有逐期自定义权限,可进入高级模式逐期填写。
3. 保证金独立计算,不参与剩余期数分摊。
4. 平台综合服务费按新订单总额重新计算。
5. 历史确认快照保留,新快照作为当前有效版本。

数据模型:

```sql
order_change_log
- log_id          bigint PK
- order_id        bigint FK
- change_type     enum -- product / sku / installments / down_payment / deposit / addon / total_price / multiple
- before_value    json
- after_value     json
- changed_by      bigint
- operator_role   enum -- merchant / merchant_staff / platform_csr
- change_reason   text
- customer_confirm_status enum -- pending / confirmed / expired / abandoned / withdrew_by_merchant
- customer_confirm_at datetime
- expires_at       datetime -- 默认 +72 小时
- withdrew_at      datetime
- withdrew_reason  text
- created_at      datetime
- updated_at      datetime
```

### 7.3 资方放款触发机制 ⭐ v1.5 整体重写

> **核心变化**:本版起,**订单结算款不再由"监管锁回调"单一事件触发**,而是由"**资方放款条件全部满足**"触发。
> - 监管锁上锁只是众多放款条件之一
> - 未来可增加任何新条件(公证 / 人脸活体 / 担保人 / 资方终审 等)
> - 不同资方可配置不同的条件清单

#### 7.3.1 商家订单(self_operate)

```
商家订单不触发资方放款
(门店自己出资,无需平台代付订单结算款)
后续由客户每月支付时的月度分账给门店
```

#### 7.3.2 联营订单 / 平台订单 — 多前置条件检查机制 ⭐

详细机制见 `资方管理/04_资方放款条件与触发机制.md`。本节摘要业务流程:

```
任何前置条件状态变化 → 触发"放款条件检查":
  - 风控审批通过
  - 资方已分配
  - 客户合同已签 (按资方对应合同模板)
  - 代扣已签约
  - 首期已付
  - 设备已交付
  - 客户已确认收货
  - 收货后验证通过(电子合同 / AI 问答 二选一)
  - 监管锁已上锁
  - 资方扩展条件(蓝海银行:征信授权 / 转让确认书 / 终审等)
  - 未来任何新条件
   ↓
系统按资方配置检查所有条件
   ├─ 全部满足 → 触发放款流程(§7.3.3)
   └─ 未全部满足 → 等待,记录日志
   ↓
事件丢失兜底:每 5 分钟定时任务复查
```

#### 7.3.3 放款流程(按 funding_source 分支)

【分支 A】funding_source = platform_self(平台自营资金)

```
① 检查平台自营授信额度是否充足
② 平台对公账户出资 = 设备价 × 出资比例
③ 实时充值通联备付金账户(< 1 分钟)
④ 充值到门店子台账(订单结算款入账)
⑤ 更新资方台账(资方=平台自营)
⑥ 更新平台自营授信额度
⑦ 写资方放款流水(funder_disbursement_log)
⑧ 推送商家端"订单结算款 ¥XXX 已到账"

例:联营 50/50,¥5,000 × 50% = ¥2,500
例:平台订单 100%,¥5,000 × 100% = ¥5,000
```

【分支 B】funding_source = funder_blue_ocean(蓝海银行赊销)

```
① 检查蓝海银行授信额度是否充足
② 平台调用蓝海银行放款 API
   POST /api/v1/blue-ocean/disbursement
   {
     "order_id": "OD2026001",
     "amount": 5000.00,
     "customer_info": {...},
     "device_info": {...}
   }
③ 蓝海银行接收 + 终审(若条件清单要求)+ 放款到平台对公账户
④ 平台对公账户实时充值通联备付金账户(< 1 分钟)
⑤ 充值到门店子台账(订单结算款入账)
⑥ 更新资方台账(资方=蓝海银行)
⑦ 更新蓝海银行授信额度
⑧ 写资方放款流水(funder_disbursement_log,带 bank_serial_no)
⑨ 推送商家端"订单结算款 ¥XXX 已到账"

例:平台订单 100%,¥5,000 = 蓝海银行授信占用 ¥5,000
```

### 7.4 关键设计

- **放款不再依赖单一事件触发**,而是多条件检查机制(§7.3.2)
- 任何前置条件状态变化都会触发一次检查 + 5 分钟定时兜底
- 不同资方可配不同的条件清单(可扩展)
- 监管锁上锁是众多条件之一,**不再是触发器**
- 商家订单不触发资方放款(门店自营)
- 联营订单 / 平台订单按出资比例计算放款金额
- 资方授信额度在放款时自动占用,完成 / 撤单时自动释放
- 任何资金动作都同步更新资方台账,确保资金链路闭环

### 7.5 资方放款异常兜底(v1.5 更新)

| 异常 | 处理 |
|---|---|
| 商家未及时上锁(超 24h)| 客服催促 |
| 商家长时间不上锁(超 7 天)| 进入主管审核 + 强制人工标记 |
| 主管手动标记"已上锁" | 触发放款条件检查 + 写"人工标记"日志 |
| 监管锁回调到达但其他条件未满足 | 仍然等待,直到全部满足才放款 |
| 平台对公账户已收钱但通联充值失败 | 系统重试 3 次 + 财务异常队列(详见 07 文档) |
| 通联充值成功但门店子台账未更新 | 系统对账兜底 |
| **蓝海银行 API 调用失败** | 系统自动重试 3 次 + 异常队列 |
| **蓝海银行终审拒绝** | 异常队列 + 客服联系客户;**可降级为平台自营**(详见资方 04 §6.4) |
| **资方授信额度不足** | 提交时风控阻拦;若已审核完成发生,异常队列 + 主管审核 |
| **条件检查任务漏跑** | 5 分钟兜底定时任务自动复查 |

### 7.6 在租期逾期费用机制(同 v1.4,字眼一致)

详细规则见 `11_逾期费用账单与规则配置.md`(v1.1 已统一字眼为"逾期费用账单与规则配置")。

#### 7.6.1 逾期费用生成

- 系统每日凌晨 1 点定时任务扫描所有"履约中"订单
- 检查每一期主账单是否逾期(到期日 + 1 < today)
- 已逾期但未生成逾期费用账单 → 新建
- 已生成逾期费用账单 → 累计天数 +1,重算金额
- 客户支付当期主账单后 → 该期逾期费用停止累计

| 平台订单 | 平台统一配置 |

#### 7.6.3 客服后台操作

详细权限矩阵 + 操作日志见 11 §10。所有减免必须填原因 + 留痕。

#### 7.6.4 客户 C 端展示

  来源期数  累计天 原始金额 减免  实付   操作

#### 7.6.5 提前结清 / 归还前的强制结清

```

检查是否有未结清逾期费用?
   ├─ 否 → 正常流程
   └─ 是 → 必须先结清逾期费用
            ├─ [一并结清后提前结清] → 金额 += 逾期费用合计 + 一笔支付

```
