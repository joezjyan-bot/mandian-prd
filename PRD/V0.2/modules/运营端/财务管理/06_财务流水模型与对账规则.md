# 财务流水模型与对账规则

> 所属端:运营端 / 财务管理
> 目标:把支付、账单、分账、钱包、提现、退款、冲正、对账拆成可开发的数据对象,确保钱流可追溯、可校验、可回滚。

> **⚠️ v1.3 历史兼容说明(2026-05-27,已被 V0.2.2 净化)**:
> - 历史文档中的 `funder_*` 字段和 `funder_receivable_ledger` 仅作为**内部资金来源台账兼容命名**保留,不得理解为客户侧、商家侧或合同侧的“内部资金来源台账”。
> - **新增撤单退款工单模型**(`refund_workflow`)+ 资金链路四方还原规则
> - settlement_entry 的 entry_type 从 7 类扩展为 10 类(新增 `refund_deduction` 撤单退款扣减、`offline_compensation` 门店线下补偿)
> - 三种合作模式商家端命名统一为商家/联营/平台订单
> - 旧版“四边对账升级为五边对账”在 V0.2.2 中调整为:通道边 + 业务边 + 钱包边 + 穿透边 + **内部资金来源台账边**。

> **⚠️ v1.2 修订(2026-05-27)**:重写为 settlement_* 流水模型(已被本版本继承)

> **⚠️ V0.2.2 全量整改补丁(2026-05-28)**:
> - 财务模型升级为“四账分离”:订单业务账、支付流水账、钱包/商家账户账、财务总账/分录账分别建模、分别验收、互相关联但不能混表替代。
> - 平台服务费默认 2%,后台可配置;商家长租和商家自营短租均按规则扣除平台服务费后分账到商家钱包。
> - 退款、打款、返点、提现、坏账核销、坏账回收必须有幂等、防重复、冲正和审计链路。

## 0. V0.2.2 财务总口径【资金/法务高风险】

### 0.1 四账分离

| 账类 | 主要用户 | 记录对象 | 不允许做的事 |
|---|---|---|---|
| 订单业务账 | 运营、客服、商家 | 订单金额、套餐、账单计划、退款责任、履约状态 | 不直接代表真实资金到账 |
| 支付流水账 | 技术、财务 | 通道交易、回调、手续费、退款请求、幂等键 | 不直接当商家钱包余额 |
| 钱包/商家账户账 | 商家、财务 | 可用余额、冻结余额、提现中、冲正、返点、分账到账 | 不直接替代财务总账 |
| 财务总账/分录账 | 财务、老板 | 平台收入、应付商家、服务费、坏账、回收、冲正 | 不直接给业务页面改状态 |

每一笔资金动作必须同时满足:

- 可追踪:能从订单追到支付、钱包、总账。
- 可审计:能看到操作人、审批人、时间、前后快照、凭证。
- 可回滚:失败或退款时能写反向流水和冲正分录。
- 可幂等:重复回调、重复点击、重复打款、重复返点不重复入账。

### 0.2 服务费和分账

| 场景 | 服务费 | 分账结果 |
|---|---|---|
| 商家长租 | 默认 2%,可配置 | 客户支付后扣服务费,商家应得进入商家钱包/结算账户 |
| 联营长租 | 默认 2%,可配置 | 按联营规则拆分商家、平台、内部台账;C 端不展示 |
| 平台长租 | 默认 2%,可配置 | 平台主导收入/内部台账;履约商家只看履约和约定结算 |
| 商家自营短租 | 默认 2%,可配置 | 客户缴费后扣服务费,剩余进入对应商家钱包 |
| 平台自营短租 | 默认 2%,可配置 | 不进入商家钱包,只入平台收入/成本台账 |

C 端不得展示服务费拆分、商家结算价、平台利润和返点规则。商家端和财务报表可以展示平台服务费、应结算款、已结算款、待结算款。

### 0.3 退款和冲正

退款必须创建退款记录和冲正链:

1. 订单业务账:记录退款原因、责任方、金额、协商单、审批结果。
2. 支付流水账:记录退款请求、通道退款号、通道状态、失败原因。
3. 钱包账:商家订单从商家应结算或钱包中扣减/冻结/挂账;联营/平台订单按协商方案冲正。
4. 总账:生成退款支出、服务费冲回、商家应付冲回、坏账/损失承担等分录。

商家订单退款默认商家承担;联营订单/平台订单需线下协商方案后执行。支付通道支持时,退款金额可由审批后的指定金额发起。

### 0.4 提现失败、重复打款和重复返点

| 风险 | 控制规则 |
|---|---|
| 提现失败 | 提现单进入失败;冻结余额释放;写提现失败回滚钱包流水和总账分录;原失败提现单不可复用为成功单 |
| 重复打款 | 打款请求必须带业务幂等键;同一订单/结算批次/收款账户/金额不得重复成功 |
| 重复返点 | 返点以订单号 + 推广关系快照 + 返点规则版本 + 期次作为唯一键 |
| 支付回调重复 | `channel_trade_no + callback_event_id + amount` 幂等;重复回调只记录日志不重复加款 |
| 退款重复 | 同一退款单只允许一个成功通道退款号;失败重试必须沿用退款单和幂等键 |

### 0.5 坏账核销和坏账回收

平台不默认兜底坏账。坏账承担按合同、证据、责任划分和审批结论确认。

坏账核销:

- 必须有审批链、原因、金额、订单/账单范围、法务意见或财务意见。
- 核销只影响总账和账单状态,不得删除原账单、支付流水、催收/法诉记录。

坏账回收:

- 法诉、催收或线下追回款项后,允许回原订单执行“回收完成”。
- 操作人选择完成方式并填写完成金额、凭证、备注。
- 系统新增回收支付流水、钱包冲正或平台收入分录、总账分录,并保留原坏账核销记录。

### 0.6 对账维度

财务必须支持按以下维度交叉对账:订单、客户、商家、门店、日期、支付渠道、业务员、推广员、业务线、订单类型、退款责任方、通道交易号、钱包流水号、总账分录号。

### 0.7 逾期费用归属与入账口径【资金/法务高风险】

当前逾期费用规则为业务默认值:0.05%/日、最低 10 元/日。逾期费用用于覆盖平台在售后、提醒、催收、客服协同和账务处理中的服务成本,不得称为罚息、利息、贷款费用。

归属口径:

1. 商家订单:逾期费用不确认为平台收入,由客户与商家按协议和商家规则承担;如系统代收,只作代收代付和流水记录。
2. 联营订单 / 平台订单:逾期费用归平台,计入平台逾期费用收入。
3. 若后续配置封顶、减免、豁免期或归属调整,必须同步更新订单业务账、账单规则、支付回调、钱包冲正、总账分录和合同/客服话术。
4. 任何逾期费用减免、冲正、补收、法诉回收均必须写入订单业务账、支付流水账、钱包账和总账分录的对应记录,不得以人工备注替代资金记录。

## 4. 分账流水模型(按合作模式)

字段:

- `allocation_id`
- `payment_id`
- `bill_id`
- `order_id`
- `cooperation_mode`:self_operate(商家)/ joint_venture(联营)/ receivables_assignment(平台)
- `funding_source`:platform_self / funder_blue_ocean / ...
- `funder_id`:关联内部资金来源(字段名兼容保留,不得外露)
- `total_amount`:客户支付总额
- `platform_service_fee_amount`:平台综合服务费(留对公户)
- `device_use_fee_amount`:设备使用费部分(穿透通联备付金)
- `merchant_share_amount`:门店分账金额
- `platform_share_amount`:平台分账金额(进平台台账)
- `internal_funding_share_amount` / `funder_share_amount`:内部资金来源回收金额(字段名可兼容保留,进入内部资金来源台账 `collected_amount`)
- `channel_commission_amount`:渠道佣金(若有)
- `rule_version_id`:分账规则版本
- `status`:pending / completed / failed

规则(按合作模式):

| 合作模式(商家端用语) | 后台标识 | 平台综合服务费 | 设备使用费部分(分账)|
|---|---|---|---|
| 商家订单 | self_operate | 留对公户 | 100% → 门店 |
| 联营订单 | joint_venture | 留对公户 | 商家/平台约定比例 × 设备使用费分别进入商家钱包、平台台账和内部资金来源台账 |
| 平台订单 | receivables_assignment | 留对公户 | 100% → 平台台账 / 内部资金来源台账,商家端只看履约和约定结算 |

- 分账失败不改门店结算账户余额
- 分账成功后才触发资金穿透到通联备付金
- 分账成功后同步更新内部资金来源台账 `collected_amount`
- 详细算法见 `财务管理/12_订单合作模式与收益分配规则.md §6.4`

## 5. 结算账户流水模型(核心)⭐

完整模型见 `财务管理/07_门店结算账户与资金穿透架构.md §3`。

### 5.1 `settlement_entry` 字段

- `entry_id`
- `account_id`(关联 merchant_settlement_account)
- `direction`:in / out
- `entry_type`:10 类入账/出账类型(详见 §5.2)
- `amount`
- `order_id` / `order_no`(若关联订单)
- `cooperation_mode`:订单合作模式快照
- `funding_source`:资金来源快照
- `funder_id`:关联内部资金来源(仅后台/财务可见)
- `source_funder_type`:assignment_payer / customer / joint_venturer / platform / merchant_offline_compensation
- `platform_account_ref`:平台对公账户过路凭证(银行流水号)
- `tonglian_topup_id`:通联充值流水号
- `tonglian_payout_id`:通联代付流水号
- `balance_before` / `balance_after`
- `description`:摘要(门店端 UI 展示)
- `status`:pending / completed / failed / reversed

### 5.2 `entry_type` 10 类枚举(v1.3 扩展)

| 枚举值 | 中文 | 触发场景 | 方向 |
|---|---|---|---|
| `order_settlement` | 订单结算款 | 联营/平台订单监管锁回调 | in |
| `monthly_split` | 月度分账 | 客户每期租金支付后(三模式都触发) | in |
| `joint_venture_profit` | 联营分润 | (并入 monthly_split,实际无独立流水) | - |
| `platform_reward` | 平台奖励 | 平台主动给的奖励 | in |
| `platform_rebate` | 平台返利 | 渠道返利 / 推广提成 | in |
| `platform_deduction` | 平台扣款 | 逾期费用扣减 / 罚款 / 服务费补扣 等 | out |
| **`refund_deduction`** ⭐ v1.3 新增 | 撤单退款扣减 | 撤单退款时,从门店结算账户扣减已分账金额(账面调整,与门店线下转账配合)| out |
| **`offline_compensation`** ⭐ v1.3 新增 | 门店线下补偿入账 | 门店通过银行对公转账给平台对公账户(罕用,撤单后门店追加补偿)| in |
| `withdrawal` | 提现 | 门店主动提现 | out |
| `financial_adjustment` | 财务调整 | 财务人工调账(经审批) | in/out |

### 5.3 规则

- 门店结算账户余额只由 settlement_entry 计算,不允许直接手改
- 余额公式:`balance_total = balance_available + balance_pending + balance_withdrawing`
- 流水状态 `pending` 时计入 balance_pending;`completed` 时进入 balance_available
- 异常 / 冲正必须写独立反向流水(`status=reversed`)或新 entry(`refund_deduction`),不删原始流水

## 6. 平台对公账户穿透日志

`platform_passthrough_log` 字段:

- `log_id`
- `direction`:in(收入金)/ out(穿透出去)
- `amount`
- `source_account`:来源(客户/受让方/联营方/通联备付金/门店线下转账)
- `target_account`:目标(平台对公/通联备付金/门店银行卡/内部资金来源清算映射/客户)
- `business_ref`:业务关联(订单 ID / 客户支付流水 ID / 退款工单等)
- `order_id`
- `entry_type`:关联的 settlement_entry.entry_type
- `platform_commission`:本次扣的平台综合服务费(若适用)
- `bank_serial_no`:银行流水号
- `tonglian_serial_no`:通联流水号(若涉及通联)
- `created_at`

规则:
- 平台对公账户每收/出一笔钱,必须写 platform_passthrough_log
- 用于日终对账"平台对公账户 ↔ 通联备付金账户"的资金穿透是否完全闭环
- 严格的 1:1 等额穿透原则,差异即异常

## 7. 内部资金来源应收台账(v1.3 兼容模型,V0.2.2 净化)⭐

详细数据模型见 `资方管理/01_资方主体与授信管理.md §2.2`。该模型仅用于后台资金来源、额度和回款核对;不得在 C 端、商家端、合同、客服话术中呈现为“内部资金来源台账”或“内部资金来源回款”。

### 7.1 `funder_receivable_ledger` 核心字段

```sql
funder_receivable_ledger
- ledger_id               bigint PK
- funder_id               bigint FK   -- 内部资金来源 ID,字段名兼容保留
- order_id                bigint FK
- order_no                varchar(32)
- cooperation_mode        enum          -- 合作模式快照
- transferred_amount      decimal(18,2) -- 内部资金来源投入 / 联营内部出资记录
- collected_amount        decimal(18,2) -- 已从客户租金回收的部分
- refunded_amount         decimal(18,2) -- 撤单/退款时内部资金来源冲回金额
- outstanding_balance     decimal(18,2) -- 应收余额
- status                  enum          -- active / collected / refunded
- created_at / updated_at
```

### 7.2 内部资金来源维度对账规则

```
对每个内部资金来源,日终对账验证:
   总投入 = SUM(transferred_amount)
   总回收 = SUM(collected_amount) + SUM(refunded_amount)
   应收余额 = SUM(outstanding_balance) = 总投入 - 总回收

   零和原则:
   ∀ 订单 i,transferred_i = collected_i + refunded_i + outstanding_i

   内部额度验证:
   credit_limit_used = SUM(outstanding_balance) where status=active
   credit_limit_total = credit_limit_used + credit_limit_left
```

### 7.3 内部资金来源全平台维度验证

```
全平台内部资金来源验证:
   内部资金来源台账总应收余额 = SUM(所有 funder_receivable_ledger.outstanding_balance)
   该余额只用于财务核对和内部额度管理,不形成客户侧或商家侧账户展示。
```

## 8. 撤单退款工单(v1.3 新增)⭐

完整规则见 `财务管理/07_门店结算账户与资金穿透架构.md §6`。

### 8.1 `refund_workflow` 字段

```sql
refund_workflow
- workflow_id             bigint PK
- order_id                bigint FK
- refund_type             enum          -- before_settlement(收货前)/ after_settlement(已分账)
- customer_refund_amount  decimal(18,2) -- 客户应退金额
- merchant_payback_amount decimal(18,2) -- 门店应退还平台金额
- internal_funding_reversal_amount / funder_refund_amount decimal(18,2) -- 内部资金来源冲回金额(字段名兼容保留)
- funder_id               bigint        -- 关联内部资金来源
- status                  enum          -- pending / merchant_paid / refunded_customer / funding_reversed / completed
- merchant_payback_at     datetime      -- 门店转账到账时间
- merchant_payback_ref    varchar(64)   -- 门店银行转账流水号
- customer_refund_at      datetime
- funding_reversal_at / funder_refund_at datetime
- created_by              bigint
- created_at / updated_at
```

### 8.2 工单状态机

```
pending(待门店转账)
   ↓ 门店银行对公转账到位
merchant_paid
   ↓ 平台执行客户退款 + 内部资金来源冲回
refunded_customer
   ↓ 内部资金来源冲回
funding_reversed
   ↓ 工单核对
completed(全闭环)
```

### 8.3 工单驱动的多方资金动作

| 工单状态 | 触发的资金动作 |
|---|---|
| pending | 系统计算各方应退/应补金额,生成工单,通知门店 |
| merchant_paid | 门店线下转账到平台对公账户;财务确认;同步反向冲销 settlement_entry(`refund_deduction`)+ 通联备付金子台账扣减 |
| refunded_customer | 平台对公账户出账给客户(Q1=B 等门店转账到位再退,详见 07 §6) |
| funding_reversed | 按协商单和审批结果执行内部资金来源冲回或台账冲正;同步更新内部资金来源台账(refunded_amount += amount,outstanding_balance -= amount,额度占用 credit_limit_used -= amount)|
| completed | 财务核对账面闭环 |

## 9. 提现与打款(门店结算账户提现)

### 9.1 `settlement_withdrawal` 字段

- `withdrawal_id`
- `account_id`
- `merchant_id`
- `amount_requested`:申请金额
- `fee_amount`:手续费(千二,可配)
- `amount_arrival`:实际到账 = amount_requested - fee_amount
- `fee_rate`:费率快照
- `bank_account_id` / `bank_account_no_masked`
- `audit_status`:pending / approved / rejected / no_audit_needed
- `audited_by` / `audited_at` / `audit_reason`
- `tonglian_payout_id`:通联代付流水号
- `payout_status` / `payout_at` / `failure_reason`
- `status`:pending / auditing / paid_processing / paid_success / paid_failed / cancelled

### 9.2 规则

- 门店自助提现按 V0.2.1 冻结规则执行:单笔 500-20000 元,每日最多 3 次;超 20000 元或异常提现进入人工审核
- 发起提现:冻结 balance_available → balance_withdrawing
- 通联代付成功:balance_withdrawing -= amount,total_withdrawn += amount_arrival
- 通联代付失败:balance_withdrawing 退回到 balance_available
- 财务驳回:同失败处理

## 10. 退款和冲正

退款适用:

- 客户取消订单(通过 `refund_workflow` 工单走)
- 审核拒绝后退款
- 售后退款
- 重复支付

冲正适用:

- 分账错误
- 结算账户入账错误
- 通道对账差异
- 人工复核后反向调整
- 订单结算款冲正(订单撤单 → 通过 refund_workflow 工单走)

要求:

- 退款影响支付流水、账单、订单状态
- 冲正影响分账流水和 settlement_entry
- 已提现金额不能直接扣回,通过 `refund_workflow` 工单要求门店线下转账补回(余额够直接扣减 `refund_deduction`;不够走 `refund_workflow` 工单)
- 退款和冲正必须二次确认和权限控制

## 11. 五账对账(v1.3 升级,V0.2.2 净化)⭐

每日日终对账:

| 对账边 | 内容 |
|---|---|
| 通道边 | 支付、退款、通联代付流水 |
| 业务边 | 订单、账单、分账流水、提现单、退款工单 |
| 钱包边 | 门店结算账户余额、settlement_entry 累计 |
| 穿透边 | 平台对公账户 ↔ 通联备付金账户(穿透是否完全闭环)|
| **内部资金来源台账边** ⭐ | 内部资金来源台账、额度占用、回收累计 |

### 11.1 平账条件

#### 通道边 ↔ 业务边

- 支付通道实收 = 支付流水成功金额
- 支付流水成功金额 = 账单已付金额
- 通联代付成功金额 = 提现单 amount_arrival 累计 + 退款工单 refunded_customer / funding_reversed 累计

#### 业务边 ↔ 钱包边

- 分账流水门店份额累计 = settlement_entry(monthly_split, direction=in)累计金额
- 订单结算款累计(联营/平台)= settlement_entry(order_settlement, direction=in)累计
- 撤单退款扣减累计 = settlement_entry(refund_deduction, direction=out)累计

#### 钱包边

- merchant_settlement_account.total_received = SUM(settlement_entry direction=in status=completed)
- merchant_settlement_account.total_withdrawn = SUM(settlement_withdrawal status=paid_success amount_arrival)
- merchant_settlement_account.balance_total = balance_available + balance_pending + balance_withdrawing

#### 穿透边

- 平台对公账户每笔"out"穿透 = 通联备付金账户对应"in"充值,严格 1:1
- 平台对公账户余额变化 = 收入金 - 平台综合服务费留存 - 穿透出去金额 - 撤单退款给客户 - 内部资金来源冲回

#### 内部资金来源台账边 ⭐

```
对每个内部资金来源:
  SUM(funder_receivable_ledger.transferred_amount) - SUM(collected_amount) - SUM(refunded_amount) = SUM(outstanding_balance)
  
  funder.credit_limit_used = SUM(outstanding_balance) where status=active
  funder.credit_limit_left = funder.credit_limit_total - funder.credit_limit_used
```

#### 通联备付金账户内部

- 通联备付金主账户余额 = 所有门店子台账之和 + 平台子台账
- 内部资金来源台账余额 = SUM(所有 active 状态资金来源的 outstanding_balance)
- 每个门店子台账余额 = merchant_settlement_account 对应账户的 balance_total

## 12. 异常处理(v1.3 更新)

| 异常 | 处理 |
|---|---|
| 通道有款,系统无支付流水 | 人工关联或补单 |
| 系统支付成功,通道无款 | 冻结订单并禁止分账 |
| 分账不平 | 进入分账异常队列 |
| 门店结算账户余额不平 | 禁止提现,财务复核 |
| 通联代付失败 | 退回 balance_available,记录失败 |
| 退款成功但账单未更新 | 重放回调或人工补偿 |
| 监管锁回调到达但订单状态不符 | 异常队列,主管核实 |
| 平台对公账户已收钱但通联充值失败 | 系统重试 + 财务异常队列 + 人工补穿 |
| 通联充值成功但门店子台账未更新 | 系统对账兜底 |
| 平台对公账户 ↔ 通联备付金账户穿透差额 | 立即告警 + 财务核实 |
| 同一订单收到重复监管锁回调 | 系统幂等校验,只处理首次 |
| 客户租金支付金额与应收金额不符 | 进入对账异常 |
| 联营订单平台出资款打款金额错误 | 系统阻止打款,人工核对 |
| **撤单退款工单超时未到位(门店未转账)** ⭐ v1.3 | 财务催办,超 7 天升级主管 |
| **内部资金来源台账不平(投入 ≠ 回收+应收余额)** ⭐ | 立即告警 + 财务核实,核对每笔订单 |
| **内部额度占用溢出**(credit_limit_used > credit_limit_total)| 立即告警 + 暂停该资金来源继续分配 |
| **内部资金来源台账汇总不平** | 立即告警 + 财务核实 |

## 13. 验收标准(v1.3 更新)

1. 任一金额变动都有流水
2. 任一流水能追到来源业务对象
3. 退款和冲正不会删除原始流水
4. 日终对账能定位差异来源
5. 门店结算账户余额可由 settlement_entry 重算
6. 通联备付金账户余额可由 platform_passthrough_log 重算
7. 每笔订单结算款入账可追溯到:订单 + 订单类型 + 内部资金来源(若有)+ 监管锁回调事件 + 分账比例 + 平台对公收款流水 + 通联充值流水
8. 每笔月度分账入账可追溯到:客户支付流水 + 平台综合服务费扣减 + 订单类型分账规则 + 通联充值流水 + 内部资金来源台账更新
9. 每笔提现可追溯到:提现申请 + 审核记录(若有)+ 通联代付流水 + 银行到账时间
10. **每笔撤单退款可追溯到**:撤单触发 + refund_workflow 工单 + 门店线下转账凭证 + 客户退款流水 + 内部资金来源冲回流水 + 内部资金来源台账还原
11. **每个内部资金来源台账闭环**:总投入 = 总回收 + 应收余额
