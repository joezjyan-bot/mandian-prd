### 3.3 统一 C 端合规口径

已完成：

- C 端目录扫描结果为 0 命中：
  - `资方|资金来源|合作模式|风控`
  - `押金|还款|买断|软钱包|租用中|在租中|门店订单|分红订单`
- C 端统一为信用/信息授权，不展示内部评分、资金来源、合作模式或资金方身份。
- 留购/提前结清按钮由后台话术模板控制，不泄露底层业务模式。

验收结论：通过。

### 3.4 统一资金和合同主线

已完成：

- 默认合同主线固定为三方租赁合同：
  - 甲方：商家/出租方
  - 乙方：客户
  - 丙方：平台
- 客户合同和 C 端不出现资金方身份。
- 银行赊销/外部资金能力仅作为后台 `funding_source`、`funder_id`、话术模板和内部台账，不改变客户侧三方租赁主链路。
- 财务主线统一为平台代收、订单结算款、我的钱包、提现、退款冲正、内部资金台账。

验收结论：通过。

### 3.5 统一提现和权限规则

已完成：

- 普通提现单笔范围：500-20000 元。
- 每日最多 3 次。
- 超 20000 元或异常提现进入人工审核。
- 商家财务人员可被授权查看钱包和发起提现。
- 商家财务人员不能修改结算账户。
- 修改结算账户必须高权限、平台审核、二次确认、完整日志。

验收结论：通过。

### 3.6 拆分长租设备和短租库存逻辑

已完成：

- 长租只在发货/交付时绑定设备识别码档案，例如 `IMEI/SN/VIN`。
- 长租不依赖短租车辆库存状态机。
- 短租车辆库存、取还车、小时/日/周租期、开关机/定位等能力全部标记为 V2 后置。
- `modules/开发设计/00_V0.2.1_核心数据模型总表.md` 明确 V1 模型和 V2 预留模型边界。

验收结论：通过。

### 3.7 发货签收和 AI 问答确认

已完成：

- `modules/运营端/发货履约/01_发货配置与物流接口.md` 已同步 V0.2.1 口径。
- V1 默认签收方式为电子确认单。
- AI 问答签收视频先补齐配置、状态、记录和异常人工复核设计，不要求 V1 接入真实 AI 服务。

## 7. 开发冻结结论

V0.2.1 可以作为 V1 长租主营链路的开发冻结版本进入下一阶段。下一阶段不建议再做大范围 PRD 重写，而是按以下顺序推进：

1. 接口级最终同步。
2. 数据库 migration 设计。
3. 后端核心链路开发。
4. 管理后台页面开发。
5. C 端长租下单和履约链路开发。
6. 按 `V1_端到端验收矩阵.md` 做联调验收。
