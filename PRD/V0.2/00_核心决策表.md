## P0 业务底座(4 项)

### P0-1 资金链路口径

**核心结论**:三类订单统一走平台代收 → 钱包结算 → 提现。

**关键决策**:
- 收款通道:**通联备付金账户**(银行卡) + **支付宝直付通分账**(支付宝);微信 V1 不优先
- 抽佣:三类订单默认 2%,后台可配,新建商户默认 2%
- **未签合同前关闭:免费**;**已签合同且 7 天内关闭:订单总租金 × 1.5%**;**7 天后只能客服处理**
- 线下支付:门店或客服可标记(联营订单**禁止**线下,因内部资金对接平台必须线上可追溯);商家订单从商家钱包扣 2%,平台订单从内部资金账户扣 2%;客服代标记需填资金核实情况、24 小时门店异议期
- 余额不足:**不阻断**业务,商家自行充值;运营有"商家欠款列表"跟进
- 内部资金账户余额不足:**待扣费队列**,等下次回款扣
- 钱包提现:单笔 500-20000,日 ≤ 3 次,手续费千分之 2 + 5 毛;> 2 万人工审核;到账时效预留快速/普通/慢速档位
- V2 升级:接分账支付通道,客户支付时直接拆分到落地账户

**详见**: `01_重构决策基线.md §9`(待补)、`modules/运营端/财务管理/*`、`modules/全局/08_线下支付标记规则.md`

### P0-2 商家与门店主体关系

**核心结论**:商家 1:N 门店(受控字典),通过门店标签和角色实现数据隔离。

**关键决策**:
- 主体:1 张营业执照 = 1 个商家主账号;多张执照**强制独立入驻**
- 商家主账号下可创建多个**门店**(受控字典,商家自行管理)
- **门店在数据库里是独立实体**(为 V2 门店级钱包/资质预留)
- 角色四级:**商家老板 / 管理员 / 店长 / 普通员工**
- 店长可看本门店所有员工数据
- 订单上必须**冻结**办单门店和员工角色快照(防员工调店导致历史数据错位)
- V1 **不支持**跨门店设备调拨(V2 再做)

**详见**: `modules/商家端/门店资料/01_门店管理.md`、`modules/商家端/门店资料/02_员工管理.md`、`modules/全局/05_账号模型与登录安全.md`

### P0-3 分账细节(退款 / 逾期费用 / 部分支付 / 渠道佣金)

**核心心智**:平台是资金代收方 + 记账方 + 工具,**不是**资金担保方,**不兜底客户**。客户履约争议仍以出租方商家为责任主体。

**关键决策**:
- **"分账有出,退款必入"**:已分账金额必须从相应钱包扣回;余额不足必须充值到位才能扣减;扣减未完成前**关闭订单/下监管锁/解锁全部锁死**
- **责任方钱包必须充值到位**,平台不垫付
- 监管锁在退款扣减未完成时**继续锁着**(作为给责任方的履约压力)
- 退款全部走"线下协商 + 线上手工执行";退款金额支持手工填写、按项目逐项拆分;责任方决定扣谁的钱
- **逾期费用**:0.05%/日,最低 10 元/日,**全部归平台**(不参与商家/内部资金账户分账)
- 部分支付:**按原比例拆**,不引入优先级
- 渠道佣金:**平台推广费用**,从平台收益账户出;只统计联营/平台订单;退款时按对应比例追回(扣不动挂"渠道应付待追")
- 渠道佣金结算:**提现申请主流程 + 月底自动批量打款兜底**
- 平台抽佣**全额退回**客户(因为已经收了 1.5% 关单补偿)
- 后台余额变动功能:支持对公转账 + 凭证上传 + 操作日志
- 充值通道:三方支付实时到账 + 对公转账后台手工入账

**详见**: `modules/运营端/财务管理/03_退款单与责任划分.md`、`modules/运营端/财务管理/02_钱包充值与提现.md`、`modules/运营端/财务管理/04_后台余额变动.md`、`modules/运营端/财务管理/05_门店欠款列表.md`、`modules/运营端/渠道管理/02_渠道佣金结算与退款冲正.md`

#### 逾期费用归属与入账口径【资金/法务高风险】

1. 当前业务默认值为 0.05%/日、最低 10 元/日,统一称为“逾期费用”,不得称为罚息、利息、贷款费用。
2. 商家订单:逾期费用不确认为平台收入,由客户与商家按协议和商家规则承担;如系统代收,只作代收代付和流水记录。
3. 联营订单 / 平台订单:逾期费用归平台,用于覆盖平台售后、提醒、催收、客服协同和账务处理成本。
4. 若后续调整费率、最低金额、封顶规则或归属方式,必须通过版本变更同步更新本文、财务规则、合同文本、C 端展示和 OpenAPI 示例。

### P0-4 中控台边界

**核心结论**:**C 混合模式**——租赁系统直连第三方,中控台异步计费 + 客户级管控。

**关键决策**:
- 第三方调用路径:租赁系统 → 第三方厂商(**直连,不经过中控台代理**)
- 客户级开关 / 套餐余量:**中控台主导**,周期性 push 到租赁系统本地缓存
- 同步频率:每 5 分钟一次,本地缓存 TTL 24 小时
- 调用前检查本地缓存:客户授权 + 余量 → 通过才调用
- 调用日志:租赁系统调用后,**异步 push** 中控台
- 实际扣费:中控台**异步扣费**
- 中控台短暂异常(< 1 小时):本地缓存还在,业务无感
- 中控台长期异常(> 7 天本地缓存过期):**降级模式默认"按已授权放行"**,保业务优先,事后对账追扣
- **所有第三方功能**(e签宝、人脸识别、OCR、风控、流水报告、监管锁、大数据)都通过中控台对接
- 月底对账机制兜底

**详见**: `modules/全局/09_中控台对接.md`

## P1 关键决策(6 项)

### P1-1 体验租短租设备状态机

**核心结论**:体验租短租进入 V1,使用独立车辆库存状态机;长租不启用短租库存状态机,只在发货节点绑定设备识别码档案。

**状态清单**:
1. 待入库
2. 在库可租
3. **预约保留**(新增,短租预约场景)
4. 已锁定(已支付待取/待发货)
5. 出租中
6. 归还待验
7. 维修中
8. **争议中**(新增,门店与客户分歧)
9. 已下架

**关键决策**:
- "已下架"用"下架原因"字段记录细节(报废/售二手/退还供应商/丢失),**不细分状态**
- **已下架设备不显示在库存列表**(避免占用列表)
- V1 长租 **不支持**跨门店调拨;体验租平台自营复杂调拨后置,所以不需要"运输中"状态阻塞 V1
- 体验租车辆码必须唯一,取车核销后绑定到具体车辆
- 设备状态流转全程留日志

**详见**: `modules/运营端/库存设备管理/01_设备状态机.md`

### P1-2 资方准入与分配规则

**核心结论**:运营线下创建资方 + 简化偏好配置 + 系统推荐人工确认。

**关键决策**:
- 入驻:**运营线下谈定后,后台人工创建资方账号**(V1 不开放自助入驻)
- 偏好配置:**品类 + 单笔限额 + 总额限额**(V1 三个维度)
- 分配方式:**系统推荐 + 人工确认**(系统按规则筛 1-3 个候选,客服选)
- 筛选规则:品类匹配 + 单笔限额 + 总额限额 + 余额 + 启用状态;按"剩余可放款额度"从高到低排序
- 超时:**2 小时未分配 → 进入"待人工介入"队列**,客服优先处理;V1 不自动降级
- **V1 不开放资方端**,所有动作由运营客服代操作
- 资方分账时机:**实时**(客户每次实付时即时分账)

**详见**: `modules/运营端/资方管理/*`(共 5 份页面级 PRD)

### P1-3 价格方案调整 & 修改账单日

**核心结论**:V1 **不做"原订单改价"**,已签合同后走"取消重下"流程;修改账单日仅运营客服可操作。

**价格调整关键决策**:
- 待付款 / 待审核(未签合同):**客服可直接修改价格方案版本**
- 已签合同后:必须**关闭原订单 + 重新下单**(走 P0-3 退款规则)
- 发起人:**仅客服**(平台主导)
- 幅度:降价 ≤ 20% 客服自主,> 20% 主管审批
- **禁止涨价**
- 客户感知:不需单独确认,签合同/收货流程会看到新价格
- 版本管理:**完整版本管理**(v1/v2/v3...)

**商家订单关闭(修订 P0-1)**:
- **未签合同前关闭:免费**
- 已签合同 + 7 天内:扣 1.5%
- 7 天后:仅客服处理

**修改账单日关键决策**:
- 适用阶段:**订单生效后 7 日内**
- 修改次数:**仅 1 次**
- **不产生逾期费用**(合规便利)
- 操作端:**仅运营客服在运营端操作**(客户/门店都看不到此功能)
- 客户感知:**不需要客户确认**(因为是后延,对客户有利),系统自动发小程序通知
- 影响范围:**包括已生成的第 1 期账单一起改**,所有未到期账单按新账单日重新生成

**详见**: `modules/运营端/订单管理/03_订单关闭与系统费用.md`、`04_订单价格调整.md`、`05_订单取消重下流程.md`、`06_修改账单日.md`

### P1-4 合同链路

**核心结论**:**三方租赁合同制固定**(甲方门店、乙方客户、丙方平台);合同模板分层管理(默认 + 后台指定模板),客户合同和 C 端不出现资方。

**关键决策**:
- 三方主体固定:甲方门店、乙方客户、丙方平台
- 资方在合同中**不出现**(避免被定性为融资租赁)
- 合同模板分层:
  - **默认合同模板**(平台维护):适用所有未匹配后台指定模板的订单
  - **后台指定模板**(按内部资金来源、业务类型或运营配置匹配):联营/平台订单完成后台配置后自动切换
  - 未配置专属模板时,继续用默认
- 电子签:**必走电子签**;支持 e签宝 / 法大大等多通道,**通道开关在系统设置中配置**
- 公证:**系统接入,合同签署完成后决定是否发起**
  - 商家订单:**门店自选**
  - 联营/平台订单:**客服决定**
- 违约责任:**按订单类型差异化**
  - 商家订单:门店自行追责,款项归门店
  - 联营订单:平台代追,款项按内部结算比例分给商家/内部资金账户
  - 平台订单:平台追责,按内部台账结算
- 合同存储:**平台必须下载 PDF 副本存档**(防通道方跑路 + 监管查阅);至少 5 年 + 业务结束后 3 年
- 模板版本管理:**快照存档**,模板更新不影响历史订单
- 门店私下融资:**V1 不考虑**

**详见**: `modules/运营端/合同公证/*`(3 份)、`modules/运营端/配置管理/05_合同模板管理.md`、`modules/运营端/配置管理/06_电子签通道配置.md`、`modules/运营端/资方管理/05_资方专属合同模板.md`

### P1-5 C 端核心流程

**核心结论**:V1 支持支付宝 + 微信 + H5 三端,后期出独立 App;断点续传 + 后台发起客户响应混合模式。

**关键决策**:
- **客户端**:V1 支持**支付宝小程序 + 微信小程序 + H5**;后期出独立 App
- **业务入口**:C 端首页分为安心用和体验租;安心用承接长租主营链路,体验租承接短租门店/车辆/取还车链路
- **三种二维码**:
  - 办单二维码 → 订单确认页(锁定方案,直接下单)
  - 商品二维码 → 商品详情页(自由选套餐规格)
  - 小程序入口码 → 小程序主页(品牌入口)
- 下单流程:**断点续传 + 后台发起客户响应混合**
  - 客户连续走 1-6(订单确认 → 资料补充 → 实名 → 信用/信息授权 → 流水报告 → 保证金/首期支付)
  - 后台发起 7-10(合同签署 → 公证 → 绑卡 → 收货确认)
- 失败处理:**分情况差异化**(实名/人脸可重试 3 次,业务审核未通过进入待审核客服决定,支付失败可换通道等)
- 门店远程协助:**身份认证/支付必须客户本人,资料填写可门店代为操作**
- 订单列表:**简洁模式**(订单+商品+状态+金额+关键待办高亮)
- 订单详情:**详尽模式**(按区块组织,客户自助操作充分)
- 详情待办按钮:**按状态动态显示**
  - 租赁模式使用"申请留购";银行赊销/内部资金来源模板可使用"提前结清"
- 超时机制:24 小时未推进进"待处理",72 小时自动关单(未签合同免费)

**详见**: `modules/C端/*`(14 份页面级 PRD)

### P1-6 IM 客服体系

**核心结论**:V1 接入 IM 客服基础能力,采用统一客服中心 + 会话上下文 + 工单转化 + 业务跳转模式。

**关键决策**:
- C 端、商家端、门店端、运营端均可进入客服,但入口必须带业务上下文。
- 客户会话按 `customer_id` 汇总,支付宝、微信、H5 和后期 App 通过 `customer_external_accounts` 统一映射。
- IM 会话必须携带业务线、订单号、订单状态、商品/车辆、门店、当前页面和触发原因。
- 客服按长租、体验租、财务、客诉等队列分流;支持在线、忙碌、小休、离线和主管接管。
- IM 内不直接执行退款、关单、改价、锁车、扣款、修改结算账户等高危动作;只能跳转到对应业务模块执行权限校验、二次确认和日志。
- 无法即时解决的问题必须转工单,工单关联客户、订单、会话、证据和 SLA。
- 客服话术受全局措辞规范约束,C 端不暴露资方、资金来源、合作模式、风控等内部概念。

**详见**: `modules/运营端/客服中心/00_IM客服体系总览.md`、`modules/全局/12_账号身份与跨端统一.md`

## P2 工程基线(4 项)

### P2-1 多租户 / 部署架构

**核心结论**:**独立部署 + 统一代码维护**(单租户独立部署模式)。

**关键决策**:
- 数据库:**每个租户一个独立数据库**(物理隔离,**不加 tenant_id 字段**)
- 服务器:**V1 初期一台服务器 + 多 Docker 容器**(成本最低);业务量爆发的租户可平滑迁移到独立服务器
- 代码:**统一主代码仓库**;不分叉
- 升级:**灰度发布**——先升 1 家灰度租户验证,1-2 周稳定后批量推送
- 数据库变更:**强制使用 migration 框架**
- 配置分层:
  - 静态配置(品牌名、Logo、客服电话、域名):`.env` 文件
  - 业务配置(抽佣比例、费率、合同模板、资方池):数据库 `platform_config` 表
- 租户级定制:**优先 feature flag**,解决不了的成为通用功能;允许付费定制
- 跨租户报表:**集中汇总数据库**(各租户实例定时同步关键数据)
- 数据隔离:数据库账号权限严格按租户隔离,跨库查询禁止

**详见**: `modules/全局/11_工程规范基线.md`

### P2-2 操作日志查询页

**核心结论**:V1 必做,按订单/操作人/时间/操作类型筛选,可导出。

**关键决策**:
- 日志维度:操作来源、操作人、操作时间、操作类型、操作对象、操作前后值、IP、设备
- 涉及资金的关键动作:**必须二次确认 + 完整日志**
- 查询入口:运营端 - 权限日志 - 操作日志查询
- 查询能力:多维度筛选 + 时间范围 + 模糊搜索 + 导出 Excel
- 权限:仅运营管理员 + 财务主管可查;普通客服只能查自己操作记录

**详见**: `modules/运营端/权限日志/01_操作日志查询页.md`

### P2-3 测试数据脱敏规范

**核心结论**:V1 必做,定义身份证/手机号/姓名/地址等测试数据池。

**关键决策**:
- 身份证号:校验位正确但身份证号是假的(避开真实公民信息)
- 手机号:用 180/170 段虚号
- 姓名:从数据库白名单池随机
- 地址:从虚构地址池随机
- 所有 PRD 文档中:**不得使用真实业务数据**(订单号/客户姓名/金额都用占位符)
- 测试环境部署时:数据全部自动脱敏

**详见**: `modules/测试验收/04_测试数据脱敏规范.md`

## 工程规范(必守)

✅ 必须遵守的工程基线:

1. **数据库 schema 变更必须走 migration 框架**(PHP / Laravel Migrations 或同类)
2. **代码 Docker 化交付**
3. **数据库连接信息走环境变量,代码不硬编码**
4. **任何"租户专属信息"不得硬编码**——走 `.env` 文件或 `platform_config` 表
5. **部署用 CI/CD 流水线**,不手工 SCP 代码
6. **灰度发布机制**:先升 1 家灰度租户,稳定后批量推送
7. **同一份代码可部署为"多 Docker 共用一台服务器"或"独立服务器"**,架构可平滑切换
8. **全程留操作日志**(尤其涉及资金的关键动作必须二次确认 + 完整日志)
9. **敏感数据展示默认脱敏**(姓名/手机号/身份证/地址),查看完整需权限 + 日志
10. **危险操作二次确认**:审核通过、审核不通过、取消订单、加黑名单、修改价格、修改租期、关闭订单、提现打款、权限变更、删除
