# UI 出图任务入口

> 目标：后续让 Claude 或设计工具出完整界面图时，按这里排顺序和验收，不再自由发挥。

---

## 1. UI 基准

1. 默认采用 `finance-zhongtai` 企业中台风格。
2. 登录页参考无界租清爽扫码/密码登录布局。
3. 订单列表参考无界租高密度订单卡片/表格混合信息密度。
4. 办单助手参考 `phone-rent` 计算器逻辑和移动端 UI。
5. 页面要服务运营和办单效率，不做营销型大屏或装饰型页面。

## 4. 订单列表出图要求

订单列表必须拆三类视图：

1. 门店订单
2. 分红订单
3. 平台订单

每个视图都要展示：

- 订单号、下单时间、订单类型、来源渠道。
- 商家、门店、办单员工。
- 商品、规格、设备价、长租租期/套餐、交付设备状态。
- 客户姓名脱敏、手机号脱敏、地址摘要。
- 已付/应付、押金、服务费、公证费、留购价。
- 审核、合同、公证、支付、发货、设备、账单状态。
- 商户备注、风控提示、回调提示、操作日志摘要。
- 当前可操作按钮。

筛选必须包含：

- 订单号、客户、手机号、商品、商家、门店、资方、渠道、员工、订单类型、订单状态、审核状态、支付状态、合同状态、发货状态、下单日期、到期日期、逾期状态。

---

## 5. 订单详情出图要求

订单详情必须是工作台式页面，至少包含：

1. 订单摘要。
2. 客户资料和授权状态。
3. 商品和报价快照。
4. 审核信息和资方分配。
5. 合同、公证、授权待办。
6. 账单和支付流水。
7. 分账和钱包流水。
8. 发货、设备、签收、归还。
9. 客诉和租后。
10. 操作日志和回调日志。

高风险按钮必须有明显二次确认样式：审核通过、拒绝、改价、关闭、退款、冲正、打款、加黑名单、锁机/解锁。

## 6. 办单助手出图要求

办单助手必须是三个入口：

1. 门店订单。
2. 分红订单。
3. 平台订单。

共通流程：

```text
选择订单类型
  -> 选择商品/规格
  -> 选择长租租期/套餐
  -> 填写或确认设备价
  -> 填首期、押金、服务费、公证费、增值服务
  -> 生成账单预览和留购价
  -> 生成二维码
```

当前第一版办单助手只出长租界面，不展示短租入口、小时租、天租、周租。

分红订单额外展示：

- 配资比例。
- 门店出资额。
- 资方出资额。
- 后台每期分账预览。

分红额外字段只展示在商家老板后台、商家 PC 端和运营后台；客户侧账单、报价图、扫码下单页和合同展示文案不得出现分红、分账、收益、配资比例、门店出资额、资方出资额。

短租不在第一批长租出图中，但进入独立体验租文档。后续单独出体验租模块/入口 UI 图，并独立处理计费、库存、设备码、预约、交付验收和归还复核。

## 7. 移动端出图要求

门店手机端：

- 首页要突出三类办单入口、待办、订单，不展示短租设备管理入口。
- 店员账号不能显示钱包金额。
- 交付材料上传要适合手机拍照。
- 发货待办要能处理发货方式选择、设备识别码填写、交付照片、签收异常。

C 端：

- 底部导航必须展示：首页、安心用、体验租、订单、我的。
- 首页可以综合展示安心用和体验租入口，但下单链路必须进入对应业务线。
- 客户订单详情要展示待办，不要让客户找不到合同、公证、支付、签收。
- 每个待办失败要有重试或联系客服。
- 客户侧不展示内部黑名单、资方分配、平台抽佣等内部信息。
- 体验租第一批只出入口、门店/车型选择、短租方案选择、订单详情状态，不把短租入口加到长租办单助手。

---

## 8. 出图验收标准

1. 不是流程图，必须是完整界面图。
2. 页面字段必须来自 PRD，不允许凭空编字段。
3. 每个页面必须有主状态、空状态、异常状态。
4. 钱相关页面必须展示来源单据和流水关系。
5. 权限差异必须体现：平台、商家老板、店员、渠道、客户看到的内容不同。
6. UI 必须能支撑日常高频操作，不要只做展示页。
7. 所有敏感数据在图里必须脱敏或使用占位。
