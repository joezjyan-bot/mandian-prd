> P0 业务文档(2026-05-27)。
> 长租订单逾期费用的核心机制:**独立账单**(与主账单分离)、**规则可配**(分商户自配 / 平台统配)、**手动减免**(客服协商场景)、**提前结清前必清**(最终结清节点)。

## 1. 核心机制概览(v1.1 字眼调整)

| 机制 | 说明 |
|---|---|
| **支付独立** | 逾期费用 = 独立账单,不付逾期费用也可以支付主账单(两条线并行)|
| **规则可配** | 后台配置逾期费用规则,支持两种算法(固定金额/天 OR 按逾期金额比例/天)|
| **配置粒度** | 商家订单 → 商家后台自配;联营 + 平台订单 → 平台后台统配 |
| **手动修改** | 运营可在后台手动调整客户的逾期费用 |
| **手动减免** | 逾期费用生成后,运营可减免(协商场景,留痕)|
| **客户 C 端展示** | 看到原始金额 + "已减免 ¥XX(运营调整)" + 实付金额 |
| **支付时机** | 平时累计;**提前结清或归还前必须清** |

## 2. 逾期费用生命周期

```
主账单某期逾期(到期日 + 1 天)
   ↓
系统定时扫描(每日凌晨 1 点)
   ↓
按订单规则计算逾期费用,生成逾期费用账单
   ↓
每日累计 +1 天,重算金额
   ↓
客户可选择:
   ├─ 支付逾期费用账单(完全自愿,平时不强制)
   └─ 暂不支付(继续累计;但提前结清或归还时必须清)
   ↓
运营可干预:
   ├─ 修改逾期费用(改原始金额)
   ├─ 手动减免(扣除一部分,实付金额降低)
   └─ 设置规则上限(单日封顶 / 总额封顶)
   ↓
客户支付主账单后(消除逾期)
   ↓
逾期费用停止累计,转为"待支付"状态
   ↓
提前结清 / 归还触发"必须结清"
   ↓
所有逾期费用账单状态变为已支付(或主管全额减免清零)
```

## 3. 数据模型(字段保留 penalty,UI 展示为"逾期费用")

### 3.1 逾期费用账单表

```sql
order_penalty
- penalty_id          bigint PK
- order_id            bigint FK
- source_period       int     -- 来源期数(第几期逾期)
- period_amount       decimal -- 当期金额(快照)
- start_date          date    -- 逾期起始日(到期日 + 1)
- end_date            date    -- 累计截止日(若已停止累计)
- rule_type           enum    -- fixed(固定金额/天)/ percent(按比例/天)
- rule_value          decimal -- rule_type=fixed:元/天;rule_type=percent:百分比/天
- rule_min_per_day    decimal -- 每日最低收(可选;比例计算时可能太少时兜底)
- rule_max_total      decimal -- 总额上限(可选)
- accumulated_days    int     -- 累计天数
- original_amount     decimal -- 系统算出的原始金额
- reduced_amount      decimal -- 累计减免金额
- final_amount        decimal -- 应付金额 = original - reduced
- paid_amount         decimal -- 已付金额
- status              enum    -- pending / partial_paid / paid / waived / closed
- rule_snapshot       json    -- 规则快照(留痕,后续改规则不影响)
- last_calc_at        datetime -- 上次计算时间
- created_at / updated_at

-- 索引建议:(order_id, status) / (status, last_calc_at)
```

### 3.2 逾期费用减免日志

```sql
order_penalty_reduction_log
- log_id              bigint PK
- penalty_id          bigint FK
- order_id            bigint FK
- reduction_amount    decimal  -- 本次减免金额
- reduction_reason    text     -- 减免原因(必填)
- reduced_by          bigint   -- 操作客服 ID
- reduced_by_role     varchar  -- 角色名(留痕)
- client_ip           varchar(45) -- 操作 IP
- created_at          datetime

-- 索引建议:(penalty_id, created_at) / (reduced_by, created_at)
```

### 3.3 逾期费用规则配置

```sql
penalty_rule_config
- rule_id             bigint PK
- owner_type          enum    -- platform(平台统配)/ merchant(商家自配)
- owner_id            bigint  -- platform 时为 NULL,merchant 时为商家 ID
- rule_name           varchar -- 规则名称
- rule_type           enum    -- fixed / percent
- rule_value          decimal -- 固定值 或 百分比
- rule_min_per_day    decimal -- 每日最低
- rule_max_total      decimal -- 总额上限
- applicable_orders   enum    -- self_operate(商家订单)/ joint_venture(联营订单)/ receivables_assignment(平台订单)/ all
- status              enum    -- active / inactive
- effective_from      date    -- 生效日期
- created_by / created_at / updated_at

-- 索引建议:(owner_type, owner_id, applicable_orders, status)
```

**重要原则**:订单生成时,逾期费用规则**快照写入** `order_penalty.rule_snapshot`,**后续平台修改规则不影响已生效订单**。

## 4. 逾期费用计算

### 4.1 触发时机

- **定时任务**(每日凌晨 1 点)扫描所有"履约中"订单
- 检查每一期是否逾期
- 已逾期但未生成逾期费用账单 → 新建
- 已生成逾期费用账单且未结清 → 累计天数 +1,重算金额

### 4.2 计算公式

```
# 按规则类型分支
if rule_type == 'fixed':
    daily_amount = rule_value           # 例:¥10/天
else (rule_type == 'percent'):
    daily_amount = period_amount × rule_value%   # 例:¥568.75 × 0.05% = ¥0.28/天

# 单日封顶(可选)
if rule_min_per_day and daily_amount < rule_min_per_day:
    daily_amount = rule_min_per_day

# 累计计算
original_amount = daily_amount × accumulated_days

# 总额封顶(可选)
if rule_max_total and original_amount > rule_max_total:
    original_amount = rule_max_total

# 实付金额
final_amount = original_amount - reduced_amount
```

### 4.3 例子

**场景**:客户第 3 期 ¥568.75 逾期 5 天,规则为"按比例 0.05%/天"

| 日期 | 累计天数 | 日金额 | 原始金额 | 减免 | 实付 |
|---|---|---|---|---|---|
| Day 1 | 1 | ¥0.28 | ¥0.28 | ¥0 | ¥0.28 |
| Day 5 | 5 | ¥0.28 | ¥1.42 | ¥0 | ¥1.42 |

**场景**:同一情况,但规则为"固定 ¥10/天"

| 日期 | 累计天数 | 日金额 | 原始金额 | 减免 | 实付 |
|---|---|---|---|---|---|
| Day 1 | 1 | ¥10.00 | ¥10.00 | ¥0 | ¥10.00 |
| Day 5 | 5 | ¥10.00 | ¥50.00 | ¥20(运营手动减免) | ¥30.00 |

## 5. 逾期费用停止累计的触发

| 触发条件 | 处理 |
|---|---|
| 客户支付了对应主账单期 | 逾期费用停止累计,`end_date = today`;状态保持"待支付"或"已支付" |
| 客户支付逾期费用(完全清)| 状态变为 `paid`,但若主账单期仍逾期,继续累计新的逾期费用 |
| 主管全额减免(waived)| 状态变为 `waived`(等同已结清)|
| 订单撤单 / 关闭 | 状态变为 `closed`,逾期费用一并核销 |
| 提前结清前结清 | 客户必须支付所有逾期费用;状态变为 `paid` |
| 归还前结清 | 同上 |

## 9. 提前结清 / 归还前的强制结清

### 9.1 提前结清触发(按内部业务话术模板差异化按钮)

客户在 C 端任意一期点 [申请留购] 或 [提前结清](按订单的话术模板)→ 系统检查:

```
检查:
  ├─ 当前期是否已付清? → 未付 提示先付
  └─ 是否有未结清逾期费用?
       ├─ 否 → 弹出提前结清明细 → 继续
       └─ 是 → 弹出"请先结清逾期费用"提示
                ├─ 未结清逾期费用合计: ¥XX
                ├─ [一并结清后提前结清] → 提前结清金额 += 逾期费用合计
                └─ [先支付逾期费用] → 跳转逾期费用支付页
```

### 9.2 归还触发

客户在 C 端点 [申请归还] → 系统同样检查逾期费用是否结清。

```
归还检查:
  ├─ 是否有未结清逾期费用?
  │    ├─ 否 → 正常归还流程
  │    └─ 是 → 必须结清逾期费用才能完成归还
  │            ├─ [立即支付逾期费用 ¥XX] → 支付完成 → 归还
  │            └─ 拒绝 → 归还流程终止
  └─ ...
```

### 9.3 例外:运营主管全额减免

如果客户与平台协商,运营主管可全额减免该订单的所有逾期费用(批量操作)→ 状态变为 `waived` → 提前结清 / 归还不再检查。

### 9.4 违约赔偿规则边界

> 本节为合规审查整改后的老板确认口径,涉及违约赔偿、设备价值损失和申请购买定性。

1. 客户逾期但设备可归还时,系统只计算:
   - 已到期未付费用;
   - 已生成且未结清的逾期费用;
   - 平台、商家或第三方实际发生且可举证的取回费用、检测费用、维修费用、物流费用等实际损失。
2. 违约不自动触发全部未到期费用加速到期,禁止配置“全部未到期费用自动一次性到期”的默认规则。
3. 客户转卖、灭失、拒不返还或客观无法返还设备时,赔偿基准按:
   - 设备价值损失 = `device_value_confirmation.confirmed_value_cent` 对应的确认价;
   - 加上已到期欠费、已生成逾期费用和可举证实际损失;
   - 不按剩余全部未到期费用直接作为赔偿金额。
4. 资方渠道如存在特殊回购、债权转让、内部结算或资方清算规则,必须单独配置和单独展示在运营端内部,不得与普通渠道标准租赁赔偿口径混判,也不得在 C 端暴露资方身份。
5. 本节赔偿口径进入合同条款、客服话术、法诉证据包和财务结算时,按本老板确认口径执行。

## 11. 异常场景

| 场景 | 处理 |
|---|---|
| 客户支付逾期费用但主账单未付 | 逾期费用标记 `paid`,但主账单期仍逾期 → 第二天继续生成新逾期费用 |
| 减免金额超过原始金额 | 系统阻止,`reduced_amount` 不可超过 `original_amount` |
| 规则修改与已生成账单冲突 | 已生成订单按 `rule_snapshot` 计算,不影响 |
| 累计天数过长(>180 天) | 进入运营主管预警,可能需要法律介入 |
| 多次减免后又恢复 | 不支持"撤销减免",只能通过 [修改逾期费用] 调整 |
| 主管批量减免误操作 | 进入异常队列,需财务复核 |
