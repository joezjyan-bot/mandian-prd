> **落地报告**:`docs/V0.2.2_SUPPLEMENT_REQUIREMENTS_REPORT.md` 记录三项需求并入主文档、OpenAPI 和开发设计的落地位置与待配置项。

> **性质**:对现有 V0.2.2 的功能补充,不是修 bug。本文给出三项新需求的完整设计,并标明各自并入/衔接的现有文档。Codex 落地时按本文新增,并在对应主文档补引用。

## 需求一:人设备合照 AI 自动核验(辅助人工,非全自动)

### 1.1 背景与痛点

当前发货/当面交付环节要上传"人设备合照(人机合照/人车合照)",但存在造假(拿别人/网图冒充)。目前纯人工核验,效率低且易漏判。目标:上传后系统自动比对,辅助人工判断合照里的人是否就是下单本人。

### 1.2 核心设计原则(必须遵守)

1. **AI 辅助 + 人工兜底,不是全自动放行。** 系统输出比对结论与置信度评分,人工据此决策;不允许"自动一致就直接放行/自动不一致就直接拒绝"。原因:门店现场合照角度/光线/距离不可控,人脸可能小、侧、糊,会有相当比例"无法判定",全自动会误伤真客户也会放过造假。
2. **接第三方持牌人脸比对 API**(如旷视/腾讯云/阿里云人脸核身类),不自建模型。
3. **隐私合规红线(个人信息保护法 / 人脸识别司法解释)**:
   - 比对通过调用第三方 API 完成,系统**只存比对结果 + 置信度评分 + 调用流水号**,**不自建可随时检索的人脸库**做留存再比对。
   - 人脸/证件影像的使用范围、存储期限、授权,必须在客户信用/信息授权环节已取得授权(见 C端/05 信用信息授权)。
   - C 端不展示"人脸比对""风控"等字眼;此功能为后台核验能力,客户侧只感知"资料审核中"。

### 1.3 比对的三组数据来源

| 比对对 | 数据来源 | 置信度 |
|---|---|---|
| A. 身份证人像 ↔ 下单实名活体人脸 | 实名认证环节(C端/04)已有的 1:1 活体核身结果 | 高(成熟) |
| B. 人设备合照中的人脸 ↔ 下单实名人脸 | 新上传的人机/人车合照 + 实名环节人脸 | 中(合照质量不可控) |
| C.(可选)人设备合照中的人脸 ↔ 身份证人像 | 合照 + 身份证人像 | 中 |

> A 在实名环节已完成,本需求重点是 B(和可选 C):把"当面交付合照里的人"和"下单本人"做比对。

### 1.4 流程

```text
门店/商家在 交付证据(订单管理/02)上传 人机合照/人车合照
   ↓
系统提取合照中人脸 → 调用第三方人脸比对 API
   ↓ 与"下单实名人脸"(及可选身份证人像)比对
返回 相似度分值 + 结论
   ↓
系统按阈值给出三档建议(阈值后台可配):
   ├ 高度匹配(≥ 高阈值)  → 标记"AI:高度匹配",可走快速通过(仍需人工最终确认按钮)
   ├ 疑似不符(< 低阈值)  → 标红"AI:疑似不符",强制人工重点核验,不可快速通过
   └ 无法判定(中间档/人脸提取失败) → 标"AI:无法判定",转人工核验
   ↓
人工在核验界面看到:合照、实名人脸、身份证人像、AI分值与结论 → 人工做最终判定(通过/驳回/要求重传)
   ↓
判定结果 + AI分值 + API调用流水号 写入订单证据链与操作日志
```

### 1.5 字段(并入 订单管理/02 交付签收照片证据)

在 §4 字段设计追加:

| 字段 | 类型 | 说明 |
|---|---|---|
| `face_match_score` | decimal | 第三方返回的相似度分值(0-100 或 0-1,按 API) |
| `face_match_result` | enum | high_match / suspect_mismatch / undetermined |
| `face_match_provider` | varchar | 第三方供应商标识 |
| `face_match_trace_no` | varchar | 第三方调用流水号(审计/复查用) |
| `face_match_checked_at` | datetime | 比对时间 |
| `manual_verify_result` | enum | pending / passed / rejected / need_reupload |
| `manual_verify_by` | bigint | 人工核验人 |
| `manual_verify_at` | datetime | 人工核验时间 |

### 1.6 配置项(后台)

| 配置 | 默认 | 位置 |
|---|---|---|
| 高度匹配阈值 | 待运营/算法定(如 ≥80) | 运营 > 风控/核验配置 |
| 疑似不符阈值 | 待定(如 <60) | 同上 |
| 是否启用合照 AI 比对 | 按类目/订单类型可配 | 同上 |
| 人脸提取失败的处理 | 转人工 | 同上 |

### 1.7 验收要点

1. 上传合照后能自动调用第三方比对并回写分值/结论。
2. 三档建议正确驱动:疑似不符不可快速通过、无法判定转人工。
3. 人工始终有最终决定权,人工判定与 AI 分值都留痕。
4. 系统不留可检索人脸库,只留结果+流水号;C 端无人脸/风控字眼。
5. 比对失败(API 超时/错误)降级为人工核验,不阻断交付流程。

## 需求二:客服上下班 + 自动轮询派单

### 2.1 目标

客户下单后,系统自动把会话/接单轮询分配给"已上班"的在线客服;客服上班需点击"上班";单子派给谁,3 分钟内只能谁操作,超时未接转下一个。

### 2.2 客服在线状态(新增)

| 状态 | 含义 | 能否被派单 |
|---|---|---|
| `off_duty` | 未上班(默认) | 否 |
| `on_duty_idle` | 已上班-空闲 | 是 |
| `on_duty_busy` | 已上班-忙(达到并发上限) | 否(满载) |
| `on_break` | 小休 | 否 |
| `off_duty_manual` | 下班 | 否 |

- 客服进工作台点【上班】→ `on_duty_idle`;点【下班】/【小休】退出派单池。
- 并发上限(同时处理会话数)后台可配;达上限自动 `on_duty_busy`,不再派新单。

### 2.3 轮询派单算法(核心)

```text
客户下单 / 发起会话 → 生成待派任务(queued)
   ↓
系统从"on_duty_idle 且未满载"的客服中,按轮询(round-robin)选一人
   ↓
派给客服 A → 任务状态 assigned_A,启动 3 分钟接单倒计时
   ↓ 在 3 分钟内:
   ├ A 点击【接单】 → active,锁定归属 A(仅 A 可操作,见 2.4)
   └ A 未接(3 分钟超时)→ 自动转派下一个 on_duty_idle 客服 B,重启 3 分钟倒计时
        (A 记一次"未接",可计入考核;可配置连续未接自动转小休)
   ↓
若一轮无人接(所有在线客服都已轮过且未接)→ 进入"无人接单"兜底:
   提醒主管 + 任务保留在队列顶部继续轮询 + 可配置升级策略
```

- 轮询超时时长默认 3 分钟,后台可配。
- 派单顺序 round-robin(轮流),保证公平;可后续扩展按技能/业务线分池。

### 2.4 单人锁定(派给谁只能谁操作)

1. 任务一旦 `assigned_X` 或 `active`(归属 X),**只有 X 能接单、回复、转接、转工单、关闭**。
2. 其他客服在会话列表能看到该单(只读),但操作按钮置灰,不能抢。
3. 例外:主管可强制改派(retransfer),改派写日志(原因、原归属、新归属、操作人)。
4. X 下班/掉线超时未处理 → 系统自动回收任务重新轮询派单,并解除 X 的锁定。

### 2.5 字段(并入 客服中心/01)

```text
customer_service_agent
- agent_id / name
- duty_status     enum   -- off_duty/on_duty_idle/on_duty_busy/on_break/off_duty_manual
- max_concurrent  int    -- 并发上限(后台可配)
- current_load    int    -- 当前处理中会话数
- on_duty_at / off_duty_at  datetime
- last_assigned_at datetime  -- 轮询排序用

im_session(在现有基础上补)
- assigned_agent_id   bigint   -- 当前归属客服(锁定)
- assign_status       enum     -- queued/assigned/active/...(扩展 2.2 会话状态)
- assign_deadline     datetime -- 当前接单倒计时截止(默认 +3 分钟)
- miss_count          int      -- 本任务被转派次数
- assign_log          json     -- 历次派给谁、是否接、超时转派记录
```

### 2.6 配置项

| 配置 | 默认 | 位置 |
|---|---|---|
| 接单超时转派时长 | 3 分钟 | 运营 > 客服配置 |
| 客服并发上限 | 待定(如 5) | 同上 |
| 连续未接自动小休阈值 | 待定(如 3 次) | 同上 |
| 无人接单升级策略 | 提醒主管 | 同上 |

### 2.7 验收要点

1. 客服不点【上班】不进派单池,收不到单。
2. 下单后自动 round-robin 派给在线空闲客服,3 分钟未接自动转下一个。
3. 派给谁只能谁操作,其他人只读不可抢;主管可强制改派并留痕。
4. 客服下班/掉线,未接任务自动回收重派。
5. 派单全过程(派给谁/是否接/超时/改派)进操作日志。

### 2.8 衔接

- 主落点:`运营端/客服中心/01_IM客服页面与接口规则.md`(§2 会话状态扩展、新增 §派单算法、§4 接口补"接单/转派/上下班"接口)。
- 总览:`客服中心/00_IM客服体系总览.md` 同步描述上下班与派单。

## 需求三:门店静默率 / 逾期率展示 + 停单 / 关低首付操作钩子

### 3.1 目标

在 **运营端店铺详情(商家管理)** 给每个门店展示两个风险指标,供运营定期检查并据此对该门店操作(停单 / 关闭低首付等):
- **静默率**:已上锁且履约中的客户里,监管锁超过 24 小时未联网(静默)的占比。
- **逾期率**:该门店提供的客户的逾期占比。

下单时也在运营后台界面给客服一个参考(展示该门店的这两个数据)。

### 3.2 指标定义(需开发与运营对齐口径)

| 指标 | 默认口径 | 备注 |
|---|---|---|
| 静默率 | 静默设备数 ÷ (已上锁且履约中的设备数) | 静默 = 监管锁超过 24 小时未联网(阈值后台可配) |
| 逾期率 | 逾期订单数 ÷ 该门店有效订单数 | "逾期"口径对齐 逾期管理/状态字典;分母口径(履约中/全部)需运营确认 |

> 静默判定依赖监管锁联网心跳数据(见 监管锁管理/01)。24 小时阈值后台可配。
> 两个比率的精确分子分母口径,开发前与运营定死并写入配置;本文给默认口径,标【需运营确认分母】。

### 3.3 展示位置(两处)

1. **运营端店铺详情(商家管理)** —— 主位置。在 §4 概览页签新增"门店风险数据"区块:
   ```
   门店数据(运营参考)
     静默率:xx.x%   (已上锁履约中客户中,锁超24h未联网占比)
     逾期率:xx.x%   (该门店客户逾期占比)
     更新时间:yyyy-mm-dd HH:MM
   ```
2. **客户下单后的运营后台订单/审核界面** —— 给客服参考:展示该订单所属门店的 静默率 / 逾期率(只读参考,不阻断)。

> C 端不展示这两个指标(内部风控参考)。

### 3.4 操作钩子(停单 / 关低首付)——在店铺详情订单能力处

在 §7 订单能力配置追加,由运营人员定期检查后手动操作(非自动):

| 操作 | 效果 | 权限 |
|---|---|---|
| 停单(暂停新增订单) | 该门店不能再办新单,历史订单继续履约 | 运营主管 |
| 关闭低首付 | 该门店办单助手禁用低首付成数(如 0.2/0.3),只留高首付 | 运营主管 |
| 恢复 | 解除上述限制 | 运营主管 |

- 这些操作与 §3.1 的静默率/逾期率联动:运营据数据手动决策,**V1 不做自动停单**(避免误伤),但预留"指标超阈值高亮告警 + 建议"。
- "关闭低首付"落到办单助手配置(门店手机端/办单助手 §3 首付成数配置):被关闭的成数在该门店办单助手不展示。
- 所有停单/关低首付/恢复操作写操作日志(原因、操作人、时间、关联指标快照)。

### 3.5 字段

```text
merchant_risk_metrics(按门店,定时计算)
- merchant_id / store_id
- silent_rate         decimal  -- 静默率
- overdue_rate        decimal  -- 逾期率
- locked_active_count int      -- 已上锁履约中设备数(静默率分母)
- silent_count        int      -- 静默设备数(分子)
- overdue_count / total_count int
- computed_at         datetime

merchant_order_control(门店管控状态)
- merchant_id / store_id
- order_suspended     bool     -- 是否停单
- low_downpay_disabled bool    -- 是否关闭低首付
- disabled_ratios     json     -- 被关闭的首付成数列表
- operated_by / operated_at / reason
```

### 3.6 配置项

| 配置 | 默认 | 位置 |
|---|---|---|
| 静默判定时长 | 24 小时未联网 | 运营 > 风控配置 |
| 指标计算频率 | 待定(如每日/每小时) | 运营 > 任务调度 |
| 静默率/逾期率告警阈值 | 待定(超阈值高亮) | 运营 > 风控配置 |
| 低首付定义(哪些成数算低) | 0.2 / 0.3 | 运营 > 风控配置 |

### 3.7 验收要点

1. 店铺详情能看到该门店静默率、逾期率及更新时间。
2. 下单后运营订单/审核界面能看到该订单门店的两个指标(只读参考)。
3. 运营可对单个门店停单 / 关低首付 / 恢复,操作即时生效于该门店办单助手与下单能力。
4. 关闭低首付后,该门店办单助手不展示被关成数;停单后不能新增订单、历史订单不受影响。
5. 所有管控操作留痕(含操作时的指标快照)。
6. 静默率依赖监管锁联网心跳;无心跳数据时指标标"数据不足"而非报 0。

## 落地优先级与提示

| 需求 | 复杂度 | 依赖 | 建议 |
|---|---|---|---|
| 一 人脸比对 | 中 | 第三方人脸 API + 授权合规 | AI 辅助+人工兜底,别做全自动;隐私口径必须守 |
| 二 轮询派单 | 中 | 现有 IM 会话机制 | 状态机扩展 + 锁定逻辑,清晰可做 |
| 三 静默/逾期率 | 中 | 监管锁心跳 + 逾期口径 | 指标口径(分母)开发前与运营定死;V1 手动操作不自动停单 |

三项均为 V0.2.2 之后的功能补充,不阻塞当前主线开发,可作为 V0.2.3 / V1.0 增量纳入排期。各项中标【需运营/算法确认】的口径(人脸阈值、指标分母、各频率)由运营定后写入配置,开发先留结构与 TODO,不编造数值。
