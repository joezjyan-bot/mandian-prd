## 一、产品总览

满点租赁系统是一套面向设备租赁业务的综合运营平台,覆盖客户下单、商家办单、平台审核、合同签署、支付、交付签收、账单履约、归还续租、申请购买、短租取还车、IM 客服、财务结算、法诉证据和经营数据分析。

系统服务三类主要业务:

- **安心用长租**:面向手机、电动车、数码设备等长期租赁场景。客户按订单周期支付每期应付,到期可选择归还设备、继续使用或申请购买。
- **体验租短租**:面向小时租、日租、周租、月租等短期租赁场景。客户按城市、门店、车型和租期下单,商家接单、核销绑车、取还车验收。
- **IM 客服基础能力**:客户、商家和平台围绕订单、账单、支付、保证金、交付、归还、售后争议进行沟通,会话沉淀为证据链。

系统建设目标:

1. 把原本散落在商家、客服、财务、风控、法务之间的流程系统化。
2. 让订单、账单、支付、钱包、总账、证据链互相可追溯。
3. 让商家订单、联营订单、平台订单三种业务模式都能共用一套底层能力。
4. 让客户侧保持简洁、合规、可理解,不暴露内部资金、利润、返点、审核等敏感信息。
5. 让技术团队能基于冻结版 PRD 进入接口、数据库、前端、测试拆分。

详见:`PRD/V0.2/00_V0.2.2_开发冻结版总PRD.md`、`docs/V0.2.2_FINAL_DOC_MAP.md`

## 二、用户与角色

### 2.1 C 端客户

客户通过小程序/H5 完成以下动作:

- 浏览安心用和体验租入口。
- 扫码下单或自主选品下单。
- 填写资料、实名认证、授权信用/信息查询。
- 确认商品、设备价值、最低服务期、协议和费用。
- 签署合同、公证或确认单。
- 支付首期、保证金、短租租金或后续账单。
- 完成交付签收、短租取还车、续租、归还、申请购买。
- 通过 IM 客服咨询和处理异常。

客户侧只展示客户应理解和确认的内容,不展示资方信息、资金来源、合作模式、风控结论、内部审核备注、商家结算价、平台利润、返点规则、黑灰名单结论。

### 2.2 商家和门店

商家是平台合作经营主体,门店是具体办单和履约网点。商家/门店主要动作:

- 入驻认证、门店资料维护、员工权限管理。
- 配置或使用授权范围内的办单方案。
- 商家订单自主审核或处理平台分配的履约待办。
- 长租设备交付、上传交付证据、发货或自提处理。
- 短租接单、核销绑车、取车确认、还车验收。
- 查看结算、钱包、提现和对账。
- 联系平台客服或财务客服。

商家可以给财务人员授权查看钱包和发起提现,但不能让财务人员修改结算账户、收款账户、主体证件、门店归属或员工权限。

### 2.3 平台运营、客服、财务、风控、法务、技术

平台内部角色按权限矩阵拆分:

- **运营**:商品、商家、订单、配置、渠道、数据、前台配置。
- **客服**:客户跟进、补资料、账单生成前改价、退款申请、异常沟通、IM 会话。
- **财务**:支付对账、结算、提现、退款、冲正、总账、报表。
- **风控**:审核、复审、黑名单、门店风险指标、监管锁异常。
- **法务**:合同模板、法诉证据包、导出审批、争议材料。
- **技术**:接口、数据、补偿任务、供应商联调、日志排查,但不得绕过业务审批直接改资金或订单结果。

详见:`PRD/V0.2/modules/运营端/权限日志/02_权限矩阵.md`

## 四、三类订单业务模式

### 4.1 商家订单

商家订单指商家自有客户、自行经营、自行承担主要交易责任的订单。平台提供系统、支付、合同、审核、客服、财务和证据链工具。客户付款后按规则扣除平台服务费,剩余进入商家钱包或结算账户。

关键规则:

- 商家可按权限配置自己的办单助手方案。
- 商家可自主审核商家订单,也可设置全部通过等审核策略。
- 商家订单退款责任由商家自行承担。
- 商家端可见自己的订单、履约、钱包和结算,但不能看到平台内部利润和其他商家数据。

### 4.2 联营订单

联营订单指平台和商家共同经营的订单。平台参与获客、审核、规则、资金或结算安排;商家参与履约、交付或服务。退款、结算、返点和责任按协议及线下协商方案执行,系统必须完整记录审批、凭证、流水和冲正。

关键规则:

- 联营订单的办单方案由运营端配置。
- 商家端可看履约视图,不得操作审核、改价、资金主控等平台主控能力。
- C 端不展示“联营”背后的内部合作关系。

### 4.3 平台订单

平台订单指平台主导获客、审核、规则和运营管理的订单。平台负责订单主控、审核、合同、结算、售后和证据链。商家如参与履约,只按权限处理交付、签收、短租取还车等现场动作。

关键规则:

- 平台订单的办单方案由运营端配置。
- 商家端只看本门店履约相关信息。
- 平台服务费、平台利润、内部结算规则不展示给 C 端。

详见:`PRD/V0.2/modules/全局/10_全局措辞规范.md`、`PRD/V0.2/modules/运营端/财务管理/12_订单合作模式与收益分配规则.md`

## 五、安心用长租完整链路

### 5.1 长租主流程

```text
客户扫码/选品
→ 订单确认
→ 资料提交
→ 实名与信用/信息授权
→ 平台/商家审核
→ 合同签署
→ 首期与保证金支付
→ 商家交付设备
→ 客户签收确认
→ 监管锁或替代验收条件校验
→ 平台结算
→ 履约中账单支付
→ 到期三选一
→ 正常结清 / 提前归还 / 申请购买 / 逾期 / 法诉 / 坏账回收
```

### 5.2 下单和订单确认

客户在 C 端看到商品、规格、设备价值、保证金、首期应付、每期应付、订单周期、最低服务期、增值服务和协议勾选。客户必须确认:

- 设备型号、成色和价值无误。
- 安心用最低服务期统一 3 个月起步。
- 已阅读协议、隐私、授权和必要告知。
- 到期可选择归还设备、继续使用或申请购买。

订单确认页要形成快照,记录客户看到的商品、费用、协议版本、页面版本、时间、IP、设备信息和确认动作。

### 5.3 审核和补资料

客户提交资料后进入审核。审核可能产生:

- 审核通过,进入签约或支付。
- 审核驳回,展示客户可理解原因。
- 待补充资料,客户补齐后重新提交。
- 人工复核,由风控或客服处理。

商家订单可由商家自主审核;联营订单和平台订单由运营端主控。审核结果、内部规则命中、黑灰名单结论不得展示给 C 端。

### 5.4 合同、公证和签约

合同模板按业务线、品类、商家、订单类型和内部规则选择。客户在 C 端签署合同,平台保存合同 PDF、副本、签署回调和页面快照。公证可按配置发起。

合同和系统必须保持一致:

- 合同中的设备信息、费用、最低服务期、归还/续租/申请购买规则要与订单快照一致。
- 合同不应把普通租赁订单写成默认购买或默认所有权转移。
- 客户主动申请购买时需要独立确认和支付。

### 5.5 首期支付和支付失败

首期支付包括保证金、首期租金、增值服务等配置项。支付成功后进入交付;支付失败保留支付单、失败码、重试入口和客服入口。支付回调必须幂等,重复通知不得重复入账。

### 5.6 交付和签收

商家选择自提或物流等发货方式,填写设备识别码,上传交付照片、人机/人车合照、配件、物流或自提凭证。客户通过电子确认单或 AI 问答确认签收。

签收方式由后台配置:

- 默认电子确认单。
- 可手动发起 AI 问答确认。
- AI 问答问题可配置,用于确认本人办理、设备、期数、每期应付、是否知晓协议等。

人设备合照 AI 核验只存比对结果、评分和流水号,不建立可检索人脸库;C 端不出现“人脸/风控”类内部字眼。

### 5.7 监管锁和结算

监管锁适用于苹果类手机、平板、手表产品。此类订单结算前必须满足:

1. 客户签收确认通过。
2. 监管锁显示已上锁。
3. 激活锁状态已激活。

其它无监管锁品类不因没有监管锁本身阻断业务,按签收确认、设备识别码归档、交付证据完整、前置检测完成进行结算。财务打款按钮需展示未满足原因,异常放行必须审批和审计。

### 5.8 履约、账单和逾期

履约中订单按账单计划生成每期应付。客户可通过 C 端支付账单。账单生成前可按权限改价并重新生成计划;账单生成后任何人不得直接改账单,只能走补收、减免、退款、冲正、坏账核销或回收完成。

逾期费用当前口径为 0.05%/日、最低 10 元/日。商家订单逾期费用不确认为平台收入;联营订单和平台订单逾期费用归平台,用于覆盖售后、提醒、催收、客服协同和账务处理成本。

### 5.9 到期三选一

到期前客户同等看到:

- 归还设备。
- 继续使用。
- 申请购买。

系统不得默认勾选申请购买,不得自动转移所有权。客户选择要记录到日志和页面快照。

### 5.10 提前归还和申请购买

客户可在履约中申请提前归还。系统展示已使用时间、已发生费用、预计检测费、物流费、折旧补偿和保证金预计退还,不强制结清全部未到期费用。归还检测凭证提供上传入口,不设置客户线上异议流程。

申请购买价一期采用 A 口径:剩余应付租金 + 保证金。折旧余值口径后期再评估。

### 5.11 法诉、坏账和回收

逾期严重或争议订单可进入外部催收 API 或法诉证据包流程。坏账核销后不删除原账单、支付、催收和法诉记录。后续追回款项时,可回原订单执行“回收完成”,选择完成方式并填写金额、凭证和备注。

详见:`PRD/V0.2/modules/运营端/订单管理/08_长租订单全生命周期与客服操作.md`、`PRD/V0.2/modules/C端/02_订单确认页.md`、`PRD/V0.2/modules/C端/12_在租期间.md`

## 六、体验租短租 V1 完整链路

### 6.1 短租主流程

```text
C 端进入体验租
→ 选择城市 / 品类 / 门店 / 车型 / 租期
→ 确认订单和协议
→ 填写骑行人信息并认证
→ 支付租金和保证金 / 免押
→ 商家接单
→ 到店核销并绑定车辆
→ 取车确认
→ 用车中
→ 续租 / 超时 / 控车异常
→ 还车预约
→ 还车验收
→ 保证金扣款或退还
→ 事故 / 违章 / 损坏 / 丢车处理
```

### 6.2 短租入口和门店选择

体验租与安心用并列,可在首页或底部导航展示。体验租首页展示城市定位、品类导航、取车地点、附近门店、租期类型和活动 Banner。客户可选择电动车等品类,再选择附近有库存的门店和车型。

### 6.3 商品、车辆和库存

短租必须有车辆/设备库存。每台车辆或设备以 VIN、SN、IMEI 或设备号唯一识别。同一设备唯一编码不能同时进入长租交付和短租可租库存。

适配短租的商品必须有统一标准商品名。商品一旦开启体验租并绑定价格方案、车辆库存或产生短租订单,标准商品名不允许直接修改。C 端可单独维护展示标题或营销标题。

### 6.4 商家接单和核销绑车

客户下单后商家接单。支付成功但商家未接单或拒单时,优先自动退款,异常或争议进入客服处理。客户到店后使用核销码或扫码取车,商家绑定唯一车辆码。车辆不一致时禁止取车。

### 6.5 用车中、续租和超时

用车中页面展示剩余时间、到期时间、续租、联系商家、联系客服和控车状态。续租生成父子订单关系。超时按价格方案计算超时费,自动扣款需满足授权、通知和扣款规则。

### 6.6 还车验收和保证金

客户还车后,门店上传验收照片、视频、维修或检测凭证。系统根据验收结果处理保证金退还、冻结、扣款或争议。保证金扣款必须有规则、证据、审批、客户通知和申诉/客服介入记录。

### 6.7 事故、违章、损坏、丢车

短租责任原则已确认:非车辆质量问题导致的事故、违章、损坏、丢车均由承租人承担。系统需要保留现场照片、车辆照片、报警/保险材料、违章通知、取还车照片、维修报价、定位/控车记录、IM 记录和商家验收记录。平台不默认兜底。

### 6.8 短租经营模式

体验租统一使用短租订单,但记录经营模式:

- 商家自营短租:商家车辆、商家接单,客户缴费后扣除平台服务费,剩余进入商家钱包。
- 平台自营短租:平台车辆或平台货源,平台结算。
- 平台货源门店履约:平台出车辆和方案,门店执行取还,门店服务费按协议和后台配置处理。

详见:`PRD/V0.2/modules/短租业务/01_体验租业务总览.md`、`PRD/V0.2/modules/短租业务/05_体验租数据模型与状态机.md`、`PRD/V0.2/modules/短租业务/11_体验租风险责任规则表.md`

## 九、门店手机端与办单助手

### 9.1 门店手机端定位

门店手机端用于现场办单和履约,包括首页待办、办单助手、订单待办、交付材料、短租待办等。它面向门店实际操作人员,必须简单、清晰、少跳转。

### 9.2 办单助手能力

办单助手不是静态计算器,而是后台可配置、全链路联通的报价和下单工具。它支持:

- 选择商品、规格、设备价。
- 选择期数、首付成数、增值服务。
- 根据后台费率表生成首期实付、每期应付、留购价、报价单。
- 生成“请扫码下单”二维码。
- 固化报价快照,客户扫码后读取服务端快照下单。

金额计算全程以分为整数运算,避免浮点误差。配置可变,订单快照不可变。

### 9.3 三方案分权

- 商家订单方案:商家自己配置。
- 联营订单方案:运营端配置。
- 平台订单方案:运营端配置。

修改费率、增值服务、期数、首付成数、报价快照属于高危操作,必须有权限、二次确认、审计和配置版本。

### 9.4 门店风险管控

运营可对门店执行停单或关闭低首付。低首付不设全局固定定义,由运营按门店禁用具体首付成数。被禁用成数在该门店办单助手和 C 端报价不可选。

详见:`PRD/V0.2/modules/门店手机端/办单助手/02_计算器字段与账单公式表.md`、`PRD/V0.2/modules/门店手机端/订单待办/01_门店待办与交付材料.md`

## 十四、财务、钱包、结算与对账

### 14.1 四账分离

系统必须采用四账分离:

| 账类 | 作用 |
|---|---|
| 订单业务账 | 看订单应收、已收、待收、状态、套餐、账单计划 |
| 支付流水账 | 看通道支付、回调、退款、手续费、失败和幂等 |
| 钱包/商家账户账 | 看商家可提现、冻结、提现中、返点、分账到账 |
| 财务总账/分录账 | 看平台收入、应付商家、服务费、冲正、坏账、回收 |

业务看订单,财务看总账,商家看钱包,技术看支付回调。四账互相关联但不能混成一张账。

### 14.2 服务费和分账

服务费当前口径为 2%,可配置。商家长租和商家自营短租客户付款后扣除平台服务费,剩余进入商家钱包。联营订单和平台订单按协议和结算规则处理。客户侧一期不展示平台服务费拆分。

### 14.3 退款和冲正

退款规则:

- 商家订单:商家自行承担。
- 联营订单/平台订单:线下协商退款方案。
- 系统支持指定金额退款。
- 退款必须记录责任方、金额、审批人、凭证、冲正对象和流水。

退款不能只改订单状态,必须同时影响订单业务账、支付流水账、钱包账和总账分录。

### 14.4 提现和回滚

商家提现单笔 500-20000 元、每日最多 3 次、手续费 0.2% + 0.5 元、超 20000 元人工审核。提现失败时释放冻结余额,写提现失败回滚钱包流水和总账分录。

### 14.5 返点和渠道佣金

返点、推广费、渠道佣金必须有推广关系快照、规则版本、订单号和期次,避免重复发放。退款关闭时按比例回滚返点。

### 14.6 坏账和回收

坏账核销需要审批链、原因、金额、订单/账单范围和凭证。核销不删除原账单和支付流水。后期追回款项可回原订单执行“回收完成”,生成回收流水和总账分录。

### 14.7 账龄和 Vintage

账龄分析与 Vintage 报表用于银行/合作方尽调和经营管理。它是只读聚合和导出能力,不产生资金流水,不反向改变订单状态、钱包余额或总账分录。

详见:`PRD/V0.2/modules/运营端/财务管理/06_财务流水模型与对账规则.md`、`PRD/V0.2/modules/运营端/财务管理/08_总账科目分录模板与报表字段.md`、`PRD/V0.2/modules/运营端/财务管理/09_账龄分析与Vintage报表.md`

## 十五、渠道、推广与佣金

渠道体系用于推广人员、门店、渠道码、佣金和数据统计。渠道端可查看推广数据、订单转化、佣金和提现。渠道佣金提现套用门店结算账户同一套限额规则,但渠道佣金账户和商家结算账户是两套账户体系,不能混表。

渠道相关原则:

- 扫渠道码跳转对应店铺或业务入口,订单应记录来源渠道。

- 退款、关闭、冲正需要回滚对应佣金。
- 推广员离职后归属按规则处理,不能影响历史订单证据和佣金流水。

## 十六、风控、监管锁与黑名单

### 16.1 风控审核

风控审核覆盖客户资料、实名、人脸、联系人、信用/信息授权、历史订单、黑名单、门店风险指标、订单金额、设备价值、品类风险、监管锁要求等。审核人员可要求补资料、驳回、通过或转人工复核。

### 16.2 门店风险指标

店铺详情展示静默率、逾期率等风险数据。静默率数据源来自监管锁联网心跳。逾期率分母为该门店所有已进入履约及之后的订单,含已完成、已逾期、已坏账,不含未起租订单。

平台可手动停单或关闭低首付。V1 不做自动停单。

### 16.3 监管锁

监管锁适用于苹果类手机、平板、手表产品。监管锁记录包括上锁、激活、定位、开关机、解锁、异常放行等。修改监管锁状态是高危动作,必须权限、审批、日志和审计。

### 16.4 黑名单和异常预警

黑名单库用于客户、手机号、证件、设备、商家、门店等匹配。黑名单结论不得展示给 C 端。运营和风控可在审核页看到只读风险提示。

详见:`PRD/V0.2/modules/运营端/监管锁管理/01_监管锁配置与订单控制.md`、`PRD/V0.2/modules/运营端/黑名单库/01_黑名单匹配与审核提示.md`、`PRD/V0.2/modules/运营端/店铺管理/02_店铺详情与商户配置.md`

## 十七、权限、日志与安全

### 17.1 权限设计

系统支持自定义角色和操作员绑定角色。所有操作必须先拆成标准权限点,再分配给角色。临时授权必须有有效期、授权人、原因和审计日志。

### 17.2 高危操作

高危操作包括:

- 修改订单金额。
- 修改账单。
- 发起/审核退款。
- 发起打款。
- 审核提现。
- 调整返点。
- 冻结/解冻商家钱包。
- 关闭订单。
- 坏账核销。
- 法诉材料导出。
- 修改监管锁状态。
- 修改结算状态。
- 强制变更订单状态。
- 删除或隐藏附件。
- 修改客户关键资料。
- 修改办单助手费率、期数、首付、增值服务、报价快照。
- 资产处置和修改设备价值。

### 17.3 日志和审计

普通操作写操作日志;高危操作写审计日志。审计日志需要记录操作人、审批人、时间、IP、设备、原因、前后快照、凭证、二次确认和复核情况。

详见:`PRD/V0.2/modules/运营端/权限日志/02_权限矩阵.md`、`PRD/V0.2/modules/全局/11_工程规范基线.md`

## 十九、OpenAPI、数据模型与工程边界

### 19.1 OpenAPI

OpenAPI 是开发契约评审基线,覆盖 C 端、商家端、运营端、财务、风控、合同、支付、退款、钱包、监管锁、IM、设备资产、补充需求等接口。接口必须落实:

- C 端 DTO 白名单和敏感字段禁返。
- 支付、退款、打款、提现、返点等幂等键。
- 错误码、权限、审计、回调、补偿任务。
- 新增接口进入 API 验收清单逐项勾选。

### 19.2 数据模型

数据模型至少覆盖订单、状态流转、账单、账单流水、支付流水、资金流水、总账分录、商家结算、平台服务费、推广关系、返点、钱包、提现、退款、合同、电子签、视频确认、设备价值、风控、操作日志、审计日志、催收、法诉材料、第三方调用、失败补偿任务、设备资产等。

### 19.3 工程边界

PRD 不创建 migration、不改业务代码。技术根据 ERD、OpenAPI 和页面契约拆分数据库、后端、前端、测试、供应商适配和部署任务。任何实现不得改变 V0.2.2 的业务口径、C 端隔离红线、四账分离和权限边界。

详见:`PRD/V0.2/modules/开发设计/17_V0.2.2_OpenAPI开发契约补丁.md`、`PRD/V0.2/modules/开发设计/18_V0.2.2_ERD字段索引与幂等键.md`、`PRD/V0.2/modules/开发设计/20_V0.2.2_API最终契约验收清单.md`
