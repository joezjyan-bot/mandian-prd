## 2. 不可改动的业务口径

1. 订单类型只有三种：门店订单、分红订单、平台订单。
2. 不再使用历史旧称作为正式枚举。
3. 普通门店端不做资金充值入口。
4. 商家 PC 端和门店手机端是同一商家主体的两个使用端。
5. 门店订单由商家/门店自己审核，平台可只读和抽佣。
6. 分红订单由运营审核、分配资方，门店出设备或部分资金，资方补足，按比例分账。
7. 平台订单由运营审核和资方/平台主控，商家只看进度和联系客服。
8. 分红订单平台抽佣按双方份额分别扣除。
9. 渠道佣金只统计分红订单和平台订单，门店订单不计佣金。
10. 默认门店发货，但保留平台、商家、资方、线上物流的链路配置。
11. 商品的全新/二手是规格，不是两个独立商品。
12. 商品、增值服务、租期、长短租、办单助手同步必须配置化。
13. 中控台是租赁系统外部系统，不并入租赁系统。
14. 所有人工、系统、回调操作必须有日志。

## 3. 开发第一阶段任务

第一阶段不要做 UI 发散，先做后端底座：

1. 建立基础目录：controller、service、domain、model、job、worker、policy、event。
2. 建立 Batch 0 到 Batch 3 的核心迁移。
3. 完成登录、角色、数据范围、操作日志、附件中心。
4. 完成商家入驻、资质、收款账户、签章授权、店铺审核。
5. 完成商品、规格、商品审核、商品复制、办单助手商品池。
6. 完成报价快照、二维码、客户扫码创建订单。
7. 完成三类订单基础状态机和审核分流。
8. 导入 `fixtures/seed-data.demo.json` 和 `fixtures/mock-callbacks.demo.json` 作为开发测试数据。

阶段验收：

- 三类订单都能从报价生成二维码，再创建订单。
- 商家入驻审核通过后能登录商家 PC 和门店手机端。
- 店员账号只能办单，不可访问钱包。
- 订单关键动作都有操作日志。

---

## 4. 第二阶段任务

1. 合同、公证、客户待办、回调 worker。
2. 支付、账单、部分支付收款码、支付回调。
3. 发货、设备绑定、交付材料、客户签收、起租。
4. 财务分账、钱包、提现、退款、冲正、对账。
5. 渠道归属、佣金、渠道 H5 基础查询。

阶段验收：

- 支付成功后账单核销、分账和钱包流水自动生成。
- 分红订单按比例分账，并分别扣平台抽佣。
- 发货签收后订单、设备、账单状态一致。
- 重复回调不会重复入账。

---

## 5. 第三阶段任务

1. 租后案件、外部催收同步预留。
2. 支付宝投诉同步、客诉处理。
3. 报表导出、字段权限、下载日志。
4. 操作手册截图版输出。
5. UI 原型按 finance 风格输出。

---

## 6. 禁止事项

1. 不要把页面做出来但不落真实数据。
2. 不要把金额只存在订单表。
3. 不要绕过账单、支付、分账、钱包流水。
4. 不要用前端权限代替后端权限。
5. 不要让回调直接修改核心状态。
6. 不要覆盖历史流水，退款和冲正必须生成反向记录。
7. 不要把三类订单混成一套审核和分账。
8. 不要把中控台功能写进租赁系统。
9. 不要把真实敏感信息写入仓库、示例、日志或截图。

---

## 7. UI 交接口径

UI 后置，但不是自由发挥：

- 默认参考 `finance-zhongtai` 企业中台风格。
- 登录页参考无界租清爽扫码/密码登录布局。
- 订单列表参考无界租高密度信息面板，但按门店订单、分红订单、平台订单拆入口。
- 办单助手参考 `phone-rent` 计算器逻辑和 UI。
- UI 原型必须先覆盖 P0 页面，再补 P1 页面。
- 每张图必须体现真实字段、按钮、状态、筛选和异常提示。
