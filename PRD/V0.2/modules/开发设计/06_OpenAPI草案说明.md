# OpenAPI 草案说明

> 所属：开发设计
> 关联文件：`PRD/V0.2/openapi/core-api.v0.2.yaml`

> **V0.2.2 全量整改补丁(2026-05-28)**: OpenAPI 必须作为开发契约补齐 DTO 白名单、敏感字段隔离、状态枚举、幂等字段、结算三条件和四账分离引用。本文件先定义整改规则;实际 YAML 可分批补全,但不得返回 C 端禁用字段。

> **研发边界声明**: 本文当前是 OpenAPI 契约规则,不是完整接口 YAML。进入编码前,必须把 §0.7 的契约化清单补入 `PRD/V0.2/openapi/core-api.v0.2.yaml` 或等价接口文档;未完成的接口不得作为“已定稿接口”交付开发。

## 0. V0.2.2 OpenAPI 开发契约补丁

### 0.1 DTO 分层

| DTO 层 | 可返回内容 | 禁止返回内容 |
|---|---|---|
| C 端 DTO | 商品、套餐、保证金、每期应付、订单周期、合同、签收、账单支付、履约状态、客服入口、保险/保单展示字段 | 资方信息、资金来源、合作模式、风控结论、内部审核备注、服务费拆分明细、商家结算价、平台利润、返点规则、内部标签、黑名单/灰名单结论 |
| 商家端 DTO | 本商家订单、履约、钱包、提现、短租待办、可见的结算/服务费信息 | 平台利润、内部资金来源明细、其他商家数据 |
| 运营端 DTO | 审核、订单、履约、配置、异常、权限范围内敏感字段 | 超越角色权限的数据 |
| 财务端 DTO | 支付流水、钱包流水、总账分录、退款、提现、结算批次 | 未授权客户完整敏感资料 |
| 风控端 DTO | 审核资料、规则命中、复审记录、黑灰名单 | C 端展示字段不得直接复用 |

### 0.2 必备枚举和展示字段

1. 三类订单必须同时提供内部枚举和展示名,例如 `order_type=merchant/joint/platform`、`order_type_label=商家订单/联营订单/平台订单`。
2. 状态响应必须区分后台状态、C 端展示状态和状态说明,例如 `status`、`status_label`、`customer_status_label`。
3. 账单、退款、钱包、打款、监管锁、法诉、坏账状态必须与 `modules/全局/02_状态字典与订单状态机.md` V0.2.2 同步。
4. 长租“提前留购/提前结清”按钮文案由后端返回 `terminology_template_id` 或展示模板配置,不得硬编码为单一业务模式。

### 0.3 幂等字段

以下写接口必须要求 `X-Idempotency-Key`,并在请求体或服务端幂等表中保存业务幂等维度:

| 场景 | 幂等维度 |
|---|---|
| 支付回调 | `channel + channel_trade_no + callback_event_id + amount` |
| 客户重复支付 | `bill_id + pay_amount + customer_id + request_key` |
| 打款/分账 | `settlement_batch_id + recipient_account + amount + request_key` |
| 退款 | `refund_id + amount + channel + request_key` |
| 返点 | `order_id + promoter_id + rule_version + period_no` |
| 提现 | `withdrawal_id + wallet_id + amount + request_key` |
| 监管锁指令 | `order_id + device_id + lock_action + request_key` |

### 0.4 结算接口前置校验

打款、结算、商家钱包入账接口必须读取结算三条件:

- 客户签收确认已通过。
- 监管锁已上锁。
- 激活锁状态已激活。

接口返回必须包含 `settlement_allowed`、`blocked_reasons[]`、`manual_review_required`。无监管锁品类替代条件已确认按推荐方案执行:客户签收确认、设备识别码归档、交付照片 / 交付证据完整;高价值订单叠加 AI 视频确认或人工复核。接口设计保留 `alternative_acceptance_rule_id` 与 `alternative_acceptance_result`。

### 0.5 退款接口契约【资金/法务高风险】

退款接口必须包含:

- `responsible_party`:商家/平台/协商/其他。
- `refund_amount`:指定退款金额。
- `negotiation_id`:联营/平台订单必填。
- `approval_id`:审批记录。
- `reversal_targets[]`:支付流水、钱包流水、总账分录、服务费、返点等冲正对象。
- `proof_files[]`:凭证附件。

无责任方、无冲正对象、无审批记录的退款接口不得进入执行态。

### 0.6 短租和 IM 接口补齐范围

体验租短租 V1 需补:门店列表、车型/价格方案、骑行人、下单、支付、保证金、商家接单、核销绑车、用车中、续租、还车验收、异常工单、保险/保单、客服上下文。

IM 客服需补:C 端发起会话、商家/门店发起会话、上下文卡片、消息发送、附件上传、会话分配、转工单、归档查询。IM 接口不得直接执行退款、扣款、锁车、关单、改账户等高危动作。

### 0.7 编码前必须补齐的接口契约清单

| 接口组 | 必补内容 | 不补的风险 |
|---|---|---|
| C 端长租 | 订单确认、资料、实名、人脸、信用/信息授权、合同、公证、首期/保证金支付、签收、账单、留购/提前结清、客服入口的请求/响应示例 | C 端误暴露内部字段或状态不可用 |
| C 端体验租 | 门店列表、车型价格、骑行人、短租订单确认、保证金/免押、支付、核销码、用车中、续租、还车、异常工单 | 短租 V1 无法独立开发 |
| 商家端 | 商家订单审核、发货/交付、体验租接单/拒单/核销/验车、钱包、提现、员工权限 | 商家端权限和履约动作不清 |
| 运营端 | 审核、合同模板、公证、结算三条件校验、监管锁异常、退款协商单、法诉证据包、强制状态变更 | 高危操作缺审批和审计 |
| 财务端 | 支付流水、钱包流水、总账分录、结算批次、打款、退款、冲正、返点、提现 | 四账无法对齐,资金重复入账/重复打款 |
| 风控端 | 审核资料、规则命中、补资料、复审、黑灰名单、人工放行 | 审核结论和客户展示混淆 |
| 回调接口 | 支付、退款、电子签、公证、监管锁、短信/通知、IM 附件处理回调 | 第三方重复/延迟回调导致状态错乱 |

每个接口在契约化时必须包含:

1. URL、method、鉴权方式、权限点。
2. 请求 DTO、响应 DTO、字段类型、是否脱敏。
3. 错误码、错误文案、是否可重试。
4. `X-Idempotency-Key` 要求和业务唯一键。
5. 审计日志字段:operator_id、role_key、approval_id、before_snapshot、after_snapshot。
6. 关联状态机:成功后状态、失败后状态、补偿任务。
7. C 端字段白名单和禁用字段扫描结果。

## 1. 当前覆盖范围

当前 OpenAPI 草案已覆盖 V0.2 主要业务链路的写接口骨架：

- 商家入驻申请、运营审核。
- 商品新增、规格配置、商品提交审核、平台商品复制给商家。
- 办单助手报价、办单二维码、客户扫码创建订单。
- 客户提交资料、订单审核通过、联营订单和平台订单分配资方。
- 发起合同、补充合同、公证。
- 创建发货/交付任务、绑定唯一设备、归还验收。
- 发起退款、财务冲正、创建对账批次。
- 发起提现。
- 渠道创建、渠道推广码创建。
- 租后案件创建。
- 客诉创建。
- 第三方回调统一入口。

## 3. 使用约束

1. 当前 `servers.url` 是占位，不代表真实部署地址。
2. 所有金额字段按分传输。
3. 写接口要求 `X-Idempotency-Key`。
4. 生产接口需要补完整权限点、错误码、字段校验、审计日志和事务边界。
5. 回调接口不直接信任第三方状态，只负责接收、验签、入库和进入处理队列。
