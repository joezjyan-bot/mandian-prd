## 2. 商业模式概述

### 2.1 业务本质

本系统支持的核心业务是 **"以租代售"** 模式的耐用品分期租赁。

- 客户分期支付租金（首付 + 月付），租期内拥有设备使用权
- 租期结束后客户可选：归还、续租、留购（买断）
- 设备所有权在租期内归出租方（门店/商家/平台/资方，按订单类型不同而不同）
- 平台通过"加价系数"（计费倍数）赚取利润，资方通过资金回报赚利润

### 2.2 资金真相

**实际模式**：

```
门店采购手机（自己进货）
       ↓
客户向门店发起租赁
       ↓
平台/资方介入，"从门店购买"这台手机（手机采购合同生效）
       ↓
平台支付采购款到门店采购账户（资方/平台垫付）
       ↓  
平台与客户签租赁合同，客户分期付款给平台
       ↓
月付收上来后，按订单类型分账
```

**关键观察**：
- "**手机采购合同**" 是商业模式的底层凭证（平台从门店买货）
- "**采购账户**" 是平台向门店付款的支付宝账号
- "**分红订单 = 平台配资**"（平台/资方出钱垫付采购款）

### 2.3 三类订单的本质区别

| 维度 | 门店订单 | 分红订单 | 平台订单 |
|---|---|---|---|
| **出资方** | 门店 100% | 门店出货 + 资方/平台配资 | 门店不出货也不出钱 |
| **审核方** | 门店自审 | 平台审核 | 平台审核 |
| **签约方** | 客户+门店+平台 | 客户+门店+平台 | 客户+平台+执行商家 |
| **客户违约损失承担** | 门店 | 资方/平台 | 平台 |

**法务关注点**：
- ⚠️ "以租代售" 模式是否触碰"融资租赁"的资质红线？需法务确认
- ⚠️ 平台代收客户款项的合规性（涉及 P2P 监管？支付牌照？）
- ⚠️ 多方签约时合同的法律效力（特别是"客户+平台+执行商家"三方组合）

### 4.3 资金流转模型

**资金链路**：客户 → 平台账户 → 按订单类型分账给资方/门店/渠道。

**法务关注点**：
- ⚠️ 平台账户的资金沉淀是否合规
- ⚠️ 跨主体分账涉及发票、税务问题
- ⚠️ 退款流程的合规边界（违约金比例是否符合法律规定）

详见完整版 PRD §4.3。

### 7.5 合同模板引擎（核心法务相关）

#### 7.5.1 模块定位

支持"出租方主体 + 合同模板"完全可配置：
- 三方组合可变（门店+客户+平台 / 商家+客户+平台 / 客户+平台+执行商家 等）
- 合同内容可变（不同业务、不同时期、不同客户用不同模板）
- 模板版本可管理

#### 7.5.2 合同类型清单（V1.0 支持）

| 合同名称 | 签署方 | 签署时机 |
|---|---|---|
| **门店入驻合同** | 门店 ↔ 平台 | 门店入驻审核通过后 |
| **手机采购合同** | 平台 ↔ 门店 | 每笔订单（分红/平台订单）成立时 |
| **租赁合同（三方）** | 客户 + 门店 + 平台 | 客户下单签约时 |
| **租赁合同（商家版）** | 客户 + 商家 + 平台 | 当出租主体为商家时 |
| **店铺合作合同** | 商家 ↔ 平台 | 商家入驻审核通过后 |
| **资方合作合同** | 资方 ↔ 平台 | 资方接入时 |
| **延期还款补充协议** | 客户 + 平台 | 客户申请改账单日且需加费时 |
| **续租补充协议** | 客户 + 平台 | 续租成立时 |
| **退款协议** | 客户 + 相关方 | 退款审核通过时 |

#### 7.5.3 模板结构

```
合同模板
├─ 模板基础信息（名称、版本、适用范围、状态）
├─ 签约方定义（甲方/乙方/丙方/丁方）
├─ 模板正文（富文本 + 变量占位符）
└─ 变量定义（必填变量、变量取值规则）
```

#### 7.5.4 变量占位符

模板内用 `{{变量名}}` 占位，签约时动态填充：客户姓名、身份证、订单号、设备金额、首付、月付、期数、出租方名称、出租方营业执照、违约条款、管辖法院等。

#### 7.5.5 出租方可切换 + 模板绑定规则

**法务关键点**：
- 系统支持订单级别指定"出租方主体"
- 默认按订单类型自动选择，运营可手动切换
- 出租方信息从对应主体的"企业资料"拉取

| 业务场景 | 默认模板 |
|---|---|
| 长租 + 门店订单 + 手机 | 长租手机三方租赁合同（客户+门店+平台）|
| 长租 + 分红订单 + 手机 | 长租手机分红三方合同（客户+门店+平台）|
| 长租 + 平台订单 + 手机 | 长租手机平台合同（客户+平台+执行商家）|
| 长租 + 任意 + 留购 | 留购合同 |
| 长租 + 任意 + 续租 | 续租合同 |
| 短租 + ... | V0.2 补充 |
| 平台 - 门店 - 入驻 | 入驻合同 |
| 平台 - 门店 - 采购 | 手机采购合同 |

#### 7.5.6 模板版本管理

- 每次模板修改保存为新版本
- 旧版本只读，已签合同永远沿用签约时的版本
- 新签合同走最新启用版本
- 版本切换可定时（如"2026-07-01 起切换 v2"）

#### 7.5.7 签约流程（e签宝）

- 系统调用 e签宝 API 创建签署任务
- 三方依次签字：客户 → 门店/商家 → 平台
- 完成后合同 PDF 自动归档到订单详情
- 签署完成后法律效力等同于纸质合同（《电子签名法》）

### 7.7 风控引擎（法务相关）

#### 7.7.1 第三方风控数据

| 数据类型 | 供应商 | 法务关注点 |
|---|---|---|
| 共债报告 | 新颜 | 数据收集授权（客户告知 + 同意）|
| 全景报告 | 新颜 | 同上 |
| 人脸识别 | 阿里/腾讯 | 生物特征信息处理合规 |
| 二要素验证 | 阿里/聚合 | 紧急联系人个人信息保护 |
| OCR | 阿里 | 身份证图像处理合规 |

**法务关注点**：
- ⚠️ 个人信息保护法（PIPL）合规
- ⚠️ 客户授权链路（隐私协议 + 单独勾选）
- ⚠️ 第三方数据使用边界

#### 7.7.2 催收 / 起诉合规

**催收**：
- 不发短信给紧急联系人（避免骚扰投诉）
- IM 沟通留痕（可作证据）
- 通话录音（电审环节）
- 不得使用威胁性语言、冒充国家机关等

**起诉**：
- 严重违约订单（逾期 90+ 天，金额 > 阈值）转起诉
- 起诉资料管理（合同、聊天记录、还款记录、风控报告、客户身份信息等证据归档）

**法务关注点**：
- ⚠️ 催收手段的法律边界（《互联网金融逾期债务催收自律公约》）
- ⚠️ 失联客户处理：联系紧急联系人的合规边界
- ⚠️ 设备远程锁定的法律依据（合同条款中需明确）
- ⚠️ 起诉证据的完整性和合法性

## 11. 重构注意事项落地（法务相关部分）

### 11.2 项目重构与悦租租过渡方案

**初步方案**：

```
Stage 1：双系统并行（2-3 个月）
├─ 新系统部署上线（仅给 1-2 个客户试用）
├─ 老系统继续维护，不开发新功能
└─ 双系统数据不互通，新订单走新系统

Stage 2：数据迁移（1 个月）
├─ 编写数据迁移脚本（老系统订单/客户/商家 → 新系统）
├─ 老系统订单冻结，全部迁移到新系统
└─ 老系统切只读模式

Stage 3：老系统下线
├─ 所有客户切换到新系统
└─ 老系统归档
```

**法务关注点**：
- ⚠️ 历史订单的合同沿用问题（按"原合同规则"处理）
- ⚠️ 数据迁移的合规性（客户授权、数据保护）
- ⚠️ 平台主体变更（如有）对历史合同的影响

**待澄清**：
- 老系统的合同存档完整性
- 客户对停机时间的容忍度
