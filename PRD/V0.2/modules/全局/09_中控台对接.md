# 中控台对接

> **来源**: P0-4 决策
> **关系模式**: C 混合模式(直连第三方 + 中控台异步计费)
> **关联项目**: `mandian-zhongkong`(独立项目,通过 HTTP API + Webhook 对接)

---

## 1. 核心原则

| 维度 | 立场 |
|---|---|
| 平台关系 | 租赁系统 + 中控台是两个独立系统,通过 HTTP API + Webhook 协作 |
| 运行时依赖 | 租赁系统**不依赖**中控台运行时存活 |
| 调用路径 | 租赁系统**直连第三方厂商**,不经过中控台代理 |
| 中控台职责 | 客户级管控 + 异步计费 + 调用记录 + 对账 + 报表 |
| 业务优先级 | **保业务可用性 > 计费精确性**(异常时降级"按已授权放行") |

## 2. 接入范围(V1 所有第三方功能)

| 第三方能力 | V1 必接 | 备注 |
|---|---|---|
| e签宝 | ✅ | 电子签主通道 |
| 法大大 | ✅ | 电子签备选 |
| 人脸识别 | ✅ | 实名认证 |
| OCR(身份证/营业执照/银行卡) | ✅ | 资料采集 |
| 风控/征信报告 | ✅ | 订单风控 |
| 流水报告 | ✅ | 大数据风控 |
| 大数据查询 | ✅ | 多头借贷/失信 |
| 监管锁 | ✅(规划) | V1 后期启用 |

**不通过中控台**:支付通道(通联、直付通)——租赁系统直接对接持牌通道。

## 3. 工作流程

### 3.1 同步流程(中控台 → 租赁系统)

```
中控台主导(每 5 分钟一次,可配):
├─ 推送租户配置变更(品类/限额/启用状态)
├─ 推送租户套餐余量更新
└─ 推送租户被暂停/恢复等关键事件(Webhook 实时)

租赁系统接收后:
├─ 更新本地缓存(tenant_config_cache 表)
├─ 标记最后同步时间
└─ 重置降级模式标志(若处于降级中)
```

### 3.2 实时调用流程(租赁系统侧)

```
业务触发(如订单需要风控查询)
   ↓
租赁系统查本地缓存:
├─ 租户是否被暂停?
├─ 该能力是否有余量?
└─ 是否处于降级模式?
   ↓
通过检查 → 直连第三方厂商
   ↓
第三方返回 → 业务继续
   ↓
异步推送调用日志到中控台(队列处理)
```

### 3.3 异步推送流程

```
1. 写入本地 usage_log_queue 表(状态: pending)
2. 异步消费者每秒批量处理:
   - 批量推送到中控台 POST /usage/batch-log
   - 成功: 状态 → success
   - 失败: 状态 → failed,记录失败次数和原因
3. 失败重试机制:1min / 5min / 15min / 1h / 6h / 24h
4. 超过 7 天未成功: 标记"对账异常",运营介入
5. 队列堆积告警:超过 1 万条未推送时告警
```

## 4. 接口规范

### 4.1 中控台提供给租赁系统的 API

| 接口 | 方法 | 用途 |
|---|---|---|
| `/v1/tenant/{tenant_id}/config` | GET | 获取租户配置和开关 |
| `/v1/tenant/{tenant_id}/quota` | GET | 获取租户套餐余量 |
| `/v1/usage/log` | POST | 推送单条调用日志 |
| `/v1/usage/batch-log` | POST | 批量推送调用日志(主要使用) |
| `/v1/audit/calls/{tenant_id}` | GET | 中控台对账用 |
| `/v1/tenant/{tenant_id}/balance` | GET | 获取租户余额 |

### 4.2 中控台 → 租赁系统的 Webhook

| Webhook 事件 | 触发时机 |
|---|---|
| `tenant.config_changed` | 租户配置变更 |
| `tenant.suspended` | 租户被暂停 |
| `tenant.resumed` | 租户恢复服务 |
| `tenant.quota_low` | 客户余量低告警 |
| `tenant.quota_exhausted` | 客户余量耗尽 |

### 4.3 租赁系统提供给中控台的 Webhook

| Webhook 事件 | 触发时机 |
|---|---|
| `tenant.usage_alert` | 异步推送堆积告警 |
| `tenant.degraded` | 进入降级模式 |
| `tenant.recovered` | 退出降级模式 |

## 5. 本地缓存机制

### 5.1 缓存表

```sql
CREATE TABLE tenant_config_cache (
    tenant_id VARCHAR(100) PRIMARY KEY,
    config_version INT,
    quota_version INT,
    suspended BOOLEAN DEFAULT FALSE,
    capabilities JSON,
    quota_by_capability JSON,
    rate_limits JSON,
    last_synced_at TIMESTAMP,
    is_degraded BOOLEAN DEFAULT FALSE,
    degraded_since TIMESTAMP NULL,
    last_zhongkong_response_status INT
);
```

### 5.2 缓存生命周期

| 阶段 | 行为 |
|---|---|
| 正常态 | 中控台每 5 分钟主动 push,本地缓存自动更新 |
| 24 小时未同步 | 主动调用 GET 接口强制刷新 |
| 7 天未同步 | **进入降级模式**,默认按"已授权 + 套餐充足"放行 |
| 中控台恢复 | 自动退出降级模式,补传积压日志 |

## 6. 降级模式详细规则

### 6.1 触发条件
- 本地缓存 last_synced_at 距离当前 > 7 天
- 主动调用中控台连续失败 N 次

### 6.2 降级表现

| 检查项 | 降级模式行为 |
|---|---|
| 租户是否被暂停 | 默认按"未暂停"放行 |
| 能力是否启用 | 默认按"已启用"放行 |
| 套餐余量 | 默认按"充足"放行 |
| 限流策略 | 默认按"无限流"放行 |

降级期间所有调用:写本地 usage_log_queue,标记 `is_degraded_call = true`;中控台恢复后批量补传。

### 6.3 恢复机制
- 中控台恢复后,租赁系统自动检测
- 拉取最新配置 + 余量
- 退出降级模式
- 推送 webhook `tenant.recovered`

## 7. 月度对账机制

### 7.1 对账目的
发现租赁系统侧统计 vs 中控台侧统计的差异。

### 7.2 对账流程
```
每月 1 日凌晨触发:
1. 租赁系统侧导出上月所有调用记录
2. 调用中控台 GET /audit/calls/{tenant_id} 拉取中控台侧记录
3. 比对:按 capability + 日期分组
4. 差异 > 0.5% 时生成对账报告
5. 推送到运营/财务,人工介入核实
```

### 7.3 对账记录表

```sql
CREATE TABLE zhongkong_reconciliation (
    reconciliation_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    period VARCHAR(7),
    tenant_id VARCHAR(100),
    capability VARCHAR(50),
    rental_system_count INT,
    zhongkong_count INT,
    diff_count INT,
    diff_percentage DECIMAL(5,2),
    diff_reason VARCHAR(500),
    handle_status ENUM('pending', 'handled', 'closed') DEFAULT 'pending',
    handler_id BIGINT,
    handled_at TIMESTAMP NULL,
    notes TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 8. 数据模型

### 8.1 调用日志队列

```sql
CREATE TABLE usage_log_queue (
    log_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    tenant_id VARCHAR(100) NOT NULL,
    capability VARCHAR(50) NOT NULL,
    call_count INT DEFAULT 1,
    success BOOLEAN NOT NULL,
    error_code VARCHAR(50),
    third_party_request_id VARCHAR(100),
    third_party_response_id VARCHAR(100),
    related_order_id VARCHAR(50),
    called_at TIMESTAMP NOT NULL,
    is_degraded_call BOOLEAN DEFAULT FALSE,

    push_status ENUM('pending', 'pushing', 'success', 'failed') DEFAULT 'pending',
    push_attempts INT DEFAULT 0,
    last_push_at TIMESTAMP NULL,
    last_push_error VARCHAR(500),
    pushed_at TIMESTAMP NULL,

    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_push_status (push_status, created_at),
    INDEX idx_tenant_cap (tenant_id, capability),
    INDEX idx_order (related_order_id)
);
```

## 9. 错误处理 & 异常补偿

### 9.1 第三方调用失败

| 场景 | 处理 |
|---|---|
| 接口超时 | 业务层重试 1 次,失败报错 |
| 返回错误码 | 业务层按错误码处理 |
| 推送失败 | 进入重试队列,业务不阻塞 |

### 9.2 中控台异常

| 场景 | 处理 |
|---|---|
| 短暂不可达(< 1 小时) | 本地缓存有效,业务正常;日志入队 |
| 长期不可达(> 7 天) | 进入降级模式,业务正常;日志继续记录 |
| 返回异常数据 | 日志告警,运营介入;本地缓存保持不变 |

## 10. 安全机制

### 10.1 Webhook 签名
HMAC-SHA256(payload + secret),防伪造

### 10.2 API 鉴权
OAuth 2.0 类似机制;每个租户独立 API key,`.env` 配置

### 10.3 限流保护
中控台对每个租户 API 调用限流

## 11. 配置项

| `zhongkong_api_base_url` | (来自 .env) | 中控台 API 根地址 |

| `zhongkong_degrade_after_days` | 7 | 几天未同步进入降级 |

| `zhongkong_push_retry_intervals` | "60,300,900,3600,21600,86400" | 推送重试间隔(秒) |

## 12. 监控告警

| 指标 | 阈值 | 告警 |
|---|---|---|
| 队列堆积 | > 10000 | 飞书/邮件 |
| 中控台 API 失败率 | > 5%(1 分钟) | 飞书 |
| 降级触发 | 立即 | 飞书 + 邮件 |
| 月度对账差异 | > 0.5% | 邮件 + 工单 |
