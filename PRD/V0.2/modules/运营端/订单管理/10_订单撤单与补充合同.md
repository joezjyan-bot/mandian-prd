# 订单撤单与补充合同

> P0 业务文档(2026-05-26)。
> 定义两个关键的"订单修正"机制:
> 1. **订单撤单**(Q3 / Q15)— 客户主动取消订单的各阶段处理
> 2. **补充合同**(Q11 / Q19)— 订单生效后需要变更某些信息(如设备编码改错)时的合同修正机制
>
> 二者是"已签约订单出问题时"的两条核心路径,**不能滥用**,都有明确触发条件与权限矩阵。

---

## 1. 撤单机制

### 1.1 撤单可点的状态

| 订单状态 | C 端 [取消订单] 按钮 | 后台撤单权限 |
|---|---|---|
| 审核中(PENDING_REVIEW)| ✅ 客户直接可取消 | ✅ |
| 待付款(未付款)| ✅ 客户直接可取消 | ✅ |
| 待付款(已付首期)| ⚠️ 需要客服同意 | ✅(客服)|
| 待签约(已付首期未签)| ⚠️ 需要客服同意 | ✅(客服)|
| **待签约(已签合同)** | ⚠️ 需要客服同意,扣 1.5% 违约金 | ✅(客服)|
| 待发货 | ❌ 不可单方取消(走售后)| ✅(客服 + 主管批准)|
| 已发货 / 待收货 | ❌ 客户不可单方取消(走售后)| ✅(主管批准)|
| 租用中 | ❌ 客户不可取消(走归还流程)| ❌(只能归还)|

### 1.2 撤单流程

#### 1.2.1 直接撤单(审核中 / 待付款未付款)

```
客户在 C 端点 [取消订单]
  ↓
弹窗确认:"确认取消订单?"
  ↓
客户填写取消原因(可选)
  ↓
[确认取消]
  ↓
订单 → 已完成(子标签:已取消)
退款:无需退款
```

#### 1.2.2 需客服同意的撤单(已付款 / 已签约)

```
客户在 C 端点 [取消订单]
  ↓
提示:"该订单已付款,需联系客服处理"
  ↓
客户填写取消原因 → 提交申请
  ↓
进入运营端"撤单申请"队列
  ↓
客服审核:
  ├─ 同意 → 计算退款金额(若已签约扣 1.5%)→ 触发退款 → 订单关闭
  └─ 拒绝 → 通知客户(给出理由)→ 订单继续
```

#### 1.2.3 后台主动撤单(已发货 / 异常)

```
运营主管在订单详情 → [关闭订单] 按钮
  ↓
弹窗:输入关闭原因 + 选择退款方案
  ↓
[确认关闭]
  ↓
订单 → 已完成(子标签:已取消)
退款按 §1.3 规则执行
```

### 1.3 退款规则

| 撤单阶段 | 退款金额 | 退款通道 |
|---|---|---|
| 审核中 | 无需退款 | - |
| 待付款(未付款)| 无需退款 | - |
| 待付款(已付首期未签)| **首期实付全额退** | 原路退回(1-3 工作日)|
| 待签约(已签合同)| **首期实付 - 订单总价 × 1.5%(违约金,平台收)** | 原路退回 |
| 已发货 / 待收货 | **首期实付 - 1.5% - 物流费 - 折旧费(按使用天数)** | 原路退回 |
| 租用中 | **走归还流程,不是撤单** | - |

### 1.4 1.5% 违约金归属(Q20)

| 项 | 说明 |
|---|---|
| 基数 | 订单总价(设备价 + 留购价)|
| 比例 | 1.5% |
| 归属 | **平台收**(非商家收)|
| 例如 | 设备价 ¥5,000 + 留购总价 ¥6,050 → 违约金 ¥165.75 → 平台收 |

### 1.5 撤单后的资产处理

| 项 | 处理 |
|---|---|
| 已发货设备 | 物流追回 / 客户原路退回 |
| 客户已下载的合同 | 合同失效,标记"已取消"|
| 客户的电子签约记录 | 保留 30 天 / 永久(审计)|
| 客户已绑定的代扣 | 自动解绑 |
| 商家已收到的采购款(若已打款)| 财务发起追款 |
| 监管锁状态(若已上锁)| 远程解锁,设备状态恢复 DEVICE_AVAILABLE |

### 1.6 撤单 SLA

| 阶段 | SLA |
|---|---|
| 客户提交撤单申请 → 客服处理 | 24 小时 |
| 客服同意 → 退款触发 | 实时 |
| 退款到账(支付宝/微信原路)| 1-3 工作日 |
| 退款到账(银行卡)| 3-7 工作日 |
| 拒绝撤单 → 通知客户 | 实时 |

### 1.7 撤单数据表

```sql
order_cancel_request
- request_id          bigint PK
- order_id            bigint FK
- requested_by_type   enum -- customer / csr / supervisor
- requested_by_id     bigint
- requested_at        datetime
- request_reason      text
- order_status_at_request enum -- 撤单时订单的状态
- approved_by         bigint
- approved_at         datetime
- approval_status     enum -- pending / approved / rejected
- penalty_amount      decimal -- 扣除的违约金
- refund_amount       decimal -- 实际退款金额
- refund_channel      varchar
- refund_status       enum -- pending / processing / refunded / failed
- refund_at           datetime
- rejection_reason    text
- created_at / updated_at
```

## 2. 补充合同机制 ⭐

### 2.1 业务场景

订单生效后(已签主合同),发现某些信息错误或需要变更,但**不撤单**。通过"补充合同"修正。

### 2.2 适用场景

| 场景 | 是否适用补充合同 |
|---|---|
| 设备编码填错(IMEI/SN 错了)| ✅ 适用(Q11 主用例)|
| 客户地址变更(发货前)| ❌ 不需要合同,只改地址 |
| 客户地址变更(发货后,搬家)| ❌ 不需要合同,客服改地址 |
| 付款方式变更(代扣卡换了)| ❌ 不需要合同,客户在 C 端重绑 |
| **客户姓名变更**(罕见,改名)| ✅ 适用(必须补充合同确认身份)|
| **商品规格变更**(发货后发现型号不对)| ✅ 适用 |
| **月供 / 留购价调整** | ❌ **严禁**(已签合同不可改金额)|
| **资方变更**(已签合同后)| ✅ 适用 |
| **履约门店变更**(主管改派)| ✅ 适用(需补合同确认新主体)|
| 商品到货发现质量问题 | ❌ 走售后流程,不走补充合同 |

### 2.3 触发权限

| 场景 | 谁可发起 |
|---|---|
| 设备编码改错 | 客服 / 客服主管 |
| 客户姓名变更 | 运营主管 |
| 商品规格变更 | 运营主管 |
| 资方变更 | 运营主管 |
| 履约门店变更 | 客服主管 / 运营主管 |
| 其他 | 运营主管 + 输入理由 |

### 2.4 补充合同状态

| 编码 | 名称 | 说明 |
|---|---|---|
| SUPP_NONE | 无补充合同 | - |
| SUPP_PENDING_REVIEW | 待审核 | 客服/主管发起,等待主管复核 |
| SUPP_PENDING_SIGN | 待签署 | 主管批准,等客户签 |
| SUPP_SIGNING | 签署中 | 客户在签 |
| SUPP_SIGNED | 已签署 | 补充合同生效 |
| SUPP_REJECTED | 已驳回 | 主管驳回或客户拒签 |

### 2.5 流程

```
1. 客服在订单详情 → [发起补充合同] 按钮
   ↓
2. 弹窗:
   - 选择变更类型(枚举:设备编码 / 客户姓名 / 商品规格 / 履约门店 / 其他)
   - 旧值(只读,自动带入)
   - 新值(必填)
   - 变更原因(必填)
   - 上传证明材料(可选)
   ↓
3. [提交] → 补充合同进入 SUPP_PENDING_REVIEW
   ↓
4. 客服主管(或运营主管,按权限)审核
   ├─ 通过 → 系统生成补充合同 PDF → SUPP_PENDING_SIGN
   └─ 驳回 → SUPP_REJECTED → 通知发起客服
   ↓
5. C 端推送通知给客户:"您的订单需要签署补充合同"
   ↓
6. 客户在 C 端点击 → 调起 e签宝 → 客户签
   ↓
7. e签宝回调:
   ├─ 签署成功 → SUPP_SIGNED → 订单字段更新(如设备编码改为新值)
   └─ 客户拒签 → SUPP_REJECTED → 通知客服
```

### 2.6 客户感知

| 场景 | C 端展示 |
|---|---|
| 补充合同发起,等客户签 | 订单详情顶部出现紫色横幅:"请签署补充合同" + [立即签署] 按钮 |
| 客户签完 | 横幅消失;订单详情新增"补充合同 - 已签署"区块 |
| 客户拒签 | 系统通知客服 |

**注意**:客户能看到补充合同**变更前后的字段对比**,以便明确知道改了什么。

```
补充合同确认                      
  变更项:  设备编码                 
  原值:    IMEI 1234567890123     
  新值:    IMEI 9876543210987     
                                    
  说明:发货时填写错误,现修正        
                                    
  [取消]  [立即签署]                  
```

### 2.7 补充合同的法律效力

- 补充合同与主合同**同等法律效力**
- 主合同条款若与补充合同冲突,**以补充合同为准**
- 一份订单可有**多份补充合同**(每次变更独立一份)
- 所有补充合同**永久保留**

### 2.8 数据表

```sql
supplement_contract
- supp_contract_id   bigint PK
- order_id           bigint FK
- supp_seq_no        int      -- 第几份补充合同(初始 1)
- variation_type     enum     -- device_code / customer_name / product_spec / funder / fulfillment_store / other
- old_value          json     -- 旧值快照
- new_value          json     -- 新值
- reason             text
- proof_url          varchar  -- 证明材料
- requested_by       bigint
- requested_at       datetime
- approved_by        bigint
- approved_at        datetime
- approval_status    enum
- contract_pdf_url   varchar  -- e签宝生成的 PDF
- esign_contract_no  varchar  -- e签宝合同编号
- sign_status        enum
- signed_at          datetime
- rejection_reason   text
- created_at / updated_at
```

### 2.9 限制

| 限制 | 规则 |
|---|---|
| 一个订单最多 N 份补充合同 | V1 无硬上限,但 5 份以上自动告警(滥用提醒)|
| 同一字段多次变更 | 允许,但每次独立一份补充合同 |
| 补充合同审核 SLA | 24 小时 |
| 客户签署 SLA | 7 天(超时自动驳回 + 提醒发起客服)|
| 补充合同失败后 | 客服可重新发起 |

## 3. 撤单 vs 补充合同的选择

| 场景 | 选撤单 还是 补充合同 |
|---|---|
| 客户改主意了不想要 | 撤单 |
| 客户对商品不满意 | 撤单 / 售后(看阶段)|
| 客户对账单价格不满意 | 撤单(已签合同后不能改金额)|
| 设备编码填错 | 补充合同 |
| 客户姓名变更 | 补充合同(必要时撤单重做)|
| 客户地址变更 | 不需要合同,后台改 |
| 履约门店改派 | 补充合同(主合同主体变了)|
| 资方变更(已签后)| 补充合同(若合同里写了资方)|

## 4. 权限矩阵

| 动作 | 客服 | 客服主管 | 运营主管 |
|---|---|---|---|
| 受理客户撤单申请(已付款)| ✅ | ✅ | ✅ |
| 拒绝客户撤单申请 | ✅(需填理由)| ✅ | ✅ |
| 后台主动撤单(已发货)| ❌ | ✅ | ✅ |
| 后台主动撤单(租用中)| ❌ | ❌ | ❌(走归还)|
| 计算违约金(1.5%)| 系统自动 | - | - |
| 发起补充合同(设备编码)| ✅ | ✅ | ✅ |
| 发起补充合同(资方/门店/规格)| ❌ | ✅ | ✅ |
| 审核补充合同 | ❌ | ✅(常规)| ✅(重大)|

## 5. 异常场景

| 异常 | 处理 |
|---|---|
| 客户撤单申请超时未处理 | 进入主管复核队列 |
| 退款失败 | 进入财务异常队列,人工跟进 |
| 补充合同客户拒签 | 通知客服,客服联系客户协商 / 撤单重做 |
| 补充合同生效后又要再改 | 允许,发起新一份补充合同(`supp_seq_no` 递增)|
| 设备编码改错后,补充合同已签,但 IMEI 仍对不上 | 启动技术排查;主管可发起紧急撤单 |
