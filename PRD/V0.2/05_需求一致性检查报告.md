# 05 V0.2 需求一致性检查报告

> 检查对象：`01_重构决策基线.md`、`02_三端模块地图.md`、`03_商品办单助手订单核心链路.md`、已拆页面级 PRD、无界租后台巡检参考和 UI 生成口径。  
> 目标：确认当前 V0.2 是否沿着已讨论的业务逻辑继续走，找出冲突、缺口和需要后续补文档的点。

---

## 1. 总体结论

V0.2 的主方向已经成立，可以继续往页面级 PRD 拆，不需要推翻重来。

当前已经统一的核心口径：

1. 订单类型统一为 `门店订单`、`分红订单`、`平台订单`，新系统页面、接口和文案不再保留历史旧名称。
2. 运营平台是总后台，负责规则、审核、资方、财务、风控、链路配置和全局数据。
3. 商家 PC 端和门店手机端是同一商家主体的两个使用入口。
4. 门店订单由门店/商家自营自审，平台主要做监管、抽佣、客诉、租后和财务对账。
5. 分红订单和平台订单进入运营端待审核，由运营端分配资方、推进合同、支付、授权、发货和分账。
6. 办单助手确定拆成三个入口：门店订单、分红订单、平台订单。
7. 办单助手逻辑和 UI 参考 `joezjyan-bot/calculator` 里的 `phone-rent`，不要沿用旧系统计算器逻辑。
8. 默认门店发货，同时保留平台发货、资方发货、顺丰线上发货等可配置能力。
9. 普通门店端不保留资金充值入口，先保留订单收益、钱包、提现和对账。
10. 分红订单抽佣顺序按已确认口径：先按比例分账，再从门店份额和资方份额分别扣平台抽佣。
11. 员工账号按最小权限设计，只能办单和看待办，不看钱包、财务、配置和全量订单。
12. 全链路必须保留人工操作、系统回调、接口回调和配置变更日志。

## 2. 需要修正或补充的地方

### 2.1 短租库存的阶段口径需要补清楚

现有文档已经把短租目标模型写得比较完整：短租商品必须绑定唯一设备，设备有仓库、状态、锁定、出租、归还验收、维修、下架等流转。

这个方向是对的，但需要补一层交付分期：

| 阶段 | 建议范围 |
|---|---|
| V1 | 先把租赁模式、计费单位、商品适配长租/短租、设备码字段预留好；短租可先做轻量试点 |
| V1.5 | 完整上线短租库存、预约锁定、仓库、归还验收、维修、超时计费 |

原因：短租库存会牵涉设备唯一性、仓库、门店交付、归还验收、赔付、超时计费和财务账单，工作量不小。文档应明确这是目标模型，避免开发误以为 V1 必须一次性全做完。

### 2.2 资方端是否 V1 独立开放还要定边界

`02_三端模块地图.md` 已经列了资方端，包括资方工作台、订单、账单、资金账户、提现等。

建议口径：

| 阶段 | 建议范围 |
|---|---|
| V1 | 资方先作为运营端配置对象，由运营端维护资方、分配订单、查看资方账单 |
| V1.5/V2 | 再开放资方登录端，给资方查看订单、回款、账单、提现申请 |

原因：当前最核心是门店、商家、运营、订单和财务分账跑通。资方端如果 V1 同步做，会增加权限、账单、提现和数据隔离复杂度。

### 2.3 门店订单合同边界需要写进核心链路

现有文档里合同、支付、授权通常作为统一后续流程出现，但门店订单不应该默认强制走平台合同。

建议补充：

- 门店订单默认由商家/门店自审，可选是否启用合同模板。
- 商家可配置自己的门店订单合同模板或使用平台提供模板。
- 平台只做合同能力提供、模板管理、签署状态记录和必要监管，不默认成为出租方。
- 分红订单和平台订单才默认由运营端主控合同、授权和资方相关签署。

### 2.4 平台订单发货主体需要落到主链路字段

UI 口径和参考文档里已经写了：默认门店发货，平台订单可配置门店发货、平台发货、资方发货。

但 `03_商品办单助手订单核心链路.md` 的订单关键字段里还没有明确 `delivery_owner` / `发货主体`。

建议在订单字段中补：

| 字段 | 说明 |
|---|---|
| delivery_owner | 门店 / 商家 / 平台 / 资方 |
| delivery_method | 门店自提 / 门店配送 / 快递发货 / 顺丰线上发货 |
| delivery_address_source | 平台默认 / 商家默认 / 门店默认 / 订单覆盖 |

### 2.5 链路配置中心应从参考文档升级为正式模块

无界租后台最值得借鉴的部分是链路配置：支付、合同、风控、发货、商户、渠道、权限都能按主体或订单类型切换。

当前这部分主要在 `references/2026-05-24_无界租后台链路配置对比与改造建议.md`，还没有成为正式页面级 PRD。

建议新增：

`modules/运营端/配置管理/02_链路配置中心.md`

核心配置范围：

1. 订单类型链路：门店订单、分红订单、平台订单分别启用哪些审核、合同、授权、支付、发货步骤。
2. 商户覆盖配置：单个商户是否使用平台默认链路，哪些能力允许覆盖。
3. 支付代扣链路：支付宝、微信、银联、租赁行业代扣、直付通等通道切换。
4. 合同签署链路：e 签宝、补充合同、授权书、短信发送、回调状态。
5. 风控链路：实名、黑名单、风控报告、流水报告、审核资料。
6. 发货链路：默认门店发货，支持平台/资方/快递配置。
7. 分账链路：抽佣比例、分账顺序、资方/门店/渠道结算规则。
8. 回调和操作日志：所有配置变更要可追溯。

### 2.6 财务分账需要尽快拆页面级 PRD

主链路已经有分账模型，但财务页面还没拆。

必须补以下页面：

1. 钱包账户：平台账户、商家钱包、资方账户、渠道账户。
2. 分账明细：按订单、账单、期数展示门店份额、资方份额、平台抽佣、渠道佣金。
3. 对账单：订单收款、退款、扣款失败、提现、打款、佣金。
4. 提现申请：商家提现、渠道提现、资方提现。
5. 财务异常队列：分账失败、打款失败、退款失败、回调异常。

这一块不能只做 UI，必须和订单、账单、钱包、提现、操作日志打通。

### 2.7 店铺入驻审核还缺正式页面级 PRD

前面已经记录了门店入驻流程：

1. 门店登录页点击 `立即入驻`。
2. 注册手机号、密码、验证码、协议。
3. 完善营业执照、法人身份证正反面、OCR 识别名称/地址/法人/身份证号。
4. 填写门店真实地址，可接地图选择。
5. 上传门头照片。
6. 填写支付宝账号或银行卡。
7. 账户需和法人实名一致，不一致需要授权。
8. 完成 e 签宝授权签章。
9. 运营后台收到店铺审核，审核通过后可登录商家/门店工作台。

建议新增：

`modules/运营端/店铺管理/01_店铺审核.md`  
`modules/商家端/入驻认证/01_门店入驻.md`

### 2.8 商品管理还需要拆“添加商品”和“复制商品给商家”

当前核心链路已经定义商品、规格、全新/二手、同步办单助手、增值服务，但页面级 PRD 还没拆。

重点要写清楚：

1. 商品不是只有审核权限，还要有添加商品总权限。
2. 全新/二手是规格，不是商品单选。
3. 同一个商品可同时存在全新、二手规格。
4. 商品和规格分别配置是否同步办单助手。
5. 运营端可以复制商品给不同商家。
6. 商品列表默认不要混在一起展示全部商家商品，必须支持按商家筛选。
7. 运营平台和商家端都可以配置增值服务。
8. 增值服务可以同步到办单助手，客服审核改套餐时也能追加。

## 3. 目前没有发现明显冲突的点

以下内容在现有文档里基本一致：

1. 订单类型已统一为 `门店订单`、`分红订单`、`平台订单`。
2. 普通门店端资金充值入口已明确不保留。
3. `小程序大礼包`、`优惠券`、营销玩法先不做，客诉管理保留。
4. 渠道统计只统计分红订单和平台订单，不统计门店自营订单。
5. 平台默认抽佣 2%，可后台配置。
6. 分红订单示例逻辑正确：设备价 5000，需求配资 80%，门店等效出资 1000，资方出资 4000，后续每笔回款按 20%/80% 分账。
7. 登录保持已写入：门店 H5 同一手机默认 30 天免登录，高风险操作二次验证。
8. 订单列表已按三类订单拆分为独立工作台，`全部订单` 只做跨类型检索和汇总。
