# 工程规范基线

> **来源**: P2-1 / P2-3 / P2-4 决策
> **适用**: 所有开发/部署/运维工作
> **强制级别**: 必守

---

## 1. 部署架构

### 1.1 部署模式
独立部署 + 统一代码维护。
- 数据库:每个租户独立数据库
- 服务器:V1 初期一台 + 多 Docker 容器
- 代码:统一仓库,不分叉
- 升级:灰度发布

### 1.2 部署架构演进

| 阶段 | 部署 | 场景 |
|---|---|---|
| V1 初期(1-5 租户) | 一台服务器 + 多 Docker | 业务量不大,成本低 |
| V1 中期(5-20 家) | 业务量大的独立,小的共用 | 视业务量分配 |
| V1 长期(20+ 家) | 重要客户独立,通用共用 | 长期运营 |

**设计原则**:同一份代码可平滑迁移。

### 1.3 数据库设计原则
- 不加 `tenant_id` 字段(每租户独立库,天然隔离)
- 连接信息走环境变量,不硬编码
- 任何租户专属信息不得硬编码

## 2. 数据库变更管理

### 2.1 Migration 强制

**所有 schema 变更走 migration 框架**(Laravel Migrations / Doctrine 或同类)。

❌ 禁止:
- 手工连数据库执行 ALTER TABLE
- 生产环境手工改表结构
- 手工改生产数据

✅ 必须:
- 先写 migration 文件,提交代码仓库
- 命名规范:`YYYY_MM_DD_HHMMSS_description.php`
- 必须实现 up() 和 down(),支持回滚
- 部署时自动执行 `php artisan migrate`

### 2.2 示例

```php
// 2026_05_25_140000_add_repayment_day_to_orders.php
class AddRepaymentDayToOrders extends Migration {
    public function up() {
        Schema::table('orders', function($table) {
            $table->integer('repayment_day')->nullable();
            $table->integer('original_repayment_day')->nullable();
            $table->integer('repayment_day_change_count')->default(0);
        });
    }
    
    public function down() {
        Schema::table('orders', function($table) {
            $table->dropColumn('repayment_day');
            $table->dropColumn('original_repayment_day');
            $table->dropColumn('repayment_day_change_count');
        });
    }
}
```

### 2.3 数据迁移规范
- 与 schema 变更分开 migration
- 大表分批处理(每次 1000-10000 行)
- 生产前先灰度验证

## 3. CI/CD 流水线

### 3.1 CI
- 代码风格检查(PHP CS Fixer)
- 静态分析(PHPStan / Psalm)
- 单元测试(PHPUnit)
- 集成测试

### 3.2 CD
```
1. 主分支合并 → 自动构建 Docker 镜像
2. 镜像推送到镜像仓库
3. 部署到灰度环境(1 家指定租户)
4. 灰度跑 1-2 周验证
5. 手动触发批量推送到所有正式租户
```

### 3.3 灰度发布要求
- 灰度租户:平台指定
- 灰度时长:至少 1 周,重大变更 2 周
- 灰度期间监控错误率、性能、资金准确性
- 任何异常立即回滚

### 3.4 回滚机制
- Docker 镜像版本化,一键切换历史版本
- migration 支持回滚(down())
- 紧急回滚 < 5 分钟

## 5. 租户级定制(Feature Flag)

### 5.1 Feature Flag 体系

优先用 feature flag 实现租户级定制:

```php
if (FeatureFlag::isEnabled('use_funder_contract_template')) {
    // 该租户启用资方专属合同模板逻辑
}
```

### 5.2 Feature Flag 表

```sql
CREATE TABLE feature_flags (
    flag_key VARCHAR(100) PRIMARY KEY,
    flag_enabled BOOLEAN DEFAULT FALSE,
    description VARCHAR(500),
    updated_at TIMESTAMP,
    updated_by BIGINT
);
```

### 5.3 升级路径

- 多个租户启用且业务稳定 → 升级为通用功能(默认开启)
- 定期审计清理不再使用的 flag

## 7. 操作日志规范

### 7.1 必记操作

| 类型 | 示例 |
|---|---|
| 资金类 | 钱包扣费、提现申请、退款、后台余额变动、抽佣变更 |
| 订单类 | 创建/状态流转/关闭/改价/修改账单支付日 |
| 审核类 | 通过/驳回/下锁解锁/加黑 |
| 合同类 | 发起/签署回调/作废 |

| 权限类 | 角色变更/员工启用禁用/授予撤销 |

### 7.2 日志字段

```sql
CREATE TABLE operation_log (
    log_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    operation_type VARCHAR(50) NOT NULL,
    operation_target_type VARCHAR(50),
    operation_target_id VARCHAR(100),
    operator_id BIGINT,
    operator_role VARCHAR(50),
    operator_name VARCHAR(100),
    operation_source VARCHAR(50),
    operation_before JSON,
    operation_after JSON,
    operation_extra JSON,
    operation_notes VARCHAR(1000),

    INDEX idx_operator (operator_id),
    INDEX idx_target (operation_target_type, operation_target_id),
    INDEX idx_time (created_at),
    INDEX idx_type (operation_type)
);
```

## 9. i18n

### 9.1 字段命名
全部英文下划线分隔:
- ✅ `customer_name`、`order_status`、`payment_method`
- ❌ 中文字段/驼峰命名

### 9.2 文案 i18n

```javascript
// ❌ 不要硬编码
<Button>申请留购</Button>

// ✅ 走 i18n
<Button>{t('order.action.apply_purchase')}</Button>
```

zh-CN.json:
```json
{
  "order.action.apply_purchase": "申请留购",
  "order.action.apply_return": "申请归还",
  "order.action.contact_store": "联系门店"
}
```

V1 只有中文,但文案必须走 i18n key,不能硬编码。

### 9.3 错误码体系

```json
{
  "code": "ORDER_4001",
  "message_zh_CN": "订单已被取消,无法继续操作",
  "message_en": "Order has been cancelled, cannot proceed",
  "message_zh_TW": "訂單已被取消,無法繼續操作"
}
```

## 10. 数据隔离 & 安全

- 每个租户独立数据库账号,限定只能访问本租户数据库
- 禁止跨库 SQL
- 跨租户查询必须经过集中汇总库
- 测试数据见《测试数据脱敏规范》

## 11. 监控与告警

### 11.1 监控指标
- 响应时间(P95 < 500ms)
- 错误率(< 0.1%)
- 数据库连接池
- 异步队列堆积
- 中控台对接异常率
- 资金类异常事件

### 11.2 告警渠道
- 邮件 / 短信 / 飞书 / 钉钉
- 资金类异常 P0:立即告警 + 多渠道
