## 1. 问题识别

`C端/13_异常与客服联系.md` 中 IM 客服仅有以下覆盖:
- 支付宝小程序:支付宝官方客服 SDK
- 微信小程序:微信客服 SDK
- H5:平台自建 IM 或第三方(Udesk 等)
- 工作时间 9:00-21:00

**这是占位级别,不是 PRD**。下列内容**完全没覆盖**。

### 1.1 客户侧(C 端)未覆盖
- 通道选型决策依据(支付宝/微信官方 SDK vs 自建 vs Udesk 等)
- 跨端会话历史是否打通
- 客服主动推送能力
- 富媒体支持(图片/视频/订单卡片)
- 转人工 vs 机器人/FAQ 策略
- 客户排队、等待时长展示
- 评价系统
- 客户上下文自动注入(进会话时自动展示当前订单、状态、操作日志等)

### 1.2 客服侧(运营端工作台)完全没写
- 客服工作台界面
- 多账号并发分配规则
- 客服在岗状态管理(在线/忙碌/小休/离线)
- 主管的会话监督、强插会话、转接
- 会话超时未回复的提醒/升级
- 一个客服并发上限
- IM 与工单系统的联动
- IM 与电话系统的联动
- 客服的快捷回复、话术库

### 1.3 业务侧未接通
- IM 里能否直接发起 / 跳转关键操作(关单、改价、留购等)
- IM 会话记录的法律效力定位
- 录音/录像/会话归档保留期
- 数据脱敏在 IM 内的策略

### 1.4 合规未覆盖
- 客服话术措辞强制(罚息→逾期违约金、买断→留购)实时拦截
- 黑名单客户提醒
- 录音录像告知

### 1.5 数据模型严重不足
- `customer_ticket`、`customer_appeal` 仅有名字,无字段
- IM 会话表完全缺失

## 2. 已讨论的方向(待用户确认)

### 2.1 跨端身份统一

**门店端**:用 staff_id / merchant_id 强统一(已有平台账号)。三端登录都是同一身份,客服后台完全同步。

**客户端**:三种方案

| 方案 | 说明 | 优点 | 缺点 |
|---|---|---|---|
| A | 实名后强制合并(身份证号) | 实名后客户完全统一 | 实名前会话无法跨端合并 |
| B | 手机号绑定统一 | 实名前也能合并 | 依赖客户主动绑手机号 |
| C | 完全不合并 | 实现简单 | 客服效率低,体验差 |

**待用户拍板**:倾向 A+B 组合(实名后用 customer_id,实名前用手机号尽量合并)。

### 2.2 客户端的"账号"三端识别问题

| 端 | 默认登录态 | 唯一标识 |
|---|---|---|
| 支付宝小程序 | 支付宝授权 | 支付宝 user_id / openid |
| 微信小程序 | 微信授权 | 微信 unionid / openid |
| H5 | 手机号 + 验证码 | 手机号 |

同一客户在三端可能是三个不同账号 ID。
关键映射表:`customer.alipay_openid`、`customer.wechat_unionid`、`customer.phone_number`、`customer.id_card_hash`(实名后)

### 2.3 客服后台统一会话窗口示例

```
[客户] 张三(13800001234,id_card_xxx)
  最近订单: DZ20260525001 (在租中)
  历史会话:
    [今天 14:20] 支付宝小程序: "支付不成功"
    [今天 14:35] H5: "试了支付宝还是不行"  ← 同一客户同一窗口
    [昨天]      微信小程序: "我什么时候可以提货"
```

而不是三个独立窗口。

### 2.4 App 端的扩展性(2026-05-25 新增问题)

后期出独立 App 是否能打通?**结论: 完全可以**,前提是 App 用统一账号体系登录:
- App 用手机号 + 验证码登录(或支付宝/微信第三方登录)
- 实名后绑定 customer_id
- 三端 + App = 四端统一身份
- 客服后台只关心 customer_id,不关心来源端
- 推送通道增加 App push(jpush / 个推)

**前提**: 账号身份层(`modules/全局/12_账号身份与跨端统一.md`)从设计时就预留 App,不写死三端。

## 3. 用户待回答的关键决策(5 个)

### 3.1 IM 通道选型
倾向:
- (1) 自建 + Udesk/容联七陌等第三方 SaaS
- (2) 用支付宝/微信官方 SDK 嵌进小程序
- (3) (1)+(2) 混合:小程序用官方,H5/App 用自建

### 3.2 客服规模与排班
- V1 上线客服人数(预估)
- 24 小时 vs 9-21 工作时间
- 排班 vs 不排班

### 3.3 客服分流策略
- 按客户类型(分红 / 平台 / 门店分开队列)
- 按问题类型(售前 / 售后 / 财务分开队列)
- 统一队列轮询

### 3.4 IM 和工单的关系
- IM 为主,工单作为离线兜底
- 工单为主,IM 仅用于售前咨询
- 平行,客户自选

### 3.5 会话归档保留期
- 30 天 / 半年 / 1 年 / 5 年(与合同同步)

### 3.6 跨端推送
- 统一推到客户最近活跃的那端
- 三端独立都推

### 3.7 V1 范围
- V1 三端全做 IM + App 接口预留
- V1 先做支付宝(主战场),微信和 H5 用电话兜底,后续补齐
- V1 不做 IM,只用电话 + 工单
