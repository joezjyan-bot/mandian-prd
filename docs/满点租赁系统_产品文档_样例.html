第 1 章长租主流程总览

安心用长租,是客户从门店租走一台设备(手机 / 电动车 / 数码),按月付费使用,到期再决定还回去、继续租、还是申请买下来的完整业务。这一章先看全局,后面几章拆开每个关键环节。

第 1 层 · 这块是干嘛的

一句话

客户扫门店的码 → 确认要租什么、花多少钱 → 实名签合同 → 付首期 → 门店把设备交给客户 → 客户签收 → 按月还款用着 → 到期选择:还 / 续 / 买。
第 2 层 · 流程图

整条主流程长这样

客户做的事 系统/平台做的事 商家/门店做的事
蓝=客户操作　绿=系统/平台　棕=商家/门店。①→⑪是一条完整的长租生命周期。
第 3 层 · 关键规则

看这条流程时,先记住三件事

  • 客户全程只跟"门店(甲方)"发生租赁关系,平台是技术通道。客户看到的是"租设备",不是"借钱"或"分期买"。
  • 每一步都留快照、留证据:看到的价格、签的合同、签收的确认,系统都存档,将来有纠纷能拿出来。
  • 到期三个选项平等展示,系统不默认帮客户选"购买",所有权不会因为租满或还清自动转移给客户。
详见:长租全生命周期 运营端/订单管理/08　·　状态字典 全局/02

第 2 章下单与订单确认

客户扫了门店办单助手生成的二维码后,看到的第一个页面。这一页决定了客户对"我要租什么、花多少钱、有什么规矩"的全部认知,也是合规和证据链最关键的一页。

第 1 层 · 这块是干嘛的

一句话

把门店给客户定好的方案(哪台设备、多少钱、租几期、什么规矩)清清楚楚摆给客户看,客户逐项确认、勾选同意,然后才能下单。
第 2 层 · 页面示意

订单确认页大概长这样

这是给设计/前端参考的"中保真示意",不是最终视觉。位置、要素、有哪些勾选项是确定的,配色字体由 UI 团队最终把控。
第 3 层 · 关键规则

这一页上,每个东西的规矩

页面元素规则
设备价值确认区必须独立成块,客户只读、不可改;型号/成色/价值都来自这一单的设备价。客户必须单独勾"确认无误"才能下单。
价格双层显示大字显示"实际您要付 ¥X",小字解释"总额−保证金抵扣"。避免客户误解为分期买断。
最低服务期显著展示"3 个月起"(安心用统一,不分品类),并说明"3个月内提前归还要结清已发生费用"。必须单独勾选确认。
到期三选一说明这一页就要告诉客户"到期可以:归还/继续用/申请购买",三个平等列出,不暗示一定要买。
下单按钮所有必勾项没勾完,点击就提示"请先阅读并同意相关协议",不放行。
⚑ 这一页为什么这么重要(给老板和法务看)
客户在这页看到、确认、勾选的一切,系统都会拍成"快照"存档(看了什么价、什么时间、什么设备、IP是多少)。将来万一打官司,这页快照就是证明"客户当初清楚自己在租设备、知道规矩"的核心证据。所以这页必须干净、清楚、留痕。
详见:订单确认页 C端/02　·　设备价值确认单 运营端/资产管理/02

第 3 章到期三选一 与 申请购买

租期快到时,客户要做一个选择。这一章是整个业务里"合规"最敏感的地方——处理得好是干净的租赁,处理不好就会被认为是"变相分期买断"。

第 1 层 · 这块是干嘛的

一句话

到期前,系统平等地给客户三个选择——把设备还回去、再续租一段、或者申请把这台设备买下来。客户主动选,系统绝不替客户默认勾"购买"。
第 2 层 · 流程图

客户选择后,各自怎么走

三条路客户自己选。"申请购买"用棕色,因为它是合规最敏感、必须最克制的一条。
第 3 层 · 关键规则

申请购买价怎么算(一期口径)

客户申请购买时,要付的钱按这个算:

申请购买价 = 当前还没付的剩余租金 + 保证金　|　客户实付 = 购买价 − 保证金抵扣
  • 这个数和办单助手当初报价单上每期那列"购买参考价"是同一个数,系统三处取同一份快照,不会各算各的。
  • 保证金可以抵扣购买价;如果保证金比购买价还高,系统自动退差额。
  • 申请购买必须客户主动发起,在独立确认页看清金额后才能支付,付清才转移所有权。
⚠ 必须如实告诉各部门(尤其法务)的一件事
现在用的"购买价 = 剩余租金 + 保证金"这个算法,我们内部叫 A 口径。它的好处是简单、系统现成、能马上开发。但它在法律上是偏敏感的——因为"付的租金累计起来约等于设备价",容易被法院往"名为租赁、实为分期买断"上靠。

所以我们明确把它标成"一期先用、待法务确认",并准备了二期的优化口径(购买价按设备折旧后的残值算,和剩余租金脱钩)。这一条是这次评审会必须请法务拍板的重点,不是已经定死、没有风险的事。
详见:在租期间 C端/12 §4.2　·　系统定版待完善清单(A口径) 开发设计/22
