### 任务 2:商家自定审批规则(决策 B)

**新增文档**:`商家端/审批配置/01_商家自动审批与风控预设.md`

**核心内容**:

#### 2.1 自动审批开关

- 商家可开启/关闭"自动审批订单"
- **仅作用于商家订单**

#### 2.2 审批规则

| 模式 | 说明 | 风险 |
|---|---|---|
| **依据风控审批** | 符合下方风控条件配置的订单自动通过 | 中 |
| **全部通过** | 所有订单全部自动通过 | ⚠️⚠️ **高危**,商家自担风险 |
| **手动审批** | 所有订单需商家手动审核 | 低 |

**"全部通过"模式必须的红字警告**:
```
⚠️ 警告:开启"全部通过"模式后,系统将不对客户做任何风控筛查,
所有订单自动进入待支付环节。由此产生的任何逾期 / 损失,
将由您自行承担,平台不为此类订单的风险背书。

[我已知晓风险,确认开启]  [取消]
```

#### 2.3 风控条件预设方案

| 方案 | 说明 |
|---|---|
| **保守型(安全优先)** | 所有 8 项风控条件全部匹配 |
| **开放型(单量优先)** | 仅匹配核心 3 项(失信人/在执案件/身份三要素) |
| **自定义** | 商家逐项设置 |

#### 2.4 风控条件清单(8 项)

每项可选"必须匹配 / 不做要求":

1. 在执行案件(无在执案件 / 不做要求)
2. 历史案件(无历史案件 / 不做要求)
3. 失信人/黑名单(非失信人/黑名单 / 不做要求)
4. 历史逾期(无历史逾期 / 不做要求)
5. 当前网贷逾期(无逾期 / 不做要求)
6. 租赁申请行为(正常 / 不做要求)
7. 身份三要素一致性(必须一致 / 不做要求)
8. 手机号状态(正常 / 不做要求)

#### 2.5 关键约束

- **仅商家订单使用本配置**
- 联营订单 / 平台订单**强制走平台统一风控**(详见 `资方管理/03_风控规则与资方分配.md`)
- 商家端无法关闭联营 / 平台订单的风控

#### 2.6 数据模型

```sql
merchant_approval_config
- config_id              bigint PK
- merchant_id            bigint FK
- auto_approval_enabled  boolean
- approval_rule          enum -- by_risk / pass_all / manual
- risk_preset            enum -- conservative / open / custom
- risk_conditions        json -- 8 项风控条件配置
- updated_by             bigint
- updated_at             datetime
```

- `运营端/库存设备管理/02_短租车辆库存.md`(若不存在)

**核心要求(摘自 Hudson 原话)**:
> 库存管理里面一个长租设备一个短租设备 两个管理列表
> 然后用户下单成功之后由商品码 那就直接把这个客户的订单信息同步到长租设备管理里面
> 只简单的带客户姓名 设备型号 设备码 下单日期 设备状态(租用中,已赎回,已留购)
> 然后后面赎回的再加一个赎回的价格和处置方式
> 这样后期赎回的就来登记一下这个表格。
> 这样对我的资产管理也比较清晰
> 当然尽量的展示在一页上面 别太多信息到时候客服操作起来也不方便。

#### 5.1 长租设备档案(单页轻量级)

**列表字段**(尽量精简,一页展示):

| 字段 | 来源 | 说明 |
|---|---|---|
| 客户姓名(脱敏) | 订单同步 | 张** |
| 设备型号 | 订单同步 | iPhone 15 Pro Max |
| 设备码(IMEI/SN/VIN) | 订单同步 | xxxx |
| 下单日期 | 订单同步 | 2026-05-26 |
| **设备状态** | 状态变化时更新 | 履约中 / 已赎回 / 已留购 / 已报废 |
| 赎回价格 | 赎回时登记 | ¥X,XXX(默认空) |
| 处置方式 | 赎回时登记 | 出售 / 出租 / 派给员工 / 报废(默认空) |
| 关联订单号 | 订单同步 | 跳转订单详情 |

**自动同步规则**:
- 订单成功(`PENDING_RETURN` 状态进入)→ 自动写入长租设备档案,状态=履约中
- 订单完成(归还/留购)→ 状态自动更新(已归还/已留购)
- 设备状态可手动调整(主管权限)

#### 5.2 赎回登记动作

**触发场景**:客户私自变卖,平台回收

**操作流程**:
```

   ↓
后续处置时手动改"处置方式"字段
```

### 任务 7:C 端独立展示价不做(决策 G)

**说明**:**不做这个功能**。

**理由**:
- 若展示价 < 实际均价,可能涉嫌虚假宣传
- 若展示价 > 实际均价,失去营销价值
- 满点坚持"实实在在做生意"原则

**替代方案**:
- C 端商品详情页**展示价格区间**(起步价 - 最高价/月)
- C 端订单详情页**展示完整付租计划**(每期具体金额)
- 不允许商家自定义"对外展示价"

| 联营订单 / 平台订单 | 高于指导价上限 | ❌ **直接拦截不允许上架** |

```sql

#### 9.3 实名认证与下单解耦(决策 M)

**修订**:`C端/01_用户实名认证.md`(若存在)

**核心流程**:
```
客户进入商品详情 → 点[立即下单]
   ↓
检查实名状态:
   ├─ 未实名 → 跳转实名认证页
   │              ↓
   │           身份证 OCR + 姓名/身份证号 + 隐私政策勾选 + 短信验证 + 人脸活体
   │              ↓
   │           实名认证成功提示弹窗
   │              ↓
   │           关闭弹窗,回到商品详情页(不自动下单)
   │              ↓
   │           客户再次点[立即下单] → 进入正常下单流程
   │
   └─ 已实名 → 直接进入下单流程

**关键**:
- 实名认证成功后**不自动下单**
- 客户需要**再次主动点[立即下单]**
- 给客户"确认要下单"的心理缓冲

### 任务 10:商品列表 4 状态(决策 I)

**修订文档**:`运营端/商品管理/01_商品列表.md` + `商家端/商品管理/01_商品列表.md`

**核心**:商品列表新增 4 个状态 Tab:

| 状态 | 说明 | 显示位置 |
|---|---|---|
| **上架中** | 已通过审核,在 C 端可见 | 商家端 + 运营端 |
| **已下架** | 商家主动下架 / 平台强制下架 | 商家端 + 运营端 |
| **待审核** | 商家提交,等待平台审核 | 商家端 + 运营端 |
| **草稿** | 商家保存草稿,未提交 | 商家端 |

**状态流转**:
```
草稿 → [立即上架] → 待审核 → [平台审核通过] → 上架中
                  └→ [平台审核驳回] → 草稿(带驳回原因)
上架中 → [商家下架] → 已下架
上架中 → [平台违规下架] → 已下架(带违规原因)
已下架 → [重新上架] → 待审核
```

**数据模型扩展**:
```sql
product
- status enum -- draft / pending_review / on_shelf / off_shelf
- review_status enum -- nullable / pending / approved / rejected
- review_reason text -- 驳回/违规原因
- off_shelf_by enum -- nullable / merchant / platform
- off_shelf_reason text
```
