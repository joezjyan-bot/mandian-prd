# 门店结算账户与资金穿透架构

> P0 业务文档(2026-05-27)。
> 配套 `订单管理/08_长租订单全生命周期与客服操作.md §7.3` 决策落地。
>
> **本文档完全替代** `07_采购账户与软钱包架构.md`(已删除)。

> **⚠️ 重大架构修订(v2.0)**
> 经合规复核(对照《门店入驻及平台服务合同》+《租赁服务协议》+《应收账款转让确认书》):
> - "采购账户 / 采购款 / 软钱包"违反合同口径,会被税务 / 监管穿透认定为融资租赁
> - 本文档**整体推翻 v1.x**,统一对齐合同口径
> - 三类订单的法律性质明确为**三模式并行**:门店自营(自营订单)/ 联营合作(联营订单)/ 应收账款受让(受让订单)
> - 门店端**只有一个钱包**(订单结算账户),所有归属门店的钱穿透到通联备付金,平台不沉淀
> - 客户租金按合同口径拆分为「设备使用费 + 平台服务费」,只有设备使用费部分穿透,平台服务费留对公账户

---

## 1. 核心定性与术语对齐

### 1.1 合同口径下的核心术语

| 旧术语(必须清理)| 新术语(对齐合同)| 法律性质 |
|---|---|---|
| 采购款 / 采购账户 | **订单结算款 / 订单结算账户** | 应收账款转让对价 / 联营出资 |
| 采购账户提现 | **订单结算账户提现** | 门店从结算账户提现到银行卡 |
| 资方出资 / 资方代付 | **应收账款受让方付款 / 联营出资** | 应收账款转让对价 / 联营投资款 |
| 客户分期款 | **客户租金**(=设备使用费+平台服务费)| 租金 |
| 平台抽佣(2%)| **平台综合服务费** | 服务费收入(交 6% 增值税) |
| 软钱包 / 我的钱包(后台) | **结算台账 / 门店结算账户** | 通联备付金账户的子台账视图 |

### 1.2 三种合作模式

订单创建时,门店选择本订单的合作模式,系统按模式适用对应的资金 / 收益规则。

| 合作模式 (`cooperation_mode`) | 业务含义 | 法律性质 | 出资责任 | 收益分配 |
|---|---|---|---|---|
| `self_operate` 自营订单 | 门店全额出资,平台提供综合服务 | 门店自主经营 + 平台服务合同 | 门店 100% | 门店拿全部租金 - 平台综合服务费 |
| `joint_venture` 联营订单 | 门店和平台按比例共同出资 | 联营投资合作 | 门店 X% / 平台 (100-X)% | 按出资比例分润 |
| `receivables_assignment` 应收账款受让订单 | 平台代付应收账款转让对价,门店履约 | 应收账款转让 | 平台 100% | 平台拿全部应收回款 + 平台综合服务费;门店赚进货差价 |

**关键设计**:
- 不管哪种合作模式,**客户合同永远是「门店出租 + 客户租赁 + 平台提供综合服务」三方租赁关系**
- 客户**完全感知不到**自营 / 联营 / 受让的内部区别
- 合作模式仅影响**门店和平台之间的内部资金流转与收益分配**

### 1.3 客户租金的法律拆分

按《租赁服务协议》定义条款:

> 【租金】乙方因接受甲方租赁服务应向甲方支付的每期应付款项的统称,**包含设备使用费及平台服务费两部分**

```
客户每期付租金 ¥500
   ↓
合同拆分:
   ├─ 设备使用费 ¥490(归门店,法律性质=应收租金)
   └─ 平台服务费 ¥10(归平台,法律性质=平台综合服务费收入)
```

**法律含义**:
- **设备使用费部分**:门店对客户享有的应收账款,需穿透通联备付金(因为这部分钱"还不是平台的")
- **平台服务费部分**:已是平台的服务收入,留在平台对公账户(不需穿透,合规)

具体拆分比例 / 服务费率由后台「合作模式费率配置」决定,详见 §10。

## 2. 资金路径总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                       │
│  应收账款受让方/联营资金     客户                                       │
│   │ 应收账款转让对价        │ 每月租金                                  │
│   │ 联营出资款              │ (设备使用费+平台服务费)                  │
│   ↓                        ↓                                          │
│  ┌──────────────────────────────────────┐                            │
│  │ 平台实体银行账户(对公户)              │                            │
│  │ - 收入金(订单生效时/客户支付)        │                            │
│  │ - 内部计算扣分(过路 < 1 分钟)         │                            │
│  └──────────────────────────────────────┘                            │
│                    │                                                   │
│         ┌──────────┼──────────┐                                       │
│         ↓ 实时穿透  ↓ 留对公户                                          │
│  ┌──────────────────────┐  ┌──────────────┐                          │
│  │ 通联备付金账户         │  │ 平台综合服务  │                          │
│  │ (平台名义,持牌通道)   │  │ 费收入(合规) │                          │
│  │ - 接收应收账款回款     │  │              │                          │
│  │ - 接收联营出资款       │  │              │                          │
│  │ - 系统子台账分账       │  │              │                          │
│  └──────────────────────┘  └──────────────┘                          │
│           │                                                            │
│           ↓ 通联代付                                                    │
│      门店银行卡 / 联营出资方银行卡                                       │
│                                                                        │
│  门店端 UI:                                                            │
│    "我的钱包"(一个账户,对应通联子台账视图)                            │
│     - 余额:¥X                                                          │
│     - 明细:订单结算款/月度分账/联营分润/平台奖励/返利等                  │
│     - 提现:一键提现 → 通联代付出款                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.1 穿透核心原则

> **平台收到任何归属门店的钱(应收账款转让对价 / 联营出资款 / 客户租金中的设备使用费部分 / 平台奖励 / 返利 等),实时(< 1 分钟)按 1:1 等额充值到通联备付金账户。平台实体账户只过路,不沉淀。**

**例外**:平台综合服务费(已是平台的服务收入)**留在平台对公账户**,这是平台的合法营业收入,不算客户资金沉淀,合规无问题。

## 3. 数据模型

### 3.1 `merchant_settlement_account` 门店结算账户主表

```sql
merchant_settlement_account
- account_id              bigint PK
- merchant_id             bigint FK     -- 商家级,一个商家一个账户(对应"我的钱包")
- account_no              varchar(32)   -- 账户编号(系统生成,展示用,如 MSA20260527001)
- balance_total           decimal(18,2) -- 余额合计
- balance_pending         decimal(18,2) -- 在途余额(平台对公账户已收钱,通联充值确认中)
- balance_available       decimal(18,2) -- 可提现余额
- balance_withdrawing     decimal(18,2) -- 提现中余额
- total_received          decimal(18,2) -- 累计入账
- total_withdrawn         decimal(18,2) -- 累计提现
- total_fee               decimal(18,2) -- 累计手续费
- status                  enum          -- normal / frozen / disabled
- bank_account_id         bigint FK     -- 绑定的收款银行账户
- tonglian_subledger_id   varchar(32)   -- 通联备付金账户中的子台账 ID
- created_at / updated_at

-- 索引:(merchant_id) 唯一约束
```

### 3.2 `settlement_entry` 结算账户流水(统一承接所有入账/出账)

```sql
settlement_entry
- entry_id                bigint PK
- account_id              bigint FK
- direction               enum          -- in(入账)/ out(出账)
- entry_type              enum          -- 详见 §3.3
- amount                  decimal(18,2)
- order_id                bigint        -- 关联订单(若适用)
- order_no                varchar(32)
- cooperation_mode        enum          -- 订单合作模式快照(self_operate/joint_venture/receivables_assignment)
- source_funder_type      enum          -- 资金来源类型:assignment_payer(受让方)/customer(客户)/joint_venturer(联营出资方)/platform(平台)
- platform_account_ref    varchar(64)   -- 平台对公账户过路凭证(银行流水号)
- tonglian_topup_id       varchar(64)   -- 通联充值流水号(入账时穿透凭证)
- tonglian_payout_id      varchar(64)   -- 通联代付流水号(出账时凭证)
- balance_before          decimal(18,2)
- balance_after           decimal(18,2)
- description             varchar(255)  -- 摘要(门店端 UI 展示)
- status                  enum          -- pending / completed / failed / reversed
- created_at / updated_at

-- 索引:(account_id, created_at) / (order_id) / (entry_type, status)
```

### 3.3 `entry_type` 枚举(7 类)

| 枚举值 | 中文名 | 触发场景 | 资金方向 |
|---|---|---|---|
| `order_settlement` | 订单结算款 | 受让订单监管锁回调 / 联营订单监管锁回调 → 转让对价/联营出资款入账 | in |
| `monthly_split` | 月度分账 | 客户每期租金支付后,门店应得部分入账 | in |
| `joint_venture_profit` | 联营分润 | 联营订单结算时,门店应得利润分成 | in |
| `platform_reward` | 平台奖励 | 平台主动给门店的奖励(扶持金 / 业绩奖 等)| in |
| `platform_rebate` | 平台返利 | 渠道返利 / 推广提成 等 | in |
| `platform_deduction` | 平台扣款 | 违约金扣减 / 罚款 / 服务费补扣 等 | out |
| `withdrawal` | 提现 | 门店主动提现 | out |
| `financial_adjustment` | 财务调整 | 财务人工调账(经审批)| in/out |

## 4. 开户流程

### 4.1 开户时机

| 节点 | 动作 |
|---|---|
| 商家入驻审核通过 | 系统不自动开户,等待商家申请 |
| 商家在 PC 端 > 财务钱包 看到 [申请开通订单结算账户] | 入驻完成后即可申请 |
| 商家点击申请 | 进入开户审核流程 |
| 财务审核通过 | 创建 `merchant_settlement_account` 记录 + 在通联备付金主账户创建子台账 |

**说明**:不分订单类型,**任何入驻商家都可申请开通**(因为门店订单也可能有平台奖励 / 返利等需要走结算账户)。

### 4.2 开户字段要求

| 字段 | 说明 |
|---|---|
| 商家主体 | 必须与营业执照一致 |
| 收款银行账户 | 户名必须与商家主体一致 |
| 法人身份证 | 必须与营业执照法人一致 |
| 实名验证状态 | 必须已实名 |
| 收款账户授权 | 收款账户非法人账户时需授权文件 |

### 4.3 开户流程

```
商家入驻审核通过
   ↓
商家在 PC 端 > 财务钱包 > 订单结算账户 > [申请开通]
   ↓
确认收款银行账户(从已提交资料中选择,或新增)
   ↓
提交申请 → 运营端财务审核
   ↓
运营端财务审核:
  ✓ 收款账户户名与商家主体一致
  ✓ 法人身份证有效
  ✓ 收款账户授权完整
   ↓
  ├─ 通过 → 创建 merchant_settlement_account + 通联子台账
  │         → 商家端通知:"订单结算账户已开通"
  └─ 驳回 → 通知商家补资料
```

### 4.4 开户接口

| 接口 | 用途 |
|---|---|
| `POST /api/v1/merchant/settlement-account/apply` | 商家发起开户申请 |
| `GET /api/v1/admin/settlement-account/applications` | 财务查看待审申请 |
| `POST /api/v1/admin/settlement-account/:id/approve` | 财务审核通过(同步创建通联子台账)|
| `POST /api/v1/admin/settlement-account/:id/reject` | 财务驳回 |
| `GET /api/v1/merchant/settlement-account` | 商家查看自己的账户 |

## 5. 入账触发(按合作模式分别处理)

### 5.1 总览:三种合作模式的入账触发

| 合作模式 | 入账类型 | 触发时点 | 资金来源 | 入账金额 |
|---|---|---|---|---|
| **联营订单** | `order_settlement` 订单结算款 | 监管锁回调成功 | 平台联营出资款(对应平台出资比例)| 设备价 × 平台出资比例 |
| **受让订单** | `order_settlement` 订单结算款 | 监管锁回调成功 | 平台支付的应收账款转让对价 | 设备价 × 100% |
| 三种模式 | `monthly_split` 月度分账 | 客户每期租金支付成功 | 客户租金 - 平台服务费,按合作模式分账 | 按分账规则计算(详见 §6)|
| 联营订单 | `joint_venture_profit` 联营分润 | 订单完成结算 | 客户支付的"租金溢价",按出资比例分配 | 按出资比例(并入月度分账)|

### 5.2 触发链路(以联营订单 / 受让订单的初始结算款为例)

```
客户 [确认收货] 完成
   ↓ 订单 PENDING_RETURN
商家给设备上锁
   ↓ 监管锁系统 Webhook 回调
平台监管锁服务收到回调
   ↓
1. 更新 lock_status = LOCKED
2. 根据订单 cooperation_mode 判断:
   ├─ self_operate(自营订单)→ 不触发入账
   ├─ joint_venture(联营订单)→ 触发"应收账款转让对价(联营出资比例部分)"穿透
   └─ receivables_assignment(受让订单)→ 触发"应收账款转让对价(全额)"穿透
   ↓
进入资金穿透流程(详见 §5.3)
```

### 5.3 资金穿透流程(实时,< 1 分钟)

```
步骤 1:平台对公账户出资(自有资金或受让方资金)
   平台对公账户 +¥X(若是受让方资金,先从受让方收款)
   ↓
步骤 2:平台对公账户实时充值通联备付金
   通联备付金账户 +¥X
   备付金子台账:门店 +¥X
   ↓
步骤 3:门店结算账户余额更新
   merchant_settlement_account.balance_available +¥X
   ↓
步骤 4:写流水
   settlement_entry:entry_type=order_settlement, amount=¥X
   platform_passthrough_log:direction=out, amount=¥X
   ↓
步骤 5:商家端 IM 推送
   "订单结算款 ¥X 已到账"
```

**关键校验**:
- 必须先有"平台对公账户实际收到 X 元"的银行流水
- 才能触发"通联备付金充值 X 元"
- 严格 1:1 等额,**绝不允许平台对公账户长期沉淀**

### 5.4 入账金额计算(按合作模式)

#### 联营订单

```python
# 平台联营出资款 = 设备价 × 平台出资比例
amount = device_price × platform_funding_ratio
```

例:设备价 ¥5,000,门店出资 50% / 平台出资 50%
→ 平台支付 ¥2,500 到门店结算账户(应收账款转让对价)
→ 门店自己也需出 ¥2,500 作为出资,但这部分不经过平台,门店自行筹措

#### 受让订单

```python
# 应收账款转让对价 = 设备价 × 100%
amount = device_price × 1.0
```

例:设备价 ¥5,000,平台 100% 受让
→ 平台支付 ¥5,000 到门店结算账户

#### 自营订单

不触发 `order_settlement` 入账(门店自己出钱采购设备,平台只在客户每月支付时做月度分账)。

### 5.5 不触发的场景

| 场景 | 处理 |
|---|---|
| 自营订单 | 不触发 order_settlement(只触发 monthly_split)|
| 订单已撤单 | 反向冲正:settlement_entry status=reversed,门店结算账户余额扣减 |
| 门店结算账户未开通 | 进入异常队列,提醒门店先开通 |
| 门店结算账户被冻结 | 暂挂入账,门店解冻后补打 |

### 5.6 异常兜底

| 异常 | 处理 |
|---|---|
| 监管锁系统超时未回调 | 客服催商家上锁(详见 08 §7.5)|
| 上锁回调到达但订单状态不对 | 异常队列,主管核实 |
| 平台对公账户已收钱但通联充值失败 | 重试 + 告警;3 次失败后人工介入 |
| 通联充值成功但子台账更新失败 | 系统自动重试 + 对账兜底 |
| 重复回调 | 系统幂等校验,只处理首次 |

## 6. 客户每期租金的分账处理 ⭐ 核心算法 C

### 6.1 分账总流程

```
客户支付 ¥500(月供)
   ↓
[步骤 1] 进平台对公账户(过路)
   ↓
[步骤 2] 按合同口径拆分:
   ├─ 平台综合服务费 ¥10(留对公户,平台收入)
   └─ 设备使用费部分 ¥490(应收账款回款)
       ↓
[步骤 3] 设备使用费部分穿透通联备付金,按合作模式分账:
   ↓
   ├─ 自营订单:¥490 全部进门店结算账户
   ├─ 联营订单(门店 50% / 平台 50%):
   │    ├─ 门店 50% × ¥490 = ¥245 → 门店结算账户
   │    └─ 平台 50% × ¥490 = ¥245 → 通联备付金子台账(平台台账,后续可代付出去给受让方/联营方)
   └─ 受让订单:¥490 全部进通联备付金子台账(平台台账)
   ↓
[步骤 4] 写流水 + 推送
```

### 6.2 三种合作模式的分账规则

#### 自营订单(self_operate)

| 客户付 | 平台服务费 | 应收账款回款部分 | 给门店 | 给平台 |
|---|---|---|---|---|
| ¥500 | ¥10(留对公户)| ¥490 | ¥490(穿透到结算账户)| ¥10(服务费收入)|

#### 联营订单(joint_venture,以 50/50 为例)

| 客户付 | 平台服务费 | 应收账款回款部分 | 给门店 | 给平台 |
|---|---|---|---|---|
| ¥500 | ¥10(留对公户)| ¥490 | ¥245(50% × ¥490,穿透到结算账户)| ¥10(服务费)+ ¥245(50% × ¥490,平台台账,作为联营出资回收+分润)|

#### 受让订单(receivables_assignment)

| 客户付 | 平台服务费 | 应收账款回款部分 | 给门店 | 给平台 |
|---|---|---|---|---|
| ¥500 | ¥10(留对公户)| ¥490 | ¥0(门店应收账款已转让)| ¥10(服务费)+ ¥490(平台台账,应收账款受让回款)|

### 6.3 关键字段

订单创建时锁定:

```sql
order
- cooperation_mode              -- self_operate / joint_venture / receivables_assignment
- merchant_funding_ratio        -- 门店出资比例(联营)/ 100%(自营)/ 0%(受让)
- platform_funding_ratio        -- 平台出资比例(联营)/ 0%(自营)/ 100%(受让)
- platform_service_fee_rate     -- 平台综合服务费率(下单时锁定快照)
- device_price                  -- 设备价
- total_rent                    -- 客户租金总额
```

### 6.4 平台综合服务费率的来源

| 合作模式 | 默认费率 | 后台可配 |
|---|---|---|
| 自营订单 | 客户租金的 2% | 是 |
| 联营订单 | 客户租金的 2% | 是 |
| 受让订单 | 客户租金的 2% | 是 |

**说明**:服务费率统一(都是 2%),不同合作模式的差异体现在"应收账款回款部分"的分账逻辑上,不体现在服务费率上。

### 6.5 分账失败 / 异常

| 场景 | 处理 |
|---|---|
| 平台对公账户已收客户钱,通联穿透失败 | 重试 + 告警 + 资金标记"待穿透" |
| 客户支付金额与应收金额不符 | 进入对账异常队列 |
| 订单已撤单但客户继续付款 | 系统阻止扣款 / 自动退款 |
| 部分支付(客户只付了一部分)| 按比例拆分服务费 + 应收账款部分,分别处理 |

## 12. 关键时序

### 12.1 完整时序(以联营订单 50/50 为例)

```
T0    客户确认收货 → 订单 PENDING_RETURN
T0+   商家给设备上锁
T1    监管锁系统 Webhook 回调
T1+1s 系统判断 cooperation_mode = joint_venture
      计算 amount = device_price(¥5,000) × platform_funding_ratio(50%) = ¥2,500
T1+1s 创建 settlement_entry(状态 pending)
T1+2s 平台对公账户已收到联营出资方 ¥2,500(若提前到账则跳过)
T1+5s 调用通联接口充值 ¥2,500 到备付金账户(门店子台账)
T1+30s 通联回调充值成功
T1+30s settlement_entry status=completed
       merchant_settlement_account.balance_available += ¥2,500
T1+30s 商家端推送:"订单结算款 ¥2,500 已到账"

(后续 1~12 个月,客户每月付租金 ¥500)
T2    客户支付 ¥500 → 平台对公账户
T2+1s 系统按合同拆分:平台服务费 ¥10 + 设备使用费 ¥490
      ¥10 留对公户(写 platform_passthrough_log,platform_commission=¥10)
      ¥490 → 通联充值
T2+5s 通联充值 ¥490 成功
T2+5s 系统按联营订单分账规则:门店 50% × ¥490 = ¥245
      门店子台账 += ¥245(merchant_settlement_account.balance_available += ¥245)
      平台子台账 += ¥245(后续可代付给联营出资方)
T2+5s 写 settlement_entry(entry_type=monthly_split, amount=¥245)
T2+5s 商家端推送:"月度分账 ¥245 已到账"
```

### 12.2 异常时序(穿透失败)

```
T0    客户支付 ¥500 → 平台对公账户
T0+1s 系统计算穿透金额 ¥490
T0+5s 通联充值失败(网络异常)
T0+5s 系统重试(1 分钟内 3 次)
T0+90s 仍失败 → 进入异常队列 + 财务告警
T0+   财务主管核实,选择:
       ├─ 重试通联接口
       ├─ 人工充值(留痕)
       └─ 退回客户(走退款流程)
```
