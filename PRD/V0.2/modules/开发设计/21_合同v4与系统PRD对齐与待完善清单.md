## 2. 必须完善的差异点(开发前处理)

### 🔴 2.1 申请购买价 / 当期参考价 计算口径(最高优先级)

**冲突描述**:
- **合同 v4**:申请购买价 = 当期参考价 = 设备折旧余值 = `max(设备价值 × 残值率, 设备价值 − (设备价值 ÷ 折旧月数) × 已使用月数)`。挂"设备折旧",不挂"已付租金"。
- **办单助手02 §1.3 现状**:当期留购价 = `剩余应付租金 + 保证金`(即 `monthly × 剩余期数 + deposit`)。挂"剩余租金"。
- **C端12 §4.2 现状**:申请购买金额 = "价格方案中的购买申请金额 / 内部 retain 计算结果"(未指明口径,实际取自办单助手的剩余租金口径)。

**为什么必须改**:这是合同 v4 与融资租赁切割的核心。"购买价=剩余租金"在法院眼里=分期还本金=名为租赁实为分期;合同已统一改为"按折旧余值定价,与已付租金无固定勾稽"。若系统仍按剩余租金算,则**产品与合同自相矛盾**,合同的去金融化努力作废。

**要做的修改**:
1. 办单助手02 §1.3「当期留购价」列,**计算公式从 `剩余租金 + 保证金` 改为 `设备折旧余值 = max(设备价值×残值率, 设备价值 − 每月折抵额×已使用月数)`**;术语从"留购价"改为"当期参考价"。
2. 办单助手02 §1.2 主公式区:`buyoutTotal 留购总价` 的概念随之调整或移除(购买不再按"保证金+全部期租金"累计,而是按申请时点折旧余值单点计算)。
3. C端12 §4.2「申请购买金额计算」明确为:`申请购买价 = 申请时点设备折旧余值;实付 = 折旧余值 − 保证金抵扣`。
4. C端02 §2.4「申请购买参考金额」、§2.5 价格双层显示,数据源指向折旧余值口径。
5. **账单的"本期应付款"不变**(仍是租金,按现有费率公式),只改"当期参考价/购买价"这一列的口径。账单与购买价是两套独立的数,务必分开。

> ⚠️ 注:折旧余值"按设备折旧不按已付租金"这一口径本身能否被当地法院认可、残值率与折旧月数取值是否合理,**属合同层,待律师确认**(见合同 v4 待确认清单)。系统侧先按此口径实现为可配置参数 + 写死公式结构;若律师最终调整口径,改配置/改公式结构版本即可。

### 🔴 2.2 删除"名义购买价1元"

**冲突描述**:
- 合同 v4 已**删除**"最后一期名义购买价1元",改由"残值保底(设备价值×残值率)"保证折旧余值不归零,从而不出现0元购买。
- 系统现状仍保留:办单助手02 §1.4「名义留购费默认1元」、§3.1⑨「名义留购费」配置项、§11.3 默认值"名义留购费1元"、全局13 §1「名义留购费(1)」。

**要做的修改**:
1. 办单助手02 §1.4 整节改写:最后一期不再用"保证金+名义留购费",而是与其他期一致,用"设备折旧余值(此时通常已等于残值保底)"。
2. 办单助手02 §3.1⑨「名义留购费」配置项移除(或保留字段但默认 0 且不参与购买价计算,并注明已废弃)。
3. 办单助手02 §11.3、全局13 §1 中"名义留购费(1)"删除或标注废弃。
4. 全局13 §5 对照示例中"留购=押金+全部期租金"这条**写死口径需更新**为新的折旧余值口径(见 §2.8)。

### ⚠️ 2.3 平台服务费客户侧展示

**合同 v4**:客户侧费用只有"设备使用费 + 增值服务费",平台服务费向门店收(11.2「乙方无需向丙方另行支付平台服务费」)。
**系统现状**:办单助手账单的"每期应付"由费率公式算出一个综合月付,未拆出/未标注"平台服务费";C端02 价格方案区列了"服务费:¥XXX"。

**要做的修改/确认**:
1. 确认 C端02 §2.4 价格方案区的"服务费:¥XXX"是否要从客户侧移除或改名——按合同 v4,客户侧不应出现"平台服务费"。客户侧总额应表述为"设备使用费(+增值服务费)"。
2. 平台服务费的收取与结算放门店侧(财务/门店结算/软钱包),不在客户账单体现。
3. **客户侧到底显示几个费用项、月付如何向支付宝及法院解释为"设备使用费"而非"分期"——属合同层,待律师 + 支付宝渠道对接人确认**(合同 v4 待确认清单第2、3条)。系统先保证"客户侧不出现平台服务费字样",具体展示项待确认后定。

### ⚠️ 2.4 折旧参数进配置中心 + 公式结构写死

**要做的修改**:
1. 配置中心新增两个可配置参数(遵循全局13 §1 费率/金额类 + §3 配置机制):
   - `depreciation_months` 折旧月数,默认 12
   - `residual_rate` 残值率,默认 30%
   - 二者支持按品类(手机/电动车/二手)覆盖(全局13 维度覆盖类)。
2. **折旧余值公式结构写死**(全局13 §2:公式结构不可配):`设备折旧余值 = max(设备价值×残值率, 设备价值 − (设备价值÷折旧月数)×已使用月数)`;`已使用月数`不足一个月按一个月计;全程按分整数运算。
3. 订单生成时快照当时的 `depreciation_months` / `residual_rate`(全局13 §3.2 快照绑定),已下单订单不随配置变动。

### ⚠️ 2.5 最低服务期:按品类 + 下单页展示留痕

**合同 v4 5.1/5.2**:最低服务期可按品类设(全新设备默认3个月),且"最低服务期及期内应结清费用,应在下单页显著展示并经乙方确认,留痕保存"。

**要做的修改/确认**:
1. 确认办单助手配置 / 商品配置中,最低服务期可按品类(手机/电动车/二手/全新/成色)配置(参数可配,默认全新3个月)。
2. 确认 C端02 订单确认页有"最低服务期 X 个月 + 期内提前归还需结清范围"的显著展示,并纳入客户勾选确认与 `order_snapshot`(现 02 §2.6 服务说明列了"服务期内可申请…",需补"最低服务期"专项展示 + 留痕)。
3. C端12 §5.1 提前归还预结算已支持"不默认收全部未到期",与合同5.3/5.4 一致,无需改。

### ⚠️ 2.6 归还检测报告凭证字段

**合同 v4 13.4/13.7**:维修/检测/物流/拖车/仓储等费用"以实际发生金额及凭证为准",并列了凭证类型(检测报告、维修发票、门店维修单、物流凭证、照片、视频等)。
**要做的修改**:确认归还检测报告数据结构包含这些凭证上传字段(检测报告、维修报价/发票/收据、门店维修单、物流/拖车/仓储凭证、照片、视频);电动车"定位取回协作费200元/次"需有取回记录/定位轨迹/凭证字段(合同13.5)。

### ⚠️ 2.7 设备无法返还赔偿计算

**合同 v4 14.4**:赔偿 = `设备折旧余值 + 已到期未付应付款 + 逾期违约金 + 取回/检测/维修/物流仓储等实际费用 − 履约保证金抵扣`。
**系统现状**:订单管理08 §0.5/§8 有逾期、坏账、回收完成、保证金抵扣,但"设备无法返还"场景的赔偿额没有落为按 14.4 公式的可算逻辑。
**要做的修改**:在坏账/法诉/回收模块,把"设备无法返还赔偿额"做成按 14.4 公式自动计算(复用 §2.4 的折旧余值)+ 实际费用按凭证累加 + 保证金抵扣;保留计算快照(合同14.4 与系统"金额计算必须保留快照"一致)。

### 🟡 2.8 全局可配置规范 §5 示例口径更新

全局13 §5 典型对照示例中,办单助手一行写"留购/月付/账单公式结构"为写死口径,但举例隐含旧的"留购=押金+全部期租金"。需把该写死口径示例更新为新的"当期参考价=设备折旧余值(残值保底)"口径,避免开发按旧示例实现。

## 5. 必须明确的边界(给开发与老板)

1. **§2.1 / §2.2 是开发前必须先改的口径冲突**,不改会导致系统与合同自相矛盾。其余多为补字段/补展示/补留痕,不阻塞主框架开发。
2. **本清单只解决"系统与合同 v4 对齐"**;合同 v4 本身仍是草案,折旧口径、残值率/折旧月数取值、平台服务费收门店的合规解释、账单列名是否被支付宝接受、最低服务期结清范围是否过重——**这些属合同/法务/渠道层,待律师 + 支付宝对接人确认**,系统先按 v4 草案口径实现为可配置参数,确认后调参或调公式结构版本即可。
3. 折旧余值口径"挂折旧不挂已付租金"是系统实现的硬约束(写死公式结构);残值率30%、折旧月数12是可配数值(默认值)。
4. 本清单为系统侧对齐说明,非法律意见。
