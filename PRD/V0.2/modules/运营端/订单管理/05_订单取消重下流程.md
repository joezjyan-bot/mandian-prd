# 订单取消重下流程

> **⚠️ V0.2 Stage 6 同步修订(2026-05-27)v1.1**:
> - 本文档仅保留历史参考；撤单、重下和补充合同闭环以 `10_订单撤单与补充合同.md` v2.0 为准。
> - 本次仅同步展示用语:商家订单 / 联营订单 / 平台订单 / 履约中 / 逾期费用;底层字段、接口、枚举不变。

> **来源**: P1-3
> **核心**: 已签合同后任何价格变更走此流程

## 1. 页面定位

运营端 · 订单管理 · 订单详情 · 取消重下入口。已签合同后,如需价格变更/客户需求变更,通过此流程实现。

## 2. 适用场景

- 已签合同 + 客户/门店协商需调价
- 已签合同 + 客户需调整规格/期数
- 已发货/在租 + 客户协商需调整(罕见,需慎重)

## 3. 业务原理

V1 不做"原订单改价",而是:
1. 关闭原订单(走 03 关闭流程 + 退款单)
2. 基于新条件创建新订单
3. 新订单链路追溯到原订单

## 4. 发起人

仅客服(操作复杂、涉及资金)

## 5. 操作流程

```
1. 客服在原订单详情点击"取消重下"
2. 弹窗收集信息:
   - 取消原因(下拉)
   - 是否立即创建新订单(选项)
   - 新订单初步信息(可选,稍后调整)
3. 客服确认 → 走 03 订单关闭流程:
   - 客户已签合同,需 1.5% 费用(若 7天内)
   - 退款单生成(责任划分)
   - 责任方钱包扣减(可能进 pending_topup)
   - 合同作废
   - 设备释放
4. 退款完成 + 设备释放后,客服在系统中创建新订单:
   - 基于原订单信息预填(客户/商品/门店保持)
   - 新价格方案
   - 新订单关联原订单 ID
5. 新订单走完整办单流程:
   - 客户重新签合同(新订单/新模板)
   - 新一次支付/分账
   - 新设备锁定/发货流程
```

## 6. 订单关联追溯

订单字段:
- `replaces_order_id`:新订单填,指向被取消的原订单
- `cancelled_and_replaced_by`:原订单填,指向新订单

订单详情显示:
- 原订单显示"已取消并替换为订单 XX"
- 新订单显示"由订单 XX 取消重下"

## 7. 风险与边界

### 7.1 已发货/在租取消重下
- 谨慎处理:需协商客户退货
- 设备需归还 + 验收 + 入库
- 客户已使用期数的费用按实际计算

### 7.2 已分账资金的处理
- 已分账给门店/资方的资金:走退款单从对应钱包扣
- 余额不足:进 pending_topup,业务锁锁定

### 7.3 监管锁
- 原订单监管锁:退款扣减未完成时继续锁
- 新订单监管锁:单独申请

## 8. 报表追踪

订单管理列表 + 数据报表可以:
- 筛选"已取消重下"的订单
- 统计取消重下比例(健康度指标)
- 关联追溯

## 9. 权限 & 日志

- 仅客服可发起
- 完整日志记录(谁/原因/原订单/新订单)
- 退款单 + 新订单的所有动作各自留日志

## 10. 数据模型

- `order.replaces_order_id`
- `order.cancelled_and_replaced_by`
- 关联 `refund_request`(原订单退款)
