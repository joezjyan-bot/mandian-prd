## 1. 给 Claude 的任务说明

请你完整阅读 `mandian-prd/PRD/V0.2` 目录下的所有文档，不要只看 README 或单个模块。阅读目标不是马上写 UI，也不是自由发挥新系统，而是先建立对满点租赁系统重构的完整理解，并找出缺口。

这套系统的核心不是普通商城租赁，而是围绕门店、商家、平台、资方、渠道、客户之间的订单、资金、风控、合同、发货、分账、租后全链路重构。

2. 三类订单 `门店订单 / 分红订单 / 平台订单` 的区别和主流程。

4. 目前 PRD 还缺的模块、页面、字段、状态、流程。
5. 哪些地方存在冲突、模糊或需要用户确认。
6. 后续开发建议按什么顺序拆。
7. 如果要做 UI 原型，应优先做哪些页面，不能自由发挥哪些地方。

## 2. 必须先理解的业务边界

### 2.1 系统角色

| 角色 | 说明 |
|---|---|
| 运营平台 | 总后台，负责平台规则、审核、资方、链路、财务、渠道、租后、权限、数据 |
| 商家 PC 端 | 入驻商家/门店老板的 PC 管理后台，可管理自己的商品、门店订单、钱包、员工等 |
| 门店手机端 | 文档里也称门店端/商家手机端，核心是办单助手、订单待办、交付上传、扫码下单 |
| C 端客户 | 扫码下单、实名授权、扫脸、签合同、公证、支付、签收、还款、归还 |
| 渠道端 | 推广门店入驻，查看推广商家、分红订单/平台订单统计、佣金和提现 |
| 资方端 | 后续可做，先由运营端主控资方管理、订单分配、账单和资金账户 |
| 中控台 | 租赁系统外部系统，不和租赁系统混为一体；主要提供设备/风控/条数等能力对接 |

### 2.2 三类订单

| 订单类型 | 说明 | 审核主体 | 资金/分账口径 |
|---|---|---|---|
| 门店订单 | 门店自有订单，门店自己审核和经营 | 门店/商家 | 平台可抽佣，剩余进入门店钱包 |
| 分红订单 | 门店出设备或部分资金，资方补足剩余资金 | 运营平台 | 按门店/资方出资比例分账，再分别扣平台抽佣 |
| 平台订单 | 门店完全推给平台，由平台匹配资方全额出资 | 运营平台 | 平台/资方主控收益，渠道可分佣，门店不主控资金 |

重要：订单类型只使用“门店订单 / 分红订单 / 平台订单”，后续不要再引入其他正式订单类型。

### 2.3 核心链路

```text
商品配置
→ 是否同步办单助手
→ 门店订单/分红订单/平台订单三入口
→ 办单助手计算价格和账单
→ 生成二维码
→ 客户扫码下单
→ 实名、扫脸、授权、资料上传
→ 后台待审核
→ 分配资方（分红/平台订单）
→ 客服审核、改套餐、补资料
→ 发起合同、公证、支付/代扣/收款码
→ 发货/门店交付/签收照片
→ 账单、还款、部分支付
→ 分账、抽佣、渠道佣金、钱包
→ 租后、逾期、催收、归还、买断
```

## 4. 阅读时必须检查的问题

### 4.1 总体一致性

| 检查项 | 目标 |
|---|---|
| 订单类型是否统一 | 全部文档必须统一为门店订单、分红订单、平台订单 |
| 门店/商家概念是否统一 | 门店入驻后就是商家主体，手机端和 PC 端是同一主体不同端 |
| 中控台边界是否清楚 | 中控台是外部能力系统，不要写成租赁系统内部模块 |
| UI 风格是否统一 | 默认参考 finance-zhongtai 企业中台风格 |
| 敏感动作边界是否清楚 | 审核、退款、打款、关闭订单、改价、分账必须有权限和日志 |

### 4.2 订单链路完整性

逐一检查三类订单是否都能从下单走到完结：

| 阶段 | 检查内容 |
|---|---|
| 商品 | 商品全新/二手规格、长租/短租、是否同步办单助手 |
| 办单助手 | 三入口、惠讯计算器逻辑、增值服务、账单、留购、二维码 |
| 客户下单 | 扫码、实名、扫脸、授权、资料、合同、公证、支付 |
| 审核 | 待审核、补资料、改价、分配资方、联系客服、风控报告 |
| 发货/交付 | 默认门店发货、线上快递可选、照片证据、设备码 |
| 财务 | 账单、部分支付、代扣、分账、抽佣、钱包、提现 |
| 租后 | 逾期、临近逾期、催收、外部系统对接、回款 |
| 完结 | 归还、买断、退款、关闭、操作日志 |

### 4.3 模块缺口

阅读后请明确哪些模块还缺页面级 PRD。重点检查：

1. 商品管理与商品审核。
2. 办单助手三入口详细页面。
3. 订单详情页、待审核页、资方分配页。
4. 客服 IM 和订单信息推送。
5. 合同、公证、授权、支付的状态详情。
6. 钱包、提现、对账、结算打款。
7. 资方管理、资方资金账户、资方账单。
8. 商家 PC 端商品、订单、员工、钱包。
9. 门店手机端办单助手、待办、员工扫码办单。
10. C 端客户订单详情、授权、签署、支付、归还。
11. 客诉管理、支付宝投诉接口。
12. 黑名单库、监管锁配置。
13. 权限、角色、操作日志。
14. 操作文档/培训手册。

## 5. 后续开发拆分建议

开发不要从 UI 随机开做，建议按“数据模型和主链路优先”的顺序。

### 5.1 第一阶段：业务模型和基础框架

| 优先级 | 内容 |
|---|---|
| P0 | 用户、角色、权限、端类型、登录态 |
| P0 | 商家/门店主体、员工账号、渠道来源 |
| P0 | 商品、规格、全新/二手、长租/短租、同步办单助手 |
| P0 | 三类订单基础模型：门店订单、分红订单、平台订单 |
| P0 | 操作日志、状态机、附件/图片证据模型 |

### 5.2 第二阶段：办单和下单主链路

| 优先级 | 内容 |
|---|---|
| P0 | 办单助手三入口 |
| P0 | 惠讯计算器逻辑迁移和配置化 |
| P0 | 二维码下单、客户订单创建 |
| P0 | 实名、扫脸、资料上传、授权状态 |
| P0 | 待审核列表、订单详情、补资料、改价 |

### 5.3 第三阶段：合同、支付、发货、分账

| 优先级 | 内容 |
|---|---|
| P0 | 合同/公证发起和状态回调 |
| P0 | 支付、代扣、部分支付收款码 |
| P0 | 默认门店发货、快递接口预留、交付照片 |
| P0 | 账单、抽佣、分红、平台订单分账、钱包 |
| P1 | 渠道佣金、资方收益、提现和对账 |

### 5.4 第四阶段：运营增强和外部能力

| 优先级 | 内容 |
|---|---|
| P1 | 链路配置中心 |
| P1 | 商家接口计费 |
| P1 | 渠道 H5 和佣金管理 |
| P1 | 租后催收、临近逾期、外部催收对接 |
| P1 | 黑名单库、监管锁配置 |
| P1 | 客诉管理和支付宝投诉接口 |

### 5.5 第五阶段：操作文档和培训交付

产品文档定稿后，需要补一套类似无界租的操作文档：

| 版本 | 对象 | 内容 |
|---|---|---|
| 内部客服运营版 | 公司内部客服、审核、租后、财务 | 查单、审核、改价、发合同、公证、部分收款、分配资方、租后 |
| 商家/门店培训版 | 门店老板、店员 | 入驻、商品、办单助手、出码、交付照片、订单、钱包、员工账号 |
| 渠道/资方/部署交付版 | 渠道、资方、外部部署客户 | 渠道推广、佣金、资方账单、系统配置、常见问题 |

## 7. 禁止事项

1. 不要再引入三类订单之外的正式订单类型。
2. 不要把中控台和租赁系统混为一个系统。
3. 不要把门店订单写成平台审核主控。
4. 不要省略资金、分账、钱包、对账、操作日志。
5. 不要做只有文字说明、没有页面结构和字段状态的 UI。
6. 不要自由发挥新风格，UI 默认参考 finance-zhongtai 企业中台风格。
7. 不要把无界租旧系统逻辑原样复制，要按满点新系统口径重构。
8. 不要在文档里写入真实订单号、手机号、身份证号、账号、密钥等敏感信息。
