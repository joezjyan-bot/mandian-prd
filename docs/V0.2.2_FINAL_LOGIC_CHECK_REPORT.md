# 满点租赁系统 V0.2.2 最终逻辑检查报告

> 检查目标: 站在真实上线运行视角,验证长租、短租、财务、权限、法诉证据链是否闭环。

## 一、长租链路

**链路**: 客户下单 → 资料提交 → 信用/信息授权 → 风控审核 → 合同签署 → 首期/保证金支付 → 商家交付 → 客户签收 → 监管锁校验 → 平台结算 → 账单生成 → 履约中 → 正常结清 / 留购或提前结清 / 逾期 / 法诉 / 坏账核销。

| 检查点 | 结果 | 说明 |
|---|---|---|
| 流程断点 | 已收口 | 08 长租全生命周期已补齐审核驳回、补资料、签约失败、首付失败、拒签、锁失败、打款失败、退款、法诉、坏账回收 |
| 客户付钱但订单不继续 | 已覆盖 | 支付流水、订单业务账、补偿任务和支付回调幂等已进入 OpenAPI/ERD/测试 |
| 商家已交付但不能结算 | 已覆盖 | 结算三条件缺一进入禁止打款或人工复核 |
| 签收/锁车/激活锁矛盾 | 已覆盖 | 监管锁失败状态和人工复核路径已纳入状态字典 |
| 订单关闭后账单继续跑 | 已覆盖 | 账单关闭、冲正、退款、坏账核销状态已补齐 |
| 退款后返点未回滚 | 已覆盖 | 退款冲正和返点回滚进入财务规则与测试矩阵 |
| 法诉后回款无法入账 | 已覆盖 | 坏账回收可回原订单操作完成,产生新支付/总账分录并保留历史核销记录 |

**结论**: 长租链路可进入研发任务拆分。无监管锁品类仍需按老板确认清单落成制度和验收配置。

## 二、短租链路

**链路**: C 端体验租入口 → 选择城市/门店/车型/租期 → 下单 → 商家接单 → 保证金处理 → 核销绑车 → 取车确认 → 用车中 → 续租/超时 → 还车预约 → 还车验收 → 保证金扣款/退还 → 事故/违章/损坏/丢车处理。

| 检查点 | 结果 | 说明 |
|---|---|---|
| 是否只有入口 | 已补齐 | 短租业务 01-11 已覆盖 C 端、商家端、运营端、数据模型、骑行人、保证金、控车、续租、逾期、风险责任 |
| 保证金冻结/扣款/退还 | 已覆盖 | 07 和 11 文档要求证据、审批、通知、申诉、流水 |
| 事故/违章/损坏/丢车 | 已覆盖结构 | 责任按保险、协议、证据划分;细则需运营/法务制度落地 |
| 客服介入 | 已覆盖 | IM 客服和异常工单承接争议 |
| 超时费和续租 | 已覆盖 | 09 续租链和 10 逾期 SLA 对应订单/账单状态 |
| 证据留存 | 已覆盖 | 取车、还车、控车、定位、照片、扣款、IM 记录进入证据链 |

**结论**: 短租 V1 已从“预留”变成可评审主链路。事故/违章/损坏/丢车的细则不再是文档结构缺口,但属于上线制度和法务审核重点。

## 三、财务链路

**链路**: 客户支付 → 支付流水 → 订单业务账 → 服务费 → 商家结算 → 钱包余额 → 提现 → 提现失败回滚 → 退款 → 冲正 → 返点回滚 → 坏账核销 → 坏账回收。

| 检查点 | 结果 | 说明 |
|---|---|---|
| 四账能否对上 | 已覆盖 | 06 财务主文档和 08 总账分录将业务账、支付流水、钱包账、总账分录拆开 |
| 重复打款 | 已覆盖 | 打款幂等键、结算三条件、审批和审计日志进入 OpenAPI/ERD/测试 |
| 重复返点 | 已覆盖 | 返点记录、幂等、退款关闭按比例回滚进入财务规则 |
| 退款未冲正 | 已覆盖 | 退款必须记录责任方、冲正对象、流水和审批 |
| 提现失败余额未回滚 | 已覆盖 | 提现失败回滚进入钱包规则和测试断言 |
| 法诉回款但坏账未恢复 | 已覆盖 | 坏账回收生成新流水和总账分录 |
| 多维对账 | 已覆盖 | 按订单、客户、商家、日期、支付渠道、业务员、推广员对账 |

**结论**: 财务逻辑已具备开发评审条件。科目编码、报表格式和供应商通道字段属于财务/技术执行阶段。

## 四、权限链路

**链路**: 角色创建 → 操作员绑定角色 → 操作权限 → 审批权限 → 高危操作 → 操作日志 → 审计日志 → 异常追责。

| 检查点 | 结果 | 说明 |
|---|---|---|
| 技术越权改业务数据 | 已设红线 | 技术账号不得自行决定业务规则,强制变更需审计和授权 |
| 客服改金额 | 已收口 | 仅账单生成前可按权限改订单金额,账单生成后不得直接改 |
| 财务单人退款/打款 | 已收口 | 高危资金动作需审批,必要时二人复核 |
| 商家越权查看 | 已收口 | 商家只看自身订单、钱包和授权数据 |
| 推广员看敏感资料 | 已收口 | 推广员只看归因和佣金,不得看客户完整敏感资料 |
| 法诉导出 | 已收口 | 法务主导,导出必须留痕,大批量需授权 |

**结论**: 权限矩阵已能进入研发拆点。具体系统角色可由技术按权限点实现,但资金/法务高危动作不能由技术自行放宽。

## 五、法诉证据链

**链路**: 订单信息 → 实名资料 → 合同 → 签收确认 → AI 视频确认 → 交付证据 → 设备价值证明 → 监管锁记录 → 账单 → 支付流水 → 催收记录 → IM 记录 → 法诉导出。

| 检查点 | 结果 | 说明 |
|---|---|---|
| 证据散落 | 已收口 | 法诉证据包要求按订单归档和一键导出 |
| 导出留痕 | 已覆盖 | 导出人、时间、范围、用途、审批、文件哈希需记录 |
| 设备价值证明 | 已覆盖 | 设备型号、编码、指导价/价值证明、交付证据进入证据包 |
| 客户本人确认 | 已覆盖 | 电子确认单和 AI 问答签收可配置 |
| 客户知晓金额和协议 | 已覆盖 | 合同、确认单、账单、支付记录、IM 记录共同证明 |

**结论**: 法诉证据链已形成闭环。合同盖章版、AI 问答最终问题和证据模板需要法务上线前终审。

## 六、仍存在的风险

| 风险 | 等级 | 处理 |
|---|---|---|
| 无监管锁品类替代验收如果配置不严,可能导致结算证据不足 | P1 | 按老板确认清单落地配置和测试 |
| 短租事故/损坏/丢车责任细则需要法务和保险条款匹配 | P1 | 上线前法务/保险专项确认 |
| 外部供应商字段可能与 PRD 适配层不完全一致 | P1 | 保持 adapter 层,联调后固化字段 |
| 历史归档文档仍可能被新成员误读 | P2 | README 和文档地图已降级,研发 onboarding 必须先读当前入口 |
