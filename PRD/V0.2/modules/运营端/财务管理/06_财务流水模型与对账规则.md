# 财务流水模型与对账规则

> 所属端：运营端 / 财务管理
> 目标：把支付、账单、分账、钱包、提现、退款、冲正、对账拆成可开发的数据对象，确保钱流可追溯、可校验、可回滚。

> **⚠️ V0.2 修订(2026-05-26)v1.1**:
> - §1 新增 procurement_inflow / procurement_withdrawal 两类流水对象
> - 新增 §11 采购账户流水模型(对接 `财务管理/07_采购账户与软钱包架构.md`)
> - §8 三边对账新增"采购账户边"对账规则
> - §9 异常处理新增采购款打款失败处置

---

## 1. 财务对象

| 对象 | 作用 |
|---|---|
| 账单 | 客户应付计划和实付状态 |
| 支付流水 | 客户通过通道支付或代扣的结果 |
| 分账流水 | 把实收金额拆给商家、资方、渠道、平台 |
| 钱包流水 | 各主体账户余额变化 |
| 提现单 | 主体从钱包提现到收款账户 |
| 打款流水 | 打款通道或线下打款结果 |
| 退款单 | 客户退款申请和财务处理 |
| 冲正单 | 已入账金额的反向调整 |
| 对账单 | 通道、业务、钱包三边核对结果 |
| **采购款入账流水(procurement_inflow)** ⭐ | 监管锁回调触发,平台代资方付款到商家采购账户 |
| **采购账户提现单(procurement_withdrawal)** ⭐ | 商家从采购账户自助提现 |

---

## 2. 账单模型

字段：

- `bill_id`
- `order_id`
- `period_no`
- `bill_type`：首期 / 租金 / 服务费 / 公证费 / 留购 / 补差价
- `amount_due`
- `amount_paid`
- `amount_refunded`
- `due_time`
- `paid_time`
- `status`
- `price_snapshot_id`
- `config_version_id`

规则：

- 首期实付和后续账单分开。
- 部分支付时账单状态为部分支付。
- 账单结清后才进入可分账流程。
- 退款和冲正不直接删除账单，必须写反向流水。

---

## 3. 支付流水模型

字段：

- `payment_id`
- `bill_id`
- `order_id`
- `channel`
- `channel_trade_no`
- `amount`
- `fee`
- `status`
- `callback_event_id`
- `paid_time`
- `raw_status`

规则：

- 支付成功必须关联账单。
- 一个账单可有多笔支付流水。
- 支付通道回调重复时不重复累加实收。
- 支付异常进入对账中心。

---

## 4. 分账流水模型

字段：

- `allocation_id`
- `payment_id`
- `bill_id`
- `order_id`
- `order_type`
- `beneficiary_type`：merchant / funder / channel / platform
- `beneficiary_id`
- `gross_amount`
- `commission_amount`
- `net_amount`
- `rule_version_id`
- `status`

规则：

- 门店订单：客户实收扣平台抽佣后进入商家钱包。
- 分红订单：按出资比例拆分后，门店和资方各自份额分别扣平台抽佣。
- 平台订单：按资方/平台配置生成收益，渠道佣金按规则生成。
- 分账失败不改钱包余额。
- 分账成功后才生成钱包流水。
- **采购款打款(订单生效一次性)不走分账流水**,走独立的 procurement_inflow(详见 §11)。

---

## 5. 钱包流水模型

字段：

- `wallet_entry_id`
- `wallet_id`
- `owner_type`
- `owner_id`
- `direction`：in / out / freeze / unfreeze
- `amount`
- `balance_before`
- `balance_after`
- `source_type`
- `source_id`
- `status`

来源类型：

- 分账入账
- 平台抽佣
- 渠道佣金
- 接口计费
- 提现冻结
- 提现成功
- 退款扣回
- 冲正调整
- 人工调账

规则：

- 钱包余额只由钱包流水计算，不允许直接手改。
- 冻结、解冻、提现中都必须有独立流水。
- 人工调账必须有审批和操作日志。
- **采购账户余额不通过钱包流水模型记录**,使用独立的 procurement_inflow / procurement_withdrawal(详见 §11)。

---

## 6. 提现与打款

提现单字段：

- `withdraw_id`
- `wallet_id`
- `owner_type`
- `owner_id`
- `amount`
- `receive_account_snapshot`
- `status`
- `reviewer_id`
- `review_note`

打款流水字段：

- `transfer_id`
- `withdraw_id`
- `channel`
- `amount`
- `channel_trade_no`
- `status`
- `callback_event_id`

规则：

- 发起提现时冻结可提现余额。
- 审核驳回时解冻。
- 打款成功后扣减提现中余额，生成已提现流水。
- 打款失败时退回可提现余额。
- **采购账户提现使用独立提现单 procurement_withdrawal**,流程与钱包提现类似但有差异(详见 §11)。

---

## 7. 退款和冲正

退款适用：

- 客户取消。
- 审核拒绝后退款。
- 售后退款。
- 重复支付。

冲正适用：

- 分账错误。
- 钱包入账错误。
- 通道对账差异。
- 人工复核后反向调整。
- **采购款打款错误**(如订单撤单后需追回采购款)。

要求：

- 退款影响支付流水、账单、订单状态。
- 冲正影响分账流水和钱包流水。
- 已提现金额不能直接扣回，需生成负向余额或后续抵扣规则。
- 退款和冲正必须二次确认和权限控制。
- **采购款冲正**:已打款采购款的冲正,优先在 procurement_inflow 写反向记录;如已提现,需财务主管处理负余额场景。

## 8. 三边对账

每日日终对账：

| 对账边 | 内容 |
|---|---|
| 通道边 | 支付、退款、打款通道流水 |
| 业务边 | 订单、账单、分账单、提现单 |
| 钱包边 | 钱包流水和余额 |
| **采购账户边** ⭐ | **采购款入账流水(procurement_inflow)+ 采购账户余额 + 采购账户提现单(procurement_withdrawal)** |

平账条件：

- 通道实收 = 支付流水成功金额。
- 支付流水成功金额 = 账单已付金额。
- 可分账金额 = 分账流水总金额。
- 分账成功金额 = 钱包入账金额 + 平台收入。
- 提现成功金额 = 钱包出账金额 = 打款成功金额。
- **采购款入账金额(procurement_inflow.amount 累计)= 采购账户 total_received**。
- **采购账户提现成功金额 = procurement_withdrawal.amount_arrival 累计 = 采购账户 total_withdrawn**。
- **每笔采购款入账金额校验**:`amount = device_price × funder_ratio`(分红订单 funder_ratio = 配资比例,平台订单 funder_ratio = 100%)。

## 9. 异常处理

| 异常 | 处理 |
|---|---|
| 通道有款，系统无支付流水 | 人工关联或补单 |
| 系统支付成功，通道无款 | 冻结订单并禁止分账 |
| 分账不平 | 进入分账异常队列 |
| 钱包余额不平 | 禁止提现，财务复核 |
| 打款失败 | 解冻提现金额，记录失败 |
| 退款成功但账单未更新 | 重放回调或人工补偿 |
| **监管锁回调到达但订单不符** ⭐ | 进入采购账户异常队列,主管核实 |
| **采购款打款失败(资金通道)** ⭐ | procurement_inflow.status = failed,主管可重试或手动补打 |
| **采购款金额计算异常**(device_price × funder_ratio ≠ amount)| 系统阻止打款,人工核对 |
| **同一订单收到重复监管锁回调** ⭐ | 系统幂等校验,只处理首次,记录重复事件 |
| **采购账户余额不平** ⭐ | 禁止提现,财务复核 |
| **采购账户提现打款失败** ⭐ | 退回 balance_available,通知商家 + 财务异常 |

---

## 10. 验收标准

1. 任一金额变动都有流水。
2. 任一流水能追到来源业务对象。
3. 退款和冲正不会删除原始流水。
4. 日终对账能定位差异来源。
5. 钱包余额可由流水重算。
6. **采购账户余额可由 procurement_inflow / procurement_withdrawal 重算**。
7. **每笔采购款入账可追溯到具体订单 + 监管锁回调事件 + 资方出资比例**。

## 11. 采购账户流水模型 ⭐ 新增

详细完整规格见 `财务管理/07_采购账户与软钱包架构.md`。本节摘要数据对象规格,与其他流水模型对齐。

### 11.1 `procurement_inflow` 采购款入账流水

字段:

- `inflow_id`
- `account_id`(关联 procurement_account)
- `order_id` / `order_no`
- `order_type`:dividend / platform
- `device_price`:订单设备价快照
- `funder_ratio`:资方出资比例(分红=订单配资%; 平台=100%)
- `amount` = `device_price × funder_ratio`(系统计算,不可改)
- `funder_id`:资金来源资方
- `trigger_source`:monitor_lock_callback / manual
- `trigger_at`
- `channel` / `channel_request_no` / `channel_response_status`
- `failure_reason`
- `status`:processing / completed / failed / reversed

规则:
- 监管锁回调触发,平台代资方付款给商家采购账户
- 不影响订单分账流水(走独立路径)
- 一笔订单最多生成一条 procurement_inflow(幂等)
- 失败时进入财务异常队列
- 订单撤单 / 异常时,可写反向冲正记录(status = reversed)

### 11.2 `procurement_withdrawal` 采购账户提现单

字段:

- `withdrawal_id`
- `account_id` / `merchant_id`
- `amount_requested`:申请金额
- `fee_amount`:手续费(千二,可配)
- `amount_arrival`:实际到账金额 = amount_requested - fee_amount
- `fee_rate`:手续费率快照
- `bank_account_id` / `bank_account_no_masked`
- `audit_status`:pending / approved / rejected / no_audit_needed
- `audited_by` / `audited_at` / `audit_reason`
- `status`:pending / auditing / paid_processing / paid_success / paid_failed / cancelled
- `channel_request_no` / `channel_response_status` / `failure_reason`

规则:
- 商家自助提现,小额(≤¥5万)系统直通,大额(>¥5万)财务主管审核
- 提现金额必须 ≤ procurement_account.balance_available
- 提现期间冻结金额(balance_available → balance_withdrawing)
- 打款成功后,balance_withdrawing -= amount, total_withdrawn += amount_arrival
- 打款失败时,balance_withdrawing 退回到 balance_available

### 11.3 `procurement_account` 采购账户主表

字段:

- `account_id`
- `merchant_id`(商家级,一个商家一个账户)
- `account_no`:账户编号(系统生成)
- `balance_total`
- `balance_pending`:在途余额(打款已发起,通道未确认)
- `balance_available`:可提现余额
- `balance_withdrawing`:提现中余额
- `total_received` / `total_withdrawn` / `total_fee`
- `status`:normal / frozen / disabled
- `bank_account_id`

规则:
- 余额变动只通过 procurement_inflow / procurement_withdrawal 触发
- balance_total = balance_pending + balance_available + balance_withdrawing
- total_received = SUM(procurement_inflow.amount WHERE status=completed)
- total_withdrawn = SUM(procurement_withdrawal.amount_arrival WHERE status=paid_success)
- total_fee = SUM(procurement_withdrawal.fee_amount WHERE status=paid_success)

### 11.4 `procurement_account_log` 操作日志

记录账户级别的操作:

- `action`:create / freeze / unfreeze / disable / bank_account_change
- `operator_id` / `operator_role`
- `previous_value` / `new_value`(JSON)
- `reason` / `client_ip` / `created_at`
