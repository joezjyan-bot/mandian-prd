> **Stage 6 术语同步(2026-05-27)**: 本文档已按 Stage 6 统一为商家、联营、平台订单、订单结算款、我的钱包、履约中、逾期费用、留购、保证金等展示术语；数据库字段、API 路径、英文枚举保持不变。

> **V0.2.2 全量整改补丁(2026-05-28)**: 权限设计升级为“角色可创建 + 操作点细化 + 操作员绑定角色 + 高危动作二次确认/二人复核/审计日志”。历史角色表可作默认模板,但 V0.2.2 开发以本补丁的高危权限点为必验范围。

## 1. 概述

V1 角色 + 关键操作的权限对照表。本表是 V1 上线时配置 RBAC 的依据。

## 1.1 V0.2.2 权限总原则

1. 系统允许创建自定义角色,但所有可操作内容必须先被拆成标准权限点。
2. 操作员只通过绑定角色获得权限;临时授权必须有有效期、授权人、原因和审计日志。
3. 高危资金/法务/客户资料动作必须支持二次确认;大额或不可逆动作必须支持二人复核。
4. 商家可自选员工权限,财务人员可以被授权查看钱包和发起提现,但不能修改结算账户、收款账户、主体证件、员工权限和门店归属。
5. 技术人员默认不拥有业务审批权。技术可处理故障、执行数据修复脚本,但涉及资金、订单关闭、退款、坏账、证据删除等业务结果必须有业务审批单。

| 操作点 | 可发起 | 可审批 | 二次确认 | 二人复核 | 必写审计日志 | 备注 |
|---|---|---|---|---|---|---|
| 修改订单金额(账单生成前) | 客服/客服主管/运营 | 客服主管(超阈值) | 是 | 超阈值是 | 是 | 账单生成后禁止直接改 |
| 修改账单 | 无 | 无 | - | - | 是 | 生成后只能走冲正/补收/减免/退款/核销 |
| 发起退款 | 客服/财务/运营 | 按金额分级 | 是 | 中大额是 | 是 | 必填责任方、金额、凭证、冲正对象 |
| 审核退款 | 财务主管/运营主管/老板 | 老板可终审 | 是 | 大额是 | 是 | 小额≤5000,中额5000-20000,大额>20000 |
| 发起打款 | 财务 | 财务主管/老板 | 是 | 大额是 | 是 | 必须通过结算三条件 |
| 审核提现 | 财务主管/老板 | 老板可终审 | 是 | 大额是 | 是 | 普通提现 500-20000,每日 3 次 |

| 解冻商家钱包 | 财务主管/运营主管 | 老板/超级管理员 | 是 | 是 | 是 | 不允许技术单人解冻 |
| 关闭订单 | 客服/运营 | 主管/规则自动 | 是 | 高风险是 | 是 | 关闭原因、退款责任必填 |
| 坏账核销 | 财务/法务 | 老板 | 是 | 是 | 是 | 核销不删除原账单 |
| 法诉材料导出 | 法务/运营主管 | 法务主管/老板 | 是 | 高敏感是 | 是 | 导出包版本和用途留痕 |

| 强制变更订单状态 | 超级管理员/运营主管 | 老板或授权主管 | 是 | 是 | 是 | 需关联审批单 |
| 删除或隐藏附件 | 法务/运营主管 | 老板或法务主管 | 是 | 是 | 是 | 原文件不得物理删除,只可标记不可见 |
| 修改客户关键资料 | 客服主管/风控 | 风控主管/运营主管 | 是 | 高敏感是 | 是 | 保留前后脱敏快照 |

### 1.4 不能交给技术自行决定

以下操作即使需要技术协助,也必须先有业务审批:退款责任变更、账单金额调整、商家钱包余额调整、坏账核销、法诉证据删除/隐藏、监管锁异常放行、订单强制关闭、商家结算账户修改、返点规则调整、短租保证金扣款、短租事故/违章/损坏/丢车责任认定。

### 3.8 商家财务权限底线

商家老板可按员工维度授权财务人员查看钱包、查看流水、导出对账、发起提现。财务人员即使获得提现权限,也不得修改结算账户、收款账户、主体证件、门店归属和员工权限。结算账户修改只能由商家老板或被单独授权的商家管理员发起,且必须二次确认、短信/OTP 校验、写入操作日志。

### 3.9 高危权限实现要求

1. “后台余额调整申请”只允许录入业务原因、凭证和冲正对象;真正影响钱包/总账必须进入审批和四账分录,不得由财务专员单人直接改余额。
2. “账单计划调整”仅允许在账单生成前操作;账单生成后任何角色都不能直接修改账单金额或账期,只能走补收、减免、冲正、退款、核销或回收完成。
3. 普通客服不得单独通过或驳回高风险审核,只能提交审核建议、补资料要求和客户跟进记录。
4. 监管锁下发、解锁、异常放行必须关联订单、设备、锁回调、操作原因和审批记录;技术排障只能处理接口重试/日志,不得单人改变业务结算结果。
5. 退款执行只能由财务或具备退款执行权限的主管角色完成;客服可以发起退款工单,不得直接执行通道退款或钱包扣减。

## 4. 二次确认要求

以下操作即使有权限,仍需二次确认 + 完整日志:
- 退款执行 / 提现打款
- 后台余额调整申请 / 审批 / 执行
- 关闭订单 / 取消重下
- 修改抽佣比例
- 角色变更 / 权限变更
- 客户加黑名单
- 删除操作(若有)
- 账单计划调整
- 商家结算账户修改
- 商家财务人员发起提现

## 5. 高频审计点

- 客服查看完整客户身份证号(展开次数)
- 客服查看完整银行卡号
- 操作日志的导出
- 大额退款发起(> X 元,X 可配)

## 6. 数据模型

权限矩阵存数据库 `permission_matrix` 表:
- role_key
- permission_key
- is_allowed
- requires_2fa
- requires_approval(谁审批)
