# 资方放款条件与触发机制

> [V0.2.2 同步声明] 本目录历史命名“资方管理”保留为内部配置入口兼容。当前业务语义统一为**内部资金来源管理**;相关信息仅运营端、财务端、风控端可见,不得进入 C 端、商家端、客服话术或合同正文。

> P0 业务文档(2026-05-27)。
> 定义**资方资金放款**的完整触发机制 — 多前置条件全部满足后才放款,支持按资方配置不同条件清单,具备扩展性。

> **设计目标**:
> - **可扩展**:未来增加任何新条件(如公证、人脸活体、征信授权终审等)只需配置,不需改代码
> - **可配置**:不同资方可有不同的放款条件清单(蓝海银行 vs 平台自营)
> - **可追溯**:每次放款条件检查都留痕,便于排查问题
> - **统一入口**:无论哪个前置条件状态变化,都触发"放款条件检查"任务

---

## 1. 核心设计原则

### 1.1 "放款条件检查"是事件驱动

```
任何前置条件状态变化 → 触发一次"放款条件检查"任务
   ↓
检查该订单是否满足该资方的所有放款条件
   ↓
  ├─ 全部满足 → 触发放款流程(详见 §5)
  └─ 未全部满足 → 等待下次条件变化触发
```

### 1.2 条件清单按资方配置,不是硬编码

```sql
funder_disbursement_precondition  -- 资方放款前置条件配置表
- funder_id              -- 资方
- condition_code         -- 条件代码(枚举)
- is_required            -- 是否必须
- priority               -- 检查优先级(用于失败时的提示)
```

**关键**:增加新条件 = 新增枚举值 + 给资方配置 + 检查代码加判断分支,不需要改主流程。

## 3. 数据模型

### 3.1 资方放款前置条件配置表

```sql
funder_disbursement_precondition
- precondition_id        bigint PK
- funder_id              bigint FK     -- 关联资方
- condition_code         enum          -- 条件代码(详见 §2)
- is_required            boolean       -- 是否必须(false = 选配,可跳过)
- priority               int           -- 排序(用于失败提示)
- effective_from         date          -- 生效日期
- effective_to           date          -- 失效日期(可空)
- status                 enum          -- active / paused
- description            text          -- 配置说明
- created_at / updated_at

-- 索引:(funder_id, status) / (condition_code)
```

### 3.2 资方放款条件检查日志

```sql
funder_disbursement_check_log
- check_log_id           bigint PK
- order_id               bigint FK
- funder_id              bigint FK
- triggered_by           enum          -- 触发源(condition_change / manual_retry / scheduled_task)
- triggered_condition    enum          -- 哪个条件变化触发的本次检查
- all_conditions_met     boolean       -- 是否全部满足
- missing_conditions     json          -- 未满足的条件清单
- result_action          enum          -- triggered_disbursement(触发放款)/ waiting(等待)/ failed(异常)
- check_at               datetime
```

### 3.3 资方放款流水

```sql
funder_disbursement_log
- disbursement_id        bigint PK
- order_id               bigint FK
- funder_id              bigint FK
- funding_source         enum          -- platform_self / funder_blue_ocean / 其他
- amount                 decimal(18,2) -- 放款金额(= 设备价 × 出资比例)
- status                 enum          -- pending(放款中)/ success(放款成功)/ failed(放款失败)/ rolled_back(已撤销,撤单时)
- request_at             datetime      -- 放款请求时间
- success_at             datetime      -- 放款成功时间
- failed_reason          text
- bank_serial_no         varchar(64)   -- 银行流水号(蓝海银行场景)
- tonglian_topup_id      varchar(64)   -- 通联充值流水号
- merchant_settlement_entry_id bigint  -- 关联门店结算账户流水
- funder_ledger_id       bigint        -- 关联资方台账
- retry_count            int           -- 重试次数
- max_retry              int           -- 最大重试次数(默认 3)
- created_at / updated_at
```

## 4. 放款条件检查触发链路

### 4.1 事件驱动机制

```
任何前置条件状态变化(任意以下事件):
  - 风控审批通过 → risk_status = PASSED
  - 客户合同已签 → contract_status = SIGNED
  - 代扣已签约 → auto_debit_status = SIGNED
  - 首期已付 → first_payment_status = PAID
  - 设备已交付 → delivery_status = COMPLETED
  - 客户已确认收货 → receive_confirmed_at = NOW()
  - 收货后验证通过 → post_delivery_verification = PASSED
  - 监管锁已上锁 → lock_status = LOCKED
  - 征信授权完成 → credit_auth_status = COMPLETED
  - ...(任何未来新增条件)
   ↓
触发"放款条件检查"任务(异步)
   ↓
查询订单关联的资方配置 → 拿到该资方的放款条件清单
   ↓
逐项检查每个条件是否满足:
  - 共通条件 + 该资方扩展条件
   ↓
  ├─ 全部满足 → 触发放款流程(§5)
  └─ 未全部满足 → 记录日志,等待下次触发
```

### 4.2 伪代码

```python
def on_condition_changed(order_id, changed_condition):
    """前置条件状态变化时调用"""
    order = Order.get(order_id)
    
    # 只对联营/平台订单做检查
    if order.cooperation_mode == 'self_operate':
        return
    
    # 若已放款,跳过
    if order.disbursement_status == 'success':
        return
    
    # 拿到该资方的前置条件配置
    preconditions = FunderDisbursementPrecondition.get_by_funder(
        funder_id=order.funder_id,
        status='active'
    )
    
    # 逐项检查
    missing = []
    for pre in preconditions:
        if not check_condition(order, pre.condition_code):
            missing.append(pre.condition_code)
    
    # 写检查日志
    log = FunderDisbursementCheckLog.create(
        order_id=order_id,
        funder_id=order.funder_id,
        triggered_by='condition_change',
        triggered_condition=changed_condition,
        all_conditions_met=(len(missing) == 0),
        missing_conditions=missing,
    )
    
    # 全部满足 → 触发放款
    if len(missing) == 0:
        log.result_action = 'triggered_disbursement'
        trigger_disbursement(order)
    else:
        log.result_action = 'waiting'
    
    log.save()


def check_condition(order, condition_code):
    """检查单个条件是否满足"""
    if condition_code == 'RISK_PASSED':
        return order.risk_status == 'PASSED'
    elif condition_code == 'CONTRACT_SIGNED':
        return order.contract_status == 'SIGNED'
    elif condition_code == 'LOCK_STATUS_LOCKED':
        return order.lock_status == 'LOCKED'
    elif condition_code == 'CREDIT_AUTH_COMPLETED':
        return order.credit_auth_status == 'COMPLETED'
    # ...(未来扩展只需要新增分支)
```

### 4.3 兜底机制

防止"事件丢失"导致永远不触发放款:

```
定时任务(每 5 分钟扫一次):
   查询所有"已满足部分条件但未放款"的订单
      ↓
   重新跑放款条件检查
```

## 5. 放款流程(条件全部满足后)

### 5.1 按 funding_source 分支

#### 分支 A:funding_source = platform_self(平台自营资金)

```
① 检查平台自营授信额度是否充足
   funder.credit_limit_left >= disbursement_amount?
   ├─ 是 → 继续
   └─ 否 → 异常队列,通知运营主管

② 平台自营资金 → 平台对公账户(账面动作,实际平台对公账户已有钱)

③ 平台对公账户实时充值通联备付金账户(< 1 分钟)
   写 platform_passthrough_log

④ 充值到门店子台账
   写 settlement_entry:entry_type=order_settlement

⑤ 更新资方台账
   funder_receivable_ledger.transferred_amount += amount
   funder.credit_limit_used += amount
   funder.credit_limit_left -= amount

⑥ 写资方放款流水
   funder_disbursement_log:status=success

⑦ 推送商家端"订单结算款 ¥XXX 已到账"
```

#### 分支 B:funding_source = funder_blue_ocean(蓝海银行赊销)

```
① 检查蓝海银行授信额度是否充足

② 平台调用蓝海银行放款 API
   POST /api/v1/blue-ocean/disbursement
   请求体:
   {
     "order_id": "OD2026001",
     "amount": 5000.00,
     "customer_info": {...},
     "device_info": {...}
   }
   ↓
   蓝海银行接收 + 终审(若需要)+ 放款到平台对公账户
   ↓
   银行回调:
   ├─ 成功 → 继续
   └─ 失败 → 异常队列(详见 §6.3)

③ 平台对公账户实时充值通联备付金账户(< 1 分钟)

④ 充值到门店子台账
   写 settlement_entry:entry_type=order_settlement

⑤ 更新资方台账
   funder_receivable_ledger.transferred_amount += amount(资方 = 蓝海银行)
   funder.credit_limit_used += amount
   funder.credit_limit_left -= amount

⑥ 写资方放款流水
   funder_disbursement_log:status=success, bank_serial_no=BL2026XXX

⑦ 推送商家端"订单结算款 ¥XXX 已到账"
```

### 5.2 放款金额计算

```python
def calculate_disbursement_amount(order):
    if order.cooperation_mode == 'joint_venture':
        # 联营订单:设备价 × 平台出资比例
        return order.device_price * order.platform_funding_ratio
    elif order.cooperation_mode == 'receivables_assignment':
        # 平台订单:设备价 × 100%
        return order.device_price * 1.0
    else:
        # 商家订单:不放款
        return 0
```

## 6. 异常处理

### 6.1 条件检查异常

| 异常 | 处理 |
|---|---|
| 订单未关联资方 | 跳过检查,提示运营先分配资方 |
| 资方配置不存在 | 用平台默认共通条件清单 |
| 条件代码未知(代码升级未同步)| 记日志告警 + 默认视为不满足 |
| 检查过程报错 | 异常队列 + 自动重试 3 次 |

### 6.2 放款异常(平台自营)

| 异常 | 处理 |
|---|---|
| 平台自营授信额度不足 | 异常队列,主管处理(调整授信 / 切换资方 / 驳回订单) |
| 平台对公账户余额不足 | 异常队列,财务紧急补款 |
| 通联充值失败 | 系统自动重试 3 次,失败后异常队列 |

### 6.3 放款异常(蓝海银行)

| 异常 | 处理 |
|---|---|
| 银行 API 调用失败 | 系统自动重试 3 次,失败后异常队列 |
| 银行终审拒绝 | 异常队列,运营客服联系客户;可降级为平台自营 |
| 银行授信额度不足 | 异常队列,通知运营调整;可降级为平台自营 |
| 银行放款延迟(超 24h)| 客服催促,7 天后升级主管 |

### 6.4 降级策略

当蓝海银行放款失败,且符合降级条件:

```
触发降级:
   蓝海银行放款失败 3 次 + 平台自营授信充足
      ↓
   运营主管确认是否降级
      ↓
   降级流程:
   ├─ 修改 order.funding_source = platform_self
   ├─ 修改 order.funder_id = 平台自营资金 ID
   ├─ 重新加载合同模板 / 话术模板
   ├─ 若合同已签 → 走补充合同流程(详见 10 文档 §4)
   └─ 重新触发放款条件检查
```

## 9. 设计延展性说明

### 9.1 增加新条件的成本

```
开发成本:
   1. 在 condition_code 枚举里加 1 个值
   2. 在 check_condition 函数里加 1 个分支
   3. 在对应订单模型加 1 个状态字段
   4. 在该状态字段变化时调用 on_condition_changed

配置成本:
   1. 在 funder_disbursement_precondition 表加 1 条记录(给指定资方配置)
   
完成。整个主流程不需要改。
```

### 9.2 不同资方差异化举例

| 资方 | 必须条件 | 选配条件 |
|---|---|---|
| 平台自营 | 10 个共通条件 | 无 |
| 蓝海银行 | 10 个共通条件 + 征信授权 + 转让确认书 | 人脸活体 / IMEI 验证 / 终审 |
| 某未来信托 | 10 个共通条件 + 公证 + 担保人确认 | 视频面签 |

### 9.3 条件检查的性能考量

- 单次检查 < 100ms
- 事件驱动 + 5 分钟兜底定时任务,确保不丢
- 检查日志按月分表
- 高频订单可缓存上次检查结果(15 分钟)
