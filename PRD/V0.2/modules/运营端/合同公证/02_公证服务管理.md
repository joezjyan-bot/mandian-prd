# 公证服务管理

> **Stage 6 术语同步(2026-05-27)**: 本文档已按 Stage 6 统一为商家、联营、平台订单、订单结算款、我的钱包、履约中、逾期费用、留购、保证金等展示术语；数据库字段、API 路径、英文枚举保持不变。

> **来源**: P1-4

## 1. 业务背景

公证为合同提供法律强执行力。系统接入公证服务,允许门店/客服选择是否给具体订单发起公证。

## 2. 系统接入

- 公证通道通过中控台对接(详见 09_中控台对接)
- 公证调用本地缓存 + 异步计费

## 3. 触发节点

**合同签署完成后,决定是否额外发起公证**

不是合同签署的强制环节,而是合同签署完成后的"附加增强"。

## 4. 谁能发起

| 订单类型 | 发起人 |
|---|---|
| 商家订单 | 门店自选(在订单详情 / 商家 PC 点击发起) |
| 联营订单 | 客服决定(业务规则要求时必发) |
| 平台订单 | 客服决定 |

## 5. 公证流程

```
1. 触发(门店或客服点击"发起公证")
2. 系统调用公证服务接口
3. 公证服务方对合同 PDF 进行公证
4. 公证完成 → 回调更新订单
5. 公证书 PDF 存档(随合同副本一起)
```

## 6. 公证费用

公证一般几百元/单(具体看通道方报价)

费用承担:按链路配置决定
- 平台承担(平台想做强保障)
- 商户承担(合同里约定)
- 客户承担(可能体现在订单总费用)
- 计入套餐(平台预付)

## 7. 公证状态

订单合同记录中:
- `notary_required`:是否需公证
- `notary_initiator`:公证发起人(门店/客服)
- `notary_status`:状态(待发起 / 发起中 / 公证完成 / 公证失败 / 不发起)
- `notary_completed_at`:完成时间
- `notary_document_url`:公证书 PDF

## 8. 失败处理

- 公证服务方拒绝:回调通知,客服核实原因(合同内容缺陷等)
- 公证超时:重新发起或人工跟进

## 9. 操作权限

- 门店主账号:对商家订单发起 / 撤销
- 客服:对所有订单类型发起 / 撤销
- 主管:审批高额公证费

## 10. 操作日志

公证发起 / 完成 / 失败:留完整日志,记发起人、时间、费用、状态、回调内容。

## 11. 数据模型

参考 `order_contract` 表中公证相关字段。
