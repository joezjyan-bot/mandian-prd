## 1. C 端订单状态(精简到 8 个 + 已完成子标签)

### 1.1 8 个 C 端状态 Tab

| Tab | 含义 |
|---|---|
| 全部订单 | 包含所有状态(含已取消)|
| 审核中 | 客户已提交,系统/客服在审核 |
| 待付款 | 客服已确认订单,客户需支付首期 |
| 待签约 | 首期已付,需签电子合同 / 代扣 |
| 已发货 | 订单已发出,等待物流送达 |
| 待收货 | 客户在 App 看到物流单号(收到设备后点确认收货)|
| 租用中 | 客户已收货确认,在租期内 |
| 已完成 | 归还 / 留购 / 已取消(含子标签)|

### 1.2 后台状态 → C 端状态映射

| 后台状态 | C 端展示 |
|---|---|
| DRAFT / PENDING_SUBMIT | (不展示)|
| PENDING_REVIEW / PENDING_FUNDER / PENDING_STORE_ASSIGN | 审核中 |
| PENDING_PAYMENT(首期未付)| 待付款 |
| PENDING_PAYMENT(首期已付,合同未签)| 待签约 |
| PENDING_PAYMENT(合同已签,代扣未签)| 待签约 |
| PENDING_SIGN | 待签约 |
| PENDING_SHIP | 审核中 |
| PENDING_RECEIVE(已发货)| 已发货 |
| PENDING_RECEIVE(客户在 App 看到物流单号后)| 待收货 |
| PENDING_RETURN / RENTING | 租用中 |
| DUE_REMINDER / OVERDUE | 租用中(详情页显示逾期标签)|
| RETURNING / RETURN_INSPECTION | 租用中(归还中)|
| PURCHASE_PENDING | 租用中(留购待支付)|
| COMPLETED | 已完成 子标签"已归还" |
| RETAINED | 已完成 子标签"已留购" |
| CLOSED | 已完成 子标签"已取消" |
| DISPUTE | 租用中(详情显示争议标签)|

核心原则:
- 审核全过程(包括分配资方/门店/调整套餐)客户都看到"审核中"
- 不显示预估时间("1-3 工作日"全部去掉)
- "已完成" Tab 通过子标签区分归还/留购/取消

### 1.3 Tab 角标数字规则(Q17)

仅非 0 的 Tab 显示数字。全部订单、已完成不显示数字。

### 1.4 已完成的子标签(Q21)

子标签颜色:
- 已归还:中性灰
- 已留购:绿色(成功)
- 已取消:浅红色

## 2. C 端订单详情页(各阶段展示规则)

### 2.1 "审核中"详情页

展示:商品基础信息 / 当前账单草案 / 联系客服 / 取消订单

不展示:商家主体 / 资方 / 履约门店 / 做单客服 / 订单类型 / 预估时间

### 2.2 "待付款"详情页

展示:商品 + 账单总览 + 首期实付明细(押金/首期租金/设备管理费等)+ [立即支付] + 取消订单

### 2.3 "待签约"详情页

展示:待办列表(首期支付 ✓ / 电子合同签署 ⏳ / 代扣签约 ⏳)+ 商品 + 账单 + 取消订单(需客服同意)

### 2.4 "已发货 / 待收货"详情页

展示:物流信息(快递公司/运单号/发货时间)+ 商品 + 账单 + [确认收货]

"已发货" → "待收货" 触发:客户首次进入订单详情页查看物流单号 → 自动改为"待收货"

### 2.5 "租用中"详情页 ⭐ 含账单瀑布流 + 违约金账单

详见 §3。

### 2.6 "已完成"详情页

展示:商品 + 设备信息 + 完整支付记录(首期 / 各期租金 / 违约金 / 留购支付)

## 3. 租用中订单的账单瀑布流 + 违约金账单 ⭐ 核心功能

### 3.1 整体布局(两个独立账单区块)

```
─── 租金账单(主账单)──────────
仅展示每期租金,押金/设备管理费/公证费 已在首期实付中收取

期数  付款日期       本期金额    留购价     操作
第1期  2026-05-25  ¥10.00 ✅  ¥6,040.00  -
第2期  2026-06-25  ¥568.75 ✅ ¥5,471.25  -
第3期  2026-07-25  ¥568.75 ⏰逾期5天 ¥4,902.50 [立即支付][申请留购]
第4期  2026-08-25  ¥568.75  ¥4,333.75 [立即支付*][申请留购*]
... 第5-9 期同上

─── 违约金账单(独立)──────────
来源期数  累计天 原始金额 减免  实付   操作
第3期    5天    ¥50    -¥20  ¥30   [立即支付][咨询客服]
       已减免 ¥20(运营调整)

合计违约金:¥30.00(可单独支付)

─── 留购总价(总览)──────────
最后到期留购价:¥6,050.00

注:*在严格顺序模式下,支付前会校验前期是否结清
```

### 3.2 任意期可支付的逻辑(Q16 + 严格顺序 A)

核心规则:任何一期都可以点 [立即支付] 按钮查看金额,但实际支付时系统强制结清前期欠款。

| 状态 | 实际行为 |
|---|---|
| 已付清 | 显示 ✅,无支付按钮 |
| 当前期未付 | 显示 [立即支付] + 当期金额 |
| 第 N 期未付,客户点第 N+M 期 | 系统弹窗:"请先支付第 N 期(¥XXX)";不允许跳期 |
| 已逾期 | 显示红色违约金提示 + 累计天数 |

支付前的校验逻辑:

```
客户点 [立即支付] 第 N 期
   ↓
系统检查:第 1 到 第 N-1 期是否都已付清?
   ├─ 是 → 直接调起支付通道
   └─ 否 → 弹窗:"您的第 X 期尚未支付,需先支付才能支付第 N 期"
            [取消]  [立即支付第 X 期]
```

预计算:办单助手生成订单时,全部期数的金额 + 对应留购价都已写入数据库。

### 3.3 支付通道(Q16)

支付方式:支付宝 / 微信支付 / 通联支付 / 信联支付 / 银行卡(已签代扣)。后台可配置开关与排序。

### 3.4 申请留购的流程(Q16 + 违约金必清)

```
客户点 [申请留购]
   ↓
检查前置条件:
  ✓ 当前期已付清?(必须先付完当期才能留购)
  ✓ 之前所有期数已付清?(严格顺序 A)
  ✓ 无未结违约金?(Q25 - 留购前必清)
   ↓
  └─ 不满足 → 提示具体未结清项
              示例:"请先支付第 3 期(¥568.75)+ 违约金 ¥30.00"
  └─ 满足 → 弹出留购支付明细
   ↓
留购明细弹窗:
   已付期数:    第 1-N 期
   留购价:      ¥X,XXX(对应第 N 期)
   押金抵扣:    -¥1,490
   ─────────────
   实付:        ¥X,XXX
   [取消]      [确认支付]
   ↓
客户确认 → 调起支付通道 → 支付完成
   ↓
订单状态:租用中 → 已完成(已留购)
设备所有权归客户
```

### 3.5 严格顺序支付(Q16 + A 决策)

| 场景 | 处理 |
|---|---|
| 第 3 期未付,客户点第 5 期 [立即支付] | 不允许,系统弹窗提示先付第 3 期 |
| 客户第 3 期 + 第 4 期一起付 | V1 不支持,客户分别点 |
| 客户提前付了第 3 期(还没到期日)| 允许,系统记录;下一期顺序变为第 4 期 |
| 第 3 期支付失败 | 客户必须重新支付,第 4 期等等 |

严格顺序的好处:
- 避免客户漏付前期产生违约金混乱
- 客户清晰知道自己当前应付的金额
- 系统记账简单(没有跳期问题)

### 3.6 留购价的计算规则

参照办单助手截图(本截图作为典型例子,详见 08 文档):

| 期数 | 本期金额 | 当期留购价 |
|---|---|---|
| 第 1 期 | ¥10.00 | ¥6,040.00 |
| 第 2 期 | ¥568.75 | ¥5,471.25 |
| 第 N 期 | ¥568.75 | 递减 |
| 第 9 期(最末)| ¥568.75 | ¥1,490.00(押金可抵扣)|

留购价的逻辑由办单助手在生成订单时预计算并写入。

### 3.7 违约金账单展示规则 ⭐ 新增

违约金账单与租金账单完全独立展示。详细规则见 `11_违约金账单与规则配置.md`。

关键展示原则:
- 独立账单区块(在租金账单瀑布流下方)
- 每笔违约金独立一条记录,显示来源期数、累计天数、原始金额、减免、实付
- 已减免的违约金显示"已减免 ¥XX(运营调整)"标识(Q24)
- 全额减免的违约金显示"已免除"+ 灰色
- 客户可单独支付违约金(不强制同时付租金)
- 客户留购或归还前必须清完违约金(Q25 - 由 §3.4 衍生检查)

违约金账单 vs 租金账单:

| 项目 | 租金账单 | 违约金账单 |
|---|---|---|
| 强制顺序 | 严格顺序 A | 独立,无顺序约束 |
| 是否影响订单状态 | 是 | 否(不阻塞订单)|
| 留购前必清 | 是 | 是(Q25)|
| 后台可修改 | 否(已锁定)| 是(可修改原始金额 + 减免)|

## 4. 客户主动撤单(Q3)

### 4.1 撤单按钮可点状态

| 订单状态 | [取消订单] 按钮 |
|---|---|
| 审核中 | ✅ 直接可取消 |
| 待付款(未付款)| ✅ 直接可取消 |
| 待付款(已付款)| ⚠️ 需要客服同意 |
| 待签约(已付款)| ⚠️ 需要客服同意 |
| 已发货 | ❌ 不可单方取消(走售后)|
| 待收货 | ❌ 不可单方取消(走售后)|
| 租用中 | ❌ 不可取消(走归还)|

### 4.2 直接可取消的撤单流程

```
[取消订单] → 客户填取消原因(可选)→ 确认
   ↓
订单 → 已完成(已取消)
押金/付款若有,按 §4.3 退款
```

### 4.3 退款规则

| 场景 | 退款 |
|---|---|
| 审核中撤单(未付款)| 无需退款 |
| 待付款撤单(未付款)| 无需退款 |
| 待付款撤单(已付首期)| 原路退回(1-3 工作日)|
| 待签约撤单(已付首期未签)| 客服同意后,原路退回 |
| 已签约撤单 | 客服同意后,扣订单总价 1.5% 违约金(平台收),余额原路退 |

### 4.4 需客服同意的撤单流程

```
[取消订单] → 提示"已付款订单需联系客服处理"
   ↓
客户填取消原因 → 提交申请
   ↓
进入运营端"取消申请"队列
   ↓
客服审核:
  ├─ 同意 → 计算扣款(若已签约) → 退款 → 订单关闭
  └─ 拒绝 → 通知客户(给出理由)→ 订单继续
```

## 5. 订单逾期 / 违约金机制

详细规则见 `11_违约金账单与规则配置.md`。本文档只列出 C 端展示要点。

### 5.1 客户 C 端感知的逾期阶段

| 时机 | C 端展示 |
|---|---|
| 到期日 | 正常无展示(客户主动支付即可)|
| 到期日 + 1 天 | 当期红色 ⏰ 标签;触达推送"账单逾期"|
| 到期日 + 1 天 | 违约金账单生成(第二天凌晨 1 点定时任务)|
| 累计逾期 | 违约金账单每天累计(详见 11 §4)|

### 5.2 客户 C 端的违约金体验

- 在租用中订单详情页的违约金账单区块可见
- 客户可单独支付违约金(不会影响租金支付)
- 已减免的违约金看到"已减免 ¥XX(运营调整)"标识
- 留购或归还前必须清完所有违约金

## 6. C 端永远不显示的信息(合规红线)

| 信息 | 隔离 |
|---|---|
| 商家主体名称 | ❌ 不展示(合同 PDF 内部除外)|
| 资方名称 | ❌ 不展示 |
| 提货门店名称 | ❌ 不展示(本轮决策)|
| 做单客服姓名 | ❌ 不展示 |
| 订单类型(门店/分红/平台)| ❌ 不展示 |
| 出资比例 / 门店出资 | ❌ 不展示 |
| 平台抽佣 / 分账 | ❌ 不展示 |
| 监管锁状态 | ❌ 不展示(后台只)|
| 软钱包打款 | ❌ 不展示 |
| 改派记录 / 工号 / 操作日志 | ❌ 不展示 |
| 客户审核记录 / 风控结论 | ❌ 不展示 |
| 违约金计算规则细节 | ❌ 不展示(只显示最终金额 + 减免)|
| 违约金减免操作日志 | ❌ 不展示(只显示"运营调整"标识)|
