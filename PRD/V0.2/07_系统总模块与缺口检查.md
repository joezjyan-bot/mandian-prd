# 系统总模块与缺口检查

> 用途：作为 V0.2 后续补 PRD、画 UI、拆开发任务的总导航。
> 状态口径：`已拆页面级 PRD` 表示已有独立页面文档；`已有总纲口径` 表示业务方向已定但还缺页面细节；`待补` 表示只在讨论中出现或仅有粗略模块名。

---

## 1. 总体结论

当前 V0.2 已经把系统主方向定住，不需要推翻重来。下一步的重点不是继续收集零散需求，而是把已确认的业务模型拆成页面级 PRD 和开发任务。

已经比较稳定的内容：

1. 三类订单：`门店订单 / 分红订单 / 平台订单`。
2. 三端主体：运营端、商家 PC 端、门店手机端。
3. 外部角色：C 端客户、渠道端、资方端。
4. 办单助手三入口。
5. 长租/短租、小时/天/周/月计费单位。
6. 默认门店发货，平台/资方/快递发货可配置。
7. 分红订单按出资比例分账，再分别扣平台抽佣。
8. 门店端取消配资充值。
9. 中控台是租赁系统外部能力系统，不并入租赁系统。
10. UI 默认参考 `finance-zhongtai` 企业中台风格，需求优先于 UI。

当前主要缺口：

1. 商品管理、商品审核、商品复制给商家还缺页面级 PRD。
2. 办单助手三入口还缺完整页面级 PRD。
3. 订单详情、待审核、资方分配、改套餐、补资料还缺页面级 PRD。
4. 财务分账、钱包、提现、对账还缺完整页面级 PRD。
5. 商家 PC 端和门店手机端除入驻外，大部分页面还缺。
6. C 端客户下单、授权、签署、支付、签收、还款还缺页面级 PRD。
7. 资方端暂时只有模块地图，还缺阶段边界。
8. 权限、角色、操作日志、回调日志需要拆全局基础能力。

## 11. 不建议现在做的事

1. 不建议先继续扩 UI 原型。UI 可以后置，需求闭环更重要。
2. 不建议先做资方独立端。先把运营端资方管理跑通。
3. 不建议先深入短租复杂库存。先把字段、设备码、状态和轻量流程定住。
4. 不建议先做营销玩法。优惠券、小程序礼包先不纳入核心。
5. 不建议把无界租旧逻辑原样复制。只借鉴页面密度、链路配置、操作文档形式。
6. 不建议把中控台混进租赁系统。两个系统后续通过接口和条数打通。
