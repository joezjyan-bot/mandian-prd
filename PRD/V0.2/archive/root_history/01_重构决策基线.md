## 1. 系统总体定位

### 1.1 租赁系统是核心业务系统

满点租赁系统负责完整租赁业务闭环：

- 商品、规格、价格、套餐、增值服务
- 长租/短租租期配置、小时租/天租/周租/月租计费单位
- 设备库存、唯一设备码、仓库、出入库、归还入库
- 门店/商家入驻、资质审核、员工账号
- 办单助手、扫码下单、订单审核、客服 IM
- 合同签署、支付、授权、发货、收货
- 账单、还款、续租、买断、归还、逾期、租后
- 财务分账、钱包、提现、对账、操作日志

### 1.2 中控台是租赁系统之外的轻量系统

中控台不应承载租赁系统的核心业务逻辑，也不应把所有厂商接口代码都集中在中控台。

确认口径：

- 租赁系统内保留具体厂商接口代码和业务适配逻辑。
- 中控台只负责租赁系统调用第三方服务时的统一资源管理：
  - 客户 token
  - 密钥凭证获取
  - 调用额度
  - 计费扣费
  - 套餐购买
  - 调用记录
  - 客户级启停
- 即使中控台短暂异常，租赁系统也应可通过本地缓存或备用密钥继续核心业务，不能让订单全链路不可用。
- 中控台和租赁系统是两个系统，后续通过 API 打通，不混为一个仓库或一个后台。

### 1.3 第三方能力拆分原则

| 能力 | 租赁系统负责 | 中控台负责 |
|---|---|---|
| e签宝合同签署 | 合同模板、签署流程、订单状态、回调处理 | 密钥、额度、调用计费、客户授权 |
| 风控/人脸/二要素 | 业务触发、报告入库、审核展示 | token、额度、计费、调用记录 |
| 支付/代扣 | 订单账单、支付状态、回调、对账 | 如需统一计费，仅记录调用或通道配置 |
| 监管锁 | 订单锁机/解锁业务动作、设备状态 | 密钥、调用额度、接口成本 |

## 2. 三端主体关系

### 2.1 运营端是总后台

运营平台负责全局管理和规则配置：

- 全平台订单、商家、门店、商品、资方、财务、渠道、客诉、配置、权限、日志
- 分红订单和平台订单的审核、资方分配、合同/支付/授权流程主控
- 平台统一商品、统一套餐、统一办单助手配置
- 平台对商家的商品审核和复制同步

### 2.2 商家 PC 端与门店手机端是同一主体

商家 PC 端和门店手机端使用同一个商家/门店主体。

- 商家 PC 端偏管理：商品、订单、财务、员工、配置。
- 门店手机端偏现场：入驻、登录、办单助手、扫码办单、待办任务。
- 入驻通过后，门店账号即成为商家主体账号，可使用 PC 端和手机端。
- 后续文档可分开记录两个端，但权限和数据主体必须统一。

### 2.3 C 端是租客端

C 端负责选品、扫码下单、资料填写、签约、支付、还款、续租、买断、归还和联系客服。

### 2.4 渠道端独立为 H5 数据看板

渠道端面向推广渠道，主要查看：

- 推广码
- 扫码入驻商家数量
- 分红订单统计
- 平台订单统计
- 在租、逾期、完成等订单状态
- 佣金明细
- 提现申请

门店自营订单不纳入渠道佣金统计。

## 3. 订单类型统一口径

后续全系统只保留三种订单：

| 订单类型 | 原系统对应 | 资金/履约本质 | 审核归属 |
|---|---|---|---|
| 门店订单 | 门店自有订单 | 门店自己出设备/资金，自行审核，自行承担风险 | 门店自审 |
| 分红订单 | 分红订单 | 门店出设备，资方补足剩余资金，收益按出资比例分账 | 运营端审核 |
| 平台订单 | 平台主控订单 | 门店不出资或不承担资金，订单推给平台，由平台匹配资方/执行方 | 运营端审核 |

后续系统字段、页面文案、接口枚举只使用 `平台订单`，不再保留历史旧名称作为正式业务名。

### 3.1 门店订单

- 门店使用自己的商品和配置。
- 门店可自审是否通过。
- 订单不需要平台审核。
- 平台可按默认 2% 抽佣，比例可后台调整。
- 抽佣后剩余进入门店钱包。
- 门店订单由商家/门店自己负责风控和履约，平台保留日志和必要监管能力。

### 3.2 分红订单

门店出设备，资方补足部分资金。

示例：

- 设备价：5000
- 门店选择需求配资比例：80%
- 资方出资：4000
- 门店等效出资：1000
- 门店收益占比：20%
- 资方/平台资金侧占比：80%

客户后续每一笔还款都按比例拆分，再按各自份额扣平台抽佣。

确认口径：

- 平台默认 2% 抽佣采用“先按 80/20 分账，再分别从资方份额和门店份额中扣除”的方式。
- 分红比例下拉建议先支持 20%-80%，后续可配置。
- 分红订单必须进入运营端审核。
- 审核客服可分配资方、改套餐、改价、发起合同/授权/支付后续流程。

### 3.3 平台订单

平台订单由门店手机端或商家端发起，但审核、资方、财务主控都在运营端。

- 门店手机端可以发起平台订单。
- 商家 PC 端可以查看自己发起的平台订单进度。
- 平台订单待审核列表支持 `联系客服`，把订单信息同步推送到 IM 客服会话。
- 客服根据客户资质审核，并可在后台修改套餐、改价、补充审核备注。
- 资方由运营端分配。
- 平台订单可产生渠道佣金，但不计入门店自营订单统计。

## 4. 办单助手统一口径

办单助手不沿用当前无界租系统逻辑，改用 GitHub `joezjyan-bot/calculator` 仓库里 `phone-rent` 的惠讯租计算器逻辑和 UI。

### 4.1 三入口确认

门店手机端办单助手拆成三个入口：

1. 门店订单办单助手
2. 分红订单办单助手
3. 平台订单办单助手

原因：

- 门店订单由商家/门店自己配置费率、增值服务、商品范围。
- 分红订单和平台订单由运营平台统一配置费率、资方、套餐、增值服务。
- 三种订单审核归属、分账、财务流向不同，拆入口更清晰。

### 4.2 配置归属

| 办单助手入口 | 商品来源 | 费率/套餐配置 | 增值服务配置 | 审核 |
|---|---|---|---|---|
| 门店订单 | 商家自己的商品 | 商家 PC 端配置 | 商家 PC 端配置 | 门店自审 |
| 分红订单 | 运营同步给商家的商品 | 运营端配置 | 运营端配置 | 运营端审核 |
| 平台订单 | 运营同步给商家的商品 | 运营端配置 | 运营端配置 | 运营端审核 |

### 4.3 二维码下单

办单助手生成价格方案后，页面底部必须带扫码下单能力：

- 展示押金、账单、服务费、公证费、留购价等所有价格项。
- 客户扫码后直接进入 C 端下单流程。
- 下单后价格方案同步到后台订单，客服不需要重复改价。
- 客服仍保留改价权限，但改价必须留操作日志。

## 5. 商品模型修订

### 5.1 商品审核改为商品管理

运营后台不应只有“商品审核”权限，应该是完整的“商品管理”模块：

- 平台自营/统一商品新增
- 商家商品审核
- 商品复制给商家
- 商品筛选
- 商品上下架
- 商品规格
- 商品同步到办单助手
- 商品增值服务

### 5.2 全新/二手是规格，不是商品单选

添加商品时，`全新`、`二手` 应作为同一商品下的规格，而不是让商家重复创建两个商品。

规则：

- 一个商品可以同时包含全新规格和二手规格。
- 每个规格有独立价格、成色、库存、图片、租赁配置。
- 同步到办单助手时，只同步已启用且有配置的规格。
- 如果某商品没有对应规格，则对应入口不展示该规格。

### 5.3 商品同步

商品需要支持多层同步：

- 运营平台商品可复制/同步给不同商家。
- 运营后台商品列表默认不展示所有商家商品，必须有商家筛选。
- 商家可决定是否把自己的商品同步到手机端门店办单助手。
- 运营平台和商家后台都可以配置增值服务。
- 增值服务也要能同步到对应办单助手。

### 5.4 长租/短租与租期单位

租期不能写死为“按月”或“按天”。系统必须把租期做成可配置模型。

确认口径：

- 平台配置租赁模式：长租、短租，后续可扩展试租、临租等模式。
- 每个租赁模式配置可用计费单位：小时、天、周、月。
- 商品新增/编辑时可以选择适配长租、短租，或同时适配。
- 商品规格可独立配置适用租赁模式和计费单位。
- 办单助手根据订单入口、商品规格、租赁模式展示可选租期。
- 订单和账单必须保存租赁模式、计费单位、租期数量、开始时间、结束时间，不能只保存“几期”。

长租示例：

- 计费单位：月
- 租期：3 期、6 期、9 期、12 期
- 适合手机、电动车长期租赁

短租示例：

- 计费单位：小时、天、周
- 租期：2 小时、8 小时、1 天、3 天、1 周
- 适合车辆、设备、临时使用场景

### 5.5 短租库存与设备管理

短租不能只靠商品库存数量。商品只是展示和价格配置，真正租出的必须是唯一设备。

确认口径：

- 商品是展示层，描述“可租什么”。
- 规格是价格和属性层，描述“以什么规格出租”。
- 设备是履约层，描述“实际租出去的是哪一台”。
- 每台短租设备必须有唯一设备码，可扩展车架号、IMEI、SN 等字段；监管锁作为接口能力，不作为固定设备编号字段。
- 短租下单后必须绑定具体设备。
- 设备出租后从可用库存变为已出租。
- 归还验收通过后设备回到仓库，可再次出租。
- 归还异常、损坏、维修、锁定中的设备不能再次出租。

设备状态至少包括：

| 状态 | 说明 |
|---|---|
| 待入库 | 已创建设备档案但未完成入库 |
| 在库可租 | 可被短租订单绑定 |
| 已锁定 | 已被订单预占，等待支付/取货 |
| 出租中 | 已交付给客户 |
| 归还待验 | 客户已归还，等待门店/仓库验收 |
| 维修中 | 设备不可租 |
| 已下架 | 不再出租 |

## 6. 入驻与账号

### 6.1 门店入驻流程

门店手机端未登录时可点击 `立即入驻`。

正常注册后进入资料完善流程：

1. 填写手机号、密码、短信验证码、邀请码（选填）
2. 上传营业执照
3. 上传法人身份证正反面
4. OCR 识别营业执照名称、统一社会信用代码、注册地址
5. OCR 识别法人姓名、身份证号
6. 填写门店真实经营地址
7. 地图选择经纬度
8. 上传门头照片
9. 填写支付宝账号或银行卡
10. 校验收款账户是否法人实名
11. 非法人账户需上传/签署授权材料
12. e签宝授权签章给平台使用
13. 提交运营后台店铺审核

运营后台店铺审核应能看到：

- 基础资料
- OCR 结果与手填结果
- 图片材料
- 收款账户实名状态
- 授权材料
- e签宝授权完成状态
- 回调状态和失败原因

### 6.2 登录保持

门店手机端同一手机登录成功后，应支持一段时间内免重新登录。

建议：

- 默认登录态 30 天
- 敏感操作二次校验，如提现、修改收款账户、分配员工账号
- 支持运营端强制踢下线/禁用账号

### 6.3 员工账号

门店老板/商家管理员可以创建员工账号。

员工账号最小权限：

- 手机号 + 密码登录
- 使用办单助手扫码办单
- 查看自己的待办任务
- 查看必要订单进度

员工账号默认不可见：

- 钱包余额
- 提现
- 财务对账
- 收款账户
- 商家配置
- 全量订单数据

## 7. 保留、删除、后置模块

### 7.1 删除/暂不考虑

- 门店端资金充值入口：本轮不保留。
- 营销管理里的小程序大礼包、优惠券：先不考虑。
- 老数据迁移：本轮重构不考虑老数据迁移。

### 7.2 保留/新增

- 客诉管理：保留，对接支付宝投诉接口，拉取投诉信息并处理。
- 租后管理：先加模块，后期可导入外部系统逾期订单。
- 黑名单库：后期接外置黑名单库，下单/审核时匹配脱敏姓名、身份证等信息，命中后弹窗提示疑似黑名单。
- 监管锁配置：后期接公司自研监管锁，可在订单中调用锁机、解锁等接口。
- 操作日志：所有人工操作、系统操作、回调操作都必须可追溯。

### 7.3 待讨论

商家是否允许作为出资方：

- 当前确认：否。
- 当前处理：普通门店不作为出资主体，只保留订单收益、提现和对账。
- 待讨论：未来是否存在“出资商户”角色。如果存在，应独立建角色和资金账户，不混进普通门店。

- `static/admin-v2/index.html`
