# Claude 体验租逻辑梳理话术

> [历史参考 / V0.2.2 口径覆盖] 本文是早期给 Claude 的检查话术,其中“门店订单 / 分红订单 / 受让 / 资方”等旧口径已被 V0.2.2 替代;后续外部审阅必须先读取 `docs/V0.2.2_P0_FIXUP_TASK.md` 和开发冻结总 PRD 顶部 V0.2.2 补丁。

把下面内容发给 Claude，用于让它基于当前 PRD 梳理体验租短租逻辑，不要自由发挥重写主业务。

```text
你是一位租赁系统产品架构顾问。请阅读 mandian-prd 仓库的 PRD/V0.2 文档，重点阅读以下文件：

1. PRD/V0.2/02_三端模块地图.md
2. PRD/V0.2/11_头部平台C端视频拆解与前端改造建议.md
3. PRD/V0.2/modules/C端/00_C端前台总入口与页面架构.md
4. PRD/V0.2/modules/短租业务/01_体验租业务总览.md
5. PRD/V0.2/modules/短租业务/02_C端体验租下单流程.md
6. PRD/V0.2/modules/短租业务/03_商家端体验租订单与车辆.md
7. PRD/V0.2/modules/短租业务/04_运营端体验租管理.md
8. PRD/V0.2/modules/短租业务/05_体验租数据模型与状态机.md
9. PRD/V0.2/modules/运营端/商品管理/01_商品列表与新增编辑.md

业务边界：

- C 端底部导航为：首页 / 安心用 / 体验租 / 订单 / 我的。
- 安心用是长租商城，平台长租订单放在安心用板块里，参考人人租小程序主界面：选商品、选规格、选套餐、确认订单、授权实名、平台审核、合同/公证、支付、发货、签收。
- 体验租是短租板块，当前重点是电动车短租。点进体验租后优先展示城市、附近门店、门店库存、车型和短租方案。
- 安心用和体验租可以共用登录、实名、协议、客服、订单中心基础能力，但商品配置、库存模型、订单状态机、履约路径必须分开。
- 门店长租办单助手不展示短租入口，短租不要混进三类长租办单助手。
- 平台自营短租也走体验租链路，只是经营主体/履约门店可以是平台直营门店或平台指定仓店。

请重点帮我梳理以下内容：

1. C 端安心用长租完整下单链路：从首页/分类/商品详情/套餐选择到订单确认、实名、风控、合同、公证、支付、发货、签收、在租。
2. C 端体验租短租完整下单链路：从城市定位、选择门店、选择车型、选择小时/天/周/月方案、骑行人资料、人脸、支付/押金/免押、商家接单、到店核销、绑定车辆、用车、续租、还车、押金处理。
3. 运营后台如何拆模块：安心用订单管理、体验租管理、商品管理、库存设备、价格方案、押金扣款、控车日志、财务对账。
4. 商家/门店端如何拆模块：长租办单助手、体验租订单、体验租车辆、钱包、员工权限。
5. 商品主数据如何设计，尤其是体验租商品标准名锁定规则：
   - 适配体验租的商品必须有统一商品标准名。
   - 商品开启体验租并绑定价格方案、车辆库存或产生订单后，商品标准名、车型编号、品牌、型号、核心类目不得直接修改。
   - C 端可以有展示标题/营销标题，但不能影响后台标准名。
   - 如果标准名确实要改，应新建标准商品或车型档案，再迁移价格方案和库存，不允许直接覆盖历史主数据。
6. 数据模型需要哪些核心表和关键字段：商品标准名、展示标题、车型编号、价格方案、门店库存、车辆唯一设备码、短租订单、押金记录、控车日志、操作日志。
7. 每个关键节点需要哪些客户告知：费用、押金/免押、定位和控车授权、人脸授权、到期/续租/还车、损坏赔付、反诈提醒。
8. 找出当前 PRD 里短租逻辑还不闭环、前后文不一致、字段缺失或权限风险的地方。

输出要求：

- 不要泛泛建议“继续完善”。
- 请按“链路图 + 后台模块 + 数据模型 + 权限风险 + 需要补充的问题清单”输出。
- 不要推翻安心用/体验租两个业务入口的命名。
- 不要把体验租混入长租办单助手。
- 不要让客户侧看到分红、资方、分账、平台抽佣等内部概念。
- 对每个问题标注涉及文档路径，并给出具体修改建议。
```
