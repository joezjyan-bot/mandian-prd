# Codex V0.2.1 开发冻结修订任务清单

> 创建时间: 2026-05-27
> 任务类型: PRD 开发前冻结修订
> 适用范围: `PRD/V0.2/`
> 优先级: P0
> 当前结论: V0.2 主体业务框架已成型,但尚未达到开发冻结标准

---

## 一、任务背景

本次检查的目标不是继续新增功能,而是把多轮讨论、Claude 修订、Codex 修订后形成的 PRD 体系收口成一版开发可以直接执行的冻结版。

当前 V0.2 已经覆盖满点租赁系统的核心框架:

- 三类订单:商家订单 / 联营订单 / 平台订单
- 长租主链路:办单助手、客户下单、审核、合同、公证、付款、发货、签收、履约、账单、留购/提前结清
- 资金主链路:平台代收、结算台账、商家钱包、资方放款/回款、退款冲正、提现
- 商家端、运营端、C 端、门店手机端、渠道端、短租体验租等模块
- 中控台、监管锁、合同模板、链路配置、发货履约、AI 问答签收等关键能力

但现在的问题是:总版、宪法、页面级 PRD、财务专项、C 端专项之间存在口径未同步,开发若直接按全部文档执行,会出现理解偏差。

---

## 二、修订原则

### 2.1 不新增大需求

本轮只做开发冻结前的收口:

- 统一术语
- 统一业务边界
- 统一状态机
- 统一资金和合同口径
- 清理旧文档和重复编号
- 补齐开发必须依赖的数据模型、权限、验收口径

不在本轮新增新的商业模式、新端口、新复杂功能。

### 2.2 以最新业务决策为准

如文档之间冲突,按以下优先级判断:

1. 用户已明确拍板的业务规则
2. `00_核心决策表.md`
3. Stage 6 后新增/修订的资金、订单、权限、状态类核心文档
4. 页面级 PRD
5. 早期总版、历史检查报告、旧交接包

### 2.3 数据库字段和枚举保持兼容

除非单独确认,本轮不改底层英文枚举、数据库字段、API 路径。

允许修订:

- UI 文案
- 文档中文描述
- C 端展示口径
- 操作按钮名称
- 管理后台中文字段名

不直接改:

- `self_operate`
- `joint_venture`
- `receivables_assignment`
- 已经进入接口或表结构草案的兼容字段

#### 历史废弃选项:B1

- 曾讨论过仅保留短租架构预留。
- 当前不采用。

#### 当前冻结选项:B2 V1 包含体验租

- 体验租进入 V1 主范围。
- 补齐以下内容:
  - C 端体验租完整下单链路
  - 商家端体验租订单处理
  - 运营端体验租管理

重点检查:

- `modules/C端/05_风控授权签署.md`
- `modules/C端/06_合同签署.md`

**修订要求**

C 端统一展示为:

- 风控授权 -> 信用/信息授权
- 风控审核 -> 业务审核 / 信用审核

**验收标准**

- C 端正式页面 PRD 不再以“风控”作为用户可见页面名。

### 任务 D:资金与合同法律主线统一

**问题**

当前文档中存在两条口径:

1. 主线:三方租赁合同,甲方商家,乙方客户,丙方平台,资方不出现在客户合同。
2. 特例:蓝海银行赊销、分期付款销售合同、客户直接取得所有权、提前结清借款。

这两条如果同时存在但不加边界,会影响合规和开发实现。

重点检查:

- `00_核心决策表.md`
- `modules/运营端/财务管理/07_门店结算账户与资金穿透架构.md`
- `modules/运营端/财务管理/08_订单合作模式与收益分配规则.md`
- `modules/运营端/资方管理/`
- `modules/C端/06_合同签署.md`
- `modules/C端/10_订单列表与详情.md`

**修订要求**

必须明确:

1. 满点主业务默认是否永远是三方租赁合同。
2. 蓝海银行赊销是否只是 `funding_source` 下的特殊资金/话术模板。
3. 如果存在银行赊销模式,C 端按钮和弹窗是否按业务模式切换:
   - 租赁模式:留购
   - 银行赊销模式:提前结清
4. 资方是否仍不出现在客户侧合同中。
5. 平台订单、联营订单、商家订单的设备所有权描述是否一致。

**建议**

默认主线仍保持:

- 客户侧不展示资方
- 客户侧不展示资金来源
- 客户看到的是设备租赁/履约/留购/账单支付
- 银行赊销如必须存在,作为内部 `funding_source` + `terminology_template_id` 控制 UI 和弹窗话术

**验收标准**

- 不同业务模式下的合同模板、按钮话术、弹窗话术都有明确配置来源。
- C 端不出现同一订单既是租赁又是借款的双重表述。

### 任务 E:提现规则统一

**问题**

不同文档存在提现规则冲突:

- 核心决策表:单笔 500-20000,超过 2 万人工审核。
- 财务钱包文档:超过 5 万财务审核。

**修订要求**

统一以下规则:

- 单笔最低金额
- 单笔最高金额
- 每日次数
- 每日总额度
- 超限处理方式
- 提现手续费
- 是否需要 2FA
- 谁可以修改结算账户

**建议**

结合资金安全:

- 普通提现:500-20000
- 每日最多 3 次
- 超过 20000 或异常提现进入人工审核,不直接提交通道
- 修改结算账户必须商家老板 + 平台审核,商家财务人员不可修改

**验收标准**

- 核心决策表、财务钱包 PRD、权限矩阵、OpenAPI 约束一致。

### 任务 G:状态字典和设备状态拆分

**问题**

`modules/全局/02_状态字典与订单状态机.md` 当前把长租、短租设备状态合并得过重。

业务真实要求:

- 长租:不做库存,发货时录入设备识别码即可。
- 短租:需要车辆库存、预约、在租、还车、维保等完整状态。

**修订要求**

拆成两层:

1. 长租设备档案:
   - 订单发货时录入 IMEI/SN/VIN/车架号等设备识别码
   - 与订单绑定
   - 可记录监管锁状态
   - 不参与库存预约

2. 短租车辆库存:
   - 可租
   - 预约中
   - 待取车
   - 在租
   - 待还车
   - 已还待核验
   - 维保中
   - 下架
   - 报废/不可用

**验收标准**

- 长租文档不再要求库存状态流转。
- 短租库存状态只服务体验租业务线。

### 任务 H:商家员工权限补齐

**问题**

商家可给员工分配财务权限,但不能让财务人员修改提现账户,这一点还没有拆细。

**修订要求**

在权限矩阵补充商家端权限:

| 权限项 | 老板 | 管理员 | 财务 | 店长 | 普通员工 |
|---|---|---|---|---|---|
| 查看我的钱包 | 是 | 可授权 | 可授权 | 否/可选 | 否 |
| 发起提现 | 是 | 可授权 | 可授权 | 否 | 否 |
| 查看提现记录 | 是 | 可授权 | 可授权 | 否/可选 | 否 |
| 修改结算账户 | 仅老板 | 否 | 否 | 否 | 否 |
| 提交结算账户变更申请 | 仅老板 | 否 | 否 | 否 | 否 |
| 审核结算账户变更 | 平台 | 平台 | 平台 | 平台 | 平台 |

**验收标准**

- 财务人员可以操作提现,但不能改收款账户。
- 修改结算账户必须二次确认、短信/2FA、平台审核、操作日志。
