# 采购账户与软钱包架构

> P0 业务文档(2026-05-26)。
> 配套 `订单管理/08_长租订单全生命周期与客服操作.md §7.3` Q13 决策落地。
> 监管锁回调成功后,平台需要把"采购款"打给商家。本文档定义这笔资金流的承接载体 — **采购账户(软钱包)**:与现有钱包体系完全分离的独立账户,商家可自助提现。
>
> **与现有财务文档的关系**:本文档是 `财务管理/02_钱包分账提现对账.md` 的**扩展**,在已有"商家/门店钱包/资方账户/渠道账户"基础上,新增第 5 类账户 — **采购账户**。

> - §1.3 / §4.2 **打款口径修正**:分红订单打**配资款**(资方出资部分),平台订单打**设备款全额**
> - 三份关联文档同步推:`02_钱包分账提现对账.md` / `06_财务流水模型与对账规则.md` / `店铺管理/02_店铺详情与商户配置.md`

## 1. 概念定义与边界

### 1.1 采购账户 vs 现有 4 类钱包

| 账户类型 | 承接资金 | 资金路径 | 提现规则 | 文档位置 |
|---|---|---|---|---|
| 商家/门店钱包(分账钱包)| 客户每月还款分账 | 客户支付 → 分账拆分 → 钱包 | 已定义,见 02 文档 | `财务管理/02` |
| 资方账户 | 分红订单资方份额 + 平台订单收益 | 客户支付 → 分账拆分 → 资方账户 | 同上 | `财务管理/02` |
| 渠道账户 | 分红/平台订单渠道佣金 | 客户支付 → 佣金计算 → 渠道账户 | 同上 | `财务管理/02` |
| 平台收入账户 | 平台抽佣、服务费、计费收入 | 各类支付分账 | 不可提现(内部记账)| `财务管理/02` |
| **采购账户(软钱包)** ⭐ | **平台代资方付给商家的设备采购款** | **资方/平台资金 → 监管锁回调 → 实时打款** | **商家自助提现(千二手续费)** | **本文档** |

### 1.2 采购账户的本质

- **业务含义**:客户已付完首期 + 合同已签 + 监管锁已上锁 → 资金方(资方)兑现承诺,**通过平台一次性把对应设备款付给商家**
- **不是分账**:不按客户还款时间累积,而是**订单生效一次性打**
- **不是垫资**:资方已承诺出资,平台代付;客户首期 + 后续每月代扣保障资方回款
- **与分账钱包并行**:同一商家可同时有分账钱包(月度租金分账)+ 采购账户(订单生效采购款)

### 1.3 适用订单类型与打款口径(v1.1 修正)⭐

| 订单类型 | 资金来源 | 是否触发采购款 | 打款金额 |
|---|---|---|---|
| 门店订单 | 门店自己出钱 | ❌ 不触发 | - |
| **分红订单** | 资方 + 门店共同出资 | ✅ 触发 | **配资款 = 设备价 × 资方出资比例**(例:设备价 ¥5,000,资方出资 80% → 平台打 ¥4,000)|
| **平台订单** | 资方全额出资(平台代付)| ✅ 触发 | **设备款全额 = 设备价 × 100%**(例:设备价 ¥5,000,资方出资 100% → 平台打 ¥5,000)|

**核心逻辑(v1.1 修正)**:
- 分红订单门店自己已经出了配资比例的部分(钱在门店手里,门店把设备给客户),**平台只需补足资方那部分**给商家
- 平台订单门店一分不出,**资方全额出资**(资金路径:资方资金 → 平台代付 → 商家采购账户)
- 平台只是"代付通道",不出自有资金;资方的钱由平台代为分发给商家

## 2. 数据模型

### 2.1 `procurement_account` 采购账户主表

```sql
procurement_account
- account_id              bigint PK
- merchant_id             bigint FK     -- 商家级账户(Q27 决策 A,一个商家一个采购账户)
- account_no              varchar(32)   -- 账户编号(系统生成,展示用)
- balance_total           decimal(18,2) -- 账户总余额
- balance_pending         decimal(18,2) -- 在途余额(打款已发起,通道未确认)
- balance_available       decimal(18,2) -- 可提现余额
- balance_withdrawing     decimal(18,2) -- 提现中余额(已申请提现,未到账)
- total_received          decimal(18,2) -- 历史累计入账(订单采购款合计)
- total_withdrawn         decimal(18,2) -- 历史累计提现(实际到账)
- total_fee               decimal(18,2) -- 历史累计手续费
- status                  enum          -- normal(正常)/ frozen(冻结)/ disabled(停用)
- bank_account_id         bigint FK     -- 绑定的收款银行账户 ID
- created_at / updated_at

-- 索引:(merchant_id) 唯一约束
```

### 2.2 `procurement_inflow` 采购款入账流水

```sql
procurement_inflow
- inflow_id               bigint PK
- account_id              bigint FK
- order_id                bigint FK     -- 关联订单
- order_no                varchar(32)
- order_type              enum          -- dividend(分红订单)/ platform(平台订单)
- device_price            decimal(18,2) -- 订单设备价快照
- funder_ratio            decimal(8,4)  -- 资方出资比例(分红=订单配资%; 平台=100%)
- amount                  decimal(18,2) -- 实际打款金额 = device_price × funder_ratio
- funder_id               bigint FK     -- 资金来源资方
- trigger_source          enum          -- monitor_lock_callback / manual
- trigger_at              datetime      -- 触发时间
- channel                 varchar(32)   -- 资金通道(平台内部 or 支付通道)
- channel_request_no      varchar(64)
- channel_response_status enum          -- pending / success / failed / unknown
- channel_response_at     datetime
- failure_reason          text
- status                  enum          -- processing / completed / failed / reversed
- created_at / updated_at

-- 索引:(account_id, created_at) / (order_id) / (status)
```

**关键字段说明**:
- `device_price`:订单设备价快照(写入时锁定)
- `funder_ratio`:资方出资比例快照(分红订单写订单的资方配资比例,如 80%;平台订单写 100%)
- `amount` = `device_price × funder_ratio`(系统自动计算,不可人工改)

### 2.3 `procurement_withdrawal` 提现申请表

```sql
procurement_withdrawal
- withdrawal_id           bigint PK
- account_id              bigint FK
- merchant_id             bigint FK
- amount_requested        decimal(18,2) -- 申请金额
- fee_amount              decimal(18,2) -- 手续费(千二)
- amount_arrival          decimal(18,2) -- 实际到账金额 = amount_requested - fee_amount
- fee_rate                decimal(8,6)  -- 手续费率快照(留痕)
- bank_account_id         bigint FK     -- 收款账户 ID
- bank_account_no_masked  varchar(32)   -- 收款账户号脱敏快照
- requested_by            bigint        -- 提现申请人 ID
- requested_at            datetime
- audit_status            enum          -- pending(待审核)/ approved(已通过)/ rejected(已驳回)/ no_audit_needed
- audited_by              bigint
- audited_at              datetime
- audit_reason            text
- channel_request_no      varchar(64)
- channel_response_status enum
- channel_response_at     datetime
- failure_reason          text
- status                  enum          -- pending / auditing / paid_processing / paid_success / paid_failed / cancelled
- created_at / updated_at

-- 索引:(account_id, created_at) / (status, requested_at)
```

### 2.4 `procurement_account_log` 操作日志

```sql
procurement_account_log
- log_id                  bigint PK
- account_id              bigint FK
- action                  enum          -- create / freeze / unfreeze / disable / bank_account_change
- operator_id             bigint
- operator_role           varchar
- previous_value          json
- new_value               json
- reason                  text
- client_ip               varchar(45)
- created_at              datetime
```

## 3. 开户流程(Q26 决策)

### 3.1 开户时机

| 节点 | 动作 |
|---|---|
| 商家入驻审核通过 | 系统不自动开户,等待商家申请 |
| 商家在 PC 端 > 财务钱包 看到"申请开通采购账户"入口 | 必要前提:商家已开通"分红订单"或"平台订单"能力 |
| 商家点击申请 | 进入开户审核流程 |
| 财务审核通过 | 创建 `procurement_account` 记录,状态 = normal |

**关键设计**:不在商家入驻时**自动开户**,而是**商家主动申请**。门店订单专属商家(没开通分红/平台订单)不需要采购账户。

### 3.2 开户字段要求

| 字段 | 说明 | 来源 |
|---|---|---|
| 商家主体 | 必须与营业执照一致 | 商家入驻资料(已审核) |
| 收款银行账户 | 户名必须与商家主体一致 | 商家提交 + 财务核验 |
| 法人身份证 | 必须与营业执照法人一致 | 入驻资料(已审核) |
| 实名验证状态 | 必须已实名 | e 签宝 |
| 收款账户授权 | 收款账户非法人账户时需授权文件 | 商家提交 |

### 3.3 开户流程

```
商家入驻审核通过 + 开通分红/平台订单能力
   ↓
商家在 PC 端 > 财务钱包 > 采购账户 > [申请开通]
   ↓
确认收款银行账户(从已提交资料中选择,或新增)
   ↓
提交申请 → 运营端财务审核
   ↓
运营端财务审核:
  ✓ 收款账户户名与商家主体一致
  ✓ 法人身份证有效
  ✓ 收款账户授权完整
   ↓
  ├─ 通过 → 创建 procurement_account 记录,状态 = normal
  │         → 商家端收到通知:"采购账户已开通"
  │         → 写入 procurement_account_log(action=create)
  └─ 驳回 → 通知商家补资料
```

### 3.4 开户接口

| 接口 | 用途 |
|---|---|
| `POST /api/v1/merchant/procurement-account/apply` | 商家发起开户申请 |
| `GET /api/v1/admin/procurement-account/applications` | 财务查看待审申请列表 |
| `POST /api/v1/admin/procurement-account/:id/approve` | 财务审核通过 |
| `POST /api/v1/admin/procurement-account/:id/reject` | 财务审核驳回 |
| `GET /api/v1/merchant/procurement-account` | 商家查看自己的采购账户 |

## 4. 入账触发(对接监管锁)

### 4.1 触发链路

```
客户 [确认收货] 完成(电子合同 / AI 问答 通过)
   ↓ 订单进入 PENDING_RETURN
商家在监管锁系统给设备上锁
   ↓ 监管锁系统 Webhook 回调到平台
平台监管锁服务收到回调
   ↓
1. 更新订单字段 lock_status = LOCKED
2. 触发采购款打款:
   ├─ 校验订单类型(必须是分红 / 平台订单)
   ├─ 校验订单状态(必须是 PENDING_RETURN 之后)
   ├─ 校验商家采购账户存在且 status = normal
   └─ 计算采购款金额(按订单类型,见 §4.2)
   ↓
创建 procurement_inflow 记录(status = processing)
   ↓
调用资金通道实际打款(平台代资方付款 → 采购账户)
   ↓
通道回调:
  ├─ 成功 → status = completed
  │         → 采购账户 balance_available += amount
  │         → 商家端 IM 推送:"采购款已到账 + 当前余额"
  └─ 失败 → status = failed
            → 进入财务异常队列
            → 主管复核,可重试或手动打款
```

### 4.2 入账金额计算(v1.1 修正)⭐

| 订单类型 | 计算公式 | 例子 |
|---|---|---|
| **分红订单** | `amount = device_price × funder_ratio`(资方出资比例)| 设备价 ¥5,000 × 80% = **¥4,000** |
| **平台订单** | `amount = device_price × 100%`(资方全额出资)| 设备价 ¥5,000 × 100% = **¥5,000** |

**逻辑解读**:
- **分红订单**:门店与资方共同出资,门店出资部分**门店自己拿设备**(钱本来就在门店手里),平台只需把资方那部分(配资款)代付给商家
- **平台订单**:资方全额出资,平台作为资金通道把资方的钱代付给商家(平台不出自有资金,只是代付)
- 采购款打款**不扣平台抽佣**(抽佣在后续月度分账时收)

**字段对应**:
| 文档字段 | 数据库字段 |
|---|---|
| 设备价 | `procurement_inflow.device_price` |
| 资方出资比例 | `procurement_inflow.funder_ratio` |
| 实际打款金额 | `procurement_inflow.amount` |

### 4.3 不触发的场景

| 场景 | 处理 |
|---|---|
| 门店订单 | 不触发(商家自己进设备)|
| 订单已撤单 | 撤销已发起的打款,从 procurement_inflow 反向冲正 |
| 商家采购账户未开通 | 进入异常队列,提醒商家先开通账户 |
| 商家采购账户被冻结 | 暂挂入账,商家解冻后补打 |

### 4.4 监管锁回调失败兜底

| 异常 | 处理 |
|---|---|
| 监管锁系统超时未回调 | 客服催商家上锁(详见 08 §7.5)|
| 上锁回调到达但订单状态不对 | 进入异常队列,主管核实 |
| 资金通道打款失败 | procurement_inflow.status = failed,主管复核 |
| 打款金额计算错误(订单已被修改)| 阻止打款,人工核对 |
| 重复回调(同一订单收到 2 次)| 系统幂等校验,只处理第一次 |

```
│  [驳回]                            [批准并打款]                 │
└──────────────────────────────────────────────────────────────┘
```

### 6.4 审核流程

```
商家提现申请提交
   ↓
系统判断:金额 > ¥50,000?
   ├─ 否 → 直接进入打款流程
   └─ 是 → 进入财务审核队列
            ↓
        财务主管审核
            ├─ 批准 → 冻结提现金额(balance_available -= amount, balance_withdrawing += amount)
            │         → 调起打款通道
            │         → 通道回调成功 → balance_withdrawing -= amount, total_withdrawn += amount_arrival
            │         → 商家端推送:"提现已到账"
            └─ 驳回 → 退回可提现余额 + 通知商家(驳回理由)
```

### 6.5 异常处理

| 异常 | 处理 |
|---|---|
| 打款通道返回失败 | 退回 balance_available,通知商家 + 财务异常 |
| 通道超时无响应 | 5 分钟内重试 3 次;仍失败进入财务异常 |
| 商家收款账户已变更 | 系统阻止,提示商家更新账户 |
| 收款账户实名校验失败 | 系统阻止 |

## 7. 风控控制(Q30 决策)

### 7.1 仅保留:打款前置硬约束

- 监管锁回调成功 → 才打款(此为唯一硬约束)
- 不冷冻、不黑名单自动冻结、不强制留存

### 7.2 删除的风控点(本轮不实施)

| 项 | 状态 |
|---|---|
| ~~3 天冷冻期~~ | ❌ 不实施 |
| ~~黑名单自动冻结提现~~ | ❌ 不实施(由风控模块单独处理) |
| ~~单日提现额度限制~~ | ❌ 不实施 |
| ~~保证金强制留存~~ | ❌ 不实施 |

**理由**:Q30 选 A,平台对商家信任度足够,只用"上锁前置"硬约束保证打款不出问题。

### 7.3 异常情况下的人工冻结

虽然系统不自动冻结,但运营/财务主管在以下场景可**手动冻结采购账户**:

| 场景 | 操作 |
|---|---|
| 商家欠款大额 | 运营主管手动冻结 |
| 商家被立案/起诉 | 运营主管手动冻结 |
| 重大风控告警 | 运营主管手动冻结 |
| 商家主动关店 | 运营主管手动冻结 |

冻结操作必须写日志(`procurement_account_log` action=freeze)。

## 11. 异常队列

以下场景进入财务异常队列,需人工处理:

| 异常 | 优先级 | 处理 |
|---|---|---|
| 监管锁回调到达但订单状态不对 | 高 | 主管核实 + 手动标记 |

| 商家采购账户冻结但有新订单 | 中 | 暂挂入账,商家解冻后补打 |
| 商家收款账户变更但提现已发起 | 高 | 暂停提现,通知财务核实 |

## 12. 关键时序

### 12.1 完整时序(从订单上锁到商家提现)

```
T0    客户确认收货 → 订单 PENDING_RETURN
T0+   商家给设备上锁
T1    监管锁系统 Webhook 回调
T1+1s 平台创建 procurement_inflow,触发打款
T1+1s 计算 amount = device_price × funder_ratio
T1+2s 资金通道实际打款(实时)
T1+5s 通道回调成功,采购账户 balance_available += amount
T1+5s 商家端 IM 推送:"采购款已到账"

(商家随时可申请提现)
T2    商家点击 [立即提现]
T2+1s 系统校验余额,生成 procurement_withdrawal
T2+   - 小额(≤ ¥5万)→ 直接进入打款流程
        - 大额(> ¥5万)→ 进入财务审核
T2+5s 通道打款(实时)
T2+   通道回调成功,提现完成
T2+   商家端推送:"提现已到账"
```

### 12.2 异常时序(打款失败)

```
T0    监管锁回调,平台触发打款
T0+1s 创建 procurement_inflow,status = processing
T0+5s 资金通道返回失败
T0+5s status = failed,进入财务异常队列
T0+10s 财务主管收到告警
T1    财务主管核实,选择:
       ├─ 重试 → 重新调用通道
       └─ 手动补打 → 写入手动操作日志
```
