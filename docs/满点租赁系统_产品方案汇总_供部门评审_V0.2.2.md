# 满点租赁系统产品方案汇总(供部门评审) V0.2.2

> 适用对象:老板、产品、技术、财务、风控、法务、运营、客服、测试。  
> 文档定位:本文件用于公司内部评审会统一理解产品范围、业务规则、风险边界和各部门关注点。细节以 V0.2.2 主线 PRD、模块文档、OpenAPI、数据模型和测试验收文档为准。  
> 重要边界:本文不展示数据库字段、API 路径、英文枚举和内部敏感字段;客户侧不得展示资方、资金来源、合作模式、风控结论、平台利润、商家结算价、返点规则等内部信息。

## 一、产品概述

满点租赁系统是面向设备租赁业务的一体化平台,覆盖客户下单、商家办单、平台审核、合同签署、交付签收、账单支付、归还续租、申请购买、短租取还车、客服沟通、财务结算、法诉证据等主流程。

系统面向三类核心用户:

- C 端客户:通过小程序/H5 完成安心用长租或体验租短租下单、实名认证、签约、付款、签收、账单支付、归还/续租/申请购买等操作。
- 商家门店:通过商家端和门店手机端办理订单、配置可授权范围内的方案、交付设备、处理短租订单、查看结算和提现。
- 平台运营:通过运营后台管理商品、商家、订单、风控审核、合同公证、财务结算、权限、客服、证据链、数据报表和系统配置。

一期范围明确为:安心用长租 + 体验租短租 V1 + IM 客服基础能力。安心用主打长租以租代用场景;体验租主打短期租车/租设备场景;IM 客服用于沉淀客户、商家、平台的沟通记录和证据链。

详见:`PRD/V0.2/00_V0.2.2_开发冻结版总PRD.md`、`PRD/V0.2/README.md`、`docs/V0.2.2_FINAL_DOC_MAP.md`

## 二、业务模式

系统内部统一支持三类订单,对外展示和部门沟通统一使用以下名称:

| 订单类型 | 业务含义 | 运营和结算原则 |
|---|---|---|
| 商家订单 | 商家自有客户、自行运营、自行承担主要交易责任 | 平台提供系统、支付、合同、审核、客服等工具能力;客户付款后按规则扣除平台服务费,其余进入商家账户 |
| 联营订单 | 平台与商家共同经营,平台参与获客、审核、规则和部分资金/结算安排 | 退款、结算、责任划分按协议和线下协商结果执行,系统记录审批、凭证、冲正和流水 |
| 平台订单 | 平台主导获客、审核、订单规则和运营管理 | 平台按系统规则处理审核、合同、交付、结算、退款、账单和证据链 |

以上三类订单是内部管理和财务经营视角。C 端客户只看到订单商品、费用、协议、签收、账单、归还、续租、申请购买和客服入口,不展示内部资金来源、合作模式、商家结算价、平台利润、返点规则等信息。

详见:`PRD/V0.2/modules/全局/10_全局措辞规范.md`、`PRD/V0.2/modules/运营端/财务管理/06_财务流水模型与对账规则.md`

## 三、核心业务流程

安心用长租主链路:

```text
客户扫码/选品下单
→ 实名认证和资料提交
→ 平台审核
→ 合同签署
→ 首期及保证金支付
→ 商家交付设备
→ 客户签收确认
→ 监管锁/交付条件校验
→ 平台结算
→ 履约中账单支付
→ 到期三选一:归还设备 / 继续使用 / 申请购买
→ 正常结清 / 提前归还 / 逾期处置 / 法诉证据导出
```

关键节点说明:

- 下单阶段:客户看到商品、套餐、保证金、首期、每期应付、最低服务期、协议和反诈提醒。客户必须确认设备型号、成色、价值和最低服务期。
- 审核阶段:平台根据客户资料、信用/信息授权、黑名单、门店风险数据、订单金额和品类规则进行审核。客户侧不展示内部审核结论和风控细节。
- 签约阶段:客户签署租赁服务协议和相关授权文件。合同与页面快照、设备价值确认、签收凭证形成证据链。
- 交付签收阶段:商家交付设备并回填设备识别信息,客户通过电子确认单或视频确认完成签收。需要监管锁回调的品类按锁状态校验;无锁品类按已确认的交付证据和前置检测完成结算。
- 履约阶段:客户按期支付账单,系统记录订单业务账、支付流水、商家账户账和财务总账分录。
- 到期阶段:客户可同等选择归还设备、继续使用或申请购买,系统不得默认引导购买或自动转移所有权。

详见:`PRD/V0.2/modules/运营端/订单管理/08_长租订单全生命周期与客服操作.md`、`PRD/V0.2/modules/C端/02_订单确认页.md`、`PRD/V0.2/modules/C端/12_在租期间.md`

## 四、关键产品规则

### 4.1 费用构成

客户侧主要看到保证金、首期应付、每期应付、增值服务费、配送/公证等可能发生的费用。系统内部还会记录平台服务费、商家结算款、推广费、返点、退款冲正等经营和财务数据,但这些不向 C 端暴露。

为什么这么定:客户侧需要清楚自己要付什么、什么时候付、到期有什么选择;内部财务需要完整核算商家结算、平台服务费和资金流水,两者要分层展示。

详见:`PRD/V0.2/modules/C端/02_订单确认页.md`、`PRD/V0.2/modules/运营端/财务管理/06_财务流水模型与对账规则.md`

### 4.2 申请购买价口径

当前采用 A 口径:申请购买价 = 剩余应付租金 + 保证金。客户实际支付时可按规则抵扣保证金。该口径用于当前开发和上线基线。

重要提示:此为当前口径,折旧余值口径为二期合规优化项,待法务确认后再决定是否切换。本文不得把 A 口径表述为“已无风险”或“法律上绝对安全”。

为什么这么定:当前 A 口径与办单助手报价、C 端展示、订单售后处理保持一致,便于先开发上线;折旧余值涉及残值率、折旧周期、合同解释和渠道确认,适合二期专项优化。

详见:`PRD/V0.2/modules/C端/12_在租期间.md`、`PRD/V0.2/modules/门店手机端/办单助手/02_计算器字段与账单公式表.md`、`PRD/V0.2/modules/开发设计/22_系统定版待完善清单(A口径).md`

### 4.3 最低服务期

安心用长租统一 3 个月起步。下单页必须显著展示最低服务期,客户需单独勾选确认,系统保留页面快照和确认留痕。

为什么这么定:最低服务期用于解释设备交付、审核、合同、客服、渠道等前置成本,也避免客户刚签收就无成本退出。但提前归还仍需提供可操作通道,不能设计成事实禁止解除。

详见:`PRD/V0.2/modules/C端/02_订单确认页.md`、`PRD/V0.2/modules/C端/12_在租期间.md`

### 4.4 到期三选一

到期前必须同等展示:归还设备、继续使用、申请购买。不得默认勾选申请购买,不得写“到期自动购买”或“自动转移所有权”。客户选择必须留痕。

为什么这么定:强化真实租赁属性,降低被误解为默认购买或变相分期购买的风险,同时保留客户主动申请购买的业务入口。

详见:`PRD/V0.2/modules/C端/12_在租期间.md`

### 4.5 提前归还

客户在履约中可以主动申请提前归还。系统展示已使用时间、已发生费用、预计检测费、物流费、折旧补偿、保证金预计退还等预结算信息。不得在客户申请提前归还时默认强制结清全部未到期费用。

为什么这么定:真实租赁应具备可操作的归还通道。提前归还需要保护平台和商家已发生成本,但不能设计成客户无论如何都必须走到申请购买。

详见:`PRD/V0.2/modules/C端/12_在租期间.md`、`PRD/V0.2/modules/运营端/订单管理/05_订单关闭退款与售后.md`

### 4.6 逾期规则

当前逾期费用口径为 0.05%/日、最低 10 元/日。逾期后进入提醒、客服跟进、必要时法诉证据沉淀流程。商家订单、联营订单、平台订单的归属和处理方式按业务协议和系统规则执行。

为什么这么定:逾期费用用于覆盖客服、催收、通知、管理和违约处置成本。费用收取和归属应在合同、系统账务和证据链中保持一致。

详见:`PRD/V0.2/modules/运营端/订单管理/11_逾期费用账单与规则配置.md`、`PRD/V0.2/modules/运营端/租后管理/03_法诉证据包与导出规则.md`

### 4.7 归还检测与费用凭证

客户归还后,平台或商家需要进行检测、维修、物流、拖车、仓储、定位取回等费用核算。费用以实际发生和凭证为准,凭证包括检测报告、照片、视频、维修单据、物流单据、拖车凭证等。

为什么这么定:避免“随口扣款”,也方便客户异议、客服解释、财务对账和法诉举证。

详见:`PRD/V0.2/modules/运营端/订单管理/05_订单关闭退款与售后.md`

### 4.8 设备价值确认与订单后资产化

下单和合同阶段要记录设备型号、成色、设备价值等信息。交付后要绑定设备识别信息并形成设备资产档案。设备价值确认用于合同附件、签收确认、归还检测、申请购买和法诉证据链。

为什么这么定:法院或争议处理中,设备真实存在、价值可证明、客户已确认,是系统证据链的关键。

详见:`PRD/V0.2/modules/运营端/资产管理/01_设备资产中心与订单后资产化.md`、`PRD/V0.2/modules/运营端/资产管理/02_设备价值确认单.md`

### 4.9 C 端隔离红线

客户侧不得展示资方信息、资金来源、合作模式、风控结论、内部审核备注、服务费拆分明细、商家结算价、平台利润、返点规则、内部标签、黑名单/灰名单结论。

为什么这么定:客户侧只需要知道自己的商品、费用、协议、签收、账单和服务入口。内部经营和资金安排不应干扰客户理解,也不应形成不必要的合规和投诉风险。

详见:`PRD/V0.2/modules/全局/10_全局措辞规范.md`、`PRD/V0.2/modules/C端/02_订单确认页.md`

## 五、各端功能概览

### C 端

C 端包括小程序和 H5,承载安心用长租和体验租短租入口。核心功能包括商品浏览、扫码办单、订单确认、实名资料、合同签署、支付、签收确认、账单支付、申请购买、归还、续租、短租取还车、客服咨询和订单记录。

详见:`PRD/V0.2/modules/C端/00_C端前台总入口与页面架构.md`

### 商家端

商家端用于商家查看订单、处理交付、查看结算、发起提现、管理员工权限、处理短租订单和查看门店经营数据。商家员工权限可由商家配置,但修改结算账户、资金类高危动作必须受控。

详见:`PRD/V0.2/modules/商家端/`、`PRD/V0.2/modules/商家端/财务钱包/01_商家钱包提现与流水.md`

### 门店手机端办单助手

办单助手是门店实际办单工具,用于选择商品、租期、首付、增值服务、生成报价和二维码。客户扫码后进入 C 端订单确认页。办单助手不是静态计算器,而是与后台配置、商品、费率、快照和订单链路联通。

详见:`PRD/V0.2/modules/门店手机端/办单助手/02_计算器字段与账单公式表.md`

### 运营后台

运营后台覆盖商品、商家、订单、审核、合同公证、发货履约、监管锁、资产、财务、权限、客服、数据、渠道、黑名单和配置中心。平台运营、客服、财务、风控、法务分别通过权限控制访问对应功能。

详见:`PRD/V0.2/modules/运营端/`

### 渠道端

渠道端用于推广人员查看数据、佣金、提现和推广效果。渠道端不应看到客户过度敏感资料和平台内部利润数据。

详见:`PRD/V0.2/modules/渠道端/01_渠道H5数据统计与提现.md`

## 六、办单助手与配置能力

办单助手由后台配置驱动,核心配置包括商品、期数、首付比例、费率、增值服务、指导价提示和报价快照。三种订单的配置权不同:

- 商家订单:商家可在授权范围内配置自己的办单方案。
- 联营订单:由平台和商家共同规则控制,关键费率和结算规则由平台配置。
- 平台订单:由平台运营配置,商家只按权限查看和执行交付。

每次办单生成的报价必须固化快照。客户扫码下单后,后续合同、账单、申请购买、客服解释都应以该报价快照为依据,避免后期配置变化影响已生成订单。

详见:`PRD/V0.2/modules/门店手机端/办单助手/02_计算器字段与账单公式表.md`、`PRD/V0.2/modules/运营端/权限日志/02_权限矩阵.md`

## 七、财务与资金

系统采用四账分离:

| 账类 | 谁关注 | 作用 |
|---|---|---|
| 订单业务账 | 运营、客服、产品 | 看订单应收、已收、待收、状态和业务动作 |
| 支付流水账 | 技术、财务 | 看每笔支付、退款、回调、失败和幂等 |
| 钱包/商家账户账 | 商家、财务 | 看可提现、冻结、提现中、提现失败回滚和商家余额 |
| 财务总账/分录账 | 财务、老板 | 看收入、成本、负债、冲正、坏账、回收和报表 |

服务费当前口径为 2%,可配置。商家订单客户付款后按规则进入商家账户并扣除平台服务费;联营订单和平台订单按协议和结算规则处理。退款必须记录责任方、退款金额、审批人、凭证、冲正对象和流水。返点、提现、打款、退款、坏账回收都必须有幂等、防重复和回滚机制。

对外融资尽调报表包括账龄分析和 Vintage 世代分析,用于银行或合作方评估应收质量。该类报表只做只读聚合和导出,不反向改变订单、钱包或总账。

详见:`PRD/V0.2/modules/运营端/财务管理/06_财务流水模型与对账规则.md`、`PRD/V0.2/modules/运营端/财务管理/09_账龄分析与Vintage报表.md`

## 八、风控与合规

平台审核包括客户资料、实名授权、信用/信息授权、订单金额、品类风险、门店风险、监管锁状态、黑名单和历史履约情况。审核结论、规则命中、内部备注只在运营和风控侧可见,不得向 C 端暴露。

监管锁用于需要锁控的品类。并非所有品类都必须有监管锁;对无锁品类,系统以签收确认、设备识别码归档、交付证据完整和前置检测完成作为替代验收条件。苹果手机等配置为需监管锁回调的品类,必须等待对应状态满足后结算。

合规边界:本系统定位为真实设备租赁,客户有归还、继续使用、申请购买的真实选择。合同文本、申请购买价、平台服务费展示、债权转让通知、提前归还费用、违约损失计算等涉及融资租赁定性或诉讼认定的事项,以法务终审为准。申请购买价当前 A 口径为开发基线,折旧余值口径列为二期合规优化项。

详见:`PRD/V0.2/modules/运营端/订单管理/08_长租订单全生命周期与客服操作.md`、`PRD/V0.2/modules/运营端/监管锁管理/01_监管锁配置与订单控制.md`、`PRD/V0.2/modules/开发设计/22_系统定版待完善清单(A口径).md`

## 九、权限与安全

系统必须支持角色创建、操作员绑定角色和细粒度权限分配。高危动作包括改金额、改账单、退款、打款、提现审核、调整返点、冻结/解冻商家账户、关闭订单、坏账核销、法诉材料导出、修改监管锁状态、修改客户关键资料、隐藏附件等。

高危动作必须写入操作日志和审计日志,必要时二次确认或二人复核。技术人员不得绕过业务权限直接修改资金、状态、账单、证据和合同相关数据。商家、推广员和渠道人员只能看与自身业务相关的信息,不得查看平台内部利润、风控结论和客户过度敏感资料。

详见:`PRD/V0.2/modules/运营端/权限日志/02_权限矩阵.md`、`PRD/V0.2/modules/全局/11_工程规范基线.md`
