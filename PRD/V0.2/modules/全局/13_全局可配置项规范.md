# 全局可配置项规范

> **版本**:v1.0
> **最后修订**:2026-05-29
> **强制级别**:必守。全项目"什么可配置、什么必须写死、配置项怎么存"的唯一基线。
> **目的**:把"尽量做成可配置"这一原则落地为可执行规范,既保证后期灵活,又避免"过度配置"导致资金/对账/审计失控。

---

## 0. 一句话原则

**「数值、阈值、开关、文案」做成可配置;「业务定义、公式结构、口径、枚举、状态机」必须唯一写死。**

可配置 = 改了不会改变"这件事是什么",只改"参数是多少"。
必须写死 = 一旦可配,不同人配出不同口径,会导致资金算错、对账不平、审计失控。

---

## 1. 必须可配置(做成活的)

以下类别一律走配置中心,带默认值,支持后台修改 + 版本号 + 审计:

| 类别 | 示例 | 说明 |
|---|---|---|
| **阈值类** | 静默判定时长(默认 24h)、人脸匹配高/低阈值、客服接单超时(默认 3 分钟)、提现限额(500-20000)、逾期判定天数、各类告警阈值 | 只改数字,不改定义 |
| **费率/金额类** | 费率表 `rates`/`ev_rates`、平台综合服务费率(默认 2%)、设备管理费档位(50/150/250)、公证费(80)、首期租金(10)、名义留购费(1)、提现手续费(0.2%+0.5)、撤单费用规则、逾期费用率(0.05%/日、最低 10 元/日) | 金额/比率可配,但计算公式结构不可配(见 §2) |
| **开关类** | 是否启用某功能、增值服务默认勾选/强制勾选、是否启用人脸 AI 比对、是否启用公证、哪些首付成数算"低"、门店停单/关低首付 | 布尔或枚举开关 |
| **频率/周期类** | 指标计算频率、兜底任务频率、结算账期、二维码/报价快照有效期、申诉窗口(72h) | 调度与时效参数 |
| **选项集合类** | 可选期数(可增删)、可选首付成数(可增删)、增值服务项列表(可增删)、设备管理费档位 | 集合内容可配,但每项的"语义"不可配 |
| **展示/文案类** | 报价单品牌名/主题色/底部提示语、通知模板、话术模板(在措辞规范允许范围内) | 不得突破 C 端红线与措辞规范 |
| **维度覆盖类** | 配置的继承层级(平台默认→城市→商家→类目→商品),允许逐层覆盖 | 覆盖关系可配,继承规则本身写死 |

---

## 2. 必须写死(禁止做成可配置)

以下一旦可配就会出事,**必须在文档里唯一定义、代码里硬约束**:

| 类别 | 为什么不能配 | 正确做法 |
|---|---|---|
| **指标的口径/分母定义** | 如静默率"算哪些订单"、逾期率分母——不同人配出不同口径,同一门店两个数,据此停单=事故 | 定义唯一写死;只有阈值(24h、告警%)可配 |
| **资金计算公式的结构** | 如"按分整数运算""四账分离""留购=押金+全部期租金""monthly 先按分四舍五入"——结构变了对账必崩 | 公式结构写死;只有公式里的费率/金额数值可配 |
| **枚举值本身** | 如 entry_type、cooperation_mode、订单状态、refund_workflow.status——可配会让对账/审计/接口失控 | 枚举在状态字典唯一定义,新增走版本迭代不走配置 |
| **状态机流转规则** | 哪个状态能到哪个状态、结算三条件、放款前置条件逻辑——可配会放过非法流转 | 流转规则写死;只有"条件清单里某条是否启用""超时时长"可配 |
| **红线与合规规则** | C 端禁用字眼、资方不外露、不全自动放行人脸、不建可检索人脸库、合同三方主体 | 合规规则写死,不给配置开关 |
| **账户体系归属** | 门店结算账户(settlement_*)vs 渠道佣金账户(wallet_entry)分属不同体系 | 写死,不可配置合并 |
| **精度基准** | 全系统金额以"分"为整数运算 | 写死,全局统一 |

> 判断口诀:如果一个东西"配错了会导致钱算错、账对不平、或放过违规",它就必须写死,不能配。

## 3. 配置中心统一机制(所有可配项必须遵守)

任何"可配置项",不是简单存个值就行,必须满足:

1. **配置版本号**:每次保存生成 `config_version_id`,记录操作人、时间、改动 diff。
2. **快照绑定**:业务单据(订单/报价/账单/提现等)生成时,快照当时的配置版本;**已生成单据永远读快照,不跟随配置变动**(见 办单助手 02 §0)。
3. **审计留痕**:涉及钱的配置(费率、服务费、提现限额、增值服务金额、逾期费用率)修改,改前/改后值入审计日志,纳入高危操作。
4. **权限分配**:谁能改哪类配置按角色授权(见各端权限矩阵);商家方案商家改,联营/平台方案运营改。
5. **默认值 + 生效范围**:每个配置项必须有默认值;支持按 §1 维度覆盖类的继承层级生效。
6. **生效时间**:配置可设即时生效或定时生效;历史单据不受影响。
7. **校验边界**:可配不等于无限制——每个配置项要有合法区间(如提现限额上限、费率上下限),防止配出异常值。

---

## 4. 落地要求(给研发/Codex)

1. 新写或修订任何含"数值/阈值/开关/频率"的功能时,默认把这些参数抽到配置中心,不硬编码在代码里。
2. 凡 PRD 中标【需运营/算法确认】的数值,先留配置结构 + TODO + 默认值占位,不编造。
3. 凡涉及"口径/定义/公式结构/枚举/状态流转"的,必须引用对应唯一来源文档(状态字典、财务 06/07、本规范),不得新开配置开关绕过。
4. 配置项的存储结构、版本、审计统一走 §3 机制;不允许某模块自己散落存配置。
5. 与本规范冲突时:涉及资金/合规/对账的,以"写死"为准;有疑义先按写死处理并标注待确认,不擅自放开为可配。

## 5. 典型对照示例

| 需求 | 可配的部分 | 必须写死的部分 |
|---|---|---|
| 静默率 | 静默时长(24h)、告警阈值% | "静默率=静默设备÷已上锁履约中设备"的口径 |
| 办单助手费率 | 费率表数值、可选期数/首付、增值服务金额 | 留购/月付/账单公式结构、按分精度 |
| 提现 | 限额(500-20000)、次数(3)、费率(0.2%+0.5)、人工审核阈值 | 提现状态机、账户体系归属 |
| 人脸核验 | 高/低阈值、是否启用、失败处理 | AI 辅助+人工兜底原则、不建人脸库、不全自动放行 |
| 客服派单 | 接单超时(3min)、并发上限、连续未接阈值 | 单人锁定规则、轮询公平性、状态机流转 |
| 逾期费用 | 费率(0.05%/日)、最低(10元/日)、是否封顶 | 是否计入留购、归属、与合同口径一致性(待法务) |
