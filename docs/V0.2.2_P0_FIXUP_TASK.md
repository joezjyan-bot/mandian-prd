# 满点租赁系统 V0.2.2 产品文档整改任务

> 本文是 V0.2.2 产品文档整改基线。V0.2.2 是当前最新产品口径,旧口径、旧章节、旧任务说明与本文冲突时以本文和本轮 V0.2.2 补丁为准。
> 本轮只允许修改 PRD、流程文档、状态字典、财务规则、权限矩阵、OpenAPI 文档、数据模型文档、测试验收文档;不修改业务代码、数据库 migration、接口实现或前端页面代码。

## 一、整改背景

本次整改基于 V0.2.1 多角色审查结论，以及老板已确认的以下规则：

1. 短租 V1 一起上：安心用长租 + 体验租短租 + IM 客服基础能力进入一期。
2. 结算触发条件：客户签收确认 + 监管锁显示已上锁 + 激活锁状态已激活，三项满足后才允许结算。
3. 无监管锁品类已确认按正常业务形态处理：仅苹果手机等配置为需监管锁回调的品类强制校验锁状态，其它无锁品类按签收确认 + 设备识别码归档 + 交付照片 / 交付证据完整 + 前置检测完成结算。
4. 退款责任：商家订单商家承担；联营订单 / 平台订单线下协商，系统支持按指定金额发起退款。
5. 三类订单命名分内部枚举和前端展示：技术代码和数据库可保留 `merchant / joint / platform` 等内部枚举；前端、客服、商家端、合同、财务报表必须统一展示为：商家订单 / 联营订单 / 平台订单。
6. 门店处罚暂不系统化，仅保留人工备注、冻结、停用等基础操作。
7. 本系统不涉及历史订单迁移，历史订单迁移不纳入 V0.2.2 整改和 V1 研发范围；本文中“数据库 migration”仅指新系统表结构变更规划。
8. 改价仅限账单生成前，由客服操作；账单根据改价后的订单报价快照生成，账单生成后任何人不得修改。
9. 退款金额审批分级：小额 ≤ 5000，中额 5000-20000，大额 > 20000。
10. 设备价超过指导价时无法下单，需联系客服人员核实。
11. 门店体验租订单客户缴费后给对应门店分账，扣除平台服务费 2%，平台服务费支持后台配置。
12. 催收不在 V1 内自建完整催收系统，后期通过三方系统 API 对接。
13. 监管锁异常允许人工放行，但必须配置对应权限、审批、原因和日志。
14. 权限体系采用“操作点细化 + 角色授权 + 操作员绑定角色”模式，先配置角色，再创建操作员并绑定单个角色。
15. 坏账或逾期订单后期追回款项时，允许回到原订单继续操作完成订单；完成方式可选，填写完成金额，类似留购 / 尾款支付完成当笔订单。

## 二、整改原则

1. 先统一口径，再改流程。
2. 先定状态机，再定接口和数据表。
3. 资金类动作必须可追踪、可审计、可回滚。
4. 资金总账不能和业务流水混在一起；订单流水、支付流水、钱包流水、总账分录必须分开设计，只通过订单号、支付单号、业务单号、幂等键等关联字段追溯。
5. 业务看订单流水，财务看总账分录，门店看钱包流水，技术看支付回调和通道流水。
6. C 端不得暴露资方、资金来源、合作模式、风控结论等敏感字段。
7. 所有退款、打款、返点、提现必须有流水和幂等控制。
8. 后续新增重大业务规则必须走变更评审并形成确认记录，不能由技术自行决定；本轮关键规则已按推荐方案确认。
9. 涉及资金、账单、退款、打款、返点、钱包、法诉证据的任务必须标记【资金/法务高风险】。
10. 本任务文档仅列修订建议和验收标准，不直接替代原 PRD。

- 具体整改内容：
  1. 明确账单范围：首期应付、周期应付、保证金 / 补充保证金、服务费、公证费、逾期费用、留购 / 提前结清金额、短租续租 / 超时 / 损坏扣款等应收项目。

  4. 定义部分支付后剩余金额、逾期费用、退款后账单状态。

- 验收标准：
  - 每笔账单状态变化都有账单流水。
